The Problem with CSV

CSV is the default format for sharing tabular data. It is human-readable, universally supported, and simple. It is also slow, bloated, and lossy. A CSV file has no concept of data types — every value is a string. When you load it into pandas, the library has to guess whether "2024-03-15" is a date, a string, or three separate integers. Columns that should be integers get cast to floats because of a single missing value. And the file itself is enormous because every digit, comma, and newline is stored as a character.

For the London House Prices dashboard, the raw Land Registry CSV files for 10 years of London transactions total ~1.5GB. The same data as a Parquet file: 5.6MB. That is a 270x compression ratio. Load time drops from minutes to under a second.

What Is Parquet

Apache Parquet is a columnar storage format. Instead of storing data row by row (like CSV), it stores each column as a contiguous block. This has three consequences:

  1. Compression: Values in the same column tend to be similar (all prices are integers in the same range, all postcodes are short strings). Columnar layout lets compression algorithms exploit this similarity. A column of prices compresses far better than a row containing a price, a date, a postcode, and a property type interleaved.
  2. Type preservation: Each column stores its data type in metadata. A date column stays a date. An integer column stays an integer. No guessing, no silent type coercion.
  3. Column pruning: If you only need the price and date columns, Parquet reads only those columns from disk. CSV has to read every byte of every row to find the columns you want.

Reading and Writing Parquet in Python

pandas has built-in Parquet support via the pyarrow backend (installed by default with modern pandas).

Writing

import pandas as pd

df = pd.read_csv("big_file.csv")
df.to_parquet("small_file.parquet", index=False)

That is the entire migration. One line to read CSV, one line to write Parquet. The resulting file is smaller, faster to load, and preserves your dtypes.

Reading

df = pd.read_parquet("small_file.parquet")

No parse_dates argument, no dtype dict, no low_memory=False. The types are already correct because Parquet stored them.

Reading specific columns

df = pd.read_parquet("small_file.parquet", columns=["price", "date"])

This reads only the price and date columns from disk. On a file with 16 columns, this is roughly 8x faster than reading the whole file.

When to Use Parquet

Use Parquet when:

  • You have more than a few thousand rows
  • The data is read more often than written
  • You want to preserve dtypes without manual configuration
  • File size matters (deployment, git, network transfer)
  • You are loading data in a dashboard or notebook where startup speed matters

Keep CSV when:

  • The file needs to be human-readable or editable in a spreadsheet
  • You are sharing with non-Python tools that do not support Parquet
  • The dataset is tiny (under 1MB) and type inference is not a problem
  • You want text-based git diffs

Parquet with Compression Options

By default, to_parquet uses Snappy compression — fast to read and write, moderate compression ratio. You can choose other algorithms:

df.to_parquet("data.parquet", compression="snappy")   # default — fast, good ratio
df.to_parquet("data.parquet", compression="gzip")      # smaller file, slower write
df.to_parquet("data.parquet", compression="zstd")      # best of both — fast + small
df.to_parquet("data.parquet", compression=None)         # no compression — fastest I/O

For a Streamlit dashboard where the file is read on every app restart, Snappy (default) is the right choice. For archival storage where you read rarely, gzip or zstd saves more space.

Parquet in the FinBytes Pipeline

The data prep script (scripts/build_london_ppd.py) downloads 10 yearly CSV files from Land Registry, filters to London postcodes, and saves the result as Parquet:

import re
import pandas as pd

LONDON_RE = re.compile(r"^(E|EC|N|NW|SE|SW|W|WC)\d", re.IGNORECASE)

frames = []
for year in range(2015, 2025):
    url = f"https://price-paid-data.publicdata.landregistry.gov.uk/pp-{year}.csv"
    df = pd.read_csv(url, header=None, names=PPD_COLS)
    df = df[df["postcode"].str.match(LONDON_RE, na=False)]
    df["postcode_district"] = df["postcode"].str.split().str[0]
    df["date"] = pd.to_datetime(df["date"])
    frames.append(df)

combined = pd.concat(frames, ignore_index=True)
combined.to_parquet("dashboard/data/london_ppd.parquet", index=False)

The dashboard then loads this single file on startup — 5.6MB, under a second, types already correct. No CSV parsing, no type guessing, no wasted memory.

Beyond pandas: Polars and DuckDB

Parquet is not a pandas-specific format. It is an open standard supported by every modern data tool:

  • Polars: pl.read_parquet("file.parquet") — often 2-5x faster than pandas for the same file
  • DuckDB: SELECT * FROM 'file.parquet' WHERE price > 500000 — SQL queries directly on Parquet files, no loading step
  • PySpark: spark.read.parquet("file.parquet") — distributed processing of large Parquet datasets
  • R (arrow): arrow::read_parquet("file.parquet")

If you write your data as Parquet once, every tool in the ecosystem can read it without conversion.
