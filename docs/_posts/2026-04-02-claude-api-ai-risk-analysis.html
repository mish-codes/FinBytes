What We're Building

A RiskAnalyst class that takes portfolio metrics (VaR, volatility, Sharpe ratio) and calls the Claude API to produce a plain-English risk summary. Think of it as automating the "so what does this mean?" conversation that happens after every risk report.

The Anthropic SDK

import anthropic

client = anthropic.Anthropic()  # reads ANTHROPIC_API_KEY from env
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=300,
    system="You are a senior risk analyst...",
    messages=[{"role": "user", "content": "Portfolio: AAPL, MSFT\nVaR: -2.15%..."}],
)
text = response.content[0].text

Anthropic() picks up the API key from the ANTHROPIC_API_KEY environment variable automatically — no need to pass it in code, and no reason to ever hardcode it. The messages.create() call is the core API. Everything else is configuration.

System Prompt vs User Prompt

The API separates two concerns:

System PromptUser Prompt
PurposeSets the persona and constraintsContains the actual request and data
Passed assystem= parametermessages=[{"role": "user", "content": ...}]
Changes between calls?Usually no — same personaYes — different data each time

Our system prompt does four things:

system_prompt = (
    "You are a senior risk analyst at an investment bank. "   # role
    "Summarize portfolio risk in 3-4 sentences for a trader. " # length
    "Be direct, specific, and flag any concerning metrics. "   # style
    "Use plain English, not jargon."                           # constraint
)

The user prompt formats the actual metrics with clear labels. Claude can read VaR (95%): -2.15% and understand exactly what it means. Structured input leads to structured thinking and better output.

Building the RiskAnalyst Class

class RiskAnalyst:
    def __init__(self, model: str = "claude-sonnet-4-6"):
        self.model = model

    def analyze(self, tickers: list[str], metrics: dict) -> str:
        user_prompt = (
            f"Portfolio: {', '.join(tickers)}\n"
            f"VaR (95%): {metrics['var_pct']:.2f}%\n"
            f"CVaR (Expected Shortfall): {metrics['cvar_pct']:.2f}%\n"
            f"Max Drawdown: {metrics['max_drawdown_pct']:.2f}%\n"
            f"Annualized Volatility: {metrics['annualized_vol']:.2f}%\n"
            f"Sharpe Ratio: {metrics['sharpe_ratio']:.2f}"
        )

        try:
            client = anthropic.Anthropic()
            response = client.messages.create(
                model=self.model,
                max_tokens=300,
                system=system_prompt,
                messages=[{"role": "user", "content": user_prompt}],
            )
            return response.content[0].text
        except Exception:
            return (
                f"Unable to generate AI analysis. Key metrics — "
                f"VaR: {metrics['var_pct']:.2f}%, "
                f"CVaR: {metrics['cvar_pct']:.2f}%, "
                f"Max Drawdown: {metrics['max_drawdown_pct']:.2f}%."
            )

The constructor takes a model name so you can swap between Sonnet (default, good balance) and Haiku (faster, cheaper for high-volume use). The try/except provides graceful degradation — if the API is down, you still get the raw metrics in a readable format.

Model Selection: Opus vs Sonnet vs Haiku

ModelSpeedCostQualityUse Case
OpusSlowestHighestBestComplex reasoning, research
SonnetMediumMediumGreatMost production tasks
HaikuFastestLowestGoodHigh-volume, simple tasks

For risk commentary, Sonnet is the sweet spot — good enough quality for financial narrative, fast enough for real-time use. If you're generating summaries for thousands of portfolios in batch, switch to Haiku. If you need deep analysis of a complex position, use Opus.

Graceful Degradation

APIs fail. Timeouts happen. Rate limits get hit. The except Exception block catches all of these and returns a fallback string with the key metrics. Your service stays up even when Claude is down. In production, you'd log the error, increment a metric, and maybe retry once — but the user always gets something useful.

Testing Without an API Key

@pytest.fixture
def mock_anthropic():
    with patch("analyst.anthropic.Anthropic") as mock_cls:
        mock_client = MagicMock()
        mock_cls.return_value = mock_client
        mock_response = MagicMock()
        mock_response.content = [MagicMock(text="The portfolio shows elevated risk.")]
        mock_client.messages.create.return_value = mock_response
        yield mock_client

All tests mock the Anthropic client. No API key needed, no network calls, tests run in milliseconds. The mock fixture patches anthropic.Anthropic so the constructor returns a mock client with a canned response. Tests verify the right arguments are passed (system prompt, model, metrics in user message) without ever calling the real API.

This means your CI/CD pipeline can run all tests without API credentials — a requirement for most enterprise environments.

API Key Management

Rules for API keys:

  • Never in code — no api_key="sk-..." in source files
  • Environment variable — export ANTHROPIC_API_KEY=sk-... or use a .env file
  • .gitignore — always include .env in your gitignore
  • Tests don't need keys — mock everything, CI works without credentials
