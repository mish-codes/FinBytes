Part 1: What We Built and Why

QuantLab Phase 1 is eight exercises (01–08) that each teach one Python tool or concept, building on the last, until they all snap together into a working application: the Stock Risk Scanner.

The scanner is a real, deployed app. You type in a list of stock tickers, and it fetches live market data, crunches a suite of risk metrics, asks an AI to write a human-readable risk narrative, saves everything to a database, and displays it on a dashboard. It lives at quantlabs.streamlit.app.

The full tech stack:

  • Frontend: Streamlit (hosted on Streamlit Cloud)
  • Backend: FastAPI (hosted on Render)
  • Database: PostgreSQL (hosted on Render)
  • AI Narratives: Claude via the Anthropic SDK
  • Data Source: Yahoo Finance via yfinance
  • Testing: pytest with 27 tests
  • CI/CD: GitHub Actions
  • Containerization: Docker + docker-compose

Why eight exercises and not one big project? Because learning one thing at a time actually works. Each exercise is small enough to finish in a day, teaches exactly one concept, and builds directly toward the capstone. By the time you reach the scanner, you've already written every piece — you're just wiring them together. No exercise is filler. Every single one contributes a real, load-bearing component to the final product. When you're done, you don't just understand the scanner — you understand the tools that built it, and you can use those tools to build anything.

Phase 1 covers the full arc from "how to test a single function" to "how to deploy a complete web application with a database, an AI integration, and automated testing." That arc — from a single test to a production system — is what separates a tutorial from real engineering education.

This post explains every single piece — every technology, every finance concept, every design decision — assuming you have never written a line of code or read a finance textbook. If you are twelve years old and curious, this is for you.

Part 2: The Python Technologies (Explain Like I'm 12)

pytest — The Homework-Checking Robot (Exercise 01)

Imagine you finish your math homework and hand it to a robot. The robot checks every single answer instantly and tells you exactly which ones are wrong. That's pytest.

You write tiny programs called "tests" that describe what your code should do. Then pytest runs them all and reports back: green checkmarks for passing, red X's for failing. If you change something and accidentally break it, the robot catches it before anyone else sees.

Here's what a real test looks like:

def test_daily_return_positive():
    prices = [100.0, 105.0]
    result = compute_daily_returns(prices)
    assert result[0] == pytest.approx(0.05)  # 5% gain

That test says: "If the price goes from 100 to 105, the daily return should be 5%." If compute_daily_returns ever returns the wrong number, this test screams red. No human has to remember to check — the robot does it every time.

Exercise 01 teaches TDD — Test-Driven Development. The trick is to write the test first, before the code. You follow a loop called red-green-refactor:

  1. Red — write a test that fails (because the code doesn't exist yet).
  2. Green — write the simplest code that makes the test pass.
  3. Refactor — clean up the code without breaking the test.

It feels backwards at first, but it forces you to think about what your code should do before you think about how. The capstone has 27 tests. Every single one was written before the code it tests.

Why does this matter? Because tests are permanent documentation. Comments get outdated. Documentation gets forgotten. But a test that says assert calculate_var(returns, 0.95) == pytest.approx(-0.032) is a guarantee that is verified every time the test suite runs. If someone changes the VaR formula and gets a different answer, the test fails immediately. The tests are the contract that the code must honor.

Pydantic — The Bouncer at the Door (Exercise 02)

Picture a bouncer at a club. Before anyone gets in, the bouncer checks: "Are you on the list? Do you meet the dress code? Is your ID valid?" If anything is off, you're turned away at the door.

Pydantic does the same thing for data. Every piece of information entering your app passes through a Pydantic model that checks the rules. Is this supposed to be a number? Is it positive? Is this date in the future? Pydantic catches bad data at the door, before it can cause chaos inside.

Exercise 02 covers three levels of validation:

  • Field validators — check one field at a time ("ticker must be uppercase").
  • Cross-field validators — check relationships between fields ("end date must be after start date").
  • Computed fields — automatically calculate values from other fields ("total = price × quantity").

In the capstone, models like ScanRequest, RiskMetrics, and ScanResult are all Pydantic. If someone tries to scan an empty ticker list or a nonsense date range, Pydantic rejects it immediately with a clear error message. No bad data gets past the bouncer.

class ScanRequest(BaseModel):
    tickers: list[str]           # ["AAPL", "MSFT", "TSLA"]
    start_date: date
    end_date: date

    @field_validator("tickers")
    def tickers_not_empty(cls, v):
        if len(v) == 0:
            raise ValueError("At least one ticker required")
        return [t.upper() for t in v]

That validator runs automatically every time a ScanRequest is created. Empty list? Rejected. Lowercase tickers? Uppercased. The model enforces the rules so the rest of the code doesn't have to worry about bad input.

FastAPI — The Post Office (Exercise 03)

Your app needs a way to receive requests from the outside world and send back answers. FastAPI is the post office.

Someone sends a letter (an HTTP request) to a specific address (a URL like /api/scan). FastAPI receives it, reads the address, routes it to the right handler function, runs the code, and sends back a reply. It's the middleman between the outside world and your Python code.

The best part: FastAPI automatically generates a documentation page at /docs. You can open it in a browser, see every endpoint, try them out with real data, and inspect the request/response formats — all for free, no extra work. This is called a Swagger UI (or OpenAPI docs), and it is one of the biggest reasons FastAPI has become so popular in the Python world. You build the API, and the documentation writes itself. No separate documentation step, no docs that drift out of sync with the code.

FastAPI also works hand-in-hand with Pydantic. When you declare that an endpoint accepts a ScanRequest model, FastAPI automatically validates incoming data against that model, returns clear error messages for invalid input, and shows the model's schema in the docs. Two libraries, working together, handling validation and documentation for free.

Exercise 03 builds a toy watchlist API with hardcoded stock data. The capstone uses the same patterns for the real scanner endpoints: POST /api/scan to start a scan and GET /api/scans/{id} to retrieve results. The toy version is a bicycle with training wheels; the capstone is the same bicycle, just without the training wheels and on a real road.

async/await — The Multi-Table Waiter (Exercise 04)

Imagine a waiter at a restaurant. The slow waiter takes an order, walks to the kitchen, stands there watching the chef cook, walks back with the food, then moves to the next table. One table at a time. Painfully slow.

The fast waiter takes an order, drops it at the kitchen, immediately goes to the next table, takes their order, drops it at the kitchen, and keeps moving. When food is ready, they grab it and deliver it. Multiple tables at once, no standing around.

That's async and await. When your code hits something slow — waiting for a website to respond, waiting for a database query — instead of freezing, it says "I'll come back for this" and goes to do other work. asyncio.gather lets you kick off many slow tasks at the same time.

Why it matters: fetching prices for 50 stocks sequentially might take 10 seconds. With asyncio.gather, all 50 requests fly out at once, and the total time drops to about 200 milliseconds. That's a 50× speedup for free.

In the capstone, every external call — price fetching, database queries, API calls to Claude — uses async. The server can handle dozens of simultaneous users without breaking a sweat, because it never stands around waiting.

This is one of the most practical performance improvements you can make in a web application. No fancy algorithms, no special hardware — just stop waiting for things you don't need to wait for. If your code spends most of its time waiting for network responses (which web apps do), async gives you a massive speedup for almost no extra complexity.

yfinance — The Pipe to Wall Street (Exercise 05)

yfinance is a Python library that connects to Yahoo Finance and downloads stock market data. Type in a ticker symbol like AAPL, pick a date range, and yfinance hands you a table of daily prices: open, high, low, close, volume.

That raw data isn't useful by itself. You need tools to wrangle it:

  • pandas — a library for working with tables of data (called DataFrames). Think of it as Excel inside Python.
  • numpy — a library for fast math on arrays of numbers. It's the calculator engine underneath pandas.

Exercise 05 fetches real stock data, handles messy real-world issues (stock splits, dividends, missing days), and computes basic statistics. Real-world data is never clean. Markets are closed on weekends and holidays. Companies split their stock (one share becomes two at half the price). Dividends create price jumps. yfinance provides "adjusted close" prices that account for all of this, so your calculations don't get tripped up by non-events.

In the capstone, this becomes market_data.py — the module that feeds live prices into the risk engine.

Claude API / Anthropic SDK — The AI Analyst (Exercise 06)

Imagine you computed a pile of risk numbers for a stock: volatility is 32%, max drawdown is −18%, Sharpe ratio is 0.45. Those numbers are accurate, but most people can't instantly feel what they mean. Now imagine handing those numbers to a smart analyst who writes: "This stock has been bouncing around a lot, lost nearly a fifth of its value at the worst point, and the return you're getting barely justifies the risk."

That's what the Claude API does. You send it the raw metrics plus a system prompt ("You are a risk analyst writing for a general audience") and a user prompt (the actual numbers), and it sends back a human-readable narrative. The system prompt defines the persona — who Claude should pretend to be. The user prompt provides the data — what Claude should analyze. Keeping these separate makes it easy to change the tone (swap the system prompt) without changing the data pipeline.

Exercise 06 teaches the Anthropic SDK: how to structure prompts, parse responses, handle rate limits, and — critically — graceful degradation. If the API is down or the key is missing, the scanner doesn't crash. It falls back to raw metrics with no narrative. The user still gets useful data.

This is a pattern you'll see again and again in professional code: plan for the API to fail, because eventually it will. The Claude API adds enormous value when it works — turning dry numbers into readable insight — but the scanner is still fully functional without it. The AI narrative is the cherry on top, not the foundation. That's a deliberate design choice.

Prompt engineering is a skill of its own. The system prompt tells Claude to be a concise risk analyst, not a creative writer. The user prompt is structured data, not free-form text. The response is parsed for specific sections. Getting consistent, useful output from an LLM requires treating it like any other API: define the contract, handle errors, test the edge cases.

Docker — The Lunchbox (Exercise 07)

You pack a perfect lunch at home: sandwich, fruit, juice box. But when you open it at school, the sandwich is squished, the juice leaked, and the fruit rolled under your desk. Docker solves this problem for code.

A Docker container is a sealed lunchbox for your application. It holds your code, every library it needs, and the exact right version of Python — all packed together. Whether you open it on your laptop, your friend's laptop, or a cloud server, everything inside is identical. No "but it works on my machine" surprises.

Exercise 07 introduces two key ideas:

  • Multi-stage builds — one stage installs all the build tools (the kitchen), another stage copies only the finished product (the lunchbox). The final container is small and clean, often 10× smaller than if you crammed everything into one stage.
  • docker-compose — when your app needs multiple containers running together (say, one for FastAPI and one for PostgreSQL), docker-compose starts them all with one command and wires them together automatically.

In the capstone, the Dockerfile builds the FastAPI backend, and docker-compose orchestrates the backend alongside a PostgreSQL database. One command — docker-compose up — and the entire stack is running on your laptop, identical to how it runs in the cloud. This is how professional teams develop: everyone runs the same containers, so "works on my machine" is never an excuse.

SQLAlchemy + PostgreSQL — The Filing System That Never Forgets (Exercise 08)

PostgreSQL is a database — think of it as a giant, organized filing cabinet that can hold millions of records, answer questions in milliseconds, and let dozens of users read and write at the same time.

But Python doesn't speak SQL (the language databases understand). SQLAlchemy is the translator. You define your data as normal Python classes, and SQLAlchemy converts them into database tables behind the scenes. When you save a Python object, it becomes a row in a table. When you query, rows come back as Python objects.

Exercise 08 also introduces:

  • Async sessions — non-blocking database calls, so the server isn't frozen while waiting for a query to finish.
  • Alembic — a migration tool. When you add a new column to your model, Alembic generates a script that updates the real database schema. Like a renovation contractor who knows exactly which wall to move.

In the capstone, db_models.py defines the scan result table, and db.py handles all reads and writes. Every scan result is persisted forever — even if the server restarts, the data is safe in PostgreSQL. Without this layer, every scan result would vanish the moment the server rebooted. With it, you can look up any scan from any date, build historical comparisons, and never lose a result.

Alembic deserves special mention. As the scanner evolves, you'll want to add new columns (maybe a "sector" field or a "benchmark" comparison). You can't just change the Python class and hope the database magically updates — the database has no idea your Python code changed. Alembic bridges that gap. It generates a migration script — a step-by-step instruction set that transforms the old database schema into the new one without losing any existing data. Run alembic upgrade head and the database catches up. It's version control for your database structure, and every production application uses something like it.

Part 3: The Finance Concepts (Explain Like I'm 12)

You don't need a finance degree to understand these concepts. They all boil down to answering one question: "How risky is this investment?" The scanner computes all of them automatically, but knowing what they mean turns a wall of numbers into real insight.

Daily Returns — Today's Report Card

If a stock was $100 yesterday and it's $102 today, it went up 2%. That's the daily return. If it dropped to $97, the return is −3%. Simple as that.

daily return = (today's price − yesterday's price) / yesterday's price

We compute these for every day in the date range, giving us a long list of small percentages — the raw material for everything that follows. Every other metric in Part 3 is derived from this list of daily returns.

Cumulative Return — The Total Score

Cumulative return answers: "If I invested $100 at the start and held on until now, how much do I have?" It chains all the daily returns together.

If the stock went up 2%, then down 1%, then up 3%, the cumulative return is not simply 2 − 1 + 3 = 4%. You have to compound: 1.02 × 0.99 × 1.03 = 1.0400 — so about 4.0%. Close in this example, but over hundreds of days and larger swings, the difference between adding and compounding becomes significant. Always compound.

The scanner shows cumulative return as a chart over time, so you can see not just the final number but the entire journey. A stock that returned 20% might have gotten there smoothly (a steady climb) or violently (a huge drop followed by a frantic recovery). The chart tells the story that the single number hides. Two stocks with identical cumulative returns can have completely different risk profiles — and the other metrics in this section (drawdown, volatility, VaR) capture exactly those differences. That is why the scanner computes all of them together. No single metric tells the whole story.

Max Drawdown — How Bad Did It Get?

Imagine you're hiking and you reach the top of a hill. Then the trail drops steeply before climbing again. Max drawdown measures the deepest valley: the biggest drop from any peak to the next trough.

If a stock hit $150, then fell to $120 before recovering, the max drawdown is ($150 − $120) / $150 = −20%. This is the worst-case "I bought at the top and panicked at the bottom" scenario. Investors care about this a lot, because a big drawdown can wipe out years of gains and test your nerve.

For context: the S&P 500's max drawdown during the 2008 financial crisis was about −57%. During COVID in March 2020, it was about −34%. A stock with a max drawdown of −15% over the same period was comparatively calm.

Volatility — How Much It Bounces

Some stocks barely move — up 0.1% one day, down 0.2% the next. Others swing wildly — up 5%, down 4%, up 6%. Volatility measures the size of those swings. Technically, it's the standard deviation of daily returns, usually annualized (multiplied by the square root of 252, the number of trading days in a year).

Higher volatility means more uncertainty. A stock with 40% annual volatility is a roller coaster. One with 10% is a gentle train ride. Neither is "good" or "bad" by itself — it depends on how much bumping around you can stomach. A retiree living off their portfolio needs low volatility. A 25-year-old with decades ahead of them can handle more.

For reference: large, boring companies (utilities, consumer staples) tend to have volatility around 15–20%. Tech stocks often sit around 25–35%. Meme stocks and crypto can easily hit 80–100%. The scanner computes this for each ticker so you can see at a glance which holdings are calm and which are volatile.

One subtlety: volatility treats upside and downside swings equally. A stock that jumps 5% up is just as "volatile" as one that drops 5% down. Most investors care more about downside swings. There's a variant called downside deviation that only counts negative returns, but standard volatility remains the industry default because it's simple, widely understood, and used in the Sharpe ratio calculation below.

VaR (Value at Risk) — The "Probably Won't Lose More Than" Number

VaR answers a specific question: "On 95% of days, how much is the most I'll lose?"

If your portfolio's daily 95% VaR is $500, it means: on a typical day, there's only a 5% chance you'll lose more than $500. It doesn't tell you what happens in that worst 5% — just where the line is.

Think of it like weather: "There's a 95% chance it won't rain more than 2 inches." Useful, but doesn't tell you about the hurricane.

Computing it is surprisingly straightforward: take all historical daily returns, sort them from worst to best, and find the value at the 5th percentile. That's your 95% VaR. Banks compute this number every single night for their entire portfolio. Regulators require it. Risk committees review it every morning. It's one of the most important numbers in finance.

CVaR (Conditional VaR) — How Bad Is the Bad Day?

CVaR picks up where VaR leaves off. It asks: "OK, so 5% of the time I lose more than $500. In those bad scenarios, what's the average loss?"

If the CVaR is $800, it means: when things go wrong (the worst 5%), you lose $800 on average. CVaR captures the severity of tail events — the stuff VaR glosses over.

Regulators love CVaR because it forces you to think about the worst-case, not just the threshold. Two portfolios can have the same VaR but wildly different CVaR. Portfolio A might lose $550 on its worst days. Portfolio B might lose $2,000. CVaR reveals that difference. The scanner computes both, because together they tell a much more complete risk story than either one alone.

Sharpe Ratio — Reward per Unit of Risk

Imagine two stocks. Both returned 15% last year. But Stock A barely moved (low volatility), while Stock B was a heart-attack roller coaster (high volatility). Which one is better? Stock A — same reward, less pain.

The Sharpe ratio makes this precise:

Sharpe = (return − risk-free rate) / volatility

The "risk-free rate" is what you'd earn by parking money in government bonds — zero risk, modest return. The Sharpe ratio subtracts that baseline so you're measuring the extra return you got for taking on the extra risk of stocks.

A higher Sharpe means you're being better compensated for the risk you're taking. Below 1.0 is mediocre — you're not getting enough reward for the volatility you endured. Between 1.0 and 2.0 is solid. Above 2.0 is excellent and rare. The scanner computes this for every ticker so you can compare apples to apples.

Portfolio Weights — How You Split Your Money

If you have $10,000 and put $5,000 in Apple, $3,000 in Google, and $2,000 in Tesla, your weights are 50%, 30%, and 20%. Weights always add up to 100%.

The scanner lets you set custom weights for each ticker. The risk metrics then apply to the combined portfolio, not just individual stocks. A well-diversified portfolio (money spread across different sectors) usually has lower volatility and smaller drawdowns than any single stock. This is the core idea behind diversification: don't put all your eggs in one basket, and the math will reward you with a smoother ride.

Part 4: The Web Stack (Explain Like I'm 12)

The capstone glues all eight exercises into a working application. Here's how the pieces connect. If Part 2 taught you the individual instruments, this section teaches you how they play together as an orchestra.

Streamlit — The Dashboard

Streamlit lets you build a web dashboard using nothing but Python. No HTML, no CSS, no JavaScript. You write st.title("Stock Risk Scanner") and a title appears. You write st.line_chart(data) and a chart appears. It's the fastest way to go from "I have some data" to "I have a web app."

Streamlit Cloud hosts it for free. Push code to GitHub, and Streamlit deploys it automatically. The dashboard lives at quantlabs.streamlit.app. If you've ever built something in a Jupyter notebook and wished you could share it as a real website, Streamlit is the answer.

FastAPI — The Backend Brain

The dashboard doesn't do the heavy lifting — it's just the face. Behind the scenes, it sends requests to a FastAPI backend that does all the real work: fetching prices, crunching risk metrics, calling Claude, saving to the database. This separation is intentional. The frontend can be rebuilt in React or Vue tomorrow without touching a single line of backend code. The API is the contract between them.

uvicorn / gunicorn — The Waiters

FastAPI defines what to do with each request, but it needs a server program to actually listen for incoming requests and deliver them. That's where uvicorn and gunicorn come in.

uvicorn is an ASGI server — it speaks the async protocol that FastAPI uses. Think of it as a single, very fast waiter who can juggle many tables because it never blocks.

gunicorn is a process manager. It spawns multiple uvicorn workers, so instead of one waiter you have four or eight, each handling requests independently. If one worker crashes, gunicorn restarts it automatically. On Render, the production command looks something like gunicorn -k uvicorn.workers.UvicornWorker app:app — gunicorn manages the workers, each worker runs uvicorn, and uvicorn runs your FastAPI code.

PostgreSQL — The Filing Cabinet

Every scan result is saved permanently. If you scanned AAPL, MSFT, and TSLA last Tuesday, that result is still in the database today. The dashboard can retrieve it anytime, even if the backend restarted ten times since then. This is the difference between an app and a toy: persistence. Without a database, every scan result vanishes the moment the server restarts. With one, your app has memory.

PostgreSQL handles concurrent access gracefully — if two users scan at the same time, both results are saved correctly without conflicts. It also supports complex queries: "show me all scans where max drawdown exceeded −25%" is a one-line SQL query.

Render — Where the Backend Lives

Render is a cloud platform that runs your backend for you. Push code to GitHub, and Render automatically builds and deploys it. It's like AWS but simpler — fewer knobs, less configuration, and a generous free tier for small projects. You don't have to set up servers, configure load balancers, or manage SSL certificates. Render handles all of that.

The scanner's FastAPI backend and PostgreSQL database both run on Render. The free tier spins down the server after 15 minutes of inactivity, which means the first request after a quiet period takes a few seconds to "wake up." That's the trade-off for free hosting — perfectly fine for a portfolio project.

One practical note: Render's free PostgreSQL databases expire after 90 days. When that happens, you create a new one and update the connection string. The data resets, but the application keeps running.

Streamlit Cloud — Where the Dashboard Lives

Same idea, but for the frontend. Streamlit Cloud watches your GitHub repo and redeploys whenever the dashboard code changes. Zero server management. You connect your GitHub account, point it at your repo, and every push to the main branch triggers a redeploy. It is the simplest possible deployment pipeline for a Python web app. No Docker required on the frontend side — Streamlit Cloud handles the Python environment for you.

GitHub Actions — The Robot Tester

Every time someone opens a pull request, a robot runs all 27 tests automatically. If any test fails, the PR is blocked. No broken code reaches production. It's a safety net that never sleeps.

The workflow is defined in a YAML file in the repository. It says: "When a PR is opened, spin up a virtual machine, install Python, install dependencies, run pytest. If any test fails, mark the PR with a red X." This is called Continuous Integration (CI) — continuously checking that new code doesn't break existing code.

Every single one of QuantLab's exercises went through this workflow: code on the working branch, open a PR to master, wait for all tests to pass, merge. Twenty pull requests, twenty green checkmarks, zero broken merges. The robot never gets tired, never forgets a test, and never merges broken code because it's Friday afternoon.

The Architecture — Putting It All Together

Now let's see how all these pieces connect. Here's how a scan request flows through the entire system, from the moment you click "Scan" to the moment results appear on screen:

User opens quantlabs.streamlit.app
       |
       v
  +------------+     POST /api/scan      +--------------+
  | Streamlit  | ----------------------> |  FastAPI on  |
  | Dashboard  |                         |   Render     |
  |            |  <---------------------- |              |
  |            |     { scan_id: "abc" }  +------+-------+
  |            |                                |
  |  polls     |                    +-----------+-----------+
  |  until     |                    v           v           v
  |  done      |              +----------+ +----------+ +----------+
  |            |              | yfinance | |  NumPy   | | Claude   |
  +------------+              | (prices) | | (risk)   | | (story)  |
                              +----------+ +----------+ +----------+
                                                |
                                                v
                                         +--------------+
                                         | PostgreSQL   |
                                         | (save result)|
                                         +--------------+

Step by step:

  1. You type tickers (like AAPL, MSFT, TSLA) and click "Scan" on the Streamlit dashboard.
  2. The dashboard sends a POST /api/scan request to the FastAPI backend on Render.
  3. FastAPI validates the input with Pydantic, immediately responds with a scan ID (HTTP 202), and starts processing in the background.
  4. The backend fetches prices from yfinance using async (all tickers in parallel), computes risk metrics with NumPy, and requests a narrative from Claude.
  5. Results are saved to PostgreSQL via SQLAlchemy.
  6. The dashboard polls GET /api/scans/{id} every few seconds until the status is "done," then displays the metrics, charts, and AI narrative.

The whole flow takes a few seconds. The user sees a progress spinner, then results. Behind the scenes, eight exercises worth of technology are working together seamlessly.

Notice how the architecture separates concerns cleanly. The dashboard knows nothing about NumPy or SQL. The risk engine knows nothing about Streamlit or HTTP. The database knows nothing about stock prices. Each piece does its job and passes the result to the next piece. This is what makes the system maintainable — you can change any one piece without rewriting the others.

This architecture also scales naturally. If the backend gets slow, you can add more workers on Render. If the database gets full, you can upgrade the PostgreSQL plan. If you want a mobile app instead of a web dashboard, you build a new client that talks to the same API. The backend doesn't care who's asking — it just processes requests and returns results.

Part 6: Key Design Patterns Worth Remembering

The capstone isn't just code that works — it's code that's designed to be maintained, extended, and debugged by future-you (who will have forgotten everything present-you knows). These patterns are the reason professional codebases survive for years while hobby projects fall apart after a few months.

HTTP 202 — Accept Now, Process Later

When the dashboard sends a scan request, the backend doesn't make it wait while it fetches fifty stock prices, crunches numbers, and calls an AI. Instead, it responds immediately with HTTP 202 ("Accepted") and a scan ID. The actual work happens in the background. The dashboard polls for updates.

# Simplified version of the pattern
@app.post("/api/scan", status_code=202)
async def start_scan(request: ScanRequest):
    scan_id = str(uuid4())
    background_tasks.add_task(run_scan, scan_id, request)
    return {"scan_id": scan_id, "status": "processing"}

This pattern is everywhere in production systems. Any time work takes more than a second or two, you accept the job, hand back a receipt, and let the client check back later. Pizza delivery works the same way: you place the order, get a confirmation number, and track it until it arrives. You don't stand at the counter watching them make it.

In the scanner, the Streamlit dashboard sends a scan request, receives the scan ID, and enters a polling loop: "Is it done yet? Is it done yet?" Every couple of seconds, it checks. When the status changes to "done," it fetches the full results and displays them. The user sees a smooth progress experience; behind the scenes, it's just a simple poll.

Graceful Degradation — Bend, Don't Break

The Claude API might be down. The API key might be missing. The rate limit might be hit. If the narrative generator fails, the scanner does not crash. It returns all the numerical metrics with a note saying the AI narrative is unavailable. The user still gets 90% of the value.

This is a fundamental principle: every external dependency can fail, and your app should have a plan for when it does. The scanner handles this at every layer:

  • yfinance returns no data? Report an error for that ticker but keep scanning the others.
  • Claude API is down? Return all numerical metrics with a note that the narrative is unavailable.
  • Database unreachable? Log the error and return results without persisting them.

Each failure is contained, never catastrophic. The user always gets something useful, even when parts of the system are misbehaving. This principle has a name in engineering: fault isolation. A failure in one subsystem should not cascade into a failure of the entire system.

Pure Functions — One Job Each

The codebase is split into modules that each do exactly one thing:

  • market_data.py — fetches prices. Nothing else.
  • risk.py — computes metrics. Takes numbers in, returns numbers out. No network calls, no database.
  • narrative.py — calls Claude. Takes metrics in, returns text out.

Each module can be tested independently, replaced independently, and understood independently. When something breaks, you know exactly where to look. Want to swap yfinance for a different data provider? Replace market_data.py — nothing else changes. Want to switch from Claude to GPT? Replace narrative.py. The rest of the system is untouched.

The Test Pyramid

Not all tests are created equal:

  • Unit tests (fast, many) — test one function in isolation using mocks. "Does risk.py compute volatility correctly given these fake prices?"
  • Integration tests (slower, fewer) — test multiple pieces working together with a real database. "Does the scan endpoint actually save results to PostgreSQL?"

You want lots of fast unit tests for quick feedback and a handful of integration tests to make sure the pieces actually connect. This pyramid keeps the test suite fast enough to run on every commit.

In the capstone, unit tests mock out yfinance (so you don't need an internet connection to test risk calculations) and mock out Claude (so you don't burn API credits on every test run). Integration tests spin up a real PostgreSQL instance and verify that scan results actually round-trip through the database correctly. The full suite runs in under 30 seconds.

Mocking is a testing technique where you replace a real dependency with a fake one. Instead of calling the real Yahoo Finance API (which is slow, requires internet, and might return different data each time), the mock instantly returns a fixed set of prices that you chose. This makes your tests fast, deterministic, and free. The real API gets tested separately, during integration tests.

Separation of Concerns

The capstone follows a clean layering:

  • Models (Pydantic) — define the shape of data.
  • Data (yfinance, market_data.py) — fetch raw information.
  • Compute (risk.py) — crunch the numbers.
  • Present (narrative.py, Streamlit) — make it human-readable.
  • Persist (SQLAlchemy, PostgreSQL) — save it permanently.

Each layer talks only to its neighbors. The risk engine doesn't know about the database. The database doesn't know about Streamlit. If you swap PostgreSQL for MySQL, only the persistence layer changes. Everything else stays untouched.

This isn't just tidy coding for its own sake. When a bug appears in the risk calculations, you know it's in risk.py. When the dashboard looks wrong, you know it's in the Streamlit code. When scan results aren't saving, you know it's in the persistence layer. Clean separation turns "something is broken somewhere" into "this specific thing is broken in this specific file." That's the difference between a 5-minute fix and a 5-hour scavenger hunt.

Part 7: How to Study This

Eight exercises plus a capstone is a lot of material. Here's a practical study plan that won't overwhelm you. The goal is not to memorize every line — it's to build mental models of how the pieces fit together. Once you have the mental model, you can always look up the syntax.

  1. Run the tests first.

    Open a terminal in any exercise folder and run pytest -v. Read the test names — they're written in plain English. test_sharpe_ratio_with_zero_volatility tells you exactly what edge case is being checked. The tests are the best documentation.

  2. Trace a scan request end to end.

    Start at the Streamlit dashboard code. Find where it calls the API. Jump to the FastAPI route handler. Follow the code into market_data.py, then risk.py, then narrative.py, then db.py. One full loop makes the entire architecture click. This is the single most valuable exercise you can do — it transforms abstract understanding into concrete knowledge.

  3. Compare the exercise to the capstone.

    Exercise 03 builds a toy watchlist API with hardcoded data. The capstone's /api/scan endpoint uses the same FastAPI patterns but with real data. Exercise 02's simple Pydantic models become the capstone's ScanRequest and RiskMetrics. Seeing the toy version next to the production version makes the jump feel small instead of intimidating. The patterns are the same — only the data is real.

  4. Change something and see what breaks.

    Delete a Pydantic validator and run the tests — watch them fail. Remove the async keyword from a function and see what error you get. Change a VaR confidence level from 0.95 to 0.50 and see the number change. Breaking things on purpose is the fastest way to understand why they exist.

  5. Draw the data flow on paper.

    Grab a pen and draw boxes for each module: market_data, risk, narrative, db. Draw arrows showing what data flows between them. Then compare your drawing to the architecture diagram in this post. If they match, you understand the system.

  6. Read the exercises in order.

    They build on each other intentionally. Exercise 04 (async) doesn't make sense without Exercise 03 (FastAPI). Exercise 08 (database) uses patterns from Exercise 04 (async sessions). Skipping ahead creates confusion that skipping back solves. If something in Exercise 06 feels confusing, the answer is probably in Exercise 02 or 03.

  7. Try building a feature.

    Once you understand the capstone, try adding something small. Maybe a new metric (like Sortino ratio, which is Sharpe but only penalizes downside volatility). You'll need to:

    • Add the field to the Pydantic model (Exercise 02 skills)
    • Write the computation in risk.py (Exercise 01 and 05 skills)
    • Display it on the Streamlit dashboard (capstone skills)
    • Write a test for it (Exercise 01 skills)

    That's a full vertical slice through the stack. The experience is worth more than reading about it ten times.
