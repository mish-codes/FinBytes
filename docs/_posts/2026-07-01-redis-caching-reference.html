Redis Is Not Just a Cache

Redis is an in-memory key-value data store. Most people meet it as a cache layer, but it is also a message broker (pub/sub), a session store, a rate limiter, a leaderboard engine (sorted sets), and a lightweight queue. It is single-threaded, which means every operation is atomic without explicit locking.

Start a local instance in seconds:

# Run Redis in Docker (Alpine image is ~13 MB)
docker run -d --name redis -p 6379:6379 redis:alpine

# Verify it is alive
docker exec redis redis-cli ping
# → PONG

# Interactive CLI
docker exec -it redis redis-cli

Python with redis-py

Install: pip install redis

import redis
import json

# --- Connection (use a pool in production) ---
pool = redis.ConnectionPool(host="localhost", port=6379, db=0,
                            decode_responses=True)
r = redis.Redis(connection_pool=pool)

# --- Strings ---
r.set("ticker:AAPL:price", "189.25")
r.setex("ticker:AAPL:price", 60, "189.25")  # expires in 60 seconds
price = r.get("ticker:AAPL:price")           # "189.25" (str)

# --- Hashes ---
r.hset("bond:US10Y", mapping={
    "coupon": "4.25",
    "maturity": "2036-03-15",
    "price": "98.50",
})
coupon = r.hget("bond:US10Y", "coupon")      # "4.25"
bond = r.hgetall("bond:US10Y")               # dict of all fields

# --- Sorted Sets (leaderboard by spread) ---
r.zadd("spreads:IG", {"AAPL": 55.2, "MSFT": 48.1, "GS": 102.7})
top3 = r.zrangebyscore("spreads:IG", 0, 100, withscores=True)
# [('MSFT', 48.1), ('AAPL', 55.2)]

Cache-Aside Pattern

The most common caching pattern. The application checks the cache first. On a miss, it computes (or fetches from DB), stores the result, and returns it.

import redis
import json
from sqlalchemy.orm import Session

r = redis.Redis(host="localhost", port=6379, db=0, decode_responses=True)
CACHE_TTL = 300  # 5 minutes

def get_yield_curve(curve_date: str, db: Session) -> dict:
    """Cache-aside: check Redis first, fall back to DB."""
    cache_key = f"yield_curve:{curve_date}"

    # 1. Check cache
    cached = r.get(cache_key)
    if cached is not None:
        return json.loads(cached)  # HIT

    # 2. Cache miss — query database
    row = db.execute(
        "SELECT data FROM yield_curves WHERE curve_date = :d",
        {"d": curve_date},
    ).fetchone()
    if row is None:
        return {}

    result = row["data"]

    # 3. Store in cache with TTL
    r.setex(cache_key, CACHE_TTL, json.dumps(result))

    # 4. Return
    return result

TTL Strategies

Different data tolerates different staleness. Set TTLs accordingly:

  • Real-time prices: 5–15 seconds (stale prices are dangerous)
  • Yield curves: 5–15 minutes (curves update once or twice per day, but you want fresh data during the window)
  • Reference data (bond metadata, issuer info): 1–24 hours
  • User sessions: 30 minutes (sliding expiration with EXPIRE)
# Different TTLs for different data
r.setex("price:AAPL", 10, "189.25")             # 10 seconds
r.setex("yield_curve:2026-06-30", 900, payload)  # 15 minutes
r.setex("ref:bond:US912828ZT09", 86400, meta)    # 24 hours

LRU Eviction

When Redis hits its memory limit, it needs to evict keys. The most common policy is allkeys-lru—evict the least-recently-used key regardless of whether it has a TTL.

# Set max memory to 256 MB with LRU eviction
# In redis.conf or via CLI:
redis-cli CONFIG SET maxmemory 256mb
redis-cli CONFIG SET maxmemory-policy allkeys-lru

# Check current memory usage
redis-cli INFO memory | grep used_memory_human
# used_memory_human:1.23M

Other policies: volatile-lru (only evict keys with a TTL), allkeys-lfu (least frequently used), noeviction (return errors when full).

lru_cache vs Redis

Python's functools.lru_cache is a process-local, in-memory cache. It is fast and requires zero infrastructure. But it breaks in production:

  • Multiple workers: Gunicorn spawns 4 workers = 4 separate caches. Each worker misses independently and queries the DB.
  • No TTL: lru_cache has no built-in expiration. Data stays cached until the process restarts or the cache fills up.
  • No invalidation: If data changes in the DB, every worker's cache is stale until eviction.
from functools import lru_cache

# Fine for single-process scripts or CLI tools
@lru_cache(maxsize=128)
def compute_spot_rate(tenor: float, par_rate: float) -> float:
    """Expensive calculation — cache locally."""
    # ... bootstrap logic ...
    return spot

# NOT fine for web apps with multiple workers.
# Use Redis instead so all workers share one cache.

Pub/Sub

Redis pub/sub is fire-and-forget: if no subscriber is listening, the message is lost. Good for real-time notifications; bad for reliable job processing (use RabbitMQ or Redis Streams for that).

# publisher.py
import redis, json, time

r = redis.Redis(host="localhost", port=6379, decode_responses=True)

while True:
    msg = {"ticker": "AAPL", "price": 189.25, "ts": time.time()}
    r.publish("prices", json.dumps(msg))
    time.sleep(1)

# subscriber.py
import redis, json

r = redis.Redis(host="localhost", port=6379, decode_responses=True)
pubsub = r.pubsub()
pubsub.subscribe("prices")

for message in pubsub.listen():
    if message["type"] == "message":
        data = json.loads(message["data"])
        print(f"{data['ticker']}: ${data['price']}")

Common Gotchas

1. Connection pooling. Creating a new redis.Redis() per request opens a new TCP connection each time. Always use a ConnectionPool or instantiate Redis once at module level.

2. Serialization. Redis stores bytes (or strings with decode_responses=True). For complex objects, serialize with json.dumps() on write and json.loads() on read. Do not use pickle—it is a security risk for untrusted data.

3. Key naming conventions. Use colons as separators: entity:id:field. Examples: user:42:session, yield_curve:2026-06-30, price:AAPL. This makes keys scannable with SCAN MATCH price:*.

4. KEYS vs SCAN. Never use KEYS * in production—it blocks Redis while scanning every key. Use SCAN with a cursor instead.

Monitoring

# Memory usage
redis-cli INFO memory

# Number of connected clients
redis-cli INFO clients | grep connected_clients

# Watch every command in real time (NEVER use in production — it slows Redis)
redis-cli MONITOR
