Part 1: What We Built and Why

QuantLab Phase 2 is twelve exercises (09–20) that take you from raw government interest rate data all the way to a portfolio credit-risk system running on Amazon Web Services. Two capstones snap everything together at the end.

The exercises split into two tracks:

  • The Rates Track (09–14) — fetching Treasury data, storing it, and doing progressively more sophisticated math on interest-rate curves.
  • The Credit Track (15–20) — using those curves to price credit risk: how likely is a company to default, what does that cost, and how does a whole portfolio behave under stress?

Along the way, each exercise introduces a real AWS service, a real piece of infrastructure tooling, or a real piece of financial math. By the end, you have a system that can ingest live data, compute dozens of risk metrics, cache results, send alerts, stream updates to a dashboard, and deploy itself — all from code.

This post explains every single piece, assuming you have never written a line of code or read a finance textbook. If you are twelve years old and curious, this is for you.

Part 2: The Cloud Technologies (Explain Like I'm 12)

AWS — The Concept

Imagine you need a computer to run a program, but instead of buying one, you rent one from a giant company that has millions of computers sitting in warehouses around the world. That's Amazon Web Services (AWS). You only pay for the time you actually use, like renting a bike by the minute instead of buying one.

AWS isn't one thing — it's a menu of dozens of services. Each one does a specific job. Here are the ones we used.

S3 — The Filing Cabinet (Exercise 10)

S3 stands for "Simple Storage Service." Think of it as a giant filing cabinet in the cloud. You can throw any file in there — a spreadsheet, a photo, a JSON blob of Treasury yield data — and it will keep it safe, forever, for almost no money.

In Exercise 10, we fetch raw Treasury par yields from the FRED API (the Federal Reserve's public data service) and drop the JSON files into an S3 bucket. That bucket is our "source of truth" — the original, untouched data. Everything downstream reads from here.

RDS / PostgreSQL — A Spreadsheet on Steroids (Exercise 11)

RDS stands for "Relational Database Service." It runs a real database — in our case, PostgreSQL — on an AWS server, so you don't have to install anything yourself.

Think of it as a spreadsheet that can hold millions of rows, answer questions in milliseconds, and let fifty people query it at the same time without breaking. In Exercise 11, we take the raw JSON from S3 and load it into structured database tables: one row per date per maturity, neatly organized.

Lambda — A Helper Who Only Exists When Needed (Exercises 12, 13, 14)

Imagine you could hire someone who appears the instant you clap your hands, does exactly the job you asked for, and then disappears. You only pay for the thirty seconds they were working. That's Lambda.

Lambda runs your code without you having to own or manage a server. You upload a function, and AWS runs it whenever something triggers it — an API request, a timer, a new file landing in S3. When it's done, the function vanishes. No idle servers burning money.

We use Lambda in Exercises 12, 13, and 14 to run our math: bootstrapping spot curves, computing forward rates, and fitting Nelson-Siegel/Svensson models. Each of these is a Lambda function that wakes up when called, crunches numbers, and returns a result.

API Gateway — The Receptionist (Exercises 12, 14)

API Gateway is the front door to your system. When someone (or some program) sends a request like "give me the 5-year spot rate for 2025-03-15," API Gateway receives that request, figures out which Lambda function should handle it, passes the request along, and returns the answer.

It's like a receptionist at a hotel: you don't walk directly into the kitchen to ask for breakfast. You tell the receptionist, and they route your request to the right person.

DynamoDB — The Dictionary Lookup (Exercise 16)

DynamoDB is a database, but a different kind from PostgreSQL. It's a "key-value" store — think of an actual dictionary. You look up a word (the key) and instantly get the definition (the value). No complex queries, no joins. Just fast lookups by key.

In Exercise 16, we use DynamoDB to store WebSocket connection IDs. When a client connects to receive live streaming data, we save their connection ID in DynamoDB so we know who to send updates to.

SQS — A To-Do List for Computers (Exercise 15)

SQS stands for "Simple Queue Service." Picture a to-do list pinned to a fridge. One person writes tasks on it; another person checks the list and works through them, one at a time. If the second person is busy or asleep, the tasks just wait on the list. Nothing gets lost.

In Exercise 15, when new credit spread data arrives, a message gets added to the SQS queue: "Hey, recalculate the spread for BBB bonds." A Lambda function picks up that message, does the calculation, and marks the task as done. If something goes wrong, the message goes back on the list and gets retried.

SNS — The Megaphone (Exercise 15)

SNS stands for "Simple Notification Service." If SQS is a to-do list, SNS is a megaphone. You shout a message, and everyone who has signed up to listen hears it — email subscribers, other queues, Lambda functions, Slack webhooks, whatever.

In Exercise 15, when a credit spread blows past a threshold (say, BBB spreads exceed 200 basis points), SNS fires an alert to every subscriber: "Warning — spreads are widening."

ElastiCache / Redis — The Sticky Note (Exercise 17)

Redis is an in-memory cache. "In-memory" means it lives in RAM, not on a hard drive, so it's absurdly fast — microseconds instead of milliseconds.

Think of it as a sticky note on your monitor. You just computed the 10-year spot rate, and you know someone is going to ask for it again in five seconds. Instead of running the whole calculation again, you scribble the answer on a sticky note. Next time someone asks, you just read the note.

ElastiCache is the AWS service that runs Redis for you. In Exercise 17, we cache bond-pricing results so repeat requests get answered instantly.

CloudWatch — The Security Camera (Exercise 19)

CloudWatch watches everything. It collects logs (what happened), metrics (how many times, how fast), and alarms (something broke). It's the security camera for your entire system.

In Exercise 19, we set up CloudWatch dashboards to monitor OAS and Z-spread calculations: how long they take, how often they fail, and whether error rates are climbing. If a Lambda function starts timing out, CloudWatch triggers an alarm.

WebSocket API — A Phone Call vs. Texting (Exercise 16)

Normal web requests are like texting: you send a message, wait for a reply, and the conversation is over. If you want an update, you have to text again.

A WebSocket is like a phone call: you dial once, and the line stays open. Both sides can talk whenever they want, with no delay. In Exercise 16, we use a WebSocket API so dashboards can receive live-streaming credit spread updates the instant they change, without refreshing or polling.

Part 3: The DevOps Technologies

Docker — A Lunchbox for Code (Exercise 14 Deploy)

You packed a great lunch at home, but when you open it at school, the sandwich is squished and the juice leaked. Docker solves this problem for code.

A Docker container is a sealed lunchbox. It holds your code, plus every library and tool your code needs, packed in exactly the right way. Whether you open it on your laptop, on a coworker's laptop, or on an AWS server, everything inside is identical. No "but it works on my machine" surprises.

In Exercise 14, we package our curve-fitting Lambda as a Docker container image, so deployment is perfectly reproducible.

Terraform — A Blueprint for Infrastructure (Exercises 14, 18)

Imagine building a house by clicking buttons on a website: "create a room," "add a door," "install plumbing." Now imagine you have to build the same house twenty times. Clicking is painful. Terraform lets you write a blueprint instead.

You describe what you want — "I need an S3 bucket, a Lambda function, an API Gateway, a database" — in a text file. Then you run terraform apply, and Terraform builds everything for you. Need to tear it down? terraform destroy. Need to change something? Edit the file and apply again. The infrastructure is code.

resource "aws_lambda_function" "curve_fitter" {
  function_name = "nelson-siegel-fitter"
  runtime       = "python3.11"
  handler       = "handler.lambda_handler"
  memory_size   = 512
  timeout       = 30
}

That block creates a Lambda function. Terraform tracks what exists and what doesn't, so it only builds what's missing and updates what changed. In Exercises 14 and 18, Terraform manages the entire stack.

GitHub Actions + CI/CD — The Robot Deployer (Exercise 13)

CI/CD stands for "Continuous Integration / Continuous Deployment." It means: every time you push code to GitHub, a robot automatically runs your tests, and if they pass, deploys the new version. No human clicking "deploy" at 2 a.m.

GitHub Actions is the robot. You write a YAML file that says: "When code is pushed to the working branch, install Python, run pytest, and if everything passes, deploy to AWS." The robot does exactly that, every single time.

OIDC — The Wristband at a Theme Park (Exercise 13)

When you buy a ticket at a theme park, they give you a wristband. Every ride checks the wristband instead of making you buy a new ticket. OIDC (OpenID Connect) is the wristband for GitHub Actions.

Instead of storing long-lived AWS passwords in GitHub (dangerous), we set up a trust relationship: "AWS, please trust anyone wearing a GitHub Actions wristband from this specific repository." GitHub presents a short-lived token, AWS verifies it, and the deploy proceeds — no secrets stored anywhere.

Git / GitHub / Pull Requests — Infinite Undo + Collaboration

Git is version control. Every time you save ("commit"), Git takes a snapshot of every file. You can go back to any snapshot, compare any two, or branch off into an experiment without touching the main code.

GitHub hosts your Git repository online and adds collaboration tools. A "Pull Request" is a proposal: "Here's what I changed. Review it, discuss it, and if it looks good, merge it in." Every exercise in QuantLab went through this workflow: work on working, PR to master, merge after review.

Testing with moto and fakeredis — Fake AWS for Tests

You don't want to hit real AWS services every time you run a test — it's slow, it costs money, and it's flaky. Instead, we use fakes.

moto is a Python library that pretends to be AWS. When your code calls "put this object in S3," moto intercepts the call and stores it in memory. Your code thinks it's talking to AWS, but it's really talking to a fake. fakeredis does the same thing for Redis.

This means every test runs in milliseconds, with zero cloud costs, and works offline.

Part 4: The Finance Math (Explain Like I'm 12)

4a. The Rates Track (Exercises 09–14)

What Is a Yield Curve?

Suppose you lend money to the U.S. government. If you lend for 1 year, they pay you 4%. If you lend for 10 years, they pay you 4.5%. If you lend for 30 years, they pay you 4.8%. Plot those rates on a graph — time on the x-axis, rate on the y-axis — and you get a yield curve.

The yield curve is one of the most important objects in all of finance. It tells you the price of time. Every mortgage rate, every corporate bond price, every swap, every pension liability — all of them start from this curve.

Par Yields — What the Treasury Publishes

The U.S. Treasury publishes "par yields" every business day. A par yield is the coupon rate at which a bond would trade at exactly its face value (par). If the 5-year par yield is 4%, it means a brand-new 5-year Treasury note paying a 4% coupon would trade at exactly $100.

These are the raw ingredients. Exercise 09 fetches them from the FRED API.

Spot Rates and Bootstrapping — Peeling the Onion

Par yields mix up two things: the coupon payments you receive along the way and the big payment at the end. A spot rate is purer: it's the interest rate for a single payment at a specific time. "What rate do I earn if I lock my money away for exactly 5 years, with zero payments in between?"

Bootstrapping is how you extract spot rates from par yields. It works like peeling an onion. You already know the 1-year spot rate (it equals the 1-year par yield, because a 1-year bond only has one payment). Now take the 2-year par bond: you know its coupon, you know the 1-year spot rate for the first coupon, so you can solve for the 2-year spot rate. Repeat for 3, 5, 7, 10, 20, 30 years — each step uses everything you've already solved.

def bootstrap_spot_rates(par_yields: dict) -> dict:
    spots = {}
    for tenor, par_rate in sorted(par_yields.items()):
        if tenor <= 1.0:
            spots[tenor] = par_rate
        else:
            # Discount all intermediate coupons using known spots
            coupon = par_rate / 2          # semiannual
            pv_coupons = sum(
                coupon / (1 + spots[t] / 2) ** (2 * t)
                for t in sorted(spots) if t < tenor
            )
            # Solve for the spot rate at this tenor
            remaining = (100 + coupon) / (100 - pv_coupons)
            n = 2 * tenor
            spots[tenor] = 2 * (remaining ** (1 / n) - 1)
    return spots

Each step "peels" one more layer, using the rates you've already found to isolate the next unknown. Exercise 12 implements this as a Lambda function.

Forward Rates — What the Market Expects

A forward rate answers: "If I've already committed to lend for 2 years, what rate does the market imply for lending from year 2 to year 5?" It's the rate that makes you indifferent between two strategies: lend for 5 years straight, or lend for 2 then reinvest for 3.

Mathematically, the forward rate between times t1 and t2 satisfies:

(1 + spot_t2)^t2 = (1 + spot_t1)^t1 × (1 + forward)^(t2 - t1)

Forward rates are like a weather forecast. They tell you what the market expects rates to be in the future. They're not always right, but they're the market's best guess today. Exercise 13 computes the full forward curve and deploys it via GitHub Actions with OIDC.

Nelson-Siegel / Svensson — Smooth Curves from a Few Numbers

Bootstrapping gives you spot rates at the specific maturities that bonds exist (1, 2, 3, 5, 7, 10, 20, 30 years). But what if you need the 4.7-year spot rate? You need a smooth curve.

Nelson-Siegel is a formula that draws a smooth curve through (or near) all your data points using just four numbers:

  • β0 — the long-term level (where rates settle in the very long run)
  • β1 — the slope (how steeply rates rise or fall as maturity increases)
  • β2 — the curvature (the hump or dip in the middle)
  • τ — the time scale (where the hump peaks)

Svensson adds two more parameters (β3 and τ2) for a second hump, giving extra flexibility. Exercise 14 fits these models using numerical optimization, packages the solver in Docker, and deploys everything via Terraform.

The beauty: instead of storing thousands of data points, you store four (or six) numbers, and you can reconstruct the entire curve at any maturity.

Cubic Spline — Connect the Dots Exactly

A cubic spline is the other approach: instead of a parametric formula, you connect the known data points with smooth polynomial segments. Think of bending a thin piece of metal through a set of nails — it passes through every nail and curves smoothly between them.

Nelson-Siegel gives you an interpretable summary; a spline gives you a perfect fit. Both are useful. We implement both.

4b. The Credit Track (Exercises 15–20)

Credit Spreads — The Fear Premium

A Treasury bond is considered "risk-free" — the U.S. government (almost) never defaults. A corporate bond, on the other hand, might default. To compensate investors for that risk, a corporate bond pays a higher rate. The difference is the credit spread:

credit spread = corporate yield − Treasury yield

Measured in basis points (1% = 100 bps). A spread of 150 bps means the company pays 1.5% more than the government. When investors get scared, spreads widen. When they feel safe, spreads tighten. Exercise 15 computes and monitors these spreads in real time using SQS and SNS.

CDS — Insurance on a Bond

A Credit Default Swap is literally insurance. You pay a quarterly premium (the "CDS spread") to a counterparty. If the company defaults, the counterparty pays you the loss. If the company doesn't default, you've just paid premiums for nothing — like car insurance when you don't crash.

CDS spreads are the market's cleanest measure of credit risk, because they strip out all the complications of actual bonds (coupon effects, call features, liquidity).

Hazard Rates and Survival Probability — The Bathtub Model

Imagine a bathtub with a slow leak. The water level represents the probability that a company has survived until now. The rate at which water drips out is the hazard rate (λ). A higher hazard rate means faster dripping — more risk.

Under a constant hazard rate:

P(survive to year T)  = exp(−λ × T)
P(default by year T)  = 1 − exp(−λ × T)

A CDS spread connects to the hazard rate through a simple relationship:

λ ≈ CDS spread / (1 − recovery rate)

If the CDS spread is 200 bps (0.02) and the recovery rate is 40%, then λ ≈ 0.02 / 0.60 ≈ 0.0333, meaning roughly a 3.3% chance of defaulting in any given year.

Piecewise Hazard Rates — Different Risk at Different Horizons

A constant hazard rate is a simplification. In reality, a company might be safe for the next year (low λ) but risky five years out (high λ). Piecewise hazard rates let us use a different λ for each time period, like different drip rates at different heights of the bathtub. Exercise 18 builds the full piecewise term structure.

Expected Loss = PD × LGD × EAD — The Banker's Formula

Banks boil credit risk down to three numbers:

  • PD (Probability of Default) — "What's the chance they don't pay?"
  • LGD (Loss Given Default) — "If they don't pay, how much do I lose?" (Usually 1 − recovery rate.)
  • EAD (Exposure at Default) — "How much do they owe me?"
Expected Loss = PD × LGD × EAD

If a company has a 3% chance of defaulting, you'd lose 60% if it did, and they owe you $1 million, then your expected loss is 0.03 × 0.60 × $1,000,000 = $18,000. Simple multiplication, but every bank in the world runs this formula every single day.

Z-Spread — One Number for "How Much Extra?"

The Z-spread is a single number (in basis points) that you add to every point on the Treasury spot curve to make a bond's calculated price match its market price. It's like asking: "How many extra basis points, applied uniformly, explain the gap between this bond's price and a risk-free bond?"

You find it by trial and error: guess a spread, price the bond, see if the price matches. Too high? Guess lower. Too low? Guess higher. A numerical solver does this in milliseconds. Exercise 19 implements this.

OAS and Binomial Trees — Strip Out the Call Option

Many corporate bonds are "callable" — the issuer can pay them off early. That call option is worth something to the issuer and costs the bondholder, which widens the Z-spread. But part of that spread is option cost, not credit risk.

The OAS (Option-Adjusted Spread) strips out the option. It builds a binomial tree — a branching diagram of possible future interest rates (rates go up or down at each step, like a coin flip) — and prices the bond at every node, accounting for whether the issuer would rationally call at that node. The OAS is the spread that, applied inside the tree, makes the model price match the market price.

OAS is the "clean" credit spread — pure credit risk, no option noise. Exercise 19 computes both Z-spread and OAS, monitored by CloudWatch.

Spread Duration / CS01 — Sensitivity to Credit Moves

CS01 answers: "If credit spreads move by 1 basis point, how much does my bond's price change?" It's the credit analogue of interest-rate duration. If your CS01 is $500, a 10 bps spread widening costs you $5,000.

Portfolio managers live and die by CS01. It tells them exactly how exposed they are to credit moves. Exercise 20 computes CS01 for every position in a portfolio.

Monte Carlo VaR with Cholesky — Correlated Coin Flips

Value at Risk (VaR) answers: "What's the worst loss I should expect over the next 10 days, with 99% confidence?"

To compute it, we simulate thousands of possible futures. Each simulation randomly wiggles every credit spread in the portfolio and reprices everything. The loss at the 99th percentile is the VaR.

But here's the catch: credit spreads are correlated. When Ford's spread widens, GM's spread probably widens too. If you simulate them independently, you underestimate the risk of everything going wrong at the same time.

Cholesky decomposition fixes this. You start with a correlation matrix (a table of how each pair of spreads moves together), and Cholesky factorizes it into a "square root" matrix. You multiply independent random numbers by this matrix, and out come correlated random numbers. Like rolling dice that are secretly linked.

import numpy as np

def simulate_correlated_spreads(corr_matrix, n_sims, current_spreads, vols):
    L = np.linalg.cholesky(corr_matrix)       # "square root" of correlation
    Z = np.random.standard_normal((n_sims, len(current_spreads)))
    correlated_Z = Z @ L.T                      # correlate the randoms
    # Each row is one simulated scenario
    simulated = current_spreads + vols * correlated_Z
    return simulated

Exercise 20 runs thousands of these simulations, reprices the portfolio each time, sorts the losses, and reports VaR at 95% and 99% confidence. This is exactly what risk desks at banks do every night.

Part 6: How to Study This

Twelve exercises is a lot. Here's how to make sense of it without drowning.

  1. Trace a single request end-to-end.

    Pick a question like "What's the 5-year spot rate for 2025-03-15?" and follow the data from FRED, through S3, into RDS, through the bootstrap Lambda, out the API Gateway, and into the Redis cache. Seeing one full loop makes the whole system click.

  2. Separate "what" from "where."

    The finance math (bootstrapping, Nelson-Siegel, hazard rates) is "what" we compute. The AWS services (Lambda, S3, SQS) are "where" we compute it. These are independent skills. You can understand bootstrapping without knowing anything about Lambda, and vice versa. Don't try to learn both at the same time.

  3. Run the tests — they're specifications with real numbers.

    Every exercise has tests with concrete inputs and expected outputs. test_bootstrap.py literally says "given these par yields, the 5-year spot rate should be 4.52%." That's more useful than any textbook explanation.

  4. Compare Exercise 15 and Exercise 18.

    Both deal with default probabilities, but Ex 15 is the simple version (constant hazard rate) and Ex 18 is the realistic version (piecewise hazard rates + Terraform infrastructure). Comparing them shows how a concept evolves from "understand the idea" to "build it for production."

  5. Think in the two tracks.

    Rates (09–14) and Credit (15–20) are parallel tracks that merge at the capstones. Study one track at a time. The rates track is the foundation — the credit track imports its curves.

  6. Draw the dependency chain.

    Everything flows from the yield curve. Par yields → spot rates → forward rates → fitted curves → bond prices → credit spreads → default probabilities → portfolio VaR. Each step consumes the previous one's output. Once you see that chain, the exercises are just stops along the way.

  7. Read the blog posts in order.

    Each FinBytes post builds on the previous one. If something in the OAS post doesn't make sense, it's probably because a concept from the bootstrapping post is missing. Go back and fill the gap.
