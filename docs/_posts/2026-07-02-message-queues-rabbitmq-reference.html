What Message Queues Solve

A message queue sits between a producer (the thing generating work) and a consumer (the thing doing the work). This decoupling buys you three things:

  • Async processing: The API returns 202 Accepted immediately. The heavy computation happens in the background.
  • Retry and durability: If the consumer crashes, the message stays in the queue. No work is lost.
  • Load leveling: A burst of 10,000 requests does not kill your backend. Messages queue up and consumers process them at their own pace.

Without a queue, a failed downstream service means a failed user request. With a queue, the message waits until the service recovers.

RabbitMQ vs Redis Pub/Sub vs SQS

Feature RabbitMQ Redis Pub/Sub AWS SQS
Delivery guarantee At-least-once (with acks) At-most-once (fire and forget) At-least-once
Persistence Yes (durable queues) No Yes (14-day retention)
Dead letter queue Built-in No Built-in
Routing Exchanges + routing keys Channel name only Queue name or SNS fan-out
Infrastructure Self-hosted or CloudAMQP Already have Redis? Free. Fully managed (AWS)
Best for Reliable task queues, complex routing Real-time broadcasts, ephemeral events AWS-native apps, serverless

Rule of thumb: If losing a message is acceptable (live price ticks, typing indicators), use Redis pub/sub. If losing a message means losing money or work, use RabbitMQ or SQS.

RabbitMQ Core Concepts

Messages in RabbitMQ flow through a three-stage pipeline:

  1. Producer publishes a message to an exchange.
  2. The exchange routes the message to one or more queues based on bindings and routing keys.
  3. A consumer pulls the message from a queue and acknowledges it when done.

Exchange types:

  • Direct: Route to queue whose binding key exactly matches the routing key.
  • Fanout: Broadcast to all bound queues (ignore routing key). Classic pub/sub.
  • Topic: Route by pattern. Routing key price.equity.AAPL matches binding price.equity.* and price.#.

Python with Pika: Work Queue

Install: pip install pika

The work queue pattern distributes tasks across multiple consumers in round-robin fashion. Each message is processed by exactly one consumer.

# producer.py — send a task to the work queue
import pika
import json

connection = pika.BlockingConnection(
    pika.ConnectionParameters("localhost")
)
channel = connection.channel()

# Declare a durable queue (survives RabbitMQ restart)
channel.queue_declare(queue="pricing_tasks", durable=True)

task = {"bond_id": "US912828ZT09", "curve_date": "2026-06-30"}

channel.basic_publish(
    exchange="",                   # default exchange (direct to queue)
    routing_key="pricing_tasks",   # queue name
    body=json.dumps(task),
    properties=pika.BasicProperties(
        delivery_mode=2,           # persistent message
        content_type="application/json",
    ),
)
print(f"Sent: {task}")
connection.close()

# consumer.py — process tasks from the queue
import pika
import json
import time

connection = pika.BlockingConnection(
    pika.ConnectionParameters("localhost")
)
channel = connection.channel()
channel.queue_declare(queue="pricing_tasks", durable=True)

# Fair dispatch: don't give a worker more than 1 message at a time
channel.basic_qos(prefetch_count=1)

def callback(ch, method, properties, body):
    task = json.loads(body)
    print(f"Processing bond {task['bond_id']}...")
    time.sleep(2)  # simulate computation
    print(f"Done: {task['bond_id']}")

    # Acknowledge: tells RabbitMQ the message is processed
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_consume(queue="pricing_tasks", on_message_callback=callback)
print("Waiting for tasks. Press Ctrl+C to exit.")
channel.start_consuming()

Run two terminals with python consumer.py to see round-robin distribution.

Pub/Sub with Fanout Exchange

Every subscriber gets every message. Useful for broadcasting price updates to multiple downstream services.

# fanout_publisher.py
import pika
import json

connection = pika.BlockingConnection(pika.ConnectionParameters("localhost"))
channel = connection.channel()

# Declare a fanout exchange
channel.exchange_declare(exchange="price_feed", exchange_type="fanout")

msg = {"ticker": "AAPL", "price": 189.25}
channel.basic_publish(
    exchange="price_feed",
    routing_key="",        # ignored for fanout
    body=json.dumps(msg),
)
print(f"Broadcast: {msg}")
connection.close()

# fanout_subscriber.py
import pika
import json

connection = pika.BlockingConnection(pika.ConnectionParameters("localhost"))
channel = connection.channel()

channel.exchange_declare(exchange="price_feed", exchange_type="fanout")

# Create an exclusive, auto-delete queue (temporary, per subscriber)
result = channel.queue_declare(queue="", exclusive=True)
queue_name = result.method.queue

# Bind the temporary queue to the fanout exchange
channel.queue_bind(exchange="price_feed", queue=queue_name)

def callback(ch, method, properties, body):
    data = json.loads(body)
    print(f"Received: {data['ticker']} @ ${data['price']}")

channel.basic_consume(queue=queue_name, on_message_callback=callback, auto_ack=True)
print("Listening for price broadcasts...")
channel.start_consuming()

Dead Letter Queues

When a consumer rejects a message (or the message expires), it goes to a dead letter exchange (DLX) instead of disappearing. This lets you inspect and retry failed messages.

import pika

connection = pika.BlockingConnection(pika.ConnectionParameters("localhost"))
channel = connection.channel()

# 1. Declare the dead letter queue
channel.queue_declare(queue="pricing_tasks_dlq", durable=True)

# 2. Declare the dead letter exchange
channel.exchange_declare(exchange="dlx", exchange_type="direct")
channel.queue_bind(queue="pricing_tasks_dlq", exchange="dlx",
                   routing_key="pricing_tasks")

# 3. Declare the main queue with DLX settings
channel.queue_declare(
    queue="pricing_tasks",
    durable=True,
    arguments={
        "x-dead-letter-exchange": "dlx",
        "x-dead-letter-routing-key": "pricing_tasks",
        "x-message-ttl": 60000,  # optional: messages expire after 60s
    },
)
connection.close()

Failed messages now land in pricing_tasks_dlq. You can inspect them in the Management UI or consume them with a separate retry worker.

Acknowledgements and Durability Checklist

To guarantee no message loss, you need all three:

  1. Durable queue: queue_declare(queue="tasks", durable=True) — queue survives broker restart.
  2. Persistent messages: delivery_mode=2 in message properties — message written to disk.
  3. Manual acknowledgement: basic_ack() after processing — message removed from queue only after consumer confirms.

If the consumer crashes before acking, RabbitMQ redelivers the message to another consumer. This means your consumer must be idempotent—processing the same message twice must produce the same result.

Common Gotchas

1. Heartbeat timeouts. Long-running tasks can cause the connection to drop. Increase the heartbeat interval or process tasks in a separate thread.

2. Queue declared differently. If a queue already exists with different settings (e.g., durable vs non-durable), RabbitMQ throws a PRECONDITION_FAILED error. Delete the queue first or use a new name.

3. Forgetting to ack. Without basic_ack(), messages pile up in "Unacked" state and are never removed. Memory usage grows until the broker crashes.

4. Connection per request. Opening a new connection for every message is expensive. Reuse connections and channels. In web apps, create one connection at startup.
