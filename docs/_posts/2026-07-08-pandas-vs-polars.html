Same Task, Two Libraries

pandas has been the default DataFrame library in Python for over a decade. polars is the challenger—written in Rust, built on Apache Arrow, and designed to be fast without requiring you to think about performance. This post runs the same six operations in both libraries so you can see the differences yourself.

The dataset: a CSV of trade records with columns ticker, side, quantity, price, and timestamp.

1. Load CSV

pandas

import pandas as pd

df = pd.read_csv("trades.csv", parse_dates=["timestamp"])

polars (eager)

import polars as pl

df = pl.read_csv("trades.csv", try_parse_dates=True)

polars (lazy)

lf = pl.scan_csv("trades.csv", try_parse_dates=True)
# Nothing executes yet. polars builds a query plan.
# Call .collect() when you want results.

Lazy mode is the key differentiator. scan_csv returns a LazyFrame. polars optimizes the entire chain of operations before touching data—predicate pushdown, projection pruning, and join reordering happen automatically.

2. Filter Rows (quantity > 100)

pandas

big_trades = df[df["quantity"] > 100]

polars (eager)

big_trades = df.filter(pl.col("quantity") > 100)

polars (lazy)

lf = lf.filter(pl.col("quantity") > 100)
# Still no execution. The filter is added to the query plan.

3. Add Computed Column (notional = quantity * price)

pandas

df["notional"] = df["quantity"] * df["price"]

polars (eager)

df = df.with_columns(
    (pl.col("quantity") * pl.col("price")).alias("notional")
)

polars (lazy)

lf = lf.with_columns(
    (pl.col("quantity") * pl.col("price")).alias("notional")
)

Notice that pandas mutates the DataFrame in place. polars returns a new frame—frames are immutable. This makes polars code easier to reason about in long pipelines.

4. Group By + Aggregate

pandas

summary = (
    df.groupby("ticker")
    .agg(
        total_qty=("quantity", "sum"),
        avg_price=("price", "mean"),
        trade_count=("ticker", "count"),
    )
    .reset_index()
)

polars (eager)

summary = df.group_by("ticker").agg(
    pl.col("quantity").sum().alias("total_qty"),
    pl.col("price").mean().alias("avg_price"),
    pl.col("ticker").count().alias("trade_count"),
)

polars (lazy)

lf_summary = lf.group_by("ticker").agg(
    pl.col("quantity").sum().alias("total_qty"),
    pl.col("price").mean().alias("avg_price"),
    pl.col("ticker").count().alias("trade_count"),
)

polars expressions are composable. Each pl.col(...).sum() is an independent expression; you can mix and match aggregations freely without the named-tuple syntax pandas requires.

5. Join with Reference Table

We have a small lookup table mapping tickers to sectors:

# pandas
sectors_pd = pd.DataFrame({
    "ticker": ["AAPL", "MSFT", "GS", "JPM", "XOM"],
    "sector": ["Tech", "Tech", "Finance", "Finance", "Energy"],
})

# polars
sectors_pl = pl.DataFrame({
    "ticker": ["AAPL", "MSFT", "GS", "JPM", "XOM"],
    "sector": ["Tech", "Tech", "Finance", "Finance", "Energy"],
})

pandas

merged = df.merge(sectors_pd, on="ticker", how="left")

polars (eager)

merged = df.join(sectors_pl, on="ticker", how="left")

polars (lazy)

lf_merged = lf.join(sectors_pl.lazy(), on="ticker", how="left")

6. Sort by Notional Descending

pandas

result = merged.sort_values("notional", ascending=False)

polars (eager)

result = merged.sort("notional", descending=True)

polars (lazy — collect the full pipeline)

result = (
    lf_merged
    .sort("notional", descending=True)
    .collect()  # NOW execution happens, with full optimization
)

When you call .collect(), polars inspects the entire plan and optimizes it. If you only selected two columns but the CSV has twenty, polars skips reading the other eighteen. pandas reads everything up front.

API Philosophy

pandas is imperative. DataFrames are mutable objects. You chain methods, but intermediate results often modify state. Index alignment is automatic and powerful but also a source of subtle bugs. The API grew organically over 15 years, so there are often multiple ways to do the same thing (df.groupby vs df.pivot_table vs df.resample).

polars is expression-based and functional. Frames are immutable. Operations are built from composable expressions (pl.col, pl.when, pl.lit). There is one way to do most things, and the type system catches errors earlier. The lack of an index is intentional—polars treats row order as data, not metadata.

Memory and Speed

polars uses Apache Arrow columnar memory under the hood. This means contiguous arrays per column, zero-copy slicing, and efficient SIMD operations. pandas uses NumPy arrays, which also columnar but with more overhead from Python object boxing (especially for strings and mixed types).

Benchmarks consistently show polars 5–50x faster than pandas on operations like groupby, join, and filter on datasets above 100 MB. The gap widens with more data because polars parallelizes across CPU cores automatically.

Error Messages

polars raises typed exceptions with clear messages. If you misspell a column name, you get a ColumnNotFoundError listing available columns. pandas often raises a generic KeyError with just the bad key.

Ecosystem

pandas wins here, and it is not close. scikit-learn expects pandas DataFrames or NumPy arrays. matplotlib and seaborn integrate directly. Every StackOverflow answer, every tutorial, every data science course uses pandas. polars is growing fast but you will hit friction when passing data to libraries that do not support it yet.

Comparison Table

Dimension pandas polars
API style Method chaining, mutable state, index-based Expression-based, immutable frames, no index
Speed Baseline 5–50x faster (auto-parallelized)
Memory NumPy-backed, object dtype overhead Arrow columnar, zero-copy, less overhead
Lazy evaluation No Yes (query plan optimization)
Ecosystem Dominant (sklearn, plotting, tutorials) Growing but gaps remain
Learning curve Gentle (vast resources) Moderate (expression DSL takes adjustment)
Error messages Generic Python exceptions Typed, descriptive, suggests fixes

When to Use What

Stay on pandas when:

  • Your data fits in memory comfortably (<1 GB)
  • You have an existing codebase with pandas everywhere
  • You need tight integration with scikit-learn, matplotlib, or seaborn
  • Your team knows pandas and the project is short-lived
  • You are prototyping in a Jupyter notebook and speed does not matter

Switch to polars when:

  • Your data exceeds 1 GB (or you expect it to grow)
  • You are starting a new project with no legacy pandas code
  • You need speed without dropping to Spark or Dask
  • You are building ETL pipelines where lazy evaluation saves I/O
  • You want stricter types and better error messages
  • You want to avoid the "which pandas API do I use?" problem

Use both:

polars has .to_pandas() and pl.from_pandas(). Do heavy lifting in polars, then convert to pandas at the boundary where you need sklearn or plotting. This is a practical middle ground for many projects.

# Heavy processing in polars
result = (
    pl.scan_csv("trades.csv")
    .filter(pl.col("quantity") > 100)
    .with_columns((pl.col("quantity") * pl.col("price")).alias("notional"))
    .group_by("ticker")
    .agg(pl.col("notional").sum().alias("total_notional"))
    .sort("total_notional", descending=True)
    .collect()
)

# Convert at the boundary for sklearn / matplotlib
result_pd = result.to_pandas()
