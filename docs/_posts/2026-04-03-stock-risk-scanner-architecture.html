System Overview

The Stock Risk Scanner is a production-style portfolio risk analysis service. It combines a Streamlit dashboard, a FastAPI backend, a PostgreSQL database, and the Claude API into a single deployable system. Here is how all the pieces fit together.

graph LR A[User Browser] --> B[Streamlit Cloud\nfinbytes.streamlit.app] B -->|POST /api/scan\nGET /api/scans/id| C[Render API\nfinbytes-scanner.onrender.com] B -->|chart data| D[yfinance\nexternal] C -->|persist scan records| E[PostgreSQL\nRender managed] C -->|generate narrative| F[Claude API\nAnthropic] F -->|fallback if unavailable| C G[GitHub Repo] -->|CI on push| H[GitHub Actions CI] G -->|auto-deploy| B G -->|auto-deploy| C

A few things worth noting about this topology:

  • yfinance is called from both Streamlit and the API — deliberately. The API fetches prices for risk calculations; Streamlit fetches prices separately for chart rendering. This keeps the API clean: no chart-data endpoints needed.
  • Claude API has a fallback path — if the API key is missing or the service is down, the scanner returns a deterministic text summary of the raw metrics. The service never fails silently.
  • Both hosting targets auto-deploy from master — merge a PR and both the API and the dashboard are updated within minutes, no manual deploys.

Request Flow

The most important design decision in this system is the async scan pattern: the HTTP POST returns immediately, and the heavy work runs in the background. Here is the full sequence from button click to rendered charts.

sequenceDiagram participant U as User participant ST as Streamlit participant API as Render API participant DB as PostgreSQL participant YF as yfinance participant CL as Claude API U->>ST: Enter tickers + weights, click Scan ST->>API: POST /api/scan {tickers, weights, period} API->>DB: INSERT ScanRecord (status: pending) API-->>ST: HTTP 202 {id: 42, status: "pending"} Note over API: Background task starts API->>YF: fetch_prices(tickers, period) YF-->>API: DataFrame of close prices API->>API: calculate_risk_metrics(prices, weights) API->>CL: generate narrative (or fallback) CL-->>API: plain-English risk summary API->>DB: UPDATE ScanRecord (status: complete, metrics, narrative) loop Poll every 2 seconds ST->>API: GET /api/scans/42 API-->>ST: {status: "pending"} or {status: "complete", ...} end ST->>YF: fetch_prices(tickers, period) for charts YF-->>ST: DataFrame of close prices ST->>U: Render metrics, narrative, Plotly charts

Why return HTTP 202 immediately? A scan takes 5–10 seconds: yfinance fetches data over the network, NumPy crunches the risk metrics, and the Claude API generates a narrative. Holding the HTTP connection open for 10 seconds would be bad UX and bad practice. Instead, the API creates a ScanRecord with status pending, fires a background task via asyncio.ensure_future, and returns the scan ID. The client polls until the status flips to complete.

Code Module Map

The backend is split into eight modules, each with a single responsibility. The split matters: it makes the code testable in isolation and makes it obvious where to look when something breaks.

graph TD M[models.py\nScanRequest · RiskMetrics · ScanResult] MD[market_data.py\nfetch_prices] R[risk.py\ncalculate_risk_metrics] N[narrative.py\nRiskNarrator] S[scanner.py\nscan orchestrator] MAIN[main.py\nFastAPI endpoints] DBM[db_models.py\nSQLAlchemy ScanRecord] DB[db.py\nCRUD operations] M --> MD M --> R M --> N M --> S MD --> S R --> S N --> S S --> MAIN DBM --> DB DB --> MAIN

Reading the diagram top-down:

  • models.py defines the data contracts — ScanRequest (input), RiskMetrics (computed numbers), ScanResult (full output). Every module imports from here, nothing else imports between modules.
  • market_data.py, risk.py, narrative.py are pure functions — take data in, return data out. No database, no FastAPI, easy to unit test in isolation.
  • scanner.py is the orchestrator — it chains the three pure modules together and returns a ScanResult. The FastAPI layer calls this in a background task.
  • db_models.py + db.py are the persistence layer — SQLAlchemy ScanRecord ORM model and the async CRUD functions that read and write it. main.py is the only consumer.

Infrastructure

Docker Multi-Stage Build

The Dockerfile uses two stages to keep the production image lean:

  • builder stage — installs all dependencies (including build tools for compiled packages). This stage is large and disposable.
  • runtime stage — copies only the installed site-packages and the application code from the builder. No build tools, no pip cache, smaller attack surface.

docker-compose.yml defines two services: scanner (FastAPI on port 8000) and postgres:16 (port 5432, with a named volume for data persistence). Running docker compose up --build -d gives you the full stack locally in one command.

Deployment Pipeline

graph LR A[git push\nworking branch] --> B[GitHub Actions CI\npytest 26 tests] B -->|all tests pass| C[Pull Request\nworking → master] C -->|merged| D[master branch] D -->|webhook| E[Render\nauto-deploy API] D -->|webhook| F[Streamlit Cloud\nauto-deploy dashboard]

The discipline here is: nothing lands on master without passing CI. The GitHub Actions workflow triggers on every push to working and on every PR to master. It checks out the code, installs dependencies, and runs pytest. If any test fails, the PR cannot merge. Once the PR merges to master, both Render and Streamlit Cloud pick up the change automatically via their GitHub integration — no manual git push to a hosting remote, no deploy scripts.

Tech Decisions — Q&A

Q: Why FastAPI over Flask?

FastAPI is async-native from the ground up. Flask can handle async routes, but it is bolted on rather than built in. FastAPI also generates OpenAPI documentation automatically from your route signatures and Pydantic models — no separate swagger setup. And Pydantic validation is integrated: request bodies are validated before your function even runs, and if validation fails you get a structured 422 response automatically. For I/O-bound financial data work where you are waiting on yfinance, PostgreSQL, and an external AI API simultaneously, async matters.

Q: Why async SQLAlchemy?

The scanner fetches market data from yfinance and calls the Claude API — both are I/O-bound network operations that can take seconds. With sync SQLAlchemy, the entire server thread blocks during a database query. With AsyncSession, the server can handle other requests while waiting for the database. In a service that might receive multiple concurrent scan requests, this is the difference between a responsive API and one that queues up behind itself.

Q: Why background tasks instead of synchronous endpoints?

A scan takes 5–10 seconds end-to-end. Holding an HTTP connection open for 10 seconds is a poor user experience and makes the API fragile under load. The POST-202-poll pattern is standard for slow operations: the server returns a scan ID immediately, the client polls for the result. This also means the server can process multiple scans concurrently — each one runs as an independent background task.

Q: Why SQLite for tests but PostgreSQL for production?

SQLite in-memory tests run in milliseconds with zero infrastructure dependencies — no Docker required to run the test suite locally or in CI. SQLAlchemy abstracts the database engine, so the same models and queries work on both. PostgreSQL for production gives strict typing, proper concurrency (SQLite has table-level write locks), and the data persistence and reliability expected of a production service.

Q: Why Streamlit over React?

Python-only — no JavaScript context switch. Streamlit is purpose-built for data apps: widgets for sliders and text inputs, built-in st.metric for KPIs, native Plotly chart rendering, and free hosting on Streamlit Cloud. The trade-off is less UI customisation: you cannot build a pixel-perfect design system in Streamlit. For a data dashboard where the numbers are the product, that trade-off is entirely acceptable.

Q: Why separate frontend and backend hosting?

Streamlit Cloud is free and runs Streamlit natively. Render is free and runs Docker containers. Combining them on a single host would require a paid tier on one or both platforms, and would mean deploying a Python web framework (FastAPI) inside a Streamlit Cloud app — which is not what it is designed for. Separate hosting also means independent scaling: if the dashboard gets traffic, the API is not affected, and vice versa. The cost is one extra environment variable (API_URL) in the Streamlit app.

Q: Why does yfinance get called from both Streamlit and the API?

The API fetches prices to calculate risk metrics — that is the authoritative data. Streamlit fetches prices separately to render historical price charts in the dashboard. If the API were to serve chart data too, you would need new endpoints, new models, and new tests for data that Streamlit can fetch directly from yfinance with three lines of code. The duplication is intentional: it keeps the API focused on risk calculation and keeps the dashboard self-sufficient for display data.

Q: How do Alembic and PostgreSQL fit together?

Think of PostgreSQL as the database that stores your data, SQLAlchemy as the Python code that talks to it, and Alembic as version control for the database schema.

Without Alembic, if you add a column to your Python model, PostgreSQL still has the old schema — your app crashes. You would have to manually write ALTER TABLE SQL. With Alembic, you run alembic revision --autogenerate and it compares your models to the actual database, generates a migration script, and applies it with alembic upgrade head.

In the scanner, we use Base.metadata.create_all() on startup instead of Alembic migrations. This works because Render’s free tier has no shell access to run Alembic commands, and create_all() is sufficient for a fresh database. The trade-off: create_all() creates missing tables but will not modify existing ones. If we ever need to add a column or change a type on a database with live data, we would need Alembic.

Rule of thumb: create_all() is fine for greenfield and demos. Alembic is essential when the schema evolves on a database that already has data.

Q: Why create_all() instead of Alembic in production?

Pragmatism. Render’s free tier does not provide shell access, so we cannot run alembic upgrade head after deployment. By calling create_all() inside the FastAPI lifespan handler, the tables are created automatically when the app starts — zero manual steps. For a demo project with a stable schema, this is the right trade-off. If the schema needed frequent changes, we would move to a paid Render tier (which provides shell access) or add an /admin/migrate endpoint protected by an API key.

Q: How do you test the Streamlit dashboard?

Streamlit provides AppTest, a headless testing framework that runs pages without a browser. Tests verify that pages load without errors, titles and tabs render, and all interactive widgets (buttons, sliders) are present. This catches regressions like missing imports, broken layouts, or removed widgets. The 8 dashboard tests run in a separate CI job parallel to the 27 backend tests — total 35 tests across both suites.
