One Model, Six Languages

The task: validate a trading request with these fields:

  • ticker — string, uppercase, 1–5 characters
  • quantity — positive integer
  • side — "buy" or "sell"
  • price — optional positive float

Every language handles this differently. Some catch errors at compile time, some at runtime, and some not at all unless you add validation manually. Here is each approach.

1. Python Pydantic

from pydantic import BaseModel, Field, field_validator
from typing import Literal, Optional

class TradeRequest(BaseModel):
    ticker: str = Field(min_length=1, max_length=5)
    quantity: int = Field(gt=0)
    side: Literal["buy", "sell"]
    price: Optional[float] = Field(default=None, gt=0)

    @field_validator("ticker")
    @classmethod
    def ticker_uppercase(cls, v: str) -> str:
        if v != v.upper():
            raise ValueError("ticker must be uppercase")
        return v

# Valid
trade = TradeRequest(ticker="AAPL", quantity=100, side="buy", price=189.50)

# Invalid — triggers ValidationError
try:
    bad = TradeRequest(ticker="aapl toolong", quantity=-5, side="hold")
except Exception as e:
    print(e)
    # 3 validation errors for TradeRequest
    # ticker: String should have at most 5 characters
    # quantity: Input should be greater than 0
    # side: Input should be 'buy' or 'sell'

Pydantic validates at runtime when you instantiate the model. It generates clear, structured error messages with field paths. It also handles JSON serialization (trade.model_dump_json()) and schema generation (TradeRequest.model_json_schema()).

2. TypeScript Zod

import { z } from "zod";

const TradeRequest = z.object({
    ticker: z.string().min(1).max(5).toUpperCase(),
    quantity: z.number().int().positive(),
    side: z.enum(["buy", "sell"]),
    price: z.number().positive().optional(),
});

type TradeRequest = z.infer<typeof TradeRequest>;

// Valid
const trade = TradeRequest.parse({
    ticker: "AAPL", quantity: 100, side: "buy", price: 189.50,
});

// Invalid — throws ZodError
try {
    TradeRequest.parse({ ticker: "", quantity: -5, side: "hold" });
} catch (e) {
    console.log(e.issues);
    // [
    //   { path: ["ticker"], message: "String must contain at least 1 character(s)" },
    //   { path: ["quantity"], message: "Number must be greater than 0" },
    //   { path: ["side"], message: "Invalid enum value..." },
    // ]
}

Zod validates at runtime but also infers TypeScript types via z.infer. You define the schema once and get both validation and type safety. The .toUpperCase() transform modifies the input during parsing—useful for normalization.

3. Rust serde + validator

use serde::Deserialize;
use validator::Validate;

#[derive(Debug, Deserialize, Validate)]
struct TradeRequest {
    #[validate(length(min = 1, max = 5))]
    ticker: String,

    #[validate(range(min = 1))]
    quantity: i64,

    side: Side,

    #[validate(range(min = 0.01))]
    price: Option<f64>,
}

#[derive(Debug, Deserialize)]
#[serde(rename_all = "lowercase")]
enum Side { Buy, Sell }

fn main() {
    let json = r#"{"ticker":"AAPL","quantity":100,"side":"buy","price":189.50}"#;

    // serde handles deserialization + type checking
    let trade: TradeRequest = serde_json::from_str(json).unwrap();

    // validator handles business rules
    trade.validate().unwrap();

    // Invalid side value fails at deserialization (serde), not validation
    let bad = r#"{"ticker":"AAPL","quantity":100,"side":"hold"}"#;
    let result: Result<TradeRequest, _> = serde_json::from_str(bad);
    // Err: unknown variant `hold`, expected `buy` or `sell`
}

Rust splits the work: serde handles deserialization and type matching (wrong type = compile error or deserialization error). The validator crate handles business rules (range, length). The enum Side restricts values at the type level—no invalid enum can exist in memory.

4. Go struct tags + validator

package main

import (
    "encoding/json"
    "fmt"
    "github.com/go-playground/validator/v10"
)

type TradeRequest struct {
    Ticker   string   `json:"ticker"   validate:"required,min=1,max=5,uppercase"`
    Quantity int      `json:"quantity" validate:"required,gt=0"`
    Side     string   `json:"side"     validate:"required,oneof=buy sell"`
    Price    *float64 `json:"price"    validate:"omitempty,gt=0"`
}

func main() {
    raw := []byte(`{"ticker":"AAPL","quantity":100,"side":"buy","price":189.50}`)
    var trade TradeRequest
    json.Unmarshal(raw, &trade)

    v := validator.New()
    err := v.Struct(trade)
    if err != nil {
        for _, e := range err.(validator.ValidationErrors) {
            fmt.Printf("Field: %s, Tag: %s, Value: %v\n", e.Field(), e.Tag(), e.Value())
        }
    }
}

Go uses struct tags for validation rules. The go-playground/validator package reads tags at runtime via reflection. The pattern is simple but error messages are less descriptive than Pydantic or Zod—you often need to map tag names to human-readable messages yourself.

5. C# DataAnnotations

using System.ComponentModel.DataAnnotations;

public class TradeRequest
{
    [Required]
    [StringLength(5, MinimumLength = 1)]
    [RegularExpression(@"^[A-Z]+$", ErrorMessage = "Ticker must be uppercase letters")]
    public string Ticker { get; set; }

    [Required]
    [Range(1, int.MaxValue, ErrorMessage = "Quantity must be positive")]
    public int Quantity { get; set; }

    [Required]
    [RegularExpression(@"^(buy|sell)$", ErrorMessage = "Side must be 'buy' or 'sell'")]
    public string Side { get; set; }

    [Range(0.01, double.MaxValue, ErrorMessage = "Price must be positive")]
    public double? Price { get; set; }
}

// Validation
var trade = new TradeRequest { Ticker = "AAPL", Quantity = 100, Side = "buy" };
var context = new ValidationContext(trade);
var results = new List<ValidationResult>();
bool isValid = Validator.TryValidateObject(trade, context, results, true);

if (!isValid)
{
    foreach (var error in results)
        Console.WriteLine($"{string.Join(",", error.MemberNames)}: {error.ErrorMessage}");
}

C# uses attributes (annotations) on properties. ASP.NET Core validates these automatically on model binding. Outside of a web framework, you call Validator.TryValidateObject manually. The approach is declarative but the regex-based enum validation is clunky compared to Zod or Pydantic.

6. C++ (Manual Validation)

#include <string>
#include <optional>
#include <stdexcept>
#include <vector>
#include <algorithm>

struct TradeRequest {
    std::string ticker;
    int quantity;
    std::string side;
    std::optional<double> price;

    std::vector<std::string> validate() const {
        std::vector<std::string> errors;

        if (ticker.empty() || ticker.size() > 5)
            errors.push_back("ticker must be 1-5 characters");

        if (!std::all_of(ticker.begin(), ticker.end(), ::isupper))
            errors.push_back("ticker must be uppercase");

        if (quantity <= 0)
            errors.push_back("quantity must be positive");

        if (side != "buy" && side != "sell")
            errors.push_back("side must be 'buy' or 'sell'");

        if (price.has_value() && price.value() <= 0.0)
            errors.push_back("price must be positive");

        return errors;
    }
};

// Usage
TradeRequest req{"AAPL", 100, "buy", 189.50};
auto errors = req.validate();
if (!errors.empty()) {
    for (const auto& e : errors)
        printf("Error: %s\n", e.c_str());
}

C++ has no standard validation framework. You write it by hand. The upside: full control, no runtime reflection overhead. The downside: every field is a manual check, and there is no schema generation, no serialization integration, and no standardized error format.

What Gets Caught Where

This is the table that matters. For each bug type, when does each tool catch it?

Bug Pydantic Zod serde+validator Go validator C# Annotations C++ manual
Wrong type (string for int) Runtime Runtime Compile Runtime (JSON) Compile Compile
Missing required field Runtime Runtime Runtime (serde) Runtime Runtime Not caught
Negative quantity Runtime Runtime Runtime (validator) Runtime Runtime Runtime (if coded)
Invalid enum ("hold") Runtime Runtime Compile (enum) / Runtime (JSON) Runtime Runtime (regex) Runtime (if coded)
Extra unknown field Ignored (configurable) Stripped (configurable) Ignored by default Ignored Ignored Not caught

When to Use What

Pydantic — Python APIs, FastAPI integration, anywhere you need runtime validation with excellent error messages and JSON Schema generation. The default choice for Python web services.

Zod — TypeScript APIs and form validation. Bridges the gap between runtime validation and compile-time types. Pairs well with tRPC, Next.js, and any TypeScript backend.

serde + validator — Rust services where you want the compiler to catch as much as possible, then validate business rules at the boundary. The two-crate split (deserialization vs validation) is slightly verbose but very explicit.

Go validator — Go APIs where struct tags keep validation close to the type definition. Simple and effective, but you will write custom error formatting.

C# DataAnnotations — ASP.NET Core APIs where model binding validates automatically. Works well in the framework; awkward outside it.

C++ manual — When you have no choice. There is no standard equivalent. Write a validate() method, return a list of errors, and accept that this is boilerplate you maintain by hand.
