Every AWS Service We Touched in QuantLab

Over the course of 20 exercises, the QuantLab project used nine AWS services to build a bond pricing engine and a credit risk platform. This page is a quick-reference for each one: what it does, the CLI commands that matter, which exercise introduced it, and what it costs.

All commands assume you have the AWS CLI configured with a named profile:

aws configure --profile quant-lab
# Access Key ID:     AKIA...
# Secret Access Key: ...
# Region:            us-east-1
# Output format:     json

# Verify identity
aws sts get-caller-identity --profile quant-lab

S3 — Object Storage

One-liner: A key-value store for files (objects), organized into buckets. Think of it as a cloud filesystem with an HTTP API.

We used S3 to store Treasury yield CSV files fetched from FRED, making them available to Lambda functions and downstream services. Introduced in Exercise 10 (S3 Par Curve Ingestion).

# Create a bucket
aws s3 mb s3://finbytes-quant-lab-data --profile quant-lab

# Upload a file
aws s3 cp par_yields_2026-03-15.csv \
  s3://finbytes-quant-lab-data/yields/par/2026-03-15.csv \
  --profile quant-lab

# List objects under a prefix
aws s3 ls s3://finbytes-quant-lab-data/yields/par/ --profile quant-lab

# Enable versioning (so overwrites keep history)
aws s3api put-bucket-versioning \
  --bucket finbytes-quant-lab-data \
  --versioning-configuration Status=Enabled \
  --profile quant-lab

# Download a file
aws s3 cp s3://finbytes-quant-lab-data/yields/par/2026-03-15.csv ./local.csv \
  --profile quant-lab

Free tier: 5 GB storage, 20,000 GET requests, 2,000 PUT requests per month for 12 months.

Lambda — Serverless Functions

One-liner: Run code without managing servers. You upload a function, AWS runs it when triggered (HTTP request, S3 event, schedule).

We deployed three Lambda functions: spot curve bootstrapping (Exercise 12), forward rate computation (Exercise 13), and Nelson-Siegel curve fitting (Exercise 14).

# Lambda handler pattern — always receives event + context
import json

def lambda_handler(event, context):
    body = json.loads(event.get("body", "{}"))
    tenors = body["tenors"]
    rates  = body["rates"]

    # ... compute forward rates ...

        "statusCode": 200,
        "headers": {"Content-Type": "application/json"},

    }

# Package a simple Lambda (no dependencies beyond stdlib)
cd exercises/13-cicd-github-actions/src
zip -j deployment.zip forward_rates.py handler.py

# Deploy
aws lambda update-function-code \
  --function-name quant-lab-forward-rates \
  --zip-file fileb://deployment.zip \
  --publish \
  --profile quant-lab

# Test invoke
aws lambda invoke \
  --function-name quant-lab-forward-rates \
  --payload '{"body": "{\"tenors\":[1,2,5],\"rates\":[4.5,4.3,4.1]}"}' \
  output.json --profile quant-lab

Free tier: 1 million requests and 400,000 GB-seconds per month (permanent, not just 12 months).

API Gateway — HTTP Front Door for Lambda

One-liner: Puts an HTTPS URL in front of Lambda functions. Two flavors: REST API (v1, more features, higher cost) and HTTP API (v2, simpler, cheaper).

We used a REST API (v1) for the bootstrap and forwards endpoints in Exercise 12, then switched to an HTTP API (v2) for curve fitting in Exercise 14 because it was simpler and cheaper.

# List your APIs
aws apigatewayv2 get-apis --profile quant-lab

# The invoke URL pattern for HTTP API:
# https://{api-id}.execute-api.us-east-1.amazonaws.com/dev/curves/fit

Free tier: HTTP APIs: 1 million API calls/month for 12 months. REST APIs: 1 million calls/month for 12 months.

RDS — Managed PostgreSQL

One-liner: A managed relational database. You pick the engine (PostgreSQL, MySQL, etc.), AWS handles backups, patching, and failover.

We used RDS PostgreSQL in Exercise 11 to store yield curve data with proper schemas, timestamps, and SQL queries. The scanner project uses Render-hosted PostgreSQL in production, but the concepts are identical.

# Create a free-tier instance
aws rds create-db-instance \
  --db-instance-identifier quant-lab-yields \
  --db-instance-class db.t3.micro \
  --engine postgres \
  --engine-version 16.1 \
  --master-username admin \
  --master-user-password "${DB_PASSWORD}" \
  --allocated-storage 20 \
  --profile quant-lab

# IMPORTANT: stop it when not using it (free tier = 750 hrs/month)
aws rds stop-db-instance \
  --db-instance-identifier quant-lab-yields \
  --profile quant-lab

# It auto-starts after 7 days. Set a calendar reminder.

Free tier: 750 hours/month of db.t3.micro, 20 GB storage, for 12 months.

DynamoDB — NoSQL Key-Value/Document Store

One-liner: A serverless NoSQL database. You define a partition key (and optional sort key), and it scales automatically. No server management, no connection pooling.

We used DynamoDB in Exercise 15 for credit spread caching where the access pattern was simple key-value lookups by issuer and date.

# Create a table with on-demand pricing
aws dynamodb create-table \
  --table-name quant-lab-credit-spreads \
  --attribute-definitions \
      AttributeName=issuer,AttributeType=S \
      AttributeName=date,AttributeType=S \
  --key-schema \
      AttributeName=issuer,KeyType=HASH \
      AttributeName=date,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST \
  --profile quant-lab

# Put an item
aws dynamodb put-item \
  --table-name quant-lab-credit-spreads \
  --item '{"issuer":{"S":"AAPL"},"date":{"S":"2026-03-15"},"spread_bps":{"N":"45"}}' \
  --profile quant-lab

Free tier: 25 GB storage, 25 WCU/RCU provisioned (or equivalent on-demand), permanent.

SQS — Message Queues

One-liner: A managed message queue. Producers send messages, consumers pull them. Decouples services so one crashing doesn't take down the others.

We used SQS in Exercise 15 to decouple credit spread calculation from downstream consumers. Standard queues (at-least-once, no ordering guarantee) vs. FIFO queues (exactly-once, ordered).

# Create a standard queue
aws sqs create-queue \
  --queue-name quant-lab-spreads \
  --profile quant-lab

# Send a message
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789/quant-lab-spreads \
  --message-body '{"issuer":"AAPL","spread_bps":45}' \
  --profile quant-lab

# Receive + delete (the two-step consumer pattern)
aws sqs receive-message --queue-url $QUEUE_URL --profile quant-lab
aws sqs delete-message --queue-url $QUEUE_URL --receipt-handle $HANDLE --profile quant-lab

# Set up a Dead Letter Queue for failed messages
aws sqs set-queue-attributes \
  --queue-url $QUEUE_URL \
  --attributes '{"RedrivePolicy":"{\"deadLetterTargetArn\":\"arn:aws:sqs:...:quant-lab-spreads-dlq\",\"maxReceiveCount\":\"3\"}"}' \
  --profile quant-lab

Free tier: 1 million requests/month, permanent.

SNS — Pub/Sub Notifications

One-liner: Publish a message to a topic, and all subscribers (email, SQS, Lambda, HTTP) get a copy. Fan-out pattern: one event triggers multiple consumers.

We used SNS in Exercise 15 to fan out credit spread updates to both an SQS queue and an email notification. Also used for the billing alarm in Exercise 09.

# Create a topic
aws sns create-topic --name quant-lab-spread-alerts --profile quant-lab

# Subscribe an email address (requires confirmation click)
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789:quant-lab-spread-alerts \
  --protocol email \
  --notification-endpoint you@example.com \
  --profile quant-lab

# Publish a message
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789:quant-lab-spread-alerts \
  --message "AAPL spread widened to 85bps" \
  --profile quant-lab

# Fan-out: subscribe an SQS queue to the same topic
aws sns subscribe \
  --topic-arn $TOPIC_ARN \
  --protocol sqs \
  --notification-endpoint $QUEUE_ARN \
  --profile quant-lab

Free tier: 1 million publishes, 100,000 HTTP deliveries, 1,000 email deliveries per month.

ElastiCache (Redis) — In-Memory Caching

One-liner: A managed Redis (or Memcached) cluster. Sub-millisecond reads. Lives inside your VPC so Lambda/EC2 can reach it but the public internet cannot.

We used ElastiCache Redis in Exercise 17 for cache-aside bond pricing. The pattern: check Redis first, if miss, compute and store the result with a TTL.

import redis

r = redis.Redis(host="quant-lab-redis.xxxxx.cache.amazonaws.com", port=6379)

def get_bond_price(isin: str) -> float:
    cached = r.get(f"bond:{isin}")
    if cached:
        return float(cached)

    price = compute_bond_price(isin)  # expensive
    r.setex(f"bond:{isin}", 3600, str(price))  # TTL = 1 hour
    return price

# Create a single-node cluster (smallest free-tier eligible)
aws elasticache create-cache-cluster \
  --cache-cluster-id quant-lab-redis \
  --engine redis \
  --cache-node-type cache.t3.micro \
  --num-cache-nodes 1 \
  --profile quant-lab

Free tier: 750 hours/month of cache.t3.micro for 12 months. After that, roughly $12/month.

CloudWatch — Monitoring and Alarms

One-liner: AWS's built-in monitoring. Collects metrics (CPU, invocations, errors), stores logs, and triggers alarms when thresholds are breached.

We used CloudWatch in Exercise 09 for a billing alarm, and in Exercise 19 for custom metrics on OAS/Z-spread computation times and Lambda error rates.

# Create a billing alarm (must be in us-east-1)
aws cloudwatch put-metric-alarm \
  --alarm-name "MonthlyBillingAlarm" \
  --metric-name EstimatedCharges \
  --namespace AWS/Billing \
  --statistic Maximum \
  --period 21600 \
  --threshold 5 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 1 \
  --alarm-actions arn:aws:sns:us-east-1:123456789:billing-alerts \
  --dimensions Name=Currency,Value=USD \
  --region us-east-1 \
  --profile quant-lab

# Publish a custom metric
aws cloudwatch put-metric-data \
  --namespace "QuantLab" \
  --metric-name "OASComputeTimeMs" \
  --value 142 \
  --unit Milliseconds \
  --profile quant-lab

# Query logs with Log Insights
aws logs start-query \
  --log-group-name "/aws/lambda/quant-lab-forward-rates" \
  --start-time $(date -d '1 hour ago' +%s) \
  --end-time $(date +%s) \
  --query-string 'fields @timestamp, @message | filter @message like /ERROR/ | limit 20' \
  --profile quant-lab

Free tier: 10 custom metrics, 10 alarms, 5 GB log ingestion, 5 GB log storage per month (permanent).

Cost Summary

For a learning project staying within free tier limits:

  • Permanently free: Lambda (1M requests), DynamoDB (25 GB), SQS (1M requests), SNS (1M publishes), CloudWatch (10 metrics + 10 alarms)
  • Free for 12 months: S3 (5 GB), RDS (750 hrs db.t3.micro), ElastiCache (750 hrs cache.t3.micro), API Gateway (1M calls)
  • Easy to accidentally spend money on: RDS left running after free tier expires (~$15/month), NAT Gateways ($32/month), ElastiCache after 12 months (~$12/month)

Set the billing alarm from Exercise 09 on day one. It costs nothing and has saved many learners from surprise bills.

Things That Tripped Me Up

  • Lambda 50 MB limit: The curve-fitting function with scipy + numpy exceeded the 50 MB zip limit. Solution: Docker container image deployment (Exercise 14). The container can be up to 10 GB.
  • RDS auto-restart: When you stop an RDS instance, AWS automatically restarts it after 7 days. You cannot prevent this. Set a weekly reminder to stop it again, or just delete it when you are done with the exercise.
  • Billing metrics only in us-east-1: CloudWatch billing data is only available in us-east-1, regardless of where your resources are. Your billing alarm must be created in us-east-1.
  • ElastiCache requires VPC: Redis on ElastiCache is not publicly accessible. Lambda functions that need Redis must be in the same VPC with proper security groups. This adds cold start latency (~5-10 seconds for VPC-attached Lambdas).
  • SQS visibility timeout: If your consumer takes longer than the visibility timeout to process a message, SQS makes the message visible again and another consumer picks it up. Set the timeout to at least 6x your expected processing time.
  • SNS email subscriptions require confirmation: You send the subscribe command, AWS sends a confirmation email, you must click the link. Until confirmed, the subscription shows as "PendingConfirmation" and no messages are delivered.
