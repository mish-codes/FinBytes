Why TDD for Financial Code?

In most software, a bug means a bad user experience. In financial code, a bug means real money lost. A wrong return calculation, a missed edge case in drawdown logic, or a division-by-zero in a price series can cascade into incorrect risk assessments, bad trades, or misleading reports.

Test-Driven Development (TDD) flips the usual workflow: you write the test before the implementation. This forces you to think about what correct behavior looks like before you write a single line of logic. In finance, that's not just good engineering — it's a safety net.

The Red-Green-Refactor Cycle

TDD follows a tight loop:

  1. Red — Write a failing test that defines expected behavior
  2. Green — Write the minimum code to make it pass
  3. Refactor — Clean up without changing behavior (tests still pass)

Let's walk through this with a portfolio returns calculator. We'll build three functions: daily_returns, cumulative_return, and max_drawdown.

Step 1: Write Failing Tests First

Before writing any implementation, we define what each function should do:

import numpy as np
import pytest
from portfolio import daily_returns, cumulative_return, max_drawdown


class TestDailyReturns:
    def test_basic_returns(self):
        prices = np.array([100.0, 105.0, 103.0])
        result = daily_returns(prices)
        expected = np.array([0.05, -0.01904762])
        np.testing.assert_allclose(result, expected, rtol=1e-5)

    def test_single_price_returns_empty(self):
        prices = np.array([100.0])
        result = daily_returns(prices)
        assert len(result) == 0

    def test_zero_price_raises(self):
        prices = np.array([100.0, 0.0, 50.0])
        with pytest.raises(ValueError, match="zero"):
            daily_returns(prices)

Running pytest -v at this point gives us Red: ModuleNotFoundError: No module named 'portfolio'. The module doesn't even exist yet. That's the point — we've defined the contract before writing the code.

Step 2: Implement Just Enough to Pass

import numpy as np


def daily_returns(prices: np.ndarray) -> np.ndarray:
    if np.any(prices == 0):
        raise ValueError(
            "Price series contains zero - cannot compute returns from zero price"
        )
    if len(prices) <= 1:
        return np.array([])
    return np.diff(prices) / prices[:-1]

The implementation is minimal. No unnecessary abstractions, no extra features. Just enough to make the tests go Green.

Key pytest Features for Financial Testing

1. pytest.approx — Floating-Point Comparison

Financial calculations almost always involve floating-point numbers. Direct equality checks (==) are dangerous because of floating-point representation errors:

>>> 0.1 + 0.2 == 0.3
False

pytest.approx handles this gracefully:

assert cumulative_return(prices) == pytest.approx(0.10, rel=1e-5)

The rel parameter sets relative tolerance. For financial code, 1e-5 (0.001%) is usually tight enough to catch real bugs while allowing for floating-point noise.

2. np.testing.assert_allclose — Array Comparison

When comparing numpy arrays, pytest.approx doesn't work well because numpy overrides the == operator. Use numpy's own assertion instead:

np.testing.assert_allclose(result, expected, rtol=1e-5)

Rule of thumb: pytest.approx for scalars, np.testing.assert_allclose for arrays.

3. @pytest.mark.parametrize — Table-Driven Tests

Financial code needs to handle many market scenarios: up, down, flat, volatile. Parametrize lets you test all of them without duplicating test functions:

class TestCumulativeReturn:
    @pytest.mark.parametrize("prices,expected", [
        (np.array([100.0, 110.0]), 0.10),    # market goes up
        (np.array([100.0, 90.0]), -0.10),    # market goes down
        (np.array([100.0, 100.0]), 0.0),     # flat market
    ])
    def test_cumulative_return(self, prices, expected):
        assert cumulative_return(prices) == pytest.approx(expected, rel=1e-5)

Each parametrize entry runs as a separate test case. If the "market goes down" case fails, pytest tells you exactly which scenario broke — you don't have to dig through a single monolithic test.

4. pytest.raises — Testing Error Handling

In finance, failing loudly on bad data is better than silently computing garbage. We test that our functions reject invalid input:

def test_zero_price_raises(self):
    prices = np.array([100.0, 0.0, 50.0])
    with pytest.raises(ValueError, match="zero"):
        daily_returns(prices)

The match parameter checks that the error message contains "zero" — so you're testing not just that an error occurs, but that it's the right error.

Test Classes vs Fixtures — When to Use Which

Test classes group related tests logically. In our example, TestDailyReturns, TestCumulativeReturn, and TestMaxDrawdown each test one function. This keeps the test file organized and makes pytest output easy to scan.

Fixtures (@pytest.fixture) are for shared setup. If multiple test classes needed the same price series, you'd use a fixture:

@pytest.fixture
def sample_prices():
    return np.array([100.0, 105.0, 103.0, 108.0, 95.0])

class TestDailyReturns:
    def test_basic(self, sample_prices):
        result = daily_returns(sample_prices)
        assert len(result) == 4

In our exercise, each test defines its own input data inline. That's the right call — don't add fixtures just to look "professional". Use them when you're genuinely repeating setup across 3+ tests.

Common Patterns for Financial Test Cases

When testing financial calculations, always cover these scenarios:

  • Normal case — typical market movement (prices go up and down)
  • Edge cases — single data point, empty input, zero values
  • Boundary conditions — flat market (no change), all-time-high (no drawdown)
  • Error conditions — invalid data should raise, not silently return wrong numbers

Our test suite covers all four. The max_drawdown tests are a good example: we test "no drawdown" (monotonically rising prices), a drawdown in the middle, and a drawdown that ends the series.
