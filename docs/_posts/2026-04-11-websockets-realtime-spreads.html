Why Real-Time Matters on a Credit Desk

For Treasury curves, "real-time" is often a stretch: FRED updates daily, Treasury auctions are on a calendar, and most of the useful analysis is comfortable with end-of-day data. Credit is different. Spreads can gap 50-100 bps in a single session during a stress event — the afternoon after the SVB headlines in March 2023, BBB spreads widened 30 bps between lunch and the close. On a trading desk, that's the difference between "interesting" and "call the risk manager."

The messaging from Exercise 15 (SQS + SNS) is pull-based or notification-based: a consumer processes a queue, or a subscriber gets an email. Neither is what you want for a dashboard that needs to reflect live spread movement. For that you want server-push — as soon as new data hits the backend, connected clients see it.

That's the WebSocket story. Exercise 16 builds the WS endpoint on AWS-native infrastructure: API Gateway WebSocket APIs, Lambda handlers, DynamoDB for connection state.

The AWS WebSocket Model

API Gateway's WebSocket APIs handle the WS protocol itself — handshake, framing, ping/pong, close. What you provide are Lambda functions on four route types:

  • $connect: fires when a client opens the WS. You get a unique connectionId.
  • $disconnect: fires on close (client-initiated, server-initiated, or network drop).
  • $default: fires on every inbound message that doesn't match a more specific route.
  • Custom routes (e.g. subscribe, unsubscribe): routed by an action field in the message body.

To push something out to a client — the whole point of the exercise — Lambda calls the API Gateway Management API's post_to_connection with the target connectionId. That's what turns this from a bidirectional socket into a fan-out broadcast tool.

Connection State: Why DynamoDB

Lambda executions are ephemeral. The container running your $connect handler may be gone by the time the next message arrives, and the message may be served by a different container entirely. There is nowhere inside the Lambda runtime to durably hold "who is currently connected."

So connections go in DynamoDB:

def handle_connect(connection_id: str, table=None) -> dict:
    tbl = table or _get_table()
    tbl.put_item(Item={
        "connectionId": connection_id,
        "connectedAt": datetime.now(UTC).isoformat(),
        "subscriptions": [],
    })
    return {"statusCode": 200}


def handle_disconnect(connection_id: str, table=None) -> dict:
    tbl = table or _get_table()
    tbl.delete_item(Key={"connectionId": connection_id})
    return {"statusCode": 200}

Pay-per-request mode (BillingMode=PAY_PER_REQUEST) is the right choice for this kind of workload: traffic is spiky (lots of activity during market hours, dead overnight), and you don't want to pay for provisioned capacity you're not using. Reads/writes cost fractions of a cent and the free tier covers 25 GB of storage plus 25 read/write capacity units.

Per-Client Subscriptions

A risk dashboard usually doesn't want every spread update — it wants updates for the ratings it cares about. Adding a subscription model is a single DynamoDB update:

def handle_message(connection_id: str, message: dict, table=None) -> dict:
    tbl = table or _get_table()
    action = message.get("action")

    if action == "subscribe":
        rating = message.get("rating", "")
        tbl.update_item(
            Key={"connectionId": connection_id},
            UpdateExpression=(
                "SET subscriptions = list_append("
                "if_not_exists(subscriptions, :empty), :rating)"
            ),
            ExpressionAttributeValues={":rating": [rating], ":empty": []},
        )
        return {"subscribed": rating}

    return {"error": f"Unknown action: {action}"}

The DynamoDB update expression there is worth dissecting: if_not_exists(subscriptions, :empty) returns the existing list if it exists, or an empty list if the attribute hasn't been set yet. list_append(..., :rating) then tacks on the new rating. That's how you idempotently grow a list in DynamoDB without a read-modify-write round trip.

Broadcasting Updates

When new spread data lands (say, from the SNS topic in Exercise 15), a Lambda fans it out to all subscribed clients:

def broadcast_spread_update(spread_data, api_url, table=None) -> int:
    clients = get_connected_clients(table)
    apigw = boto3.client("apigatewaymanagementapi", endpoint_url=api_url)
    sent = 0

    for conn_id in clients:
        try:
            apigw.post_to_connection(
                ConnectionId=conn_id,
                Data=json.dumps(spread_data).encode(),
            )
            sent += 1
        except Exception:
            handle_disconnect(conn_id, table)

    return sent

Two details matter here:

  • Stale-connection pruning. Clients vanish all the time — laptop lids close, wifi drops, tabs get killed. API Gateway's GoneException tells you a connection is dead. The handler above treats any exception as "connection gone" and removes it from DynamoDB, so the table doesn't grow unbounded.
  • The endpoint_url for the management API isn't the WS URL (wss://...) — it's the HTTPS management endpoint, which looks like https://abc123.execute-api.us-east-1.amazonaws.com/prod. The stage matters: if you're posting to prod you need prod in the URL.

Testing WebSocket Infrastructure Without WebSockets

The functions above all take DynamoDB tables as optional parameters. That's what makes the test suite small and fast:

@pytest.fixture
def dynamodb_table():
    with mock_aws():
        dynamodb = boto3.resource("dynamodb", region_name="us-east-1")
        table = dynamodb.create_table(
            TableName="ws-connections",
            KeySchema=[{"AttributeName": "connectionId", "KeyType": "HASH"}],
            AttributeDefinitions=[{"AttributeName": "connectionId", "AttributeType": "S"}],
            BillingMode="PAY_PER_REQUEST",
        )
        yield table


def test_connect_stores_connection(dynamodb_table):
    handle_connect("conn-123", table=dynamodb_table)
    clients = get_connected_clients(table=dynamodb_table)
    assert "conn-123" in clients

moto mocks DynamoDB in memory, so the tests run in ~1s, don't touch AWS, and don't cost anything. What they don't test is the WS protocol itself — the Lambda entry points (lambda_connect, lambda_disconnect, lambda_default) are thin wrappers that extract connectionId from the API Gateway event and delegate. Those are easy to eyeball; the non-trivial logic (connection tracking, subscriptions, broadcast) is all in the tested core.

Lambda Entry Points

API Gateway passes WS route events in a well-known shape:

def lambda_connect(event, context):
    connection_id = event["requestContext"]["connectionId"]
    return handle_connect(connection_id)


def lambda_default(event, context):
    connection_id = event["requestContext"]["connectionId"]
    body = json.loads(event.get("body", "{}"))
    result = handle_message(connection_id, body)
    return {"statusCode": 200, "body": json.dumps(result)}

Each route is a separate Lambda invocation (or, more practically, the same Lambda with a route dispatcher). A $connect handler that returns {"statusCode": 200} lets the handshake succeed; return a 4xx and the client's onopen will never fire.

Cost Reality Check

API Gateway WebSocket APIs are billed on connection-minutes plus messages: $0.25 per million connection-minutes, $1.00 per million messages. A dashboard with 100 connected users all day, receiving one update per second during a 6.5-hour market session, is ~2.3M messages per day — about $70/month if you keep it open every trading day. Not free, but not unreasonable for something a team relies on.

DynamoDB on pay-per-request for this workload is effectively free: low row counts (current connections only), tiny items, bursty access.
