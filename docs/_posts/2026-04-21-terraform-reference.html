Why Terraform

Clicking through the AWS console is fine for learning, but it does not scale. You forget what you changed, you cannot review changes before applying them, and you cannot tear everything down cleanly. Terraform solves this: you declare what infrastructure you want in .tf files, and Terraform figures out how to create, update, or destroy it.

In QuantLab, we introduced Terraform in Exercise 14 (Nelson-Siegel Curve Fitting) and extended it in Exercise 18 (Advanced Terraform, Default Probabilities).

The Core Workflow

Every Terraform session follows the same four commands:

# 1. Initialize — downloads provider plugins, sets up backend
terraform init

# 2. Plan — shows what will change, creates nothing
terraform plan

# 3. Apply — creates/updates resources (prompts for confirmation)
terraform apply

# 4. Destroy — tears everything down
terraform destroy

plan is the safety net. Always read the plan output before typing "yes" on apply. It shows exactly which resources will be created (+), changed (~), or destroyed (-).

HCL Syntax Basics

Terraform uses HCL (HashiCorp Configuration Language). It looks like JSON but with less punctuation:

# Provider block — which cloud and region
provider "aws" {
  region  = var.aws_region
  profile = var.aws_profile
}

# Resource block — one piece of infrastructure
resource "aws_lambda_function" "curve_fitting" {
  function_name = "${var.project_name}-curve-fitting"
  runtime       = "python3.11"
  handler       = "handler.lambda_handler"
  role          = var.lambda_role_arn
  timeout       = 60
  memory_size   = 512

  filename         = "${path.module}/../src/deployment.zip"
  source_code_hash = filebase64sha256("${path.module}/../src/deployment.zip")
}

The pattern is always resource "TYPE" "LOCAL_NAME" { ... }. The type determines which API Terraform calls. The local name is how you reference it elsewhere in your config.

Resources vs Data Sources

Resources are things Terraform creates and manages. Data sources are things that already exist and you just want to read.

# Resource — Terraform creates this
resource "aws_apigatewayv2_api" "curves" {
  name          = "${var.project_name}-curves-http"
  protocol_type = "HTTP"
}

# Data source — Terraform reads this (it already exists)
data "aws_iam_role" "lambda_role" {
  name = "quant-lab-lambda-role"
}

# Reference the data source
resource "aws_lambda_function" "example" {
  role = data.aws_iam_role.lambda_role.arn
  # ...
}

Variables and Outputs

Variables go in variables.tf. Outputs go in outputs.tf. This separation is convention, not requirement.

# variables.tf
variable "aws_region" {
  description = "AWS region for all resources"
  type        = string
  default     = "us-east-1"
}

variable "project_name" {
  description = "Prefix for resource names"
  type        = string
  default     = "quant-lab"
}

variable "lambda_role_arn" {
  description = "ARN of the IAM role for Lambda"
  type        = string
  # No default — must be provided at plan/apply time
}

# outputs.tf
output "api_url" {
  description = "Base URL of the HTTP API"
  value       = "${aws_apigatewayv2_api.curves.api_endpoint}/dev"
}

output "lambda_function_name" {
  value = aws_lambda_function.curve_fitting.function_name
}

Pass variables at apply time:

terraform apply -var="lambda_role_arn=arn:aws:iam::123456789:role/quant-lab-lambda-role"

# Or use a .tfvars file (do NOT commit secrets)
terraform apply -var-file="dev.tfvars"

State Management

Terraform tracks what it has created in a state file (terraform.tfstate). This is how it knows the difference between "create new" and "update existing".

Local State

By default, state is a JSON file on your local disk. Fine for learning, dangerous for teams (two people applying at once can corrupt state).

Remote State (S3)

# In the terraform block
terraform {
  backend "s3" {
    bucket = "finbytes-quant-lab-tfstate"
    key    = "exercises/14-curve-fitting/terraform.tfstate"
    region = "us-east-1"
  }
}

Remote state means your team can collaborate safely, and you cannot accidentally delete the state file. The S3 bucket must already exist before you run terraform init.

The source_code_hash Trick

Lambda functions are deployed from a zip file. But Terraform only updates the function if something in the config changes. If you rebuild the zip with new code but the Terraform config is identical, Terraform skips the update.

The fix: source_code_hash. It computes a hash of the zip file. When the code changes, the hash changes, and Terraform knows to re-deploy.

resource "aws_lambda_function" "curve_fitting" {
  filename         = "${path.module}/../src/deployment.zip"
  source_code_hash = filebase64sha256("${path.module}/../src/deployment.zip")
  # ...
}

Without this line, you would need to taint the resource manually (terraform taint aws_lambda_function.curve_fitting) to force a re-deploy.

Scoping Blast Radius

In QuantLab, we had resources created manually in earlier exercises (the S3 bucket, the IAM role, the REST API). We deliberately kept those outside Terraform's state. The rule from Exercise 14:

This config only manages resources that did not exist before. The existing S3 bucket, the shared IAM role, and the existing REST API were created manually in earlier exercises and stay outside Terraform's state.

This is a real-world pattern. You do not need to import everything into Terraform on day one. Start with new resources, and import old ones later if you choose.

HTTP API vs REST API in Terraform

We created an HTTP API (v2) for the curve-fitting endpoint because it is simpler and cheaper than the REST API (v1) used in earlier exercises:

resource "aws_apigatewayv2_api" "curves" {
  name          = "${var.project_name}-curves-http"
  protocol_type = "HTTP"
  description   = "HTTP API for curve fitting endpoints"
}

resource "aws_apigatewayv2_stage" "dev" {
  api_id      = aws_apigatewayv2_api.curves.id
  name        = "dev"
  auto_deploy = true
}

resource "aws_apigatewayv2_integration" "curve_fitting" {
  api_id                 = aws_apigatewayv2_api.curves.id
  integration_type       = "AWS_PROXY"
  integration_uri        = aws_lambda_function.curve_fitting.invoke_arn
  payload_format_version = "2.0"
}

resource "aws_apigatewayv2_route" "fit" {
  api_id    = aws_apigatewayv2_api.curves.id
  route_key = "POST /curves/fit"
  target    = "integrations/${aws_apigatewayv2_integration.curve_fitting.id}"
}

# Permission for API Gateway to invoke Lambda
resource "aws_lambda_permission" "apigw" {
  statement_id  = "AllowAPIGWInvoke"
  action        = "lambda:InvokeFunction"
  function_name = aws_lambda_function.curve_fitting.function_name
  principal     = "apigateway.amazonaws.com"
  source_arn    = "${aws_apigatewayv2_api.curves.execution_arn}/*/*"
}

The Docker Packaging Story

The curve-fitting Lambda used scipy and numpy for Nelson-Siegel optimization. The standard zip deployment hit the 50 MB limit. The solution was Docker container deployment:

FROM public.ecr.aws/lambda/python:3.11
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY src/ ${LAMBDA_TASK_ROOT}/
CMD ["handler.lambda_handler"]

# Build and push to ECR
docker build -t quant-lab-curve-fitting .
aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin 123456789.dkr.ecr.us-east-1.amazonaws.com
docker tag quant-lab-curve-fitting:latest 123456789.dkr.ecr.us-east-1.amazonaws.com/quant-lab-curve-fitting:latest
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/quant-lab-curve-fitting:latest

In Terraform, you reference the ECR image instead of a zip file:

resource "aws_lambda_function" "curve_fitting" {
  function_name = "${var.project_name}-curve-fitting"
  package_type  = "Image"
  image_uri     = "${aws_ecr_repository.curve_fitting.repository_url}:latest"
  role          = var.lambda_role_arn
  timeout       = 60
  memory_size   = 512
}

Useful Commands Beyond the Basics

# Show current state
terraform show

# List all resources in state
terraform state list

# Remove a resource from state (does NOT delete the resource)
terraform state rm aws_lambda_function.curve_fitting

# Import an existing resource into state
terraform import aws_s3_bucket.data finbytes-quant-lab-data

# Format all .tf files
terraform fmt -recursive

# Validate syntax without accessing cloud APIs
terraform validate

Things That Tripped Me Up

  • State file contains secrets: terraform.tfstate stores connection strings, passwords, and ARNs in plain text. Never commit it to git. Add *.tfstate and *.tfstate.backup to .gitignore.
  • Backend changes require reinit: If you switch from local to S3 backend, you must run terraform init -migrate-state. Running just terraform init will error or silently start with empty state.
  • Destroy is real: terraform destroy deletes real resources with real data. There is no undo. Always run terraform plan -destroy first to see what will be removed.
  • Provider version pinning: Without version = "~> 5.0" in the provider block, Terraform downloads the latest version. A major version bump can break your config. Always pin.
  • Lambda permission ordering: If you create a Lambda + API Gateway route but forget the aws_lambda_permission resource, the API returns 500 errors. The Lambda exists and works when invoked directly, but API Gateway is not allowed to call it.
  • zip changes are invisible: If you forget source_code_hash, Terraform will not re-deploy your Lambda even after you rebuild the zip. You will be debugging "why didn't my code change?" for an hour.
