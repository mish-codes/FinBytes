Docker: Images, Containers, and Layers

An image is a read-only blueprint (filesystem + metadata). A container is a running instance of that image with its own writable layer on top. You can run many containers from one image, and each is isolated.

A Dockerfile defines how to build an image. Each instruction (FROM, RUN, COPY) creates a layer. Docker caches layers, so order matters: put things that change rarely (system packages) at the top and things that change often (your code) at the bottom.

Dockerfile Best Practices

This is the pattern used in QuantLab Exercise 07:

# Stage 1: Build dependencies
FROM python:3.11-slim AS builder

WORKDIR /app

# Copy requirements first — this layer is cached until requirements change
COPY requirements.txt .
RUN pip install --no-cache-dir --target=/deps -r requirements.txt

# Stage 2: Runtime image (smaller, no build tools)
FROM python:3.11-slim

WORKDIR /app

# Copy installed packages from builder stage
COPY --from=builder /deps /usr/local/lib/python3.11/site-packages

# Copy application code (changes most often — last layer)
COPY src/ ./src/
COPY config/ ./config/

# Non-root user (security)
RUN useradd --create-home appuser
USER appuser

EXPOSE 8000
CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]

Key practices:

  • Multi-stage build: The builder stage installs packages. The runtime stage copies only the result. Final image has no pip, no gcc, no build artifacts.
  • COPY requirements first: Docker caches this layer. If requirements.txt has not changed, pip install is skipped entirely on rebuild.
  • Non-root user: Containers should not run as root. If someone exploits your app, they get limited permissions.
  • .dockerignore: Exclude files that should not enter the build context.

Docker Compose: Multi-Service Setup

For local development, you rarely run one container in isolation. A typical setup: your app + a database + Redis.

# docker-compose.yml
version: "3.9"

services:
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql://quant:quant@db:5432/quantlab
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: quant
      POSTGRES_PASSWORD: quant
      POSTGRES_DB: quantlab
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U quant"]
      interval: 5s
      retries: 5

  redis:
    image: redis:alpine
    ports:
      - "6379:6379"

volumes:
  pgdata:

# Start everything
docker compose up -d

# Rebuild after code changes
docker compose up -d --build

# View logs
docker compose logs -f app

# Stop and remove containers (keep volumes)
docker compose down

# Stop and remove everything including volumes
docker compose down -v

Essential Docker Commands

# Build an image
docker build -t quantlab-app:latest .

# Run a container
docker run -d --name app -p 8000:8000 quantlab-app:latest

# Run with environment variables
docker run -d --name app -p 8000:8000 \
  -e DATABASE_URL=postgresql://... \
  quantlab-app:latest

# Execute a command in a running container
docker exec -it app bash
docker exec app python -c "import src; print(src.__version__)"

# View logs
docker logs -f app

# List running containers
docker ps

# List all containers (including stopped)
docker ps -a

# Stop and remove
docker stop app && docker rm app

# Clean up unused images, containers, networks, volumes
docker system prune -a
# WARNING: removes ALL unused images. Use with care.

CI/CD with GitHub Actions

A GitHub Actions workflow is a YAML file in .github/workflows/. It runs on triggers (push, PR, schedule) and executes jobs on GitHub-hosted runners.

# .github/workflows/ci.yml
name: CI

on:
  pull_request:
    branches: [master]
  push:
    branches: [master]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: pip

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run tests
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/testdb
        run: pytest tests/ -v --tb=short

      - name: Lint
        run: |
          pip install ruff
          ruff check src/

  deploy:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/master' && github.event_name == 'push'

    steps:
      - uses: actions/checkout@v4

      - name: Deploy to Render
        env:
          RENDER_API_KEY: ${{ secrets.RENDER_API_KEY }}
        run: |
          curl -X POST "https://api.render.com/v1/services/${{ secrets.RENDER_SERVICE_ID }}/deploys" \
            -H "Authorization: Bearer $RENDER_API_KEY"

How this works in QuantLab:

  • Every PR to master triggers the test job: spin up Postgres, install deps, run pytest, lint with ruff.
  • On merge to master, the deploy job triggers a Render deployment.
  • Secrets (API keys, service IDs) are stored in GitHub → Settings → Secrets → Actions. Never hardcode them in YAML.

Workflow YAML Structure

Key Purpose
name Workflow name shown in GitHub UI
on Triggers: push, pull_request, schedule, workflow_dispatch
jobs Map of job IDs to job definitions. Jobs run in parallel by default.
needs Job dependency. deploy runs only after test passes.
runs-on Runner OS: ubuntu-latest, windows-latest, macos-latest
services Sidecar containers (Postgres, Redis) available to the job
steps Sequential commands within a job. Each step is an action or shell command.
secrets Accessed via ${{ secrets.NAME }}. Set in repo settings.

Kubernetes: Just Enough to Understand

Kubernetes (K8s) orchestrates containers at scale. You will not need to run it for small projects, but you should know the vocabulary because it appears in every infrastructure discussion.

  • Pod: The smallest deployable unit. Usually one container, sometimes a sidecar (e.g., app + log collector). Pods are ephemeral—they can be killed and recreated.
  • Deployment: Declares "I want 3 replicas of this pod." Kubernetes maintains that count. If a pod crashes, a new one is created.
  • Service: A stable network endpoint (DNS name + IP) that routes traffic to pods. Pods come and go; the Service address does not change.
  • ConfigMap / Secret: Inject configuration and credentials into pods without baking them into the image.
  • Namespace: Logical isolation within a cluster (e.g., dev, staging, prod).
# deployment.yml — what a K8s deployment looks like
apiVersion: apps/v1
kind: Deployment
metadata:
  name: quantlab-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: quantlab-api
  template:
    metadata:
      labels:
        app: quantlab-api
    spec:
      containers:
        - name: api
          image: quantlab-app:latest
          ports:
            - containerPort: 8000
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: quantlab-secrets
                  key: database-url

When to Use What

Tool Use Case
Docker Package your app into a portable, reproducible image
Docker Compose Local development with multiple services (app + DB + cache)
GitHub Actions CI (test on every PR) and CD (deploy on merge to master)
Render / Railway Deploy containers without managing servers (PaaS)
Kubernetes Production orchestration at scale (multiple replicas, rolling deploys, auto-scaling)

For most projects (including QuantLab), the stack is: Docker for packaging, Compose for local dev, GitHub Actions for CI, Render for hosting. Kubernetes is for when you outgrow single-server deployments.
