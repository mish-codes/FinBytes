What We're Building

A ScanRecord model that stores risk scan results (tickers, VaR, narrative) using async SQLAlchemy 2.0. We'll write two functions — save_scan and get_recent_scans — test them with SQLite in-memory, and run PostgreSQL via Docker Compose.

SQLAlchemy 2.0 Style

SQLAlchemy 2.0 introduced a modern way to define models using Python type hints:

from datetime import datetime, UTC
from sqlalchemy import String, Float, DateTime
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column


class Base(DeclarativeBase):
    pass


class ScanRecord(Base):
    __tablename__ = "scan_records"

    id: Mapped[int] = mapped_column(primary_key=True)
    tickers: Mapped[str] = mapped_column(String(200))
    var_pct: Mapped[float] = mapped_column(Float)
    narrative: Mapped[str] = mapped_column(String(2000))
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), default=lambda: datetime.now(UTC)
    )

The key difference from legacy SQLAlchemy: Mapped[int] tells both Python and SQLAlchemy the column type. Your IDE now understands that record.var_pct is a float — autocomplete, type checking, and refactoring all work properly.

Async Functions

Every database call uses await because queries go over the network. While waiting for a response, Python can handle other requests instead of blocking.

from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession
from src.models import ScanRecord


async def save_scan(
    session: AsyncSession,
    tickers: list[str],
    var_pct: float,
    narrative: str,
) -> ScanRecord:
    record = ScanRecord(
        tickers=",".join(tickers),
        var_pct=var_pct,
        narrative=narrative,
    )
    session.add(record)
    await session.commit()
    await session.refresh(record)
    return record


async def get_recent_scans(session: AsyncSession, limit: int = 10) -> list[ScanRecord]:
    stmt = (
        select(ScanRecord)
        .order_by(ScanRecord.created_at.desc(), ScanRecord.id.desc())
        .limit(limit)
    )
    result = await session.execute(stmt)
    return list(result.scalars().all())

save_scan joins the ticker list with commas, creates the record, commits it, and refreshes to get the auto-generated id and created_at. get_recent_scans returns the most recent records, newest first, with a secondary sort on id to break ties when timestamps match.

Testing with SQLite In-Memory

Tests run against an in-memory SQLite database — no Docker, no setup, millisecond-fast. This works because SQLAlchemy abstracts the database: the same model and queries run on SQLite, PostgreSQL, or MySQL.

import pytest
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker


@pytest.fixture
async def session():
    from src.models import Base

    engine = create_async_engine("sqlite+aiosqlite:///:memory:")
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    async_session = async_sessionmaker(engine, expire_on_commit=False)
    async with async_session() as session:
        yield session
    await engine.dispose()

The fixture creates a fresh database for each test, runs Base.metadata.create_all to create the tables, yields a session, then cleans up. Each test gets an isolated, empty database.

class TestScanRecord:
    async def test_save_and_retrieve(self, session):
        from src.db import save_scan, get_recent_scans

        await save_scan(session, tickers=["AAPL", "MSFT"], var_pct=-2.15, narrative="Test narrative")
        scans = await get_recent_scans(session, limit=10)
        assert len(scans) == 1
        assert scans[0].tickers == "AAPL,MSFT"
        assert scans[0].var_pct == pytest.approx(-2.15)

    async def test_recent_scans_ordered_by_date(self, session):
        from src.db import save_scan, get_recent_scans

        await save_scan(session, tickers=["AAPL"], var_pct=-1.0, narrative="First")
        await save_scan(session, tickers=["MSFT"], var_pct=-2.0, narrative="Second")
        scans = await get_recent_scans(session, limit=10)
        assert scans[0].narrative == "Second"  # most recent first

    async def test_limit_works(self, session):
        from src.db import save_scan, get_recent_scans

        for i in range(5):
            await save_scan(session, tickers=[f"T{i}"], var_pct=-float(i), narrative=f"Scan {i}")
        scans = await get_recent_scans(session, limit=3)
        assert len(scans) == 3

Three tests, all async, all passing in under a second. The asyncio_mode = "auto" setting in pyproject.toml means pytest handles the async event loop automatically.

Docker Compose for PostgreSQL

For real-world use, we run PostgreSQL via Docker Compose with a named volume for data persistence:

services:
  db:
    image: postgres:16
    ports:
      - "5432:5432"
    environment:
      POSTGRES_USER: quantlab
      POSTGRES_PASSWORD: quantlab_dev
      POSTGRES_DB: quantlab
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:

The named volume pgdata means your data survives docker compose down and docker compose up cycles. Without it, you'd lose your entire database every restart. To start it:

docker compose up -d
docker compose exec db psql -U quantlab -c "SELECT 1;"
# Verify it's running, then stop:
docker compose down

What Is Alembic?

Right now, tests create tables with Base.metadata.create_all() — it reads your models and creates matching tables from scratch. This works for fresh databases but breaks in production.

The problem: you deploy with a ScanRecord table. Next week, you add a sharpe_ratio column. create_all() won't modify existing tables — it only creates missing ones. Your production database is stuck with the old schema.

Alembic solves this with migrations — versioned scripts that describe schema changes step by step. It compares your SQLAlchemy models to the actual database, figures out the difference, and generates a migration script:

# Set up Alembic in your project
alembic init alembic

# Compare models to DB, generate migration
alembic revision --autogenerate -m "add sharpe ratio"

# Apply all pending migrations
alembic upgrade head

# Roll back the last migration
alembic downgrade -1

The --autogenerate flag is the key feature: Alembic reads your models, compares them to the database, and writes the migration for you. You review it, then apply it. Think of it as git for your database schema — every change is tracked, reversible, and shareable.

Rule of thumb: create_all() is for tests. Alembic is for real databases.

Key Takeaways

  • SQLAlchemy 2.0 — use Mapped, mapped_column, and DeclarativeBase for type-safe models.
  • Async SQLAlchemy — create_async_engine + async_sessionmaker + await on every DB call. Non-blocking under load.
  • Test with SQLite — fast, no setup, good enough for standard SQL. Use PostgreSQL integration tests for database-specific features.
  • Docker Compose — run PostgreSQL with one command. Named volumes persist data across restarts.
  • Connection pooling — SQLAlchemy pools connections by default (5 ready, 10 overflow). Reuses connections instead of opening new ones per request.
  • Alembic — version-controlled migrations for production databases. Autogenerates from your models.
