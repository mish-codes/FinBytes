The Stack

PostgreSQL is the database. SQLAlchemy is the Python ORM that maps Python classes to database tables. Alembic handles schema migrations (adding columns, creating tables) without dropping data. All three work together, and understanding how they connect saves hours of debugging.

In QuantLab, this stack was introduced in Exercise 08 (PostgreSQL + SQLAlchemy), used in Exercise 11 (RDS Yield Schema) for AWS deployment, and became the persistence layer for the Stock Risk Scanner.

SQLAlchemy 2.0 Mapped Types

SQLAlchemy 2.0 introduced Mapped type annotations, replacing the older Column() style. Here is the actual scan record model from the scanner:

from datetime import datetime, UTC
from typing import Optional
from sqlalchemy import String, Float, DateTime
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column


class Base(DeclarativeBase):
    pass


class ScanRecord(Base):
    __tablename__ = "scan_records"

    id: Mapped[int] = mapped_column(primary_key=True)
    tickers: Mapped[str] = mapped_column(String(200))
    weights: Mapped[str] = mapped_column(String(200))
    period: Mapped[str] = mapped_column(String(10))
    status: Mapped[str] = mapped_column(String(20))
    var_pct: Mapped[Optional[float]] = mapped_column(Float, nullable=True)
    cvar_pct: Mapped[Optional[float]] = mapped_column(Float, nullable=True)
    max_drawdown_pct: Mapped[Optional[float]] = mapped_column(Float, nullable=True)
    volatility_pct: Mapped[Optional[float]] = mapped_column(Float, nullable=True)
    sharpe_ratio: Mapped[Optional[float]] = mapped_column(Float, nullable=True)
    narrative: Mapped[Optional[str]] = mapped_column(String(2000), nullable=True)
    error_message: Mapped[Optional[str]] = mapped_column(String(500), nullable=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), default=lambda: datetime.now(UTC)
    )
    completed_at: Mapped[Optional[datetime]] = mapped_column(
        DateTime(timezone=True), nullable=True
    )

Key patterns: Mapped[str] means the column is NOT NULL. Mapped[Optional[float]] means it is nullable. The type annotation drives both Python type checking and database schema generation.

Async Engine and Session

The scanner uses async SQLAlchemy throughout. The setup happens in the FastAPI lifespan:

from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine

DATABASE_URL = os.getenv("DATABASE_URL", "sqlite+aiosqlite:///scanner.db")

engine = create_async_engine(DATABASE_URL)
session_factory = async_sessionmaker(engine, expire_on_commit=False)

The connection URL determines the driver:

  • sqlite+aiosqlite:///scanner.db — local SQLite with async driver (development, tests)
  • postgresql+asyncpg://user:pass@host:5432/dbname — PostgreSQL with asyncpg (production)

expire_on_commit=False means you can still access attributes on a model instance after committing. Without it, accessing record.id after commit triggers a lazy load, which fails in async context.

CRUD Operations

The scanner's database layer shows the four core operations:

Create

async def create_pending_scan(session: AsyncSession, request: ScanRequest) -> ScanRecord:
    record = ScanRecord(
        tickers=",".join(request.tickers),
        weights=",".join(str(w) for w in request.weights),
        period=request.period,
        status="pending",
    )
    session.add(record)
    await session.commit()
    await session.refresh(record)  # reload to get the auto-generated id
    return record

Read (by primary key)

async def get_scan(session: AsyncSession, scan_id: int) -> ScanRecord | None:
    return await session.get(ScanRecord, scan_id)

Read (query with filters)

from sqlalchemy import select

async def get_recent_scans(session: AsyncSession, limit: int = 10) -> list[ScanRecord]:
    stmt = (
        select(ScanRecord)
        .where(ScanRecord.status == "complete")
        .order_by(ScanRecord.created_at.desc(), ScanRecord.id.desc())
        .limit(limit)
    )
    result = await session.execute(stmt)
    return list(result.scalars().all())

Update

async def complete_scan(session: AsyncSession, scan_id: int, result: ScanResult) -> None:
    record = await session.get(ScanRecord, scan_id)
    if record is None:
        raise ValueError(f"Scan record {scan_id} not found")
    record.status = "complete"
    record.var_pct = result.metrics.var_pct
    record.cvar_pct = result.metrics.cvar_pct
    record.max_drawdown_pct = result.metrics.max_drawdown_pct
    record.volatility_pct = result.metrics.volatility_pct
    record.sharpe_ratio = result.metrics.sharpe_ratio
    record.narrative = result.narrative
    record.completed_at = datetime.now(UTC)
    await session.commit()

Delete

async def delete_scan(session: AsyncSession, scan_id: int) -> None:
    record = await session.get(ScanRecord, scan_id)
    if record:
        await session.delete(record)
        await session.commit()

Alembic Migrations

Alembic tracks schema changes as versioned migration files. Instead of dropping and recreating tables, you apply incremental changes.

Setup

# Initialize Alembic in the project
alembic init alembic

# Edit alembic.ini to set sqlalchemy.url
# Or better: set it dynamically in alembic/env.py

Create a Migration

# Auto-generate from model changes
alembic revision --autogenerate -m "create scan_records table"

# This creates a file like:
# alembic/versions/002878ddee4c_create_scan_records_table.py

The generated migration from the scanner:

def upgrade() -> None:
    op.create_table('scan_records',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('tickers', sa.String(length=200), nullable=False),
        sa.Column('weights', sa.String(length=200), nullable=False),
        sa.Column('period', sa.String(length=10), nullable=False),
        sa.Column('status', sa.String(length=20), nullable=False),
        sa.Column('var_pct', sa.Float(), nullable=True),
        sa.Column('cvar_pct', sa.Float(), nullable=True),
        sa.Column('max_drawdown_pct', sa.Float(), nullable=True),
        sa.Column('volatility_pct', sa.Float(), nullable=True),
        sa.Column('sharpe_ratio', sa.Float(), nullable=True),
        sa.Column('narrative', sa.String(length=2000), nullable=True),
        sa.Column('error_message', sa.String(length=500), nullable=True),
        sa.Column('created_at', sa.DateTime(timezone=True), nullable=False),
        sa.Column('completed_at', sa.DateTime(timezone=True), nullable=True),
        sa.PrimaryKeyConstraint('id')
    )

def downgrade() -> None:
    op.drop_table('scan_records')

Apply and Rollback

# Apply all pending migrations
alembic upgrade head

# Rollback one version
alembic downgrade -1

# See current version
alembic current

# See migration history
alembic history --verbose

Testing with In-Memory SQLite

The scanner tests use SQLite in-memory instead of a real PostgreSQL instance. This makes tests fast and isolated:

@pytest.fixture
async def app():
    from src.scanner.db_models import Base
    from src.scanner.main import create_app

    engine = create_async_engine("sqlite+aiosqlite:///:memory:")
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    factory = async_sessionmaker(engine, expire_on_commit=False)

    app = create_app()
    app.state.session_factory = factory
    yield app
    await engine.dispose()

The trick: Base.metadata.create_all creates all tables from the ORM models, bypassing Alembic entirely. This is fine for tests because you always want a fresh schema. In production, always use Alembic migrations.

Docker Compose for Local PostgreSQL

The scanner's docker-compose.yml runs both the API and PostgreSQL locally:

services:

      - "8000:8000"

      - "5432:5432"
    env_file:
      - .env
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:

The .env file contains:

POSTGRES_USER=scanner
POSTGRES_PASSWORD=localdev
POSTGRES_DB=scanner
DATABASE_URL=postgresql+asyncpg://scanner:localdev@db:5432/scanner

# Start everything
docker compose up -d

# Check logs
docker compose logs -f scanner

# Tear down (keeps data in volume)
docker compose down

# Tear down and delete data
docker compose down -v

RDS on AWS

For Exercise 11, we deployed PostgreSQL on RDS. The key differences from local Docker:

  • Security groups: You must allow inbound traffic on port 5432 from your IP (for local development) or from the Lambda/EC2 security group (for production).
  • Connection string: postgresql+asyncpg://admin:password@quant-lab-yields.xxxxxx.us-east-1.rds.amazonaws.com:5432/yields
  • SSL: RDS requires SSL by default. Add ?ssl=require to the connection string or configure via connect_args.
# Connect via psql for debugging
psql "host=quant-lab-yields.xxxxxx.us-east-1.rds.amazonaws.com \
      port=5432 dbname=yields user=admin sslmode=require"

The Yield Schema (Exercise 11)

The yield curve exercise used a different schema from the scanner, showing how SQLAlchemy models map to financial data:

class YieldObservation(Base):
    __tablename__ = "yield_observations"

    id: Mapped[int] = mapped_column(primary_key=True)
    observation_date: Mapped[date] = mapped_column(Date, index=True)
    maturity_label: Mapped[str] = mapped_column(String(5))   # "1M", "2Y", "10Y"
    yield_pct: Mapped[float] = mapped_column(Float)
    curve_type: Mapped[str] = mapped_column(String(10))       # "par", "spot", "forward"
    source: Mapped[str] = mapped_column(String(20))           # "FRED"
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), default=lambda: datetime.now(UTC)
    )

Things That Tripped Me Up

  • expire_on_commit: Without expire_on_commit=False, accessing any attribute on a model after await session.commit() triggers a lazy load. In async context, this raises MissingGreenlet error. Always set it to False for async sessions.
  • session.refresh() after commit: After inserting a new record, the auto-generated id is not available in Python until you call await session.refresh(record). Without it, record.id is None.
  • Alembic autogenerate is not perfect: It does not detect column type changes (e.g., String(100) to String(200)). It does not detect renamed columns. Always review generated migrations before applying.
  • SQLite vs PostgreSQL differences: SQLite ignores String length limits, has no native ARRAY type, and handles datetime differently. Tests pass on SQLite but can fail on PostgreSQL. Test against real PostgreSQL before deploying.
  • Connection pool exhaustion: If you open sessions but do not close them (missing async with or forgotten await session.close()), the connection pool fills up and new requests hang. The async with factory() as session pattern prevents this.
  • Alembic with async: The default alembic/env.py uses synchronous connections. For async engines, you need to modify env.py to use run_async. The Alembic docs have a recipe for this, but it is easy to miss.
  • RDS auto-restart: Stopped RDS instances restart after 7 days. You cannot prevent this. Either delete the instance when done, or set a recurring reminder to stop it again.
