One Task, Eight Implementations

The task: fetch 50 URLs concurrently and collect the results. This is I/O-bound work—the CPU waits for network responses. Every modern language has a concurrency model for this, but they differ in overhead, scalability, and mental model. Here is each one, side by side.

1. Python asyncio (aiohttp)

import asyncio
import aiohttp

async def fetch(session: aiohttp.ClientSession, url: str) -> str:
    async with session.get(url, timeout=aiohttp.ClientTimeout(total=10)) as resp:
        return await resp.text()

async def main(urls: list[str]) -> list[str]:
    async with aiohttp.ClientSession() as session:
        tasks = [fetch(session, url) for url in urls]
        return await asyncio.gather(*tasks)

results = asyncio.run(main(urls))

Single-threaded event loop. All 50 requests run on one thread, yielding control at every await. No parallelism, but no thread-safety bugs either.

2. Python threading (ThreadPoolExecutor)

from concurrent.futures import ThreadPoolExecutor
import requests

def fetch(url: str) -> str:
    return requests.get(url, timeout=10).text

with ThreadPoolExecutor(max_workers=20) as pool:
    results = list(pool.map(fetch, urls))

Uses OS threads. The GIL (Global Interpreter Lock) prevents true parallel execution of Python bytecode, but it releases during I/O system calls. So this works fine for network-bound tasks. It does not help for CPU-bound work.

3. Python multiprocessing (ProcessPoolExecutor)

from concurrent.futures import ProcessPoolExecutor
import requests

def fetch(url: str) -> str:
    return requests.get(url, timeout=10).text

with ProcessPoolExecutor(max_workers=8) as pool:
    results = list(pool.map(fetch, urls))

Each worker is a separate OS process with its own Python interpreter and its own GIL. This bypasses the GIL entirely. The cost: high memory overhead (each process copies the interpreter state) and serialization overhead when passing data between processes. Use this for CPU-bound work, not I/O.

The GIL, Explained

CPython has a mutex called the Global Interpreter Lock. Only one thread can execute Python bytecode at a time. The GIL exists because CPython's memory management (reference counting) is not thread-safe.

  • I/O-bound: The GIL is released during system calls (network, file, sleep). Threading works fine.
  • CPU-bound: The GIL means threads run one at a time. You get no speedup. Use multiprocessing or write the hot loop in C/Rust.
  • asyncio: Single thread, no GIL contention at all. Cooperative multitasking via await.

4. Go (goroutines + WaitGroup + channel)

package main

import (
    "fmt"
    "io"
    "net/http"
    "sync"
    "time"
)

func fetch(url string, ch chan<- string, wg *sync.WaitGroup) {
    defer wg.Done()
    client := &http.Client{Timeout: 10 * time.Second}
    resp, err := client.Get(url)
    if err != nil {
        ch <- fmt.Sprintf("ERROR: %s", err)
        return
    }
    defer resp.Body.Close()
    body, _ := io.ReadAll(resp.Body)
    ch <- string(body)
}

func main() {
    urls := []string{ /* ... 50 URLs ... */ }
    ch := make(chan string, len(urls))
    var wg sync.WaitGroup

    for _, url := range urls {
        wg.Add(1)
        go fetch(url, ch, &wg)
    }

    go func() {
        wg.Wait()
        close(ch)
    }()

    var results []string
    for body := range ch {
        results = append(results, body)
    }
    fmt.Printf("Fetched %d pages\n", len(results))
}

Goroutines are lightweight green threads managed by the Go runtime. They start at ~2 KB of stack (vs ~1 MB for OS threads). The runtime multiplexes thousands of goroutines onto a small pool of OS threads. Channels provide safe communication without shared mutable state.

5. Rust (tokio + reqwest)

use reqwest::Client;
use tokio::task::JoinSet;
use std::time::Duration;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let urls: Vec<&str> = vec![ /* ... 50 URLs ... */ ];
    let client = Client::builder()
        .timeout(Duration::from_secs(10))
        .build()?;

    let mut set = JoinSet::new();
    for url in urls {
        let client = client.clone();
        let url = url.to_string();
        set.spawn(async move {
            client.get(&url).send().await?.text().await
        });
    }

    let mut results = Vec::new();
    while let Some(res) = set.join_next().await {
        match res? {
            Ok(body) => results.push(body),
            Err(e) => eprintln!("Error: {e}"),
        }
    }
    println!("Fetched {} pages", results.len());
    Ok(())
}

tokio is an async runtime for Rust. Tasks are lightweight (similar to goroutines). The compiler enforces thread safety at compile time—you cannot accidentally share mutable state across tasks without explicit synchronization.

6. TypeScript (Promise.all + fetch)

async function fetchUrl(url: string): Promise<string> {
    const controller = new AbortController();
    const timeout = setTimeout(() => controller.abort(), 10_000);
    try {
        const resp = await fetch(url, { signal: controller.signal });
        return await resp.text();
    } finally {
        clearTimeout(timeout);
    }
}

const urls: string[] = [ /* ... 50 URLs ... */ ];
const results = await Promise.all(urls.map(fetchUrl));
console.log(`Fetched ${results.length} pages`);

JavaScript/TypeScript has a single-threaded event loop, like Python asyncio. Promise.all fires all requests concurrently and resolves when all complete. No threads, no GIL—just callbacks under the hood.

7. C# (Task.WhenAll + HttpClient)

using System.Net.Http;

var client = new HttpClient { Timeout = TimeSpan.FromSeconds(10) };
var urls = new string[] { /* ... 50 URLs ... */ };

var tasks = urls.Select(async url => {
    var response = await client.GetAsync(url);
    return await response.Content.ReadAsStringAsync();
});

string[] results = await Task.WhenAll(tasks);
Console.WriteLine($"Fetched {results.Length} pages");

C# async/await compiles to state machines. The runtime schedules continuations on a thread pool. HttpClient is designed to be reused (do not create one per request—that exhausts socket handles).

8. C++ (std::async + std::future)

#include <future>
#include <vector>
#include <string>
#include "httplib.h"  // cpp-httplib header-only library

std::string fetch(const std::string& host, const std::string& path) {
    httplib::Client cli(host);
    cli.set_read_timeout(10);
    auto res = cli.Get(path);
    if (res && res->status == 200) return res->body;
    return "ERROR";
}

int main() {
    std::vector<std::pair<std::string, std::string>> urls = {
        {"http://example.com", "/page1"},
        // ... 50 entries ...
    };

    std::vector<std::future<std::string>> futures;
    for (auto& [host, path] : urls) {
        futures.push_back(std::async(std::launch::async, fetch, host, path));
    }

    std::vector<std::string> results;
    for (auto& f : futures) {
        results.push_back(f.get());
    }
    printf("Fetched %zu pages\n", results.size());
}

std::async with std::launch::async spawns OS threads. There is no built-in async runtime in the C++ standard library. For production, you would use Boost.Asio, libuv, or a custom event loop.

When to Use What

I/O-bound work (API calls, database queries, file reads)

Use async in whatever language you are already using. Python asyncio, TypeScript Promise.all, C# Task.WhenAll, and Go goroutines all handle this well. Python threading also works but uses more memory per task.

CPU-bound work (data processing, compression, math)

Avoid Python threads—the GIL makes them useless here. Use multiprocessing, or better, write the hot path in Rust/C++ and call it from Python. Go and Rust handle CPU-bound concurrency natively without workarounds.

Mixed workloads

Go is the simplest choice: goroutines handle both I/O and CPU work without a separate model. Rust (tokio + rayon) is faster but more complex. In Python, combine asyncio for I/O with ProcessPoolExecutor for CPU-heavy stages via loop.run_in_executor.

Maximum throughput

Rust (tokio) and Go can sustain millions of concurrent tasks on modest hardware. C# is close behind. Python and TypeScript top out earlier due to single-threaded event loops and higher per-task overhead.
