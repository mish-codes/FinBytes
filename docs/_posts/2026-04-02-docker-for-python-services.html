What We're Building

A minimal FastAPI app containerized with Docker. We'll write a multi-stage Dockerfile, a docker-compose file, and a .dockerignore — the core building blocks for running any Python service in a container.

Installing Docker Desktop on Windows

Before we can build containers, we need Docker installed. On Windows, Docker runs inside WSL 2 (Windows Subsystem for Linux).

Step 1: Install WSL 2

Open PowerShell as Administrator and run:

wsl --install

This installs WSL 2 with Ubuntu as the default distribution. Restart your computer when prompted. After the restart, a terminal window will open asking you to create a Unix username and password for Ubuntu — pick something you'll remember.

Step 2: Install Docker Desktop

Open a terminal and run:

winget install Docker.DockerDesktop

Restart your computer again. Docker Desktop needs a fresh start to finish setup.

Step 3: Verify

After restarting, open a terminal and check both are installed:

docker --version
# Docker version 29.x.x

docker compose version
# Docker Compose version v5.x.x

If both commands return version numbers, you're good to go.

What Is Docker and Why Use It

Docker packages your application and all its dependencies into a container — a lightweight, isolated environment that runs the same way everywhere. Unlike virtual machines, containers share the host OS kernel, so they start in seconds and use a fraction of the resources.

For financial services, this matters because:

  • Reproducibility — the same image runs identically on your laptop, in CI, and in production. No "works on my machine" surprises.
  • Consistent environments — every team member runs the exact same stack, down to the Python version and package versions.
  • Deployment confidence — you test the container, you ship the container. What you tested is what runs in prod.

The FastAPI App

We start with a tiny app — two endpoints, no external dependencies beyond FastAPI and Uvicorn. The goal is to learn Docker, not build a complex API.

import os
from fastapi import FastAPI

app = FastAPI(title="Dockerized API")


@app.get("/health")
def health():
    return {"status": "ok", "environment": os.getenv("APP_ENV", "development")}


@app.get("/")
def root():
    return {"message": "Hello from Docker!"}

The /health endpoint reads APP_ENV from the environment. Locally it defaults to "development". Inside Docker, we'll set it to "docker" — a quick way to confirm you're actually running in a container.

The Dockerfile

A Dockerfile is a recipe for building a container image. Each instruction creates a layer.

# Stage 1: Build
FROM python:3.11-slim AS builder

WORKDIR /app
COPY pyproject.toml .
RUN pip install --no-cache-dir .

# Stage 2: Runtime
FROM python:3.11-slim

WORKDIR /app
COPY --from=builder /usr/local/lib/python3.11/site-packages /usr/local/lib/python3.11/site-packages
COPY --from=builder /usr/local/bin/uvicorn /usr/local/bin/uvicorn
COPY src/ src/

EXPOSE 8000
CMD ["uvicorn", "src.app:app", "--host", "0.0.0.0", "--port", "8000"]

Line by line

  • FROM python:3.11-slim AS builder — start from a minimal Python image. AS builder names this stage so we can reference it later.
  • WORKDIR /app — set the working directory inside the container (creates it if it doesn't exist).
  • COPY pyproject.toml . — copy just the project config first. Dependencies change rarely, so this layer gets cached.
  • RUN pip install --no-cache-dir . — install dependencies. --no-cache-dir keeps the layer smaller.
  • Second FROM python:3.11-slim — start a fresh image for the runtime stage. None of the build tools carry over.
  • COPY --from=builder — cherry-pick only the installed packages and the uvicorn binary from the builder stage.
  • COPY src/ src/ — add the application code.
  • EXPOSE 8000 — documents the port (doesn't actually open it — that's what ports: does in compose).
  • CMD — the default command when the container starts.

Why multi-stage?

The builder stage has pip, build tools, and wheel caches. The runtime stage has none of that — just Python, your packages, and your code. Result: a smaller, faster, more secure image.

docker-compose.yml

Instead of typing a long docker run command with flags, we declare the configuration in YAML:

services:
  api:
    build: .
    ports:
      - "8000:8000"
    environment:
      - APP_ENV=docker
    restart: unless-stopped
  • build: . — build the image from the Dockerfile in the current directory.
  • ports: "8000:8000" — map host port 8000 to container port 8000 (left is host, right is container).
  • environment: APP_ENV=docker — inject an environment variable. This is how /health knows it's running in Docker.
  • restart: unless-stopped — auto-restart on crash, but not if you manually stop it.

.dockerignore

When you run docker build, your entire project folder gets sent to the Docker daemon. A .dockerignore file filters out files you don't need in the image:

__pycache__
*.pyc
.git
.env
.venv

This keeps builds faster (less data to transfer), images smaller (no junk files), and more secure (no .env secrets baked into the image).

Running It

Build and start:

cd exercises/07-docker
docker compose up --build -d

Verify:

curl http://localhost:8000/health
# {"status":"ok","environment":"docker"}

curl http://localhost:8000/
# {"message":"Hello from Docker!"}

The environment field says "docker" instead of "development" — proof that the compose env var is being injected.

Stop:

docker compose down

Key Takeaways

  • Dockerfile — a recipe for building an image. Each instruction is a cached layer.
  • Multi-stage builds — separate build dependencies from runtime. Smaller, safer images.
  • docker-compose — declarative config for running services. One command to start everything.
  • .dockerignore — keep builds fast and images clean. Never bake secrets into images.
  • Environment variables — configure containers at runtime, not build time. Same image, different behavior.
