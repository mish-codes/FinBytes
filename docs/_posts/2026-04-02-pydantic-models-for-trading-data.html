Why Pydantic for Financial Data?

Financial data is full of constraints. A trade quantity can't be negative. A ticker must exist. Weights in a portfolio must sum to 1.0. You can enforce these rules with manual if statements scattered through your code, or you can declare them once in a model and let Pydantic validate automatically.

Pydantic v2 is a data validation library built on a Rust core. You define Python classes with type annotations, and Pydantic handles parsing, validation, and serialization. It's also the foundation of FastAPI — every request body and response model in FastAPI is a Pydantic model.

Three Models for Trading

We'll build three models that represent a real trading workflow: a single trade, an aggregated position, and a portfolio analysis request.

Model 1: Trade — Field Validators and Computed Fields

from datetime import date
from typing import Literal
from pydantic import BaseModel, Field, field_validator, computed_field


class Trade(BaseModel):
    ticker: str = Field(..., min_length=1, max_length=10)
    side: Literal["buy", "sell"]
    quantity: int = Field(..., gt=0)
    price: float = Field(..., gt=0)
    trade_date: date

    @field_validator("ticker")
    @classmethod
    def uppercase_ticker(cls, v: str) -> str:
        return v.upper()

    @field_validator("trade_date")
    @classmethod
    def not_future(cls, v: date) -> date:
        if v > date.today():
            raise ValueError("trade_date cannot be in the future")
        return v

    @computed_field
    @property
    def notional(self) -> float:
        return self.quantity * self.price

Several things happening here:

  • Literal["buy", "sell"] — constrains side to exactly two values. Passing "hold" raises a ValidationError. This is better than a plain str because you get validation, documentation, and IDE autocomplete in one declaration.
  • Field(..., gt=0) — the gt (greater than) constraint rejects zero or negative quantities at the model level. No manual checks needed.
  • @field_validator — runs on a single field. The ticker validator transforms input ("aapl" becomes "AAPL"). The date validator rejects future dates. Both run automatically on construction.
  • @computed_field — notional (quantity × price) is derived, not stored. It appears in model_dump() output but can't be passed as input. Perfect for derived financial values you don't want anyone manually overriding.

Model 2: Position — Nested Models

class Position(BaseModel):
    ticker: str
    trades: list[Trade]

    @computed_field
    @property
    def net_quantity(self) -> int:
        total = 0
        for t in self.trades:
            total += t.quantity if t.side == "buy" else -t.quantity
        return total

    @computed_field
    @property
    def avg_price(self) -> float:
        buys = [t for t in self.trades if t.side == "buy"]
        if not buys:
            return 0.0
        total_cost = sum(t.quantity * t.price for t in buys)
        total_qty = sum(t.quantity for t in buys)
        return total_cost / total_qty

trades: list[Trade] is a nested model — Pydantic validates each trade in the list against the Trade model. If any trade has a negative quantity or invalid side, the entire Position construction fails with a clear error pointing to the exact trade and field.

The computed fields aggregate across child data: net_quantity sums buys and subtracts sells, avg_price calculates the weighted average of buy prices. These update automatically whenever you access them.

Model 3: PortfolioRequest — Model Validators

from pydantic import model_validator

class PortfolioRequest(BaseModel):
    tickers: list[str] = Field(..., min_length=1, max_length=10)
    weights: list[float] = Field(..., min_length=1, max_length=10)
    period: str = Field(default="1y", pattern=r"^\d+(d|mo|y)$")
    confidence: float = Field(default=0.95, ge=0.9, le=0.99)

    @model_validator(mode="after")
    def validate_portfolio(self):
        if len(self.tickers) != len(self.weights):
            raise ValueError(
                f"tickers length ({len(self.tickers)}) must match "
                f"weights length ({len(self.weights)})"
            )
        if abs(sum(self.weights) - 1.0) > 0.01:
            raise ValueError(
                f"weights must sum to 1.0, got {sum(self.weights):.4f}"
            )
        return self

@model_validator(mode="after") is the key difference from @field_validator. Field validators see one field at a time. Model validators see the entire model after all fields are validated, so you can check relationships between fields.

Here we enforce two cross-field rules: tickers and weights must have the same length, and weights must sum to 1.0. These constraints can't be expressed with field-level validation alone.

The period field shows Field(default=..., pattern=...) — a default value with a regex constraint. Valid periods look like "30d", "6mo", or "1y".

Serialization: model_dump and model_validate_json

Pydantic models convert to and from dicts and JSON:

# Model → dict (includes computed fields)
data = trade.model_dump()
# {'ticker': 'AAPL', 'side': 'buy', 'quantity': 100,
#  'price': 150.25, 'trade_date': datetime.date(2026, 4, 1),
#  'notional': 15025.0}

# Dict → model (re-validates)
trade2 = Trade(**data)

# JSON string → model (what APIs actually receive)
trade3 = Trade.model_validate_json(
    '{"ticker":"AAPL","side":"buy","quantity":100,"price":150.25,"trade_date":"2026-04-01"}'
)

This is the bridge to web APIs. When FastAPI receives a JSON request body, it calls model_validate_json() under the hood. When it sends a response, it calls model_dump(). The PortfolioRequest model we wrote here will become the FastAPI request body in the next exercise — zero extra work needed.

v2 vs v1: What Changed

If you've used Pydantic v1, the main changes:

  • @validator → @field_validator (plus @classmethod decorator)
  • @root_validator → @model_validator(mode="after")
  • .dict() → .model_dump()
  • .json() → .model_dump_json()
  • parse_raw() → model_validate_json()
  • Validation core rewritten in Rust — 5-50x faster

The naming is more consistent and the model_ prefix avoids clashing with your own method names.
