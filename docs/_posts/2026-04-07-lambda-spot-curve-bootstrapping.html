From Par Yields to Spot Rates

The Treasury publishes par yields — the coupon rates that make a hypothetical bond price at 100. But for discounting arbitrary cash flows (a coupon at year 7.5, a callable bond's exercise date, an options payoff), you need spot rates: the zero-coupon yield for each future date.

Converting one to the other is called bootstrapping, and it's one of the first non-trivial calculations every fixed income quant learns. In this post we implement it, package it as an AWS Lambda function, expose it through API Gateway, and get a live HTTPS endpoint that anyone can POST a par curve to.

The Bootstrap Math

A par bond trades at face value (100). For an annual-coupon bond paying coupon c with maturity n years:

100 = c/(1+s₁) + c/(1+s₂)² + ... + (c + 100)/(1+sₙ)ⁿ

The sᵢ are the spot rates we want to solve for. The trick is to do it sequentially:

  1. A 1-year par bond has only one cash flow at year 1, so s₁ = par yield at 1Y.
  2. For the 2-year bond, s₁ is already known — we rearrange and solve algebraically for s₂.
  3. For the 3-year bond, both s₁ and s₂ are known, so we solve for s₃. And so on.

At each step:

sₙ = ((c + 1) / (1 - Σ c/(1+sᵢ)ⁱ))^(1/n) − 1

where the sum runs over all prior spot rates and cash flows are scaled to per unit of face value.

The Gappy Tenor Problem

Treasury par yields come at 11 tenors: 1M, 3M, 6M, 1Y, 2Y, 3Y, 5Y, 7Y, 10Y, 20Y, 30Y. Notice the gaps — there's no 4Y, no 6Y, no 8Y/9Y/11Y...19Y. But the standard bootstrap formula needs a coupon (and a spot) at every integer year from 1 to n, because that's how the par bond's cash flows are defined.

A naïve implementation that only uses the tenors it was given produces wildly wrong spot rates. For a flat par curve (every tenor at 5%), the correct spot curve is also flat at 5%. Running a gap-skipping bootstrap on that same input gave back a 5Y spot of 2.97% — clearly broken.

The fix is to linearly interpolate the par yields onto the dense annual grid first, then bootstrap year-by-year, then return only the spots at the originally-requested tenors:

def bootstrap_spot_curve(par_yields: dict[str, float]) -> dict[str, float]:
    ordered_labels = [m for m in MATURITY_ORDER if m in par_yields]
    par_decimal = {m: par_yields[m] / 100.0 for m in ordered_labels}
    spots: dict[str, float] = {}

    # Sub-year tenors are single-cashflow zeros: spot = par
    for m in ordered_labels:
        if MATURITY_YEARS[m] < 1.0:
            spots[m] = par_decimal[m]

    # Annual tenors: densify par curve, bootstrap year-by-year
    annual_labels = [m for m in ordered_labels if MATURITY_YEARS[m] >= 1.0]
    known_years = [int(MATURITY_YEARS[m]) for m in annual_labels]
    known_rates = [par_decimal[m] for m in annual_labels]
    dense_par = _interpolate_par_to_annual_grid(known_years, known_rates)

    dense_spots: dict[int, float] = {}
    for y in range(1, known_years[-1] + 1):
        c = dense_par[y]
        if y == 1:
            dense_spots[1] = c
            continue
        pv_coupons = sum(c / ((1 + dense_spots[k]) ** k) for k in range(1, y))
        dense_spots[y] = ((c + 1.0) / (1.0 - pv_coupons)) ** (1.0 / y) - 1.0

    for m in annual_labels:
        spots[m] = dense_spots[int(MATURITY_YEARS[m])]

    return {m: round(spots[m] * 100, 4) for m in ordered_labels}

This passes every test case: flat curves stay flat, normal upward-sloping curves produce spots slightly above par at the long end (as they should), and sub-year tenors pass through unchanged.

Wrapping it as a Lambda Function

The handler is intentionally tiny — parse JSON, validate, compute, respond:

import json
from bootstrap import bootstrap_spot_curve

def _response(status: int, body: dict) -> dict:
    return {
        "statusCode": status,
        "headers": {"Content-Type": "application/json"},
        "body": json.dumps(body),
    }

def lambda_handler(event, context):
    try:
        body = json.loads(event.get("body") or "{}")
        par_yields = body.get("par_yields")
        if not isinstance(par_yields, dict) or not par_yields:
            return _response(400, {"error": "par_yields must be a non-empty object"})
        return _response(200, {"spot_curve": bootstrap_spot_curve(par_yields)})
    except json.JSONDecodeError as exc:
        return _response(500, {"error": f"invalid JSON: {exc}"})
    except Exception as exc:
        return _response(500, {"error": str(exc)})

No external dependencies — just standard library and our own module. The deployment zip is about 2.4KB. You could inline this in the Lambda console and it'd still work.

Deploying via CLI

The full deployment takes eight AWS CLI calls. In order:

  1. aws lambda create-function — upload the zip, attach the execution role
  2. aws apigateway create-rest-api — a new REST API container
  3. aws apigateway create-resource x2 — create /curves then /curves/bootstrap
  4. aws apigateway put-method — declare the POST method
  5. aws apigateway put-integration — wire POST to Lambda via AWS_PROXY integration
  6. aws lambda add-permission — let API Gateway invoke the function
  7. aws apigateway create-deployment — publish to a /dev stage

Missing step 6 is a classic pitfall: the deployment succeeds, the URL responds, and you get a 500 with "Internal server error" forever because Lambda silently refuses the invocation. The API Gateway test console gives a clearer error. Always add the resource-based permission on the Lambda.

Testing the Live Endpoint

Once deployed:

curl -X POST https://<API_ID>.execute-api.us-east-1.amazonaws.com/dev/curves/bootstrap \
  -H "Content-Type: application/json" \
  -d '{"par_yields":{"1M":5.25,"3M":5.30,"6M":5.20,"1Y":4.80,"2Y":4.50,"5Y":4.20,"10Y":4.30}}'

Response:

{
  "spot_curve": {
    "1M": 5.25, "3M": 5.30, "6M": 5.20,
    "1Y": 4.80, "2Y": 4.4933, "5Y": 4.1855, "10Y": 4.3053
  }
}

The 10Y spot (4.31%) is slightly above the 10Y par (4.30%) — exactly what theory predicts for an upward-sloping yield curve.

Cost

The free tier gives 1 million Lambda requests and 400,000 GB-seconds of compute per month, plus 1 million API Gateway calls. For a learning project that gets curled a few times a day, effective cost is zero — and there's no "running instance" to remember to turn off. Serverless is a genuinely different cost model from RDS.
