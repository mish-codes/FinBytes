The QuantLab Workflow

This is the actual workflow used across QuantLab and FinBytes. It is simple and it works:

  1. Work on the working branch. All development happens here.
  2. PR to master. When a feature or exercise is complete, open a pull request from working to master.
  3. Merge and sync. After the PR merges, pull master into working to stay current.

No feature branches, no release branches, no gitflow. For a solo developer or small team, this is the right amount of process.

# Daily workflow
git checkout working
# ... make changes ...
git add src/pricing.py tests/test_pricing.py
git commit -m "feat(pricing): add OAS calculation"
git push origin working

# Open PR
gh pr create --base master --head working \
  --title "feat(pricing): add OAS calculation" \
  --body "Adds OAS/Z-spread with binomial tree pricing."

# After PR merges on GitHub
git checkout master
git pull origin master
git checkout working
git merge master
git push origin working

Branch Protection

Protect master so nobody (including you at 2 AM) can push directly:

  1. GitHub → Settings → Branches → Add rule
  2. Branch name pattern: master
  3. Enable: Require a pull request before merging
  4. Enable: Require status checks to pass before merging (once you have CI)
  5. Optional: Require conversation resolution before merging

With protection enabled, git push origin master is rejected. All changes must go through a PR.

Daily Commands

# Check what has changed
git status                     # modified, staged, untracked files
git diff                       # unstaged changes (working tree vs index)
git diff --staged              # staged changes (index vs last commit)

# Stage and commit
git add src/pricing.py         # stage specific files (preferred)
git add -A                     # stage everything (use carefully)
git commit -m "fix(curves): handle missing 30Y tenor"

# Push to remote
git push origin working

# Pull latest from remote
git pull origin working        # fetch + merge
git pull --rebase origin working  # fetch + rebase (cleaner history)

Branching

# Create and switch to a new branch
git checkout -b feature/dlq-handler

# List branches
git branch              # local
git branch -r           # remote
git branch -a           # all

# Switch branches
git checkout working
git switch working      # newer syntax (Git 2.23+)

# Delete a merged branch
git branch -d feature/dlq-handler        # safe (refuses if unmerged)
git branch -D feature/dlq-handler        # force (even if unmerged)

# Merge a branch into current
git checkout working
git merge feature/dlq-handler

Pull Requests with gh

The GitHub CLI (gh) is faster than the browser for creating PRs:

# Install: https://cli.github.com/
# Authenticate once:
gh auth login

# Create a PR
gh pr create --base master --head working \
  --title "feat(exercises): 20 integration testing + portfolio credit VaR" \
  --body "Monte Carlo simulation for portfolio credit VaR with integration tests."

# List open PRs
gh pr list

# View PR details
gh pr view 47

# Merge a PR
gh pr merge 47 --merge

# Check CI status on a PR
gh pr checks 47

Merge Conflicts

Conflicts happen when two branches modify the same lines. Git marks the conflict in the file:

<<<<<<< HEAD
spread = calculate_oas(bond, curve)
=======
spread = calculate_z_spread(bond, curve)
>>>>>>> working

To resolve: edit the file to keep the correct version, remove the conflict markers, then:

git add src/pricing.py
git commit -m "fix: resolve merge conflict in pricing.py"

To abort a merge that has gone wrong: git merge --abort

Useful Commands

# Compact log
git log --oneline -20
# afb3f7e Merge pull request #47 from MishCodesFinBytes/working
# 039ccdd feat(exercises): 20 integration testing + portfolio credit VaR

# See what changed in a specific commit
git show afb3f7e

# Compare branches
git diff master...working

# Stash uncommitted changes (switch branches without committing)
git stash
git checkout master
# ... do something ...
git checkout working
git stash pop

# List stashes
git stash list

# Undo the last commit (keep changes staged)
git reset --soft HEAD~1

# Amend the last commit message (only if not pushed)
git commit --amend -m "fix(curves): correct tenor interpolation"

Commit Message Conventions

Use a prefix that describes the type of change:

  • feat: — new feature or capability
  • fix: — bug fix
  • chore: — maintenance (deps, config, CI)
  • refactor: — code restructuring (no behavior change)
  • docs: — documentation only
  • test: — adding or fixing tests

Optional scope in parentheses: feat(pricing): add OAS calculation

For AI-assisted commits, add a co-author trailer:

feat(exercises): 20 integration testing + portfolio credit VaR

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>

Common Mistakes

1. Committing secrets. An .env file with API keys pushed to a public repo is compromised forever (even if you delete it later, it is in git history). Use .gitignore and check git status before every commit.

2. Force-pushing to master. git push --force origin master rewrites history for everyone. With branch protection enabled, GitHub blocks this. Never do it.

3. Not pulling before pushing. If your local branch is behind the remote, git push is rejected. Always git pull first.

4. Giant commits. A commit that touches 40 files across 5 features is impossible to review or revert. Commit early, commit often, one logical change per commit.

5. Forgetting to switch branches. Accidentally committing to master when you meant working. Fix with git reset --soft HEAD~1, switch branches, then recommit.
