Why Async Matters for Financial Data

Imagine you need to fetch prices for 50 tickers. With synchronous code, each HTTP request blocks until it returns — 50 requests at 200ms each means 10 seconds of waiting. With async, all 50 requests fire at once. Total time: ~200ms. Same single thread, same machine, 50x faster.

Async doesn't make individual requests faster. It eliminates the waiting between them. For financial services that pull data from multiple sources, this is the difference between a responsive API and a sluggish one.

The Event Loop Mental Model

Think of the event loop as a single waiter serving multiple tables. When table 1 orders food, the waiter doesn't stand there waiting for the kitchen — they go take table 2's order. When the kitchen rings the bell, the waiter delivers the food.

async def defines a coroutine — a function that can pause and resume. At each await, the function says "I'm waiting for something — go do other work." The event loop switches to another coroutine. When the awaited operation completes, the original function resumes exactly where it left off.

Key insight: async is faster for I/O-bound work (network calls, file reads, database queries). For CPU-bound work (number crunching), async doesn't help — use multiprocessing instead.

Fetching a Single Price with aiohttp

import aiohttp

async def fetch_price(ticker: str) -> float:
    url = f"https://query1.finance.yahoo.com/v8/finance/chart/{ticker}"
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as response:
            response.raise_for_status()
            data = await response.json()
            return float(data["chart"]["result"][0]["meta"]["regularMarketPrice"])

Two levels of async with here:

  • aiohttp.ClientSession() — manages a connection pool. Creating one per request is wasteful; in production you'd share a session across calls.
  • session.get(url) — sends the request and streams the response. The async with ensures the response body is properly closed even if an error occurs.

await response.json() is where the magic happens — the function pauses here while the response body is read from the network, and the event loop is free to run other coroutines in the meantime.

Fetching Many Prices Concurrently

import asyncio

async def fetch_many_prices(tickers: list[str]) -> dict[str, float]:
    async def safe_fetch(ticker: str) -> tuple[str, float | None]:
        try:
            price = await fetch_price(ticker)
            return ticker, price
        except Exception:
            return ticker, None

    tasks = [safe_fetch(t) for t in tickers]
    results = await asyncio.gather(*tasks)
    return {ticker: price for ticker, price in results if price is not None}

asyncio.gather takes multiple coroutines and runs them concurrently on the same event loop. All start immediately. Results come back in the same order as the input, even if some finish faster.

The safe_fetch wrapper is a graceful failure pattern: if one ticker fails (network error, invalid symbol), the others still return successfully. You don't want to lose 49 good results because ticker #37 had a hiccup. The failed ticker is simply excluded from the output.

Critical rule: never call requests.get() inside an async function. It blocks the entire event loop — all other concurrent tasks freeze until the HTTP response arrives. Always use aiohttp (or httpx with async) inside async functions.

Testing Async Code

import pytest
from unittest.mock import AsyncMock, patch, MagicMock

class TestFetchPrice:
    async def test_returns_float(self):
        mock_response = AsyncMock()
        mock_response.json = AsyncMock(return_value={
            "chart": {"result": [{"meta": {"regularMarketPrice": 150.25}}]}
        })
        mock_response.raise_for_status = MagicMock()
        mock_response.__aenter__ = AsyncMock(return_value=mock_response)
        mock_response.__aexit__ = AsyncMock(return_value=False)

        mock_session = AsyncMock()
        mock_session.get = MagicMock(return_value=mock_response)
        mock_session.__aenter__ = AsyncMock(return_value=mock_session)
        mock_session.__aexit__ = AsyncMock(return_value=False)

        with patch("async_prices.aiohttp.ClientSession", return_value=mock_session):
            price = await fetch_price("AAPL")
            assert price == 150.25

Mocking async code is more involved than sync code. You need to mock the async context manager protocol (__aenter__/__aexit__) for both the session and the response. AsyncMock handles await-able return values; MagicMock is used for sync methods like raise_for_status and for session.get (which returns a context manager, not a coroutine).

With pytest-asyncio and asyncio_mode = "auto" in your config, test functions can be async def directly — no @pytest.mark.asyncio decorator needed.

How FastAPI Uses Async

FastAPI runs on Starlette, which runs on asyncio. When you write async def endpoints, the server handles other requests while yours is waiting for I/O:

@app.get("/prices/{ticker}")
async def get_price(ticker: str):
    price = await fetch_price(ticker)  # non-blocking
    return {"ticker": ticker, "price": price}

With sync def, FastAPI runs your endpoint in a thread pool — still concurrent, but less efficient. For I/O-heavy endpoints (database queries, external API calls), async def is the way to go.
