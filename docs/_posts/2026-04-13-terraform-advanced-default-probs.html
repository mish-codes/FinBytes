From Constant to Piecewise Hazard Rates

Exercise 15 used a single constant hazard rate λ = s / (1 − R), which implies the same instantaneous default intensity at every horizon. That's fine for a quick read, but CDS term structures are rarely flat. A BBB name might trade at 80 bps for 1Y and 150 bps for 5Y — the market is saying default risk is back-loaded, probably because the company has enough cash to survive the near term but faces refinancing walls or competitive threats later.

Piecewise-constant hazard rates capture this. For each successive tenor interval, we solve for the interval-specific λ that makes the cumulative hazard integrate to the level implied by the observed CDS spread:

def bootstrap_hazard_rates(cds_spreads, recovery_rate=0.40):
    lgd = 1.0 - recovery_rate
    ordered = [(t, cds_spreads[t]) for t in TENOR_ORDER if t in cds_spreads]

    hazard_rates = {}
    cum_hazard = 0.0
    prev_years = 0.0

    for tenor, spread_bps in ordered:
        years = CDS_TENOR_YEARS[tenor]
        dt = years - prev_years

        lambda_avg = (spread_bps / 10000.0) / lgd
        cum_hazard_target = lambda_avg * years

        lambda_i = (cum_hazard_target - cum_hazard) / dt if dt > 0 else lambda_avg
        lambda_i = max(lambda_i, 0.0)

        interval = f"{int(prev_years)}Y-{tenor}" if prev_years > 0 else f"0-{tenor}"
        hazard_rates[interval] = round(lambda_i, 6)

        cum_hazard = cum_hazard_target
        prev_years = years

    return hazard_rates

The logic: the CDS spread at tenor T implies an average hazard rate over [0, T]. If you already know the cumulative hazard up to the previous tenor, the interval-specific λ is whatever fills the gap. On a steepening curve, later intervals have higher λ. On an inverted curve (rare, but possible for distressed names with near-term default risk), early intervals have higher λ and we floor later intervals at 0.

Reading the Output

For a hypothetical BBB name with spreads {1Y: 80, 3Y: 120, 5Y: 150} and 40% recovery:

Interval    λ          Interpretation
0-1Y        0.0133     Low near-term risk: company has runway
1Y-3Y       0.0200     Rising risk: market sees refinancing pressure
3Y-5Y       0.0250     Highest interval risk: structural concerns

A flat curve would give the same λ everywhere. The piecewise structure is what lets you see when the market thinks the risk lives.

Survival Curves

The survival probability is the complement of the cumulative default probability:

S(T) = exp(−λ_avg × T)
P(default by T) = 1 − S(T)

Plotted over time, survival curves are always decreasing and always start at 1.0 (you're alive today with certainty). Their shape tells you about the risk profile:

  • Convex (drops steeply early, flattens later): front-loaded risk. The dangerous period is now. If the entity survives the near term, it's probably fine.
  • Concave (drops slowly early, steepens later): back-loaded risk. Safe now, trouble ahead. This is the pattern for companies burning cash or facing debt maturity walls.
  • Approximately linear: constant hazard. Risk is evenly distributed. This is what a flat CDS curve implies.

A credit analyst looking at two names with the same 5Y CDS spread but different curve shapes would treat them very differently. Same headline risk, very different stories.

Expected Loss: PD × LGD × EAD

The expected loss formula is the foundation of credit risk capital calculations (Basel II/III). Three inputs:

  • PD (Probability of Default): from the CDS-implied default probability.
  • LGD (Loss Given Default): 1 − recovery rate. Senior unsecured: ~60%. Secured: ~40%. Subordinated: ~80%.
  • EAD (Exposure at Default): dollar amount at risk. For a bond, typically par value.
def expected_loss(default_prob, lgd, exposure):
    return round(default_prob * lgd * exposure, 2)

Example: a $1M senior unsecured bond with a 5% implied default probability:

EL = 0.05 × 0.60 × $1,000,000 = $30,000

$30K expected loss on a $1M position. That's the number that drives reserve calculations, credit limits, and portfolio allocation decisions.

Terraform: Modules and Environments

The Terraform side of this exercise is about organising infrastructure code for a growing system. Through exercises 12 and 14 we wrote flat .tf files — fine for one or two resources, but the full credit-risk platform now has Lambda functions, API Gateways (REST and HTTP), RDS, SQS, SNS, DynamoDB, and eventually ElastiCache. A single flat config would be unreadable.

Modules

Terraform modules are reusable blocks of infrastructure. A lambda-api module might take a function name, handler path, and route key, and create the Lambda + API Gateway integration + route + permission. You call it once per endpoint:

module "curve_fitting" {
  source        = "./modules/lambda-api"
  function_name = "quant-lab-curve-fitting"
  handler       = "handler.lambda_handler"
  route_key     = "POST /curves/fit"
}

module "cds_endpoint" {
  source        = "./modules/lambda-api"
  function_name = "quant-lab-cds"
  handler       = "handler.lambda_handler"
  route_key     = "POST /credit/cds"
}

Same infrastructure pattern, different parameters. The module encapsulates the wiring so each new endpoint is 5 lines instead of 30.

Environments via Variable Files

Rather than Terraform workspaces (which share state but use different variable values), the common pattern is environment-specific .tfvars files:

terraform plan -var-file=environments/dev.tfvars
terraform plan -var-file=environments/prod.tfvars

Dev might use db.t3.micro and a single-AZ RDS; prod uses db.r6g.large with Multi-AZ. Same config, different inputs.

Remote State

Local state (terraform.tfstate on disk) works for solo development but breaks the moment two people run terraform apply simultaneously — they'll both read the same state, plan independently, and one will clobber the other's changes. The fix is an S3 backend with DynamoDB locking:

backend "s3" {
  bucket         = "finbytes-quant-lab-tfstate"
  key            = "credit-risk/terraform.tfstate"
  region         = "us-east-1"
  dynamodb_table = "terraform-locks"
}

DynamoDB provides a distributed lock: before writing state, Terraform acquires a row-level lock. If someone else is mid-apply, yours waits. This is a one-time setup that prevents an entire class of infrastructure bugs.
