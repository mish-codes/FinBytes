Not Everything Needs to Be Computed Fresh

By this point in the project the pipeline has several computationally expensive steps: bootstrapping a spot curve from par yields, fitting Nelson-Siegel/Svensson parameters via Nelder-Mead, pricing bonds off the fitted curve. Most of these produce the same result if called twice with the same inputs on the same day. Without a cache, every API request recomputes from scratch. With one, the first request pays the cost and every subsequent request gets the precomputed answer in sub-millisecond time.

That's what Redis (and AWS's managed version, ElastiCache) is for: an in-memory key-value store that sits between your application and the expensive computation, holding results that are still fresh enough to use.

The Cache-Aside Pattern

The pattern is simple and widely used:

  1. Check cache. Look up the key. If it's there (cache hit), return it immediately.
  2. On miss, compute. Run the expensive operation (fit the curve, price the bond, run the simulation).
  3. Store in cache with a TTL. Set the result with a time-to-live so it auto-expires.
  4. Return result.

The caller owns step 2 — the cache module only handles 1, 3, and 4. This keeps the cache layer thin and ignorant of the business logic. It doesn't know what a yield curve is; it just stores and retrieves JSON blobs by key.

TTL Strategy: Staleness Tolerances Vary

Not all cached data goes stale at the same rate. The right TTL depends on what drives the underlying data and how wrong you can afford to be:

Data typeTTLRationale
Par/spot/forward/fitted curves24 hoursSource is daily FRED data. Curves don't change until the next publish.
Bond pricing5 minutesDepends on intraday market inputs (spreads, rates). Stale pricing is actively dangerous.
Monte Carlo VaR1 hourExpensive (thousands of simulations). Inputs shift slowly intraday. A 45-minute-old VaR is still useful; a 4-hour-old one isn't.

These TTLs are defaults. The caller can override on any individual set_curve or set_pricing call if a particular scenario warrants it (stress testing might want a 0-second TTL to force fresh computation every time).

Key Schema

Redis keys are strings. A good key schema is self-documenting and collision-proof:

curve:par:2026-04-04         → par yield curve for 4 April
curve:fitted:2026-04-04      → Nelson-Siegel fitted curve
pricing:bond-123:2026-04-04  → pricing result for bond 123

The colon-separated namespace convention is a Redis idiom — not enforced by Redis, just useful for humans and for pattern-based operations like SCAN curve:*.

Implementation

class CurveCache:
    DEFAULT_TTLS = {
        "par": 86400, "spot": 86400, "forward": 86400, "fitted": 86400,
        "pricing": 300, "var": 3600,
    }

    def __init__(self, redis_client: redis.Redis):
        self._r = redis_client

    def set_curve(self, curve_type, date_str, data, ttl=None):
        key = f"curve:{curve_type}:{date_str}"
        ttl = ttl or self.DEFAULT_TTLS.get(curve_type, 3600)
        self._safe(lambda: self._r.setex(key, ttl, json.dumps(data)))

    def get_curve(self, curve_type, date_str):
        key = f"curve:{curve_type}:{date_str}"
        result = self._safe(lambda: self._r.get(key))
        return json.loads(result) if result else None

Every operation routes through _safe, which catches Redis errors and returns a default. That's the graceful-degradation story.

Graceful Degradation: The Cache Is Not the System

This is the most important design decision in the module. If Redis goes down — ElastiCache maintenance window, network partition, config error — the application must keep working. It will be slower (every request recomputes from scratch), but it must not crash.

def _safe(self, fn, default=None):
    try:
        return fn()
    except (redis.ConnectionError, redis.TimeoutError, Exception):
        return default

Every get returns None on error (same as a cache miss). Every set silently succeeds. The caller doesn't need to know whether Redis is healthy; its cache-aside logic already handles the miss path.

The broad Exception catch looks aggressive, but it's intentional. Redis client libraries can raise a range of errors (ResponseError, DataError, AuthenticationError, etc.) and you don't want an unexpected one to take down your pricing API. Log it in production (a topic for Exercise 19 with CloudWatch), but don't raise.

Cache Invalidation

Phil Karlton's famous "two hard things in computer science" quote names cache invalidation as one of them. For this use case it's actually tractable because the invalidation trigger is well-defined: new curve data for a date.

When the daily FRED ingest lands new par yields for 2026-04-06, you invalidate all derived curves for that date in one call:

def invalidate_date(self, date_str: str):
    for curve_type in ("par", "spot", "forward", "fitted"):
        self.invalidate_curve(curve_type, date_str)

This is conservative: it wipes everything for that date, even if only the par yields changed and the fitted curve might still be valid. The cost of recomputing a Nelson-Siegel fit once is negligible compared to the risk of serving stale derived data.

Testing Without Redis

fakeredis provides an in-memory Python implementation of the Redis protocol. Tests run in ~0.1s, no Docker or server needed:

@pytest.fixture
def redis_client():
    return fakeredis.FakeRedis(decode_responses=True)

@pytest.fixture
def cache(redis_client):
    return CurveCache(redis_client)

def test_set_and_get(cache):
    cache.set_curve("par", "2026-04-04", {"1Y": 4.0, "2Y": 4.5})
    assert cache.get_curve("par", "2026-04-04") == {"1Y": 4.0, "2Y": 4.5}

The graceful-fallback test uses FakeRedis(connected=False) to simulate an unreachable Redis — cleaner than killing a real server mid-test.

ElastiCache in Production

On AWS, the managed Redis service is ElastiCache. For a lab project, a single-node cache.t3.micro (512 MB RAM, free tier eligible for 12 months) is plenty. The provisioning considerations that matter more than size:

  • VPC placement. ElastiCache nodes live inside a VPC and are not internet-accessible. Lambda functions that need to reach Redis must be VPC-attached (which means configuring subnets and security groups on the Lambda).
  • Cold-start penalty. VPC-attached Lambdas historically had ~10s cold starts. AWS fixed this with Hyperplane ENI sharing in 2019, but the first invocation in a new execution environment still takes 1-2s longer than a non-VPC Lambda.
  • Encryption in transit. ElastiCache supports TLS. Turn it on — even in a private VPC, defense in depth is worth the trivial latency cost.
