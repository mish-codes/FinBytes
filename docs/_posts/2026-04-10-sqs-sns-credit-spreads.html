Switching Lanes: From Curves to Credit

Exercises 09–14 built out the Treasury-curve side of the stack: ingestion, storage, bootstrapping, forwards, parametric fitting. Exercise 15 shifts to credit, and along the way picks up two of AWS's most-used messaging primitives: SQS (queues) and SNS (pub/sub).

The combination is natural. Credit data arrives as a stream of events — new quotes, new CDS spreads, rating changes, index updates. Most of those events need to trigger downstream work (recalculate portfolio spreads, re-run VaR, send an alert) that you don't want blocking whatever produced the event in the first place. That's what SQS and SNS are for.

Credit Spreads in One Line

A credit spread is the extra yield a corporate bond pays over a Treasury of the same maturity:

spread = y_corp − y_treasury

Measured in basis points. 1% = 100 bps. It's the market's real-time price on credit risk for that issuer (or rating bucket): how much extra yield investors demand to hold a bond that could default.

Spread behaviour follows a hierarchy that tracks the rating scale:

  • AAA: trades tight. 50–80 bps in normal markets. Almost no credit risk priced in.
  • BBB: the lowest investment-grade bucket. 120–200 bps typically. Sensitive to the cycle.
  • BB/B: high yield. 300–700 bps in calm markets; can blow out past 1000 bps in a crisis.

When the market gets scared, spreads widen across the board — investors demand more compensation to hold anything risky. The BBB spread is a classic "fear gauge" for credit. When BBBs blow past 200 bps, something is off.

The calculation itself is trivial:

def calculate_credit_spread(corporate_yield: float, treasury_yield: float) -> float:
    return round(corporate_yield - treasury_yield, 4)

The interesting part is where the inputs come from (ICE BofA indices from FRED) and what you do with the output (alerts, re-pricing, stress scenarios).

CDS and the Hazard-Rate Model

A Credit Default Swap is an insurance contract on a bond. The buyer pays a quarterly premium (the "CDS spread"); if the reference entity defaults, the seller pays out par minus recovery. The CDS spread is therefore a direct market price on default risk, stripped of the complications bond yields carry (coupon, callability, liquidity).

To translate a CDS spread into a probability of default, the standard tool is the hazard-rate model. Under a constant hazard λ (the instantaneous probability of defaulting in the next instant, given no default yet), survival and default follow an exponential:

P(survive to T) = exp(−λT)
P(default by T) = 1 − exp(−λT)

And the CDS spread relates to λ through a simple arbitrage argument:

CDS_spread ≈ λ × (1 − recovery_rate)

       CDS_spread_bps / 10000
  λ ≈ ─────────────────────────
            1 − R

Intuition: the protection seller, per unit time, expects to pay out λ × (1 − R) (probability of default times loss given default). To break even, the protection buyer's premium has to equal that. If you observe the premium, you can invert for λ.

In code:

def bootstrap_default_probabilities(
    cds_spreads: dict[str, float],
    recovery_rate: float = 0.40,
) -> dict[str, float]:
    lgd = 1.0 - recovery_rate
    probs = {}
    for tenor, spread_bps in cds_spreads.items():
        years = CDS_TENOR_YEARS.get(tenor, float(tenor.replace("Y", "")))
        hazard_rate = (spread_bps / 10000.0) / lgd
        cum_default_prob = 1.0 - np.exp(-hazard_rate * years)
        probs[tenor] = round(float(cum_default_prob), 6)
    return probs

Quick sanity check with a 100 bps CDS spread and 40% recovery (industry-convention for senior unsecured corporates):

λ = 0.01 / 0.60 ≈ 0.0167   → 1.67% annual hazard
P(default in 1Y) = 1 − exp(−0.0167 × 1) ≈ 1.65%
P(default in 5Y) = 1 − exp(−0.0167 × 5) ≈ 8.0%

The feature: default probabilities compound. A 1.67% annual hazard isn't scary; the same hazard sustained over 5 years is.

Caveats

The constant-λ approximation is what trading desks use for a quick read. For anything serious you'd want:

  • A term structure of hazards, bootstrapped from the full CDS curve (1Y, 3Y, 5Y, 7Y, 10Y) — typically piecewise-constant λ between tenors.
  • Exact cash-flow matching of the CDS premium leg and protection leg, rather than the s ≈ λ(1−R) shortcut.
  • Risky-discount integration of the premium accrual and the loss payout.

Exercise 18 will tighten this up. For now, the approximation gets us to a number we can reason about.

SQS: Decoupling Producers from Consumers

SQS is a managed queue. Producers put messages in, consumers pull them out, and AWS handles delivery, retries, and at-least-once semantics. You get two flavours:

  • Standard queues: at-least-once delivery, best-effort ordering, nearly unlimited throughput.
  • FIFO queues: exactly-once delivery, strict ordering, throughput capped at ~300 msg/s per message-group (3000 with batching).

For a credit-spread recalculation job, standard is fine — occasional duplicate recomputation isn't a correctness issue, and exact ordering across ratings doesn't matter.

Producing a message is a single boto3 call:

def send_to_queue(queue_url: str, message: dict, sqs_client=None) -> dict:
    sqs = sqs_client or boto3.client("sqs")
    return sqs.send_message(
        QueueUrl=queue_url,
        MessageBody=json.dumps(message),
    )

The consumer side (not in this exercise — next time with Lambda triggers) polls for messages, processes them, and deletes them on success. Anything that fails repeatedly lands in a Dead Letter Queue after maxReceiveCount attempts, so you can investigate without the message clogging the main queue forever.

SNS: Fan-Out Pub/Sub

SNS is the pub/sub counterpart. Publishers push messages to a topic; all subscribers receive them. Subscribers can be email addresses, SMS, HTTP endpoints, SQS queues, Lambdas.

The classic use case is fan-out: one event needs to trigger several independent consumers. You publish once to SNS, and SNS delivers to each subscribed SQS queue. Each consumer team owns their own queue, their own retry policy, their own DLQ — without the publisher knowing anything about them.

For credit-spread alerts, I started simple — one SNS topic, one email subscription:

def publish_alert(topic_arn: str, message: str, sns_client=None) -> dict:
    sns = sns_client or boto3.client("sns")
    return sns.publish(
        TopicArn=topic_arn,
        Subject="Credit Spread Alert",
        Message=message,
    )

The threshold check that triggers it is equally terse:

def check_spread_threshold(spread_bps: float, threshold: float) -> bool:
    return spread_bps > threshold

Wired together: fetch ICE BofA BBB spread from FRED → compare to threshold (say, 150 bps) → if over, publish to SNS → email lands in my inbox. Later you can add a second SQS subscriber to the same topic for a recalc pipeline, a third for a Slack bot, without touching the publisher.

Testing Against AWS Without Touching AWS

One of the best side effects of boto3's client interface is that it mocks cleanly. moto patches the underlying HTTP calls so boto3.client("sqs") talks to an in-memory fake instead of real AWS. Tests run in milliseconds, don't need credentials, don't cost anything, and exercise real serialisation paths.

from moto import mock_aws

class TestSQS:
    @mock_aws
    def test_send_to_queue(self):
        sqs = boto3.client("sqs", region_name="us-east-1")
        queue = sqs.create_queue(QueueName="credit-calc-queue")
        queue_url = queue["QueueUrl"]

        send_to_queue(
            queue_url=queue_url,
            message={"rating": "BBB", "spread": 180},
            sqs_client=sqs,
        )

        msgs = sqs.receive_message(QueueUrl=queue_url, MaxNumberOfMessages=1)
        body = json.loads(msgs["Messages"][0]["Body"])
        assert body["rating"] == "BBB"
        assert body["spread"] == 180

The dependency-injection detail matters: send_to_queue accepts an optional sqs_client, defaulting to boto3.client("sqs"). In production nothing changes; in tests, we pass the moto-patched client explicitly so the function and the test agree on which fake they're talking to.
