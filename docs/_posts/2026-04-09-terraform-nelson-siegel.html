Two Things in One Exercise

This post covers two very different pieces of quant infrastructure, bundled into a single exercise because they come up together in practice: Terraform for describing AWS resources as code, and parametric yield curve fitting (Nelson-Siegel and Svensson) for smoothing noisy market data into economically interpretable curves.

The underlying theme is the same: replace point-in-time artefacts with reusable, versionable descriptions. Terraform turns one-off AWS CLI commands into a repeatable config. Nelson-Siegel turns a scatter of quoted yields into four or six numbers that describe the entire term structure.

Nelson-Siegel: Four Parameters, Whole Curve

The Nelson-Siegel model (1987) represents a yield curve as:

y(τ) = β₀ + β₁ · [(1 − e^(−τ/λ)) / (τ/λ)]
           + β₂ · [(1 − e^(−τ/λ)) / (τ/λ) − e^(−τ/λ)]

Every parameter has an economic meaning:

  • β₀ — the long-term level. As τ → ∞, y(τ) → β₀. This is the market's long-run inflation + real-rate expectation.
  • β₁ — the slope. As τ → 0, y(τ) → β₀ + β₁. A negative β₁ means a normal (upward) curve.
  • β₂ — the curvature. Creates a hump at intermediate maturities. Driven by uncertainty about medium-term rates.
  • λ — the decay factor. Controls where the hump peaks. A smaller λ moves the hump to shorter maturities.

You fit it by minimising the sum of squared errors between the model's predicted yields and your observed yields:

def fit_nelson_siegel(maturities, yields):
    def objective(params):
        b0, b1, b2, lam = params
        if lam <= 0.01:
            return 1e10
        fitted = np.array([nelson_siegel(t, b0, b1, b2, lam) for t in maturities])
        return np.sum((fitted - yields) ** 2)

    b0_init = float(yields[-1])                # level ≈ long rate
    b1_init = float(yields[0] - yields[-1])    # slope ≈ short - long
    x0 = [b0_init, b1_init, 0.0, 1.5]

    result = minimize(objective, x0, method="Nelder-Mead",
                      options={"maxiter": 10000, "xatol": 1e-8})
    ...

Nelder-Mead is a gradient-free simplex search — slow but robust on noisy four-dimensional objective surfaces. For daily fitting it's fine; for a research codebase doing thousands of fits you'd switch to a faster method with gradients.

Svensson: One More Hump

Svensson (1994) adds a second hump term, giving 6 parameters total. The extra flexibility helps fit long-end twists in real Treasury curves:

y(τ) = NS(τ) + β₃ · [(1 − e^(−τ/λ₂)) / (τ/λ₂) − e^(−τ/λ₂)]

When β₃ = 0, Svensson reduces to Nelson-Siegel exactly — so fitting Svensson on data where NS is sufficient just pushes β₃ toward 0 and wastes two degrees of freedom. Use NS first; only switch to NSS if the residuals show a systematic long-end pattern.

The ECB and the Bundesbank publish NSS parameters daily for the euro-area yield curve. Analysts track β₀ over time to see long-run rate expectations evolve — something that's hard to do with a raw yield curve.

Cubic Spline: The Non-Parametric Alternative

Cubic splines pass exactly through every observed point. No residuals, no fit quality — it just interpolates. Useful when you need "the yield at 7.5 years" without caring about the underlying model:

from scipy.interpolate import CubicSpline

def cubic_spline_interpolate(maturities, yields, target_maturities):
    cs = CubicSpline(maturities, yields)
    return cs(target_maturities)

The trade-off: splines can oscillate between knots, especially with sparse data. Nelson-Siegel smooths through noise because the functional form is constrained. A spline will happily fit every bump and wiggle, including the ones caused by a single thinly-traded bond.

Terraform: Infrastructure as Code

Through exercises 10–13 we built AWS resources with the CLI: aws s3 mb, aws lambda create-function, aws apigateway create-rest-api, aws apigateway put-method, etc. Each command was a one-off action — there's no record in the repo of what was created, what parameters were used, or how to recreate it.

Terraform flips this around. You describe the desired state in .tf files, and Terraform figures out the diff against what exists and makes it happen. The config becomes the source of truth.

Scoping the blast radius

A tempting mistake with Terraform on an existing account: import everything. Write .tf files for all existing resources, import them into state, and now Terraform "manages" everything. Sounds clean — but if you run terraform destroy in the wrong directory, you delete everything, including the database you still care about.

For this exercise I scoped Terraform narrowly: only the new curve-fitting Lambda and its HTTP API are Terraform-managed. Existing resources (S3 bucket, shared IAM role, the REST API with bootstrap/forwards routes) stay outside Terraform state. Going forward, all new infrastructure is Terraform-managed.

The config:

resource "aws_lambda_function" "curve_fitting" {
  function_name = "${var.project_name}-curve-fitting"
  runtime       = "python3.11"
  handler       = "handler.lambda_handler"
  role          = var.lambda_role_arn   # EXISTING role, referenced by ARN
  timeout       = 60
  memory_size   = 512

  filename         = "${path.module}/../src/deployment.zip"
  source_code_hash = filebase64sha256("${path.module}/../src/deployment.zip")
}

resource "aws_apigatewayv2_api" "curves" {
  name          = "${var.project_name}-curves-http"
  protocol_type = "HTTP"
}

resource "aws_apigatewayv2_route" "fit" {
  api_id    = aws_apigatewayv2_api.curves.id
  route_key = "POST /curves/fit"
  target    = "integrations/${aws_apigatewayv2_integration.curve_fitting.id}"
}

The source_code_hash is the trick that makes Terraform detect when you've changed the Lambda code — if the zip changes, the hash changes, and terraform apply pushes the new code.

HTTP API vs REST API

I used an HTTP API (apigatewayv2) here rather than the REST API (apigatewayv1) from exercise 12. HTTP APIs are cheaper, simpler, and the default recommendation for new APIs. The only reasons to stick with REST are older features (custom domains with edge-optimised distributions, request validators, AWS WAF integration).

For production I'd probably standardise on one — mixing HTTP + REST is extra surface area. This project keeps both for learning purposes, which is fine in a lab.

Packaging scipy for Lambda (the friction log)

The one painful bit: scipy's compiled wheels need to match Lambda's runtime (Linux, x86_64, manylinux). If you pip install scipy -t package/ on Windows, you ship Windows wheels and the Lambda crashes at import with "invalid ELF". The clean fix is to package inside the official Lambda Python image — but actually running that command on a Windows / Git Bash host turned up four gotchas the tutorials never mention.

Gotcha 1: The Lambda image has an entrypoint

public.ecr.aws/lambda/python:3.11 expects the Lambda handler name as its first argument. A plain bash -c "..." fails with "entrypoint requires the handler name to be the first argument." You have to override the entrypoint explicitly:

docker run --rm --entrypoint bash -v "$PWD":/out public.ecr.aws/lambda/python:3.11 -c "..."

Gotcha 2: Git Bash path translation silently breaks the mount

On Git Bash for Windows, $PWD returns a POSIX path like /c/codebase/quant_lab/.../src, which MSYS then mangles when passed to Docker. The container starts, the mount "succeeds", and then /out is empty inside the container. No error — your copies just silently fail.

The fix is two steps: use pwd -W for the Windows-style path, and disable MSYS path conversion:

WINPATH=$(pwd -W)   # C:/codebase/quant_lab/.../src
MSYS_NO_PATHCONV=1 docker run --rm --entrypoint bash \
  -v "${WINPATH}:/out" public.ecr.aws/lambda/python:3.11 -c "..."

Gotcha 3: pip will build numpy from source if the version is too new

The naive pip install numpy scipy grabs the latest numpy, which may not have a cp311 manylinux wheel yet. pip then falls back to building from source, which fails because the Lambda base image has no C compiler. You'll see a Meson error about missing cc, gcc, clang.

Force binary-only installs and pin versions known to have wheels for your runtime:

pip install --only-binary=:all: 'numpy<2' 'scipy<1.14' -t /out/package

That lands numpy-1.26.4 + scipy-1.13.1, both with manylinux_2_17_x86_64 wheels.

Gotcha 4: The container has no zip binary

The Lambda base image is minimal — no zip. Use Python to create the archive, which is always available:

python -c "import shutil; shutil.make_archive('/out/deployment', 'zip', '.')"

Gotcha 5: 50 MB is a hard ceiling for direct upload

First build weighed 66 MB, which exceeds Lambda's 50 MB limit for direct zip upload (the 250 MB limit is for unzipped code, or zips uploaded via S3). Terraform's filename = attribute does a direct upload, so the apply would have failed at the Lambda creation step.

Trimming the usual suspects got it down to 48.3 MiB:

cd /out/package
find . -type d -name 'tests' -exec rm -rf {} +
find . -type d -name '__pycache__' -exec rm -rf {} +
find . -type d -name '*.dist-info' -exec rm -rf {} +
find . -name '*.pyc' -delete
find . -name '*.pyi' -delete

scipy's tests/ trees alone carry around 15 MB of fixture data you'll never use at runtime. If a future scipy/numpy bump pushes the zip back over 50 MB, the next step is switching Terraform to upload via S3 (s3_bucket + s3_key attributes) or moving scipy into a Lambda layer and referencing it from multiple functions.

The full build command, end to end

WINPATH=$(pwd -W)
MSYS_NO_PATHCONV=1 docker run --rm --entrypoint bash \
  -v "${WINPATH}:/out" public.ecr.aws/lambda/python:3.11 \
  -c "pip install --only-binary=:all: 'numpy<2' 'scipy<1.14' -t /out/package -q && \
      cp /out/handler.py /out/curve_fitting.py /out/package/ && \
      cd /out/package && \
      find . -type d -name 'tests' -exec rm -rf {} + 2>/dev/null; \
      find . -type d -name '__pycache__' -exec rm -rf {} + 2>/dev/null; \
      find . -type d -name '*.dist-info' -exec rm -rf {} + 2>/dev/null; \
      find . -name '*.pyc' -delete; find . -name '*.pyi' -delete; \
      python -c \"import shutil; shutil.make_archive('/out/deployment','zip','.')\""

Deployment: terraform apply

Once the zip exists, the rest is unsurprising:

cd terraform
terraform init      # pulls hashicorp/aws 5.x provider
terraform plan      # review: 6 resources to add, 0 to change, 0 to destroy
terraform apply

Six resources come up in about 20 seconds — the Lambda takes most of the time (zip upload + cold init), the API Gateway v2 resources are nearly instant. Terraform's output block prints the live endpoint:

api_endpoint = "https://rckgnjrj4c.execute-api.us-east-1.amazonaws.com/dev/curves/fit"

Smoke-testing the Svensson model on a 10-point US Treasury-shaped curve:

curl -X POST $API_ENDPOINT -H "Content-Type: application/json" -d '{
  "model":"svensson",
  "maturities":[0.25,0.5,1.0,2.0,3.0,5.0,7.0,10.0,20.0,30.0],
  "yields":    [4.80,4.90,5.10,5.20,5.15,4.90,4.70,4.50,4.40,4.45]
}'

{
  "model": "svensson",
  "params": {
    "beta0": 5.1174, "beta1": -0.509, "beta2": -2.4821,
    "beta3": 2.655, "lambda1": 9.0585, "lambda2": 1.4405
  },
  "fitted": [4.7872, 4.923, 5.0964, 5.1967, 5.1387, 4.9066, 4.7016, 4.5088, 4.3757, 4.4652],
  "rmse": 0.0134
}

RMSE of 0.0134 — 1.34 basis points across the curve. β₀ ≈ 5.12% is the long-run level, β₁ ≈ −0.51% is the (mild) inverted slope at the short end, and the two curvature terms (β₂, β₃) capture the belly dip and the 20s hump. Exactly the sort of summary you'd want to track day-over-day as market expectations shift.

Teardown is one command (terraform destroy), and because Terraform state is scoped only to the six new resources, the existing S3 bucket / shared IAM role / REST API are untouched.

Testing Pattern: Avoiding Module Collisions

Small engineering note: exercises 12, 13, and 14 each define a top-level handler module. When all three are installed as editable packages, import handler in test code might resolve to whichever one Python discovers first — typically not what you want.

The fix is a one-line addition to pyproject.toml:

[tool.pytest.ini_options]
testpaths = ["tests"]
pythonpath = ["src"]

pythonpath prepends the exercise's own src/ directory to sys.path before test collection, so local modules win regardless of install order. Simple, but easy to miss until imports start resolving strangely.
