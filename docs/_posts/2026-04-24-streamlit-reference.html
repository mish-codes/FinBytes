Why Streamlit Over React

The QuantLab scanner needed a frontend. The options were: build a React app (requires JavaScript, npm, webpack, a separate hosting service), or use Streamlit (pure Python, free hosting on Streamlit Cloud, built-in charting). For a data-focused dashboard built by a Python developer, Streamlit wins on every dimension that matters.

Trade-offs: Streamlit reruns the entire script on every interaction. You cannot build complex SPAs. The styling options are limited. But for a portfolio risk scanner dashboard, none of that matters.

Basic Layout

A Streamlit app is a single Python script. Streamlit reads it top to bottom and renders each call as a UI element:

import streamlit as st

st.set_page_config(
    page_title="FinBytes QuantLabs",
    page_icon="chart_with_upwards_trend",
    layout="wide",
)

st.title("Stock Risk Scanner")
st.markdown("Analyze portfolio risk metrics for any combination of tickers.")

Columns

col1, col2, col3, col4 = st.columns(4)
col1.metric("VaR (95%)", f"{var_pct:.2f}%")

col3.metric("Max Drawdown", f"{max_dd:.2f}%")

Session State

Because Streamlit reruns the entire script on every interaction, you lose all local variables. Session state persists data across reruns:

# Initialize (only runs once)
if "scan_count" not in st.session_state:
    st.session_state.scan_count = 0
    st.session_state.scan_reset_time = datetime.now()

# Update
st.session_state.scan_count += 1

# Read
st.write(f"Scans this session: {st.session_state.scan_count}")

The scanner dashboard uses session state for rate limiting (maximum scans per minute) and caching the latest scan result so it does not disappear when the user interacts with a widget.

Secrets Management

Streamlit Cloud has built-in secrets management. You define secrets in the Streamlit Cloud dashboard, and access them in code:

# Access a secret
API_URL = st.secrets.get("API_URL", "http://localhost:8000")

For local development, create a .streamlit/secrets.toml file:

# .streamlit/secrets.toml (add to .gitignore!)
API_URL = "http://localhost:8000"

In CI/CD tests, you can create the file on the fly:

echo 'API_URL = "http://localhost:8000"' > .streamlit/secrets.toml

Multi-Page Apps

Streamlit supports multiple pages by placing Python files in a pages/ directory. The filename determines the page order and title:

dashboard/
    app.py                        # home page (System Health)
    pages/
        1_Stock_Risk_Scanner.py   # main scanner page
        99_Admin.py               # admin page
    lib/
        api_client.py
        charts.py
    assets/
        logo.png
    requirements.txt

The number prefix controls ordering in the sidebar. The underscore in the filename becomes a space in the page title. 1_Stock_Risk_Scanner.py appears as "Stock Risk Scanner".

You can customize the sidebar navigation with explicit page links:

# In app.py (or any page)
st.sidebar.page_link("pages/1_Stock_Risk_Scanner.py", label="Stock Risk Scanner", icon="chart_with_upwards_trend")
st.sidebar.page_link("app.py", label="System Health", icon="stethoscope")

Calling External APIs

The dashboard calls the FastAPI scanner backend using the requests library. The polling pattern for background scans:

import requests
import time

def submit_scan(api_url: str, tickers: list, weights: list, period: str) -> dict:
    resp = requests.post(
        f"{api_url}/api/scan",
        json={"tickers": tickers, "weights": weights, "period": period},
        timeout=10,
    )
    resp.raise_for_status()
    return resp.json()  # {"id": 42, "status": "pending"}


def poll_scan(api_url: str, scan_id: int, max_wait: int = 60) -> dict:
    for _ in range(max_wait):
        resp = requests.get(f"{api_url}/api/scans/{scan_id}", timeout=10)
        data = resp.json()
        if data["status"] in ("complete", "failed"):
            return data
        time.sleep(1)
    raise TimeoutError(f"Scan {scan_id} did not complete in {max_wait}s")

Deployment to Streamlit Cloud

Deployment is free and takes under five minutes:

  1. Push your code to a public GitHub repository.
  2. Go to share.streamlit.io and sign in with GitHub.
  3. Select the repo, branch, and the main file (dashboard/app.py).
  4. Add secrets in the Streamlit Cloud dashboard (Settings → Secrets).
  5. Click Deploy.

Streamlit Cloud watches the branch and auto-redeploys on every push. The free tier gives you one public app with reasonable resources.

Health Checks and Status Display

The scanner dashboard's home page shows system health by querying GitHub Actions and the API:

st.title("System Health")

# Check API health
try:
    r = requests.get(f"{API_URL}/health", timeout=5)
    if r.status_code == 200:
        st.success("API: Connected")
    else:
        st.error(f"API: Status {r.status_code}")
except requests.exceptions.ConnectionError:
    st.error("API: Unreachable")

# Check GitHub CI status
try:
    r = requests.get(
        f"https://api.github.com/repos/{GITHUB_REPO}/actions/runs",
        params={"per_page": 3},
        timeout=10,
    )
    runs = r.json().get("workflow_runs", [])
    if runs:
        latest = runs[0]
        if latest.get("conclusion") == "success":
            st.success(f"CI: Passing ({latest['name']})")
        else:
            st.warning(f"CI: {latest.get('conclusion', 'running')}")
except Exception:
    st.warning("CI: Could not reach GitHub API")

Running Locally

# Install dependencies
pip install -r requirements.txt

# Create local secrets
mkdir -p .streamlit
echo 'API_URL = "http://localhost:8000"' > .streamlit/secrets.toml

# Run the app
streamlit run app.py

# Opens browser at http://localhost:8501

Things That Tripped Me Up

  • Full script rerun: Every button click, slider move, or text input change reruns the entire script top to bottom. If you have an expensive API call at the top level, it fires on every interaction. Wrap expensive calls in @st.cache_data or guard them with if st.button().
  • st.cache_data vs st.cache_resource: Use @st.cache_data for data (DataFrames, API responses). Use @st.cache_resource for connections (database engines, HTTP clients). Mixing them up causes serialization errors.
  • Button state is ephemeral: st.button() returns True only on the click event, then False on the next rerun. If you need to remember that a button was clicked, store the state in st.session_state.
  • File paths on Streamlit Cloud: __file__ resolves differently on Cloud vs local. Use Path(__file__).parent for relative paths to assets, not hardcoded paths.
  • Secrets format: .streamlit/secrets.toml uses TOML, not JSON or YAML. String values need quotes. Nested sections use [section] headers. If the format is wrong, Streamlit silently returns empty values instead of raising an error.
  • Sleep in Streamlit blocks everything: time.sleep(1) in a polling loop freezes the entire UI. Use st.spinner() to show a loading indicator, and keep the total wait time reasonable.
  • Auto-deploy surprises: Streamlit Cloud redeploys on every push to the tracked branch. If you push broken code, the live app goes down immediately. Use a separate branch for development and only merge to the tracked branch when ready.

Testing with AppTest

Streamlit provides AppTest in streamlit.testing.v1 — a headless runner that executes your page script and lets you inspect the rendered elements without a browser. Combined with pytest, it gives you fast, deterministic frontend tests.

Basic Smoke Test

The simplest test: load the page and check it does not crash.

from streamlit.testing.v1 import AppTest

class TestStockAnalysis:
    def _run(self):
        at = AppTest.from_file("pages/22_Stock_Analysis.py", default_timeout=15)
        at.run()
        return at

    def test_loads_without_error(self):
        at = self._run()
        assert not at.exception, f"Page crashed: {at.exception}"

AppTest.from_file takes the path relative to the working directory (usually dashboard/). Set default_timeout high enough for pages that do computation on load. If the page raises any exception, at.exception will contain it.

Inspecting Elements

After at.run(), you can query every Streamlit element by type:

def test_shows_title(self):
    at = self._run()
    assert any("Technical Analysis" in t.value for t in at.title)

def test_has_ticker_input(self):
    at = self._run()
    ti = [w for w in at.text_input if w.value == "AAPL"]
    assert ti, "Expected a ticker text_input defaulting to AAPL"

def test_has_period_selectbox(self):
    at = self._run()
    labels = [s.label for s in at.selectbox]
    assert any("Period" in lbl for lbl in labels)

def test_has_confidence_slider(self):
    at = self._run()
    sliders = [s for s in at.slider if s.min == 90 and s.max == 99]
    assert sliders, "Expected a confidence slider with range 90-99"

def test_has_tabs(self):
    at = self._run()
    assert len(at.tabs) >= 2

def test_checkbox_checked_by_default(self):
    at = self._run()
    sma_cb = [c for c in at.checkbox if "SMA" in c.label][0]
    assert sma_cb.value is True

Available element lists: at.title, at.markdown, at.text_input, at.selectbox, at.slider, at.button, at.checkbox, at.radio, at.tabs, at.dataframe, at.number_input, at.exception, and more. Each item exposes .value, .label, and type-specific attributes.

Mocking External Dependencies

AppTest runs the actual page script, which means it will call yfinance, requests, boto3, and anything else your page imports. You need autouse fixtures in conftest.py to intercept these:

# conftest.py
import pytest
from unittest.mock import patch, MagicMock
import pandas as pd
import numpy as np

@pytest.fixture(autouse=True)
def mock_set_page_config():
    """Make st.set_page_config a no-op so pages don't crash in AppTest."""
    with patch("streamlit.set_page_config"):
        yield

@pytest.fixture(autouse=True)
def mock_yfinance(sample_ohlcv_df):
    """Patch yfinance.download so no test hits the network."""
    def fake_download(tickers, **kwargs):
        return sample_ohlcv_df
    with patch("yfinance.download", side_effect=fake_download):
        yield

@pytest.fixture(autouse=True)
def mock_requests():
    """Patch requests.get for all external HTTP calls."""
    def fake_get(url, **kwargs):
        resp = MagicMock()
        resp.status_code = 200
        resp.json.return_value = {}
        return resp
    with patch("requests.get", side_effect=fake_get):
        yield

Key points:

  • st.set_page_config must be patched — it raises an error if called more than once in a session, and AppTest triggers it on every at.run()
  • Use autouse=True so every test gets the mocks automatically. Pages import at the module level, so if a mock is missing, the page crashes before your test even starts.
  • Match the mock to the URL — if your page calls different APIs, use the URL in fake_get to return appropriate fake data for each one
  • Provide realistic fake data — a 252-row OHLCV DataFrame with realistic price ranges prevents NaN and division-by-zero errors in pages that compute indicators

Testing Pattern per Page

The QuantLab dashboard follows a consistent pattern: one test class per page, with a shared _run() helper that calls AppTest. Tests then check for expected elements:

class TestVaRCVaR:
    def _run(self):
        at = AppTest.from_file("pages/30_VaR_CVaR.py", default_timeout=15)
        at.run()
        return at

    def test_loads_without_error(self):
        at = self._run()
        assert not at.exception, f"Page crashed: {at.exception}"

    def test_shows_title(self):
        at = self._run()
        assert any("Value at Risk" in t.value for t in at.title)

    def test_has_ticker_input(self):
        at = self._run()
        ti = [w for w in at.text_input if w.value == "AAPL"]
        assert ti, "Expected a ticker text_input defaulting to AAPL"

Every page gets at least a smoke test (test_loads_without_error) and a title check. Pages with inputs get widget assertion tests. This catches two categories of bugs: import/runtime errors, and broken UI contracts (e.g., a renamed widget that breaks downstream logic).

CI Integration

The QuantLab CI runs AppTest tests on every push and generates a JSON report that the dashboard itself displays in a Tests tab per page:

# In GitHub Actions
pip install pytest pytest-json-report
echo 'API_URL = "http://localhost:8000"' > .streamlit/secrets.toml
python -m pytest tests/ -v --tb=short --json-report --json-report-file=test_results.json

The test_results.json file is committed back via a CI-generated PR, so the deployed dashboard always shows the latest test results without running tests at runtime.

Things That Tripped Me Up

  • st.set_page_config crashes: If you forget to patch it, AppTest fails with "can only be called once." Add the autouse fixture and forget about it.
  • Sidebar elements are separate: at.text_input includes both main area and sidebar widgets. If you need to distinguish, check .label rather than position.
  • Tabs render all content: AppTest executes all tab bodies, not just the active one. This is good for testing but means all tabs need their mocks in place.
  • Slow pages need higher timeouts: The default timeout is 3 seconds. Pages with heavy computation or multiple chart renders need default_timeout=15 or 30.
  • Bokeh version mismatch: st.bokeh_chart only supports Bokeh 2.4.x. If you use a newer Bokeh, render via components.html(file_html(fig, CDN)) instead — AppTest handles this fine.
