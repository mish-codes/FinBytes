Moving Beyond S3 JSON Files

In the previous post, we wrote daily yield curves to S3 as dated JSON objects. That works for immutable snapshots, but it doesn't help when you want to answer questions like "how did the 10Y yield evolve in Q1?" or "give me every curve between two dates". For those, we need a database.

This post covers setting up AWS RDS PostgreSQL, designing the schema for yield curves and bonds, and building async SQLAlchemy models with an in-memory SQLite test suite that runs without AWS.

Why RDS Over Self-Managed PostgreSQL

RDS (Relational Database Service) is AWS's managed database service. You still design the schema, write queries, and manage indexes — but AWS handles patching, backups, monitoring, and failover. For a learning project, the practical benefits are:

  • Free tier: 750 hours/month of db.t3.micro for the first 12 months
  • Automated backups: daily snapshots retained for 1-35 days
  • One-click scaling: change instance size without rebuilding

The trade-off is cost per hour once you're past the free tier (about $15/month for a t3.micro running 24/7). One lesson learned: stop or delete the instance when you're not actively using it, especially if you're running multiple RDS databases in parallel.

A small helper: scripts/rds.py

"Remember to stop RDS" is one of those things every AWS guide tells you, and that everyone forgets. A tiny Python CLI using boto3 makes it one command:

python scripts/rds.py status         # show current state
python scripts/rds.py stop --wait    # stop and block until stopped
python scripts/rds.py start --wait   # start and block until available

The script wraps describe_db_instances, stop_db_instance, and start_db_instance. It checks current status before acting (so calling stop on an already-stopped instance is a no-op, not an error), and optionally polls until the target state is reached. Config comes from env vars with sensible defaults — no command-line flags needed for the normal case.

def _describe(rds, instance_id):
    try:
        response = rds.describe_db_instances(DBInstanceIdentifier=instance_id)
        return response["DBInstances"][0]
    except ClientError as e:
        if e.response["Error"]["Code"] == "DBInstanceNotFound":
            return None
        raise


def cmd_stop(args):
    rds = _client(args.profile, args.region)
    instance = _describe(rds, args.instance)
    status = instance["DBInstanceStatus"]

    if status in ("stopped", "stopping"):
        print(f"Instance is already {status}. No action.")
        return 0
    if status != "available":
        print(f"Cannot stop — current status is {status!r}.")
        return 1

    rds.stop_db_instance(DBInstanceIdentifier=args.instance)
    print(f"Stop requested for {args.instance}.")
    if args.wait:
        _wait_for_status(rds, args.instance, "stopped", timeout_min=10)

Start and stop operations on RDS are asynchronous — the API returns immediately and the instance transitions through intermediate states (stopping → stopped, starting → available). The --wait flag polls every 15 seconds so scripts can sequence operations without racing.

This feels like a small thing, but at $0.017/hr for a t3.micro, 24 hours of forgetting to stop adds up to $0.40/day. Over a month of on-and-off learning, that's the difference between free-tier and a surprise bill.

  --db-instance-identifier quant-lab-db \
  --db-instance-class db.t3.micro \

  --master-username quantlab \

  --publicly-accessible \
  --backup-retention-period 1 \
  --no-multi-az \

  --profile quant-lab

A security group controls who can reach the database on port 5432. For learning, we allow access from a single developer IP; production setups would restrict it to Lambda's VPC or private subnets only.

Schema Design for Time-Series Financial Data

Fixed income data has specific characteristics that shape schema decisions:

  1. Point-in-time queries dominate: "give me the curve for 2026-04-04"
  2. No look-ahead bias: a query as of April 1 must not see data that only existed on April 5
  3. Curves are compound objects: 11 maturities per day per curve type

The first two points argue strongly for a unique index on the date column. The third raises a design question: do we store one row per (date, maturity), or one row per date with a JSON payload?

JSON vs Normalised

DimensionJSON blobOne row per maturity
SimplicityWinExtra join per read
Schema evolutionAdd maturities anytimeNeeds migration
AggregationsPost-process in PythonNative SQL aggregates
StorageSlightly largerMore compact

For a tiny payload (11 numbers per day), the simplicity of JSON wins. If we were doing cross-maturity aggregations in SQL ("average 10Y yield by month"), the normalised version would be better. For our use case — point-in-time reads — JSON is the right call.

The Models

Six tables cover the domain: one for each curve type, plus bond definitions and pricing results:

from sqlalchemy import String, Float, Integer, Date, DateTime, Text
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column


class Base(DeclarativeBase):
    pass


class ParYieldRecord(Base):
    __tablename__ = "par_yields"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    curve_date: Mapped[date] = mapped_column(Date, unique=True, index=True)
    yields_json: Mapped[str] = mapped_column(Text)
    source: Mapped[str] = mapped_column(String(20), default="FRED")
    created_at: Mapped[datetime] = mapped_column(
        DateTime, default=lambda: datetime.now(UTC)
    )

The unique=True on curve_date enforces "one curve per day" at the database level — no ambiguous duplicates. The index=True makes date lookups fast (which is every read).

BondRecord is keyed by ISIN (the global 12-character identifier for a security):

class BondRecord(Base):
    __tablename__ = "bonds"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    isin: Mapped[str] = mapped_column(String(12), unique=True, index=True)
    coupon: Mapped[float] = mapped_column(Float)
    maturity: Mapped[date] = mapped_column(Date)
    face_value: Mapped[float] = mapped_column(Float, default=1000.0)
    issue_date: Mapped[date] = mapped_column(Date)
    frequency: Mapped[int] = mapped_column(Integer, default=2)  # coupons per year
    bond_type: Mapped[str] = mapped_column(String(20), default="treasury")
    is_callable: Mapped[bool] = mapped_column(Boolean, default=False)
    created_at: Mapped[datetime] = mapped_column(
        DateTime, default=lambda: datetime.now(UTC)
    )

Note is_callable rather than callable — callable() is a Python built-in we don't want to shadow.

Async CRUD with Upsert

Saving yields is an upsert — if the date exists, update; otherwise insert. This matches the real-world workflow where the daily ingestion script might re-run for the same date:

async def save_par_yields(
    session: AsyncSession, curve_date: date, yields: dict[str, float]
) -> ParYieldRecord:
    existing = await session.execute(
        select(ParYieldRecord).where(ParYieldRecord.curve_date == curve_date)
    )
    record = existing.scalar_one_or_none()

    if record:
        record.yields_json = json.dumps(yields)
    else:
        record = ParYieldRecord(
            curve_date=curve_date, yields_json=json.dumps(yields)
        )
        session.add(record)

    await session.commit()
    return record

Range queries are a straightforward WHERE-BETWEEN with ordering:

async def get_par_yields_range(
    session: AsyncSession, start: date, end: date
) -> list[dict]:
    result = await session.execute(
        select(ParYieldRecord)
        .where(ParYieldRecord.curve_date >= start)
        .where(ParYieldRecord.curve_date <= end)
        .order_by(ParYieldRecord.curve_date)
    )
    records = result.scalars().all()
    return [
        {"date": r.curve_date, "yields": json.loads(r.yields_json)}
        for r in records
    ]

Testing Without RDS: In-Memory SQLite

We don't need RDS to test our async SQL code. SQLAlchemy's async engine supports SQLite via aiosqlite, and :memory: gives us a throwaway database per test fixture:

@pytest.fixture
async def session():
    engine = create_async_engine("sqlite+aiosqlite:///:memory:")
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    factory = async_sessionmaker(engine, expire_on_commit=False)
    async with factory() as s:
        yield s
    await engine.dispose()

Every test gets a fresh database. The schema is created from the same Base.metadata that'd be used in production. The same SQLAlchemy code runs against SQLite in tests and PostgreSQL in production — the async engine abstracts the difference.

One caveat: SQLite doesn't enforce every constraint PostgreSQL would (e.g. some NOT NULL coercions differ). For most ORM-level tests this is fine; for constraint-specific tests, run against a real PostgreSQL via a Docker container.

Module Collision Lesson

Exercise 08 also had models.py and db.py files. Installing both exercises as editable Python packages caused import collisions — tests from exercise 11 were finding exercise 08's db module first.

The fix: scope each exercise's code under a package with a unique name. Exercise 11 lives at src/yield_store/, so imports become from yield_store.db import save_par_yields. This is a good general principle — modules named db or models are practically guaranteed to collide in a monorepo.
