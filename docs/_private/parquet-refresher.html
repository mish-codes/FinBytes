A fast refresher on Apache Parquet — what it is, why it's fast, and the bits that come up in interviews. Parquet is an open, columnar, binary on-disk format for analytical data: store columns together, not rows, so analytical queries read only what they need and compress hard.

Why columnar

  • Projection pushdown — read only the columns a query touches (skip the rest entirely). A row format (CSV, JSON, Avro) must read every row in full.
  • Compression — a column holds one data type with similar values, so encodings and codecs work far better than on mixed rows.
  • Vectorized reads — contiguous same-type values feed columnar engines (Arrow, DuckDB, Spark) efficiently.

Physical layout

A Parquet file is a sequence of row groups, each split into one column chunk per column, each chunk made of pages, with a footer holding the schema and all metadata at the end of the file:

File
 ├─ Row group 0
 │   ├─ Column chunk: col_a  → pages (+ min/max stats)
 │   ├─ Column chunk: col_b  → pages
 │   └─ ...
 ├─ Row group 1
 │   └─ ...
 └─ Footer  (schema, row-group offsets, per-column statistics)  + magic "PAR1"
  • Row group — a horizontal slice of rows (commonly ~128 MB). The unit of parallelism and of skipping.
  • Column chunk — one column's data within a row group.
  • Page — the smallest encoded/compressed unit; carries its own encoding.
  • Footer — read first (it's at the end): gives the schema and the offsets/stats so a reader can jump straight to the columns and row groups it needs.

What makes it small & fast

  • Encodings — dictionary (map repeated values to small ints; great for low-cardinality strings), run-length + bit-packing (RLE), and delta encodings. Chosen per column/page.
  • Compression codecs — applied on top of encoding: snappy (default, fast), zstd (better ratio, now common), gzip, lz4.
  • Statistics + predicate pushdown — each row group/page stores min/max/null counts per column. A filter like WHERE date > X lets the reader skip whole row groups whose max < X without decoding them. (Sort your data on the filter column to make this powerful.)

Partitioning

Hive-style directory partitioning encodes a column's value in the path, so the engine prunes whole folders before opening any file:

data/year=2026/month=06/part-0.parquet

Partition on low-cardinality columns you filter on (date, region). Over-partitioning creates the small-files problem — thousands of tiny files kill read performance (per-file overhead, tiny row groups). Aim for reasonably large files/row groups.

Nested data

Parquet stores nested/repeated structures (structs, lists, maps) by shredding them into columns using definition levels (is this optional field null, and at what depth) and repetition levels (where a repeated element starts) — the Dremel model. You rarely touch this directly, but it's why Parquet handles JSON-like data without storing rows.

Schema & evolution

  • Schema lives in the footer; types are explicit (unlike CSV).
  • Evolution: adding columns and reading a subset is easy; renames/type changes are not safe. Readers match by name.
  • Parquet is immutable — no in-place updates. "Updates" mean rewriting files; that's what table formats (Delta Lake, Iceberg, Hudi) add on top of Parquet.

Reading/writing (pyarrow + pandas)

import pyarrow.parquet as pq
import pandas as pd

# write
df.to_parquet("trades.parquet", engine="pyarrow",
              compression="zstd", index=False)

# read only two columns, with a row-group-skipping filter
df = pd.read_parquet("trades.parquet",
                     columns=["symbol", "pnl"],
                     filters=[("date", ">=", "2026-01-01")])

# inspect without loading data
md = pq.read_metadata("trades.parquet")
print(md.num_rows, md.num_row_groups)
print(md.row_group(0).column(0).statistics)   # min/max/nulls

Parquet vs … (one-liners)

  • vs CSV/JSON — typed, columnar, compressed, with stats; orders of magnitude less I/O for analytics. CSV wins only for tiny/human-readable data.
  • vs Avro — Avro is row-oriented, great for streaming/write-heavy and record-at-a-time; Parquet is for columnar analytical reads.
  • vs ORC — sibling columnar format (Hadoop/Hive world); very similar trade-offs, Parquet has the wider ecosystem.
  • vs Arrow — Arrow is the in-memory columnar format; Parquet is the on-disk one. They're designed to convert cheaply.

Interview pitfalls

  • Thinking compression alone makes it fast — the bigger win is reading fewer columns and fewer row groups.
  • Forgetting that sort order drives how well min/max stats prune row groups.
  • The small-files problem from over-partitioning.
  • Expecting row-level updates — Parquet files are immutable; use a table format for ACID/upserts.
  • Assuming the footer is at the start — it's at the end (so readers do a seek-to-end first).
