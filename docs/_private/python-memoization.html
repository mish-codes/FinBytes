Memoization caches a function's return value keyed by its arguments, so a repeat call with the same arguments returns the stored result instead of recomputing. It trades memory for speed, and only makes sense for pure functions (same inputs → same output, no side effects).

functools gives you three tools:

  • @cache (3.9+) — unbounded memoization; the simplest option. Exactly lru_cache(maxsize=None).
  • @lru_cache(maxsize=128) — a bounded cache that evicts the least-recently-used entry when full. Exposes .cache_info() (hits/misses/maxsize/currsize) and .cache_clear(). typed=True keys 3 and 3.0 separately.
  • @cached_property — computes a property once per instance and stores the result in the instance's __dict__ (so it won't work with __slots__ unless __dict__ is present).
from functools import cache, lru_cache, cached_property

@cache                      # unbounded — remembers every (n) it ever sees
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)

@lru_cache(maxsize=1024)    # bounded — evicts least-recently-used
def price(symbol, date):
    return load_from_db(symbol, date)

price("BRN", "2026-06-16")
price.cache_info()          # CacheInfo(hits=0, misses=1, maxsize=1024, currsize=1)
price.cache_clear()         # the only built-in invalidation

class Dataset:
    @cached_property
    def stats(self):        # runs once, then cached on the instance
        return expensive_compute(self.rows)

Key constraints & foot-guns:

  • Arguments must be hashable — the args form the cache key. Passing a list/dict raises TypeError; convert to tuple/frozenset first.
  • Memory & lifetimes — the cache holds references to both arguments and results. @cache grows without bound; and caching a method keys on self, so the cache keeps every instance alive (a classic leak). Prefer cached_property, a bounded lru_cache, or a weakref-based cache for methods.
  • Purity — never memoize functions that depend on time, randomness, I/O, or mutable global state; you'll serve stale answers.
  • Concurrency — lru_cache bookkeeping is thread-safe, but it has no per-key locking, so the wrapped function can still run twice for the same key under a race.
  • No TTL — there's no built-in expiry. For time-based eviction reach for cachetools (TTLCache, LRUCache, LFUCache), and for cross-process caching an external store like Redis.

Rolling your own is a few lines when you need full control:

def memoize(fn):
    store = {}
    def wrapper(*args):
        if args not in store:
            store[args] = fn(*args)
        return store[args]
    return wrapper

Memoization just means remembering answers so you don't redo the work. Like homework: once you've figured out 7 × 8 = 56, you write it down — next time you just read it instead of working it out again.

Python hands you a sticker for that. Put @cache above a function and it remembers each answer by its inputs:

  • @cache — remember everything, forever (careful: the notebook keeps filling up).
  • @lru_cache(maxsize=128) — a notebook with limited pages; when it's full, rub out the answer you haven't used for the longest time.
  • @cached_property — work something out about an object once, then keep it.
from functools import cache

@cache
def slow_double(n):
    print("working...")
    return n * 2

slow_double(10)   # prints "working..." then 20
slow_double(10)   # just 20 — no "working...", it remembered

Three rules: the inputs must be simple, fixed things (numbers, strings, tuples — not lists); only cache functions that always give the same answer for the same inputs (no dice-rolling, no "what time is it?"); and remember the cache uses memory — you're trading space for speed.

The trick to remember: @cache is just @lru_cache(maxsize=None) — unbounded. If the inputs keep changing and never stop, use lru_cache with a maxsize so the notebook can't grow forever.
