A fast refresher on RabbitMQ — the model, the durability/reliability knobs, and the things that come up in interviews. RabbitMQ is a broker that speaks AMQP 0-9-1: a "smart broker, dumb consumer" design where routing logic lives in the broker.

The core model

A producer never publishes straight to a queue. It publishes to an exchange, which routes to one or more queues via bindings, and consumers read from queues:

producer → exchange → (binding + routing key) → queue(s) → consumer(s)

Exchange types

  • direct — route to queues whose binding key exactly equals the message's routing key.
  • topic — pattern match on dotted routing keys with wildcards: * = one word, # = zero or more. e.g. binding orders.*.eu matches orders.new.eu.
  • fanout — ignore the routing key, broadcast to every bound queue (pub/sub).
  • headers — route on message header attributes instead of the routing key (rarely used).

Durability & what actually survives a restart

Three independent things must all be set or messages are lost on broker restart:

  • Durable queue — the queue definition survives.
  • Durable exchange — the exchange survives.
  • Persistent message — delivery_mode=2 so the message is written to disk.

A persistent message in a non-durable queue still dies with the queue. Persistence is also not a hard guarantee on its own — a message can be in flight (in memory) when the node dies. Pair it with publisher confirms for real assurance.

Reliability knobs

  • Acknowledgements — consumers ack after processing. nack/reject can requeue or discard. Auto-ack is dangerous: the broker drops the message the moment it's delivered, so a crash mid-processing loses it. Use manual ack.
  • Prefetch (QoS) — basic.qos(prefetch_count=N) caps unacked messages per consumer, giving fair dispatch instead of one greedy consumer hoarding the queue. A small prefetch (e.g. 1–20) is the usual fix for uneven load.
  • Publisher confirms — broker asynchronously acks the publisher once it has taken responsibility for a message. This is how you get reliable publishing.
  • Dead-letter exchange (DLX) — messages that are rejected, expire (TTL), or exceed length limits get routed to a DLX. The basis for retry/parking-lot patterns and handling poison messages.
  • TTL — per-message or per-queue time-to-live; expired messages go to the DLX (a common delayed-retry trick).

Delivery semantics

RabbitMQ gives at-least-once delivery with manual acks (a redelivery can happen after a crash before the ack lands), so consumers must be idempotent. Ordering is only roughly preserved and breaks under requeues, multiple consumers, or prefetch > 1 — don't rely on strict global ordering.

Availability

  • Quorum queues — the modern, Raft-based replicated queue type for HA; prefer these. Classic mirrored queues are deprecated.
  • Clustering — nodes share metadata; queues live on (or are replicated across) nodes. Use a load balancer in front for client connectivity.

Minimal consumer (pika)

import pika

conn = pika.BlockingConnection(pika.ConnectionParameters("localhost"))
ch = conn.channel()
ch.queue_declare(queue="tasks", durable=True)
ch.basic_qos(prefetch_count=10)            # fair dispatch

def on_msg(ch, method, props, body):
    try:
        handle(body)                       # must be idempotent
        ch.basic_ack(method.delivery_tag)
    except Exception:
        ch.basic_nack(method.delivery_tag, requeue=False)  # → DLX

ch.basic_consume(queue="tasks", on_message_callback=on_msg)
ch.start_consuming()

RabbitMQ vs Kafka (one-liner)

RabbitMQ is a message broker — rich per-message routing, acks, and queues; messages are removed once consumed. Kafka is a distributed log — consumers track their own offset, messages are retained and replayable, and throughput/streaming scale is the priority. Reach for RabbitMQ for task queues, RPC, and complex routing; Kafka for event streaming and replay.

Interview pitfalls

  • Forgetting that durability needs all three (durable queue + durable exchange + persistent message).
  • Using auto-ack and claiming "guaranteed delivery".
  • No idempotency despite at-least-once semantics.
  • No DLX/retry plan for poison messages — they requeue forever and block the queue.
  • Assuming strict ordering with multiple consumers or prefetch > 1.
