asyncio runs on a single-threaded event loop. The loop maintains a queue of tasks — scheduled coroutines. On each iteration it picks a ready task, runs it until it hits an await, then suspends it and moves to the next ready task. There is no thread switching and no parallelism; concurrency comes from coroutines voluntarily yielding control.

A coroutine is a function defined with async def. Calling it does not execute it — it returns a coroutine object. To run it you either await it from another coroutine, or wrap it in a task with asyncio.create_task(), which schedules it on the event loop and lets it run concurrently with the caller. await suspends the current coroutine and hands control back to the event loop until the awaited thing completes.

The critical pitfall is blocking the event loop. Any call that does not release control — time.sleep(), synchronous file I/O, a requests.get(), a CPU-heavy loop — freezes every other coroutine for its entire duration. The event loop is cooperative; it cannot preempt you. Use asyncio.sleep() in place of time.sleep(), and loop.run_in_executor() to push genuinely blocking work into a thread pool.

import asyncio

async def fetch(n):
    print(f"start {n}")
    await asyncio.sleep(1)   # yields control — other tasks run here
    print(f"done {n}")
    return n

async def main():
    # gather runs all three concurrently; total wall time ~1s, not 3s
    results = await asyncio.gather(fetch(1), fetch(2), fetch(3))
    print(results)   # [1, 2, 3]

asyncio.run(main())

Watch out: asyncio.gather() runs coroutines concurrently within the event loop, but they still execute one at a time — if any coroutine contains a blocking call with no await, it starves the others for the entire duration of that call.

threading — concurrent I/O with real threads

threading.Thread runs a Python function in a separate OS thread. Unlike asyncio, thread switching is handled by the OS and the GIL — you don't write await. Create a thread with target and optional args, call start() to launch it, and join() to block until it finishes.

Daemon threads. A thread created with daemon=True is killed automatically when the main thread exits — you don't wait for it. Use this for background housekeeping (heartbeat pings, log flushers, cache warmers) where it is acceptable for the work to be cut short. Non-daemon threads keep the process alive until they finish.

The GIL is released during I/O waits, so threading delivers real concurrency for network-bound or file-bound work. For CPU-bound work it provides no speedup — threads still take turns on the GIL. Use ThreadPoolExecutor from concurrent.futures for a higher-level API that handles pool sizing and result collection.

import threading
import time
from concurrent.futures import ThreadPoolExecutor

def fetch(n):
    print(f"start {n}")
    time.sleep(1)          # simulates I/O — GIL released here
    print(f"done {n}")
    return n

# Manual threads
threads = [threading.Thread(target=fetch, args=(i,)) for i in range(3)]
for t in threads:
    t.start()
for t in threads:
    t.join()               # wait for all to finish; ~1s wall time, not 3s

# Daemon thread — lives in the background, dies when main thread exits
def heartbeat():
    while True:
        print("ping")
        time.sleep(5)

t = threading.Thread(target=heartbeat, daemon=True)
t.start()                  # no join needed; killed automatically on exit

# ThreadPoolExecutor — same concurrency, higher-level API
with ThreadPoolExecutor(max_workers=4) as pool:
    results = list(pool.map(fetch, range(4)))

Watch out: join() without a timeout will block forever if a thread hangs. Pass timeout= seconds and check t.is_alive() afterwards to detect stalls.

Synchronisation — Lock, RLock, Semaphore

When multiple threads share mutable state, you get race conditions: two threads read-modify-write the same value and one overwrites the other's result. Synchronisation primitives coordinate access.

Lock — a mutex: only one thread holds it at a time. Always use the context manager (with lock:) so the lock is released even if an exception is raised. A thread that calls lock.acquire() while another holds it blocks until the lock is released.

RLock (reentrant lock) — like Lock but the same thread can acquire it multiple times without deadlocking. Useful when a locked method calls another method that acquires the same lock on the same object.

Semaphore — allows up to N threads through simultaneously. The canonical use is rate-limiting: cap concurrent API calls or database connections to avoid overwhelming a downstream resource.

Deadlock — thread A holds lock 1 and waits for lock 2; thread B holds lock 2 and waits for lock 1. Neither can proceed. Prevention: always acquire multiple locks in the same order across all threads, or use lock.acquire(timeout=5) and back off on failure.

import threading
import time

# Lock — fix a race condition on a shared counter
counter = 0
lock = threading.Lock()

def increment():
    global counter
    for _ in range(100_000):
        with lock:           # only one thread inside at a time
            counter += 1     # safe — no lost updates

threads = [threading.Thread(target=increment) for _ in range(5)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)               # always 500_000

# Semaphore — limit to 3 concurrent API calls
sem = threading.Semaphore(3)

def call_api(n):
    with sem:                # blocks if 3 threads are already inside
        print(f"calling {n}")
        time.sleep(0.5)

threads = [threading.Thread(target=call_api, args=(i,)) for i in range(10)]
for t in threads: t.start()
for t in threads: t.join()

Watch out: acquiring two locks in different orders in different code paths will eventually deadlock. If you need multiple locks, always acquire them in the same fixed order everywhere.

multiprocessing — true parallelism for CPU-bound work

Each multiprocessing.Process is a separate OS process with its own Python interpreter and its own GIL. There is no shared GIL — processes run truly in parallel across CPU cores. Use multiprocessing for CPU-bound work (number crunching, image processing, data transformation) where threads would be serialised by the GIL.

Pool.map(fn, iterable) distributes work across a pool of worker processes and returns results in input order. ProcessPoolExecutor from concurrent.futures is the equivalent higher-level API.

For inter-process communication, use multiprocessing.Queue (safe for multiple producers and consumers) or Pipe (two-endpoint channel for one producer / one consumer). Data crossing process boundaries must be picklable — lambdas, closures, and open file handles cannot be pickled.

The costs: each process copies the parent's memory space at startup (fork on Linux/macOS, spawn on Windows), startup is slower than a thread, and passing large objects between processes carries serialisation overhead. For small tasks the overhead exceeds any gain from parallelism.

import multiprocessing
import time
from concurrent.futures import ProcessPoolExecutor

def crunch(n):
    return sum(i * i for i in range(n))

if __name__ == "__main__":   # required on Windows — guards the spawn entry point

    numbers = [10_000_000] * 8

    # Sequential — one core
    start = time.time()
    results = [crunch(n) for n in numbers]
    print(f"sequential: {time.time() - start:.2f}s")

    # Pool — distribute across all cores, truly parallel
    start = time.time()
    with multiprocessing.Pool() as pool:
        results = pool.map(crunch, numbers)
    print(f"pool: {time.time() - start:.2f}s")   # ~8x faster on 8 cores

    # ProcessPoolExecutor — same result, futures API
    with ProcessPoolExecutor() as exe:
        results = list(exe.map(crunch, numbers))

Watch out: always guard multiprocessing entry points with if __name__ == "__main__":. On Windows, each worker imports the module from scratch — without the guard, spawning workers triggers infinite recursive spawning.

Imagine one waiter serving many tables. The waiter doesn't clone themselves — they take an order, go put it in, then immediately serve another table while the kitchen works. They're always moving; they never stand waiting. That's asyncio: one thread, many tasks, switching whenever a task is waiting for something.

The key rule is that tasks have to choose to hand control back — they're not interrupted. An await is the waiter saying "I'm done with this table for now, let me go do something else." A task that never awaits freezes every other task for its entire duration, because nobody else gets a turn.

import asyncio

async def fetch(n):
    print(f"start {n}")
    await asyncio.sleep(1)   # hands control back — other tasks run here
    print(f"done {n}")

async def main():
    # gather runs all three "at the same time"
    # total time ~1s because all three wait concurrently
    await asyncio.gather(fetch(1), fetch(2), fetch(3))

asyncio.run(main())
# start 1, start 2, start 3 ... (all start before any finish)
# done 1, done 2, done 3
The trick to remember: if a function doesn't await anything, it blocks the whole event loop — use asyncio.sleep(0) to yield briefly, or push blocking work to a thread pool with run_in_executor.
