Nodes and pointers — the simplest dynamic structure

A singly-linked list is a chain of nodes. Each node holds data and a pointer to the next node. The last node points to nothing (null, None, nil).

Operations: push_front is O(1) — allocate a node, point it at the current head, update head. pop_front is O(1). Search is O(n) — walk from head to tail. There is no random access.

In garbage-collected languages (Python, Go, C#, TypeScript), a linked list is trivial: create nodes, assign references, and the GC cleans up. In C++, unique_ptr gives clear ownership. In Rust, things get interesting.

Rust's ownership model makes singly-linked lists possible with Option<Box<Node<T>>>, but doubly-linked lists require shared ownership (Rc<RefCell<>>) or unsafe. This post focuses on the singly-linked case — the doubly-linked version is a separate adventure.

Python — trivial with GC

from __future__ import annotations
from dataclasses import dataclass
from typing import Optional

@dataclass
class Node:
    data: object
    next: Optional[Node] = None

class LinkedList:
    def __init__(self):
        self.head: Optional[Node] = None
        self._len = 0

    def push_front(self, val):
        self.head = Node(data=val, next=self.head)
        self._len += 1

    def pop_front(self):
        if self.head is None:
            raise IndexError("pop from empty list")
        val = self.head.data
        self.head = self.head.next
        self._len -= 1
        return val

    def contains(self, val) -> bool:
        cur = self.head
        while cur:
            if cur.data == val:
                return True
            cur = cur.next
        return False

    def __iter__(self):
        cur = self.head
        while cur:
            yield cur.data
            cur = cur.next

    def __len__(self):
        return self._len

    def __repr__(self):
        return " -> ".join(str(x) for x in self) + " -> None"

ll = LinkedList()
for x in [3, 2, 1]:
    ll.push_front(x)
print(ll)            # 1 -> 2 -> 3 -> None
print(ll.contains(2))  # True
ll.pop_front()
print(ll)            # 2 -> 3 -> None
Python's GC handles the old head node automatically when pop_front drops the reference. No manual free, no ownership headaches. This is why linked lists feel trivial in GC languages.

C# — generic singly-linked list

public class SLinkedList<T> where T : IEquatable<T>
{
    private class Node
    {
        public T Data;
        public Node? Next;
        public Node(T data, Node? next = null) { Data = data; Next = next; }
    }

    private Node? _head;
    private int _count;

    public int Count => _count;

    public void PushFront(T val)
    {
        _head = new Node(val, _head);
        _count++;
    }

    public T PopFront()
    {
        if (_head is null) throw new InvalidOperationException("empty");
        T val = _head.Data;
        _head = _head.Next;
        _count--;
        return val;
    }

    public bool Contains(T val)
    {
        for (var cur = _head; cur is not null; cur = cur.Next)
            if (cur.Data.Equals(val)) return true;
        return false;
    }

    public override string ToString()
    {
        var parts = new List<string>();
        for (var cur = _head; cur is not null; cur = cur.Next)
            parts.Add(cur.Data?.ToString() ?? "null");
        parts.Add("None");
        return string.Join(" -> ", parts);
    }
}

var ll = new SLinkedList<int>();
ll.PushFront(3); ll.PushFront(2); ll.PushFront(1);
Console.WriteLine(ll);            // 1 -> 2 -> 3 -> None
Console.WriteLine(ll.Contains(2)); // True
The where T : IEquatable<T> constraint ensures we can call .Equals() for Contains. Without it, we would fall back to the base object.Equals which boxes value types.

C++ — unique_ptr ownership, raw pointer traversal

#include <memory>
#include <iostream>
#include <sstream>

template<typename T>
class LinkedList {
    struct Node {
        T data;
        std::unique_ptr<Node> next;
        Node(T d, std::unique_ptr<Node> n = nullptr)
            : data(std::move(d)), next(std::move(n)) {}
    };

    std::unique_ptr<Node> head_;
    size_t len_ = 0;

public:
    void push_front(T val) {
        head_ = std::make_unique<Node>(std::move(val), std::move(head_));
        ++len_;
    }

    T pop_front() {
        if (!head_) throw std::runtime_error("empty");
        T val = std::move(head_->data);
        head_ = std::move(head_->next);
        --len_;
        return val;
    }

    bool contains(const T& val) const {
        for (const Node* cur = head_.get(); cur; cur = cur->next.get())
            if (cur->data == val) return true;
        return false;
    }

    size_t size() const { return len_; }

    friend std::ostream& operator<<(std::ostream& os, const LinkedList& ll) {
        for (const Node* cur = ll.head_.get(); cur; cur = cur->next.get())
            os << cur->data << " -> ";
        return os << "None";
    }
};
unique_ptr owns each node. When the list destructs, the chain of unique_ptr destructors fires automatically. For traversal, we use .get() to borrow a raw pointer — no ownership transfer. A very long list may stack-overflow on destruction due to recursive destructor calls; production code uses an iterative destructor.

Rust — Option<Box<Node<T>>>

use std::fmt;

pub struct LinkedList<T> {
    head: Option<Box<Node<T>>>,
    len: usize,
}

struct Node<T> {
    data: T,
    next: Option<Box<Node<T>>>,
}

impl<T> LinkedList<T> {
    pub fn new() -> Self {
        Self { head: None, len: 0 }
    }

    pub fn push_front(&mut self, val: T) {
        let node = Box::new(Node {
            data: val,
            next: self.head.take(),   // take() moves old head out
        });
        self.head = Some(node);
        self.len += 1;
    }

    pub fn pop_front(&mut self) -> Option<T> {
        self.head.take().map(|node| {
            self.head = node.next;
            self.len -= 1;
            node.data
        })
    }

    pub fn contains(&self, val: &T) -> bool
    where T: PartialEq
    {
        let mut cur = &self.head;
        while let Some(node) = cur {
            if &node.data == val { return true; }
            cur = &node.next;
        }
        false
    }

    pub fn len(&self) -> usize { self.len }
}

impl<T: fmt::Display> fmt::Display for LinkedList<T> {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        let mut cur = &self.head;
        while let Some(node) = cur {
            write!(f, "{} -> ", node.data)?;
            cur = &node.next;
        }
        write!(f, "None")
    }
}

fn main() {
    let mut ll = LinkedList::new();
    ll.push_front(3); ll.push_front(2); ll.push_front(1);
    println!("{}", ll);              // 1 -> 2 -> 3 -> None
    println!("{}", ll.contains(&2)); // true
    ll.pop_front();
    println!("{}", ll);              // 2 -> 3 -> None
}
Option<Box<Node<T>>> works because each node has exactly one owner. The .take() method replaces the Option with None and returns the old value — this is how you “move” the head out without violating the borrow checker. A doubly-linked list needs shared ownership (Rc<RefCell<>>) or raw pointers with unsafe, because a node would be owned by both its predecessor and its successor.

Go — pointer-based, GC-managed

package main

import (
    "fmt"
    "strings"
)

type Node[T comparable] struct {
    Data T
    Next *Node[T]
}

type LinkedList[T comparable] struct {
    head *Node[T]
    len  int
}

func (ll *LinkedList[T]) PushFront(val T) {
    ll.head = &Node[T]{Data: val, Next: ll.head}
    ll.len++
}

func (ll *LinkedList[T]) PopFront() (T, bool) {
    if ll.head == nil {
        var zero T
        return zero, false
    }
    val := ll.head.Data
    ll.head = ll.head.Next
    ll.len--
    return val, true
}

func (ll *LinkedList[T]) Contains(val T) bool {
    for cur := ll.head; cur != nil; cur = cur.Next {
        if cur.Data == val {
            return true
        }
    }
    return false
}

func (ll *LinkedList[T]) Len() int { return ll.len }

func (ll *LinkedList[T]) String() string {
    var parts []string
    for cur := ll.head; cur != nil; cur = cur.Next {
        parts = append(parts, fmt.Sprintf("%v", cur.Data))
    }
    parts = append(parts, "None")
    return strings.Join(parts, " -> ")
}

func main() {
    ll := &LinkedList[int]{}
    ll.PushFront(3); ll.PushFront(2); ll.PushFront(1)
    fmt.Println(ll)             // 1 -> 2 -> 3 -> None
    fmt.Println(ll.Contains(2)) // true
    ll.PopFront()
    fmt.Println(ll)             // 2 -> 3 -> None
}
Go's garbage collector handles the dropped nodes. The comparable constraint lets us use == in Contains. For non-comparable types, you would pass a comparison function. The standard library's container/list is a doubly-linked list with interface{} values (no generics).

TypeScript — class-based with generics

class LNode<T> {
  constructor(public data: T, public next: LNode<T> | null = null) {}
}

class LinkedList<T> {
  private head: LNode<T> | null = null;
  private _len = 0;

  pushFront(val: T): void {
    this.head = new LNode(val, this.head);
    this._len++;
  }

  popFront(): T | undefined {
    if (!this.head) return undefined;
    const val = this.head.data;
    this.head = this.head.next;
    this._len--;
    return val;
  }

  contains(val: T): boolean {
    let cur = this.head;
    while (cur) {
      if (cur.data === val) return true;
      cur = cur.next;
    }
    return false;
  }

  get length(): number { return this._len; }

  toString(): string {
    const parts: string[] = [];
    let cur = this.head;
    while (cur) {
      parts.push(String(cur.data));
      cur = cur.next;
    }
    parts.push("None");
    return parts.join(" -> ");
  }
}

const ll = new LinkedList<number>();
ll.pushFront(3); ll.pushFront(2); ll.pushFront(1);
console.log(ll.toString());    // 1 -> 2 -> 3 -> None
console.log(ll.contains(2));   // true
ll.popFront();
console.log(ll.toString());    // 2 -> 3 -> None
LNode<T> | null is TypeScript's way of saying “maybe a node, maybe nothing.” The === check in contains works for primitives but uses reference equality for objects — for deep equality you would need a custom comparator.

Key observations

  • Rust makes singly-linked lists possible but doubly-linked lists painful. A singly-linked list has clear single ownership: each node owns the next. A doubly-linked list creates shared ownership (two pointers to the same node), which requires Rc<RefCell<>> or unsafe. This is by design — the compiler is telling you that doubly-linked lists are inherently tricky with memory safety.
  • GC languages make this trivial. Python, Go, C#, and TypeScript all follow the same pattern: create a node, point to it, let GC handle cleanup. The implementations are nearly identical across all four.
  • C++ unique_ptr is the middle ground. It gives ownership semantics without a borrow checker. The risk: a long list causes recursive destructor calls that can overflow the stack. Production code adds an iterative ~LinkedList().
  • Linked lists are rarely the right choice in practice. Arrays and vectors have better cache locality. Use a linked list when you need O(1) insert/remove at known positions (with an iterator) and never need random access.

Decision guide

  • Interview or teaching: implement in the language you are studying to understand pointers/references
  • Production singly-linked (stack, undo log): any language works; Rust's version is safe and efficient
  • Production doubly-linked: use the stdlib (std::list, container/list, LinkedList<T>) or accept unsafe in Rust
  • Need fast iteration: use a dynamic array instead — cache locality dominates for sequential access
