Contiguous memory that grows on demand

A dynamic array stores elements in a contiguous block of memory. When the block is full, it allocates a new, larger block, copies everything over, and frees the old one. This gives O(1) random access and amortized O(1) append.

The key insight is the growth factor. If you double the capacity each time, the total cost of N appends is O(N) — each element is copied at most O(log N) times across all resizes. A factor of 1.5 wastes less memory but copies slightly more often.

Two numbers matter: length (how many elements are stored) and capacity (how many the current allocation can hold). When length equals capacity, the next push triggers a resize.

Every language in this series has a built-in dynamic array (Python list, C++ vector, Rust Vec, Go slice). The goal here is to build one from scratch to understand the allocation, copy, and ownership mechanics underneath.

Python — ctypes raw array, not list

import ctypes

class DynArray:
    def __init__(self):
        self._len = 0
        self._cap = 1
        self._arr = self._make(self._cap)

    @staticmethod
    def _make(cap):
        return (cap * ctypes.py_object)()

    def __len__(self):
        return self._len

    def get(self, index):
        if index < 0 or index >= self._len:
            raise IndexError("index out of range")
        return self._arr[index]

    def push(self, val):
        if self._len == self._cap:
            self._resize(self._cap * 2)
        self._arr[self._len] = val
        self._len += 1

    def pop(self):
        if self._len == 0:
            raise IndexError("pop from empty array")
        self._len -= 1
        val = self._arr[self._len]
        self._arr[self._len] = None
        return val

    def _resize(self, new_cap):
        new_arr = self._make(new_cap)
        for i in range(self._len):
            new_arr[i] = self._arr[i]
        self._arr = new_arr
        self._cap = new_cap

# Usage
a = DynArray()
for i in range(10):
    a.push(i)
print(len(a), a.get(7))  # 10  7
We use ctypes.py_object to get a fixed-size C array of Python object references. This is as close to raw memory as CPython lets you get without writing a C extension. The real list does exactly this internally, but in C.

C# — generic DynArray<T>

public class DynArray<T>
{
    private T[] _buf;
    private int _len;

    public DynArray(int initialCap = 4)
    {
        _buf = new T[initialCap];
        _len = 0;
    }

    public int Count => _len;
    public int Capacity => _buf.Length;

    public T this[int i]
    {
        get
        {
            if (i < 0 || i >= _len) throw new IndexOutOfRangeException();
            return _buf[i];
        }
    }

    public void Push(T val)
    {
        if (_len == _buf.Length) Resize(_buf.Length * 2);
        _buf[_len++] = val;
    }

    public T Pop()
    {
        if (_len == 0) throw new InvalidOperationException("empty");
        T val = _buf[--_len];
        _buf[_len] = default!;
        return val;
    }

    private void Resize(int newCap)
    {
        var newBuf = new T[newCap];
        Array.Copy(_buf, newBuf, _len);
        _buf = newBuf;
    }
}

// Usage
var a = new DynArray<int>();
for (int i = 0; i < 10; i++) a.Push(i);
Console.WriteLine($"{a.Count} {a[7]}"); // 10 7
C#'s List<T> is implemented almost identically — an internal T[] with a doubling strategy. The runtime's GC handles the old array; no manual free needed.

C++ — raw new/delete with placement

#include <cstddef>
#include <stdexcept>
#include <utility>
#include <new>

template<typename T>
class DynArray {
    T*     buf_;
    size_t len_;
    size_t cap_;

    void grow() {
        size_t new_cap = cap_ == 0 ? 1 : cap_ * 2;
        T* new_buf = static_cast<T*>(::operator new(new_cap * sizeof(T)));
        for (size_t i = 0; i < len_; ++i)
            ::new (new_buf + i) T(std::move(buf_[i]));
        for (size_t i = len_; i-- > 0;)
            buf_[i].~T();
        ::operator delete(buf_);
        buf_ = new_buf;
        cap_ = new_cap;
    }

public:
    DynArray() : buf_(nullptr), len_(0), cap_(0) {}
    ~DynArray() {
        for (size_t i = 0; i < len_; ++i) buf_[i].~T();
        ::operator delete(buf_);
    }

    size_t size()     const { return len_; }
    size_t capacity() const { return cap_; }

    void push(const T& val) {
        if (len_ == cap_) grow();
        ::new (buf_ + len_) T(val);
        ++len_;
    }

    T pop() {
        if (len_ == 0) throw std::out_of_range("empty");
        --len_;
        T val = std::move(buf_[len_]);
        buf_[len_].~T();
        return val;
    }

    const T& operator[](size_t i) const {
        if (i >= len_) throw std::out_of_range("index");
        return buf_[i];
    }
};
We separate allocation (::operator new) from construction (placement new) just like std::vector does internally. This avoids default-constructing elements in unused capacity — important when T has expensive constructors.

Rust — raw pointer + capacity + len

use std::alloc::{self, Layout};
use std::ptr;

pub struct DynArray<T> {
    ptr: *mut T,
    len: usize,
    cap: usize,
}

impl<T> DynArray<T> {
    pub fn new() -> Self {
        Self { ptr: ptr::null_mut(), len: 0, cap: 0 }
    }

    pub fn len(&self) -> usize { self.len }

    pub fn push(&mut self, val: T) {
        if self.len == self.cap { self.grow(); }
        unsafe { ptr::write(self.ptr.add(self.len), val); }
        self.len += 1;
    }

    pub fn pop(&mut self) -> Option<T> {
        if self.len == 0 { return None; }
        self.len -= 1;
        unsafe { Some(ptr::read(self.ptr.add(self.len))) }
    }

    pub fn get(&self, i: usize) -> Option<&T> {
        if i >= self.len { return None; }
        unsafe { Some(&*self.ptr.add(i)) }
    }

    fn grow(&mut self) {
        let new_cap = if self.cap == 0 { 1 } else { self.cap * 2 };
        let layout = Layout::array::<T>(new_cap).unwrap();
        let new_ptr = unsafe { alloc::alloc(layout) as *mut T };
        if !self.ptr.is_null() {
            unsafe {
                ptr::copy_nonoverlapping(self.ptr, new_ptr, self.len);
                let old_layout = Layout::array::<T>(self.cap).unwrap();
                alloc::dealloc(self.ptr as *mut u8, old_layout);
            }
        }
        self.ptr = new_ptr;
        self.cap = new_cap;
    }
}

impl<T> Drop for DynArray<T> {
    fn drop(&mut self) {
        if !self.ptr.is_null() {
            for i in 0..self.len {
                unsafe { ptr::drop_in_place(self.ptr.add(i)); }
            }
            let layout = Layout::array::<T>(self.cap).unwrap();
            unsafe { alloc::dealloc(self.ptr as *mut u8, layout); }
        }
    }
}
This mirrors how the real Vec<T> works internally. The unsafe blocks are unavoidable — raw pointer arithmetic is inherently unsafe in Rust. The safe API (push, pop, get) encapsulates all unsafety so callers never touch pointers.

Go — slice internals (make, append, cap vs len)

package main

import "fmt"

type DynArray[T any] struct {
    buf []T
    len int
}

func NewDynArray[T any]() *DynArray[T] {
    return &DynArray[T]{buf: make([]T, 1), len: 0}
}

func (a *DynArray[T]) Len() int { return a.len }
func (a *DynArray[T]) Cap() int { return len(a.buf) }

func (a *DynArray[T]) Push(val T) {
    if a.len == len(a.buf) {
        a.grow()
    }
    a.buf[a.len] = val
    a.len++
}

func (a *DynArray[T]) Pop() (T, bool) {
    var zero T
    if a.len == 0 {
        return zero, false
    }
    a.len--
    val := a.buf[a.len]
    a.buf[a.len] = zero
    return val, true
}

func (a *DynArray[T]) Get(i int) (T, bool) {
    var zero T
    if i < 0 || i >= a.len {
        return zero, false
    }
    return a.buf[i], true
}

func (a *DynArray[T]) grow() {
    newBuf := make([]T, len(a.buf)*2)
    copy(newBuf, a.buf[:a.len])
    a.buf = newBuf
}

func main() {
    a := NewDynArray[int]()
    for i := 0; i < 10; i++ {
        a.Push(i)
    }
    v, _ := a.Get(7)
    fmt.Println(a.Len(), v) // 10 7
}
Go's built-in append does exactly this under the hood. The runtime's slice header is a struct with pointer, length, and capacity — the same three fields we declared. Growth factor is roughly 2x for small slices, tapering to 1.25x for large ones.

TypeScript — array-backed with manual capacity

class DynArray<T> {
  private buf: (T | undefined)[];
  private _len = 0;

  constructor(initialCap = 4) {
    this.buf = new Array(initialCap);
  }

  get length(): number { return this._len; }
  get capacity(): number { return this.buf.length; }

  push(val: T): void {
    if (this._len === this.buf.length) this.grow();
    this.buf[this._len++] = val;
  }

  pop(): T | undefined {
    if (this._len === 0) return undefined;
    const val = this.buf[--this._len];
    this.buf[this._len] = undefined;
    return val;
  }

  get(i: number): T {
    if (i < 0 || i >= this._len) throw new RangeError("index out of range");
    return this.buf[i] as T;
  }

  private grow(): void {
    const newBuf = new Array(this.buf.length * 2);
    for (let i = 0; i < this._len; i++) newBuf[i] = this.buf[i];
    this.buf = newBuf;
  }
}

// Usage
const a = new DynArray<number>();
for (let i = 0; i < 10; i++) a.push(i);
console.log(a.length, a.get(7)); // 10 7
JavaScript arrays are already dynamic and sparse, so this exercise is somewhat academic in TS. The value is understanding that under the hood, V8 optimizes dense integer arrays into contiguous C-style buffers — very similar to what we built manually in C++ and Rust.

Key observations

  • C++ and Rust require separating allocation from construction. In GC languages, the runtime handles uninitialized slots. In C++ you use placement new; in Rust you use ptr::write. Getting this wrong causes undefined behaviour.
  • Go's slice is the closest to a “language-level” dynamic array. The slice header (pointer, length, capacity) is a value type baked into the language, and append is a builtin — not a method on a type.
  • Growth factor matters more than you think. Python's ~1.125 factor minimizes wasted memory but copies more often. C++ / Rust doubling wastes up to 50% capacity but gives fewer allocations. For large arrays (millions of elements), the difference is measurable.
  • Rust's approach prevents iterator invalidation at compile time. You cannot hold a reference into a Vec and simultaneously push to it — the borrow checker rejects it. C++ vector silently invalidates iterators on reallocation, a classic source of bugs.

Decision guide

  • Need contiguous memory for FFI or SIMD: C++ vector or Rust Vec (guaranteed contiguous)
  • Predictable memory usage: call reserve / with_capacity upfront to avoid repeated reallocations
  • GC language, append-heavy workload: stdlib is fine — the GC reclaims old buffers efficiently
  • Teaching or interviews: implement from scratch in C++ or Rust to understand placement new, unsafe, and RAII
