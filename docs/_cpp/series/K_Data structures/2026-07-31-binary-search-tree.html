Ordered data with O(log n) average lookup

A binary search tree (BST) is a binary tree where every node satisfies: left child < node < right child. This invariant lets you find any value by comparing and going left or right — halving the search space at each step.

Insert: walk down comparing, then attach the new node as a leaf. Search: same walk, return true if found. In-order traversal: visit left, then root, then right — this yields elements in sorted order.

Average case for insert and search is O(log n), but a sorted input creates a degenerate tree (effectively a linked list) with O(n) operations. Balanced BSTs (AVL, red-black) fix this but add complexity.

The interesting language comparison is how each represents “child might not exist” — Python uses None, Rust uses Option<Box<>>, Go uses *Node (nil), and so on. This simple question reveals each language's null-safety philosophy.

Python — recursive BST

from __future__ import annotations
from typing import Optional, List

class Node:
    __slots__ = ("val", "left", "right")
    def __init__(self, val):
        self.val = val
        self.left: Optional[Node] = None
        self.right: Optional[Node] = None

class BST:
    def __init__(self):
        self.root: Optional[Node] = None

    def insert(self, val):
        self.root = self._insert(self.root, val)

    def _insert(self, node: Optional[Node], val) -> Node:
        if node is None:
            return Node(val)
        if val < node.val:
            node.left = self._insert(node.left, val)
        elif val > node.val:
            node.right = self._insert(node.right, val)
        return node  # duplicate — no-op

    def search(self, val) -> bool:
        cur = self.root
        while cur:
            if val == cur.val:
                return True
            cur = cur.left if val < cur.val else cur.right
        return False

    def in_order(self) -> List:
        result = []
        def walk(node):
            if node:
                walk(node.left)
                result.append(node.val)
                walk(node.right)
        walk(self.root)
        return result

tree = BST()
for v in [5, 3, 7, 1, 4]:
    tree.insert(v)
print(tree.in_order())    # [1, 3, 4, 5, 7]
print(tree.search(4))     # True
print(tree.search(6))     # False
Insert is recursive (cleaner), search is iterative (avoids stack depth issues on deep trees). Python's None acts as the null sentinel — no special type wrapping needed.

C# — nullable reference for children

public class BST<T> where T : IComparable<T>
{
    private class Node
    {
        public T Val;
        public Node? Left, Right;
        public Node(T val) => Val = val;
    }

    private Node? _root;

    public void Insert(T val) => _root = Insert(_root, val);

    private Node Insert(Node? node, T val)
    {
        if (node is null) return new Node(val);
        int cmp = val.CompareTo(node.Val);
        if (cmp < 0) node.Left = Insert(node.Left, val);
        else if (cmp > 0) node.Right = Insert(node.Right, val);
        return node;
    }

    public bool Search(T val)
    {
        var cur = _root;
        while (cur is not null)
        {
            int cmp = val.CompareTo(cur.Val);
            if (cmp == 0) return true;
            cur = cmp < 0 ? cur.Left : cur.Right;
        }
        return false;
    }

    public List<T> InOrder()
    {
        var result = new List<T>();
        void Walk(Node? n)
        {
            if (n is null) return;
            Walk(n.Left);
            result.Add(n.Val);
            Walk(n.Right);
        }
        Walk(_root);
        return result;
    }
}

var tree = new BST<int>();
foreach (int v in new[] { 5, 3, 7, 1, 4 }) tree.Insert(v);
Console.WriteLine(string.Join(", ", tree.InOrder())); // 1, 3, 4, 5, 7
Console.WriteLine(tree.Search(4)); // True
IComparable<T> gives us CompareTo which returns negative, zero, or positive. The nullable reference (Node?) with is null / is not null pattern matching keeps the code clean and null-safe.

C++ — unique_ptr for child ownership

#include <memory>
#include <vector>
#include <iostream>

template<typename T>
class BST {
    struct Node {
        T val;
        std::unique_ptr<Node> left, right;
        explicit Node(T v) : val(std::move(v)) {}
    };

    std::unique_ptr<Node> root_;

    std::unique_ptr<Node>& find_slot(std::unique_ptr<Node>& node, const T& val) {
        if (!node || val == node->val) return node;
        return val < node->val
            ? find_slot(node->left, val)
            : find_slot(node->right, val);
    }

public:
    void insert(T val) {
        auto& slot = find_slot(root_, val);
        if (!slot) slot = std::make_unique<Node>(std::move(val));
    }

    bool search(const T& val) const {
        const Node* cur = root_.get();
        while (cur) {
            if (val == cur->val) return true;
            cur = (val < cur->val ? cur->left : cur->right).get();
        }
        return false;
    }

    std::vector<T> in_order() const {
        std::vector<T> result;
        auto walk = [&](auto&& self, const Node* n) -> void {
            if (!n) return;
            self(self, n->left.get());
            result.push_back(n->val);
            self(self, n->right.get());
        };
        walk(walk, root_.get());
        return result;
    }
};
The find_slot trick returns a reference to the unique_ptr where the node should live. If the slot is empty, we construct in place. This eliminates the “parent pointer” dance that raw-pointer BSTs often require. The C++23 deducing-this would make the const/non-const overloads cleaner.

Rust — Option<Box<Node<T>>> for children

pub struct BST<T: Ord> {
    root: Option<Box<Node<T>>>,
}

struct Node<T> {
    val: T,
    left: Option<Box<Node<T>>>,
    right: Option<Box<Node<T>>>,
}

impl<T: Ord> BST<T> {
    pub fn new() -> Self { Self { root: None } }

    pub fn insert(&mut self, val: T) {
        fn go<T: Ord>(slot: &mut Option<Box<Node<T>>>, val: T) {
            match slot {
                None => {
                    *slot = Some(Box::new(Node { val, left: None, right: None }));
                }
                Some(node) => {
                    use std::cmp::Ordering::*;
                    match val.cmp(&node.val) {
                        Less    => go(&mut node.left, val),
                        Greater => go(&mut node.right, val),
                        Equal   => {} // duplicate — no-op
                    }
                }
            }
        }
        go(&mut self.root, val);
    }

    pub fn search(&self, val: &T) -> bool {
        let mut cur = &self.root;
        while let Some(node) = cur {
            use std::cmp::Ordering::*;
            match val.cmp(&node.val) {
                Equal   => return true,
                Less    => cur = &node.left,
                Greater => cur = &node.right,
            }
        }
        false
    }

    pub fn in_order(&self) -> Vec<T>
    where T: Clone
    {
        let mut result = Vec::new();
        fn walk<T: Ord + Clone>(node: &Option<Box<Node<T>>>, out: &mut Vec<T>) {
            if let Some(n) = node {
                walk(&n.left, out);
                out.push(n.val.clone());
                walk(&n.right, out);
            }
        }
        walk(&self.root, &mut result);
        result
    }
}

fn main() {
    let mut tree = BST::new();
    for v in [5, 3, 7, 1, 4] { tree.insert(v); }
    println!("{:?}", tree.in_order()); // [1, 3, 4, 5, 7]
    println!("{}", tree.search(&4));   // true
}
Option<Box<Node<T>>> is Rust's way of saying “maybe a heap-allocated child node.” The Ord trait bound ensures keys are totally ordered. Pattern matching on Ordering is idiomatic and the compiler checks you handle all three cases.

Go — nil pointer for absent children

package main

import (
    "fmt"
    "golang.org/x/exp/constraints"
)

type Node[T constraints.Ordered] struct {
    Val         T
    Left, Right *Node[T]
}

type BST[T constraints.Ordered] struct {
    root *Node[T]
}

func (t *BST[T]) Insert(val T) {
    t.root = insert(t.root, val)
}

func insert[T constraints.Ordered](n *Node[T], val T) *Node[T] {
    if n == nil {
        return &Node[T]{Val: val}
    }
    if val < n.Val {
        n.Left = insert(n.Left, val)
    } else if val > n.Val {
        n.Right = insert(n.Right, val)
    }
    return n
}

func (t *BST[T]) Search(val T) bool {
    cur := t.root
    for cur != nil {
        if val == cur.Val {
            return true
        } else if val < cur.Val {
            cur = cur.Left
        } else {
            cur = cur.Right
        }
    }
    return false
}

func (t *BST[T]) InOrder() []T {
    var result []T
    var walk func(*Node[T])
    walk = func(n *Node[T]) {
        if n == nil { return }
        walk(n.Left)
        result = append(result, n.Val)
        walk(n.Right)
    }
    walk(t.root)
    return result
}

func main() {
    t := &BST[int]{}
    for _, v := range []int{5, 3, 7, 1, 4} {
        t.Insert(v)
    }
    fmt.Println(t.InOrder())  // [1 3 4 5 7]
    fmt.Println(t.Search(4))  // true
}
Go uses *Node where nil means “no child.” The constraints.Ordered type set (from golang.org/x/exp) restricts T to types that support <, >, ==. In Go 1.21+ you can use cmp.Ordered from the standard library instead.

TypeScript — union with null

class TNode<T> {
  val: T;
  left: TNode<T> | null = null;
  right: TNode<T> | null = null;
  constructor(val: T) { this.val = val; }
}

class BST<T> {
  private root: TNode<T> | null = null;

  constructor(private cmp: (a: T, b: T) => number) {}

  insert(val: T): void {
    this.root = this._insert(this.root, val);
  }

  private _insert(node: TNode<T> | null, val: T): TNode<T> {
    if (!node) return new TNode(val);
    const c = this.cmp(val, node.val);
    if (c < 0) node.left = this._insert(node.left, val);
    else if (c > 0) node.right = this._insert(node.right, val);
    return node;
  }

  search(val: T): boolean {
    let cur = this.root;
    while (cur) {
      const c = this.cmp(val, cur.val);
      if (c === 0) return true;
      cur = c < 0 ? cur.left : cur.right;
    }
    return false;
  }

  inOrder(): T[] {
    const result: T[] = [];
    const walk = (n: TNode<T> | null) => {
      if (!n) return;
      walk(n.left);
      result.push(n.val);
      walk(n.right);
    };
    walk(this.root);
    return result;
  }
}

const numCmp = (a: number, b: number) => a - b;
const tree = new BST<number>(numCmp);
for (const v of [5, 3, 7, 1, 4]) tree.insert(v);
console.log(tree.inOrder());  // [1, 3, 4, 5, 7]
console.log(tree.search(4));  // true
TypeScript has no Comparable interface, so we pass a comparator function to the constructor. The | null union type is checked by the compiler with strictNullChecks — you cannot access .val on a TNode<T> | null without a null guard.

Key observations

  • Rust's Option<Box<>> makes missing children explicit and safe. You cannot accidentally dereference a None child — the compiler forces you to pattern-match. Other languages rely on runtime null checks or trust the programmer.
  • C++'s reference-to-unique_ptr trick is elegant. By returning a reference to the slot where a node should live, insert becomes a single assignment. This avoids the “parent pointer” tracking that raw-pointer implementations need.
  • Go and TypeScript require an external ordering mechanism. Go's constraints.Ordered only covers built-in types. TypeScript has no ordering interface at all — you must pass a comparator. Python, C#, C++, and Rust all have standard ordering protocols.
  • In-order traversal is almost identical everywhere. Recursive tree traversal is one of the few algorithms that looks nearly the same in every language. The only variation is how you check for “null child.”

Decision guide

  • Need a sorted container in production: use the stdlib balanced tree (C++ std::set, Rust BTreeSet, C# SortedSet, Java TreeSet). A plain BST degenerates on sorted input.
  • Teaching or interviews: implement a plain BST to understand recursion and tree structure, then discuss how balancing fixes the O(n) worst case
  • Need fast lookups but not sorted order: use a hash map instead — O(1) average beats O(log n)
  • Frequent range queries (all values between A and B): BST or B-tree, not a hash map
