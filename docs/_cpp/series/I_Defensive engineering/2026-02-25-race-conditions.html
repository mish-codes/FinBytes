The hardest class of bug

Race conditions are non-deterministic, timing-dependent, and often impossible to reproduce on demand. A race that corrupts a position once per day in production might never appear in a month of testing.

What causes races: two threads accessing shared mutable state with no synchronisation. In finance: a position object updated by the order feed thread while being read by the risk engine thread. No mutex, no atomic - undefined behaviour.

Prevention strategies: immutable data (if threads only read, there is no race); message passing (threads communicate via queues rather than shared state); lock hierarchy (always acquire locks in defined order); thread confinement (each object is owned by exactly one thread).

Detection: ThreadSanitiser (TSan) - -fsanitize=thread in C++. Catches races at runtime with ~5x overhead. Run the full test suite under TSan regularly.

Python thread safety

import threading
from threading import Lock
from queue import Queue

class UnsafePosition:
    def __init__(self): self.quantity = 0.0
    def add(self, qty: float) -> None:
        self.quantity += qty  # NOT thread-safe: read-modify-write is three ops

class SafePosition:
    def __init__(self): self._qty = 0.0; self._lock = Lock()
    def add(self, qty: float) -> None:
        with self._lock: self._qty += qty
    @property
    def quantity(self) -> float:
        with self._lock: return self._qty

# Message passing: avoid shared state entirely
class PositionManager:
    def __init__(self):
        self._queue    = Queue()
        self._position = {}
        threading.Thread(target=self._run, daemon=True).start()

    def add(self, symbol: str, qty: float) -> None:
        self._queue.put(("add", symbol, qty))  # queue is thread-safe

    def _run(self) -> None:  # single thread owns the state
        while True:
            msg = self._queue.get()
            if msg[0] == "add":
                self._position[msg[1]] = self._position.get(msg[1], 0) + msg[2]

C++ thread safety

#include <mutex>
#include <shared_mutex>
#include <atomic>
#include <unordered_map>

// Build with TSan: g++ -fsanitize=thread -g -O1

// Atomic for simple values
std::atomic<double> last_price{0.0};
// Writer: last_price.store(p, std::memory_order_release);
// Reader: auto p = last_price.load(std::memory_order_acquire);

// Shared mutex: many readers, one writer
class PositionCache {
    mutable std::shared_mutex mutex_;
    std::unordered_map<std::string, double> data_;
public:
    double get(const std::string& sym) const {
        std::shared_lock lock(mutex_);
        auto it = data_.find(sym);
        return it != data_.end() ? it->second : 0.0;
    }
    void set(const std::string& sym, double qty) {
        std::unique_lock lock(mutex_);
        data_[sym] = qty;
    }
};
The best way to avoid race conditions is to avoid shared mutable state. Thread confinement (each object owned by exactly one thread, communication via queues) eliminates the entire class of problem. It is the design principle behind actor systems and most high-performance market data systems.

Rust thread safety

The borrow checker prevents data races at compile time

use std::sync::{Arc, Mutex, RwLock};
use std::thread;

// This will NOT compile - Rust prevents shared mutable access:
// let mut position = 0.0;
// thread::spawn(|| position += 100.0);  // ERROR: cannot borrow as mutable
// println!("{position}");                // while also borrowed here

// Fix: wrap in Arc<Mutex> for shared mutable state
fn safe_position_update() {
    let position = Arc::new(Mutex::new(0.0_f64));

    let handles: Vec<_> = (0..10).map(|_| {
        let pos = Arc::clone(&position);
        thread::spawn(move || {
            let mut guard = pos.lock().unwrap();
            *guard += 100.0;
            // guard dropped here - lock released automatically (RAII)
        })
    }).collect();

    for h in handles { h.join().unwrap(); }
    println!("Final: {}", *position.lock().unwrap());  // always 1000.0
}

// RwLock for read-heavy workloads
struct PositionCache {
    data: RwLock<std::collections::HashMap<String, f64>>,
}

impl PositionCache {
    fn get(&self, sym: &str) -> Option<f64> {
        let guard = self.data.read().unwrap();  // many readers OK
        guard.get(sym).copied()
    }
    fn set(&self, sym: String, qty: f64) {
        let mut guard = self.data.write().unwrap();  // exclusive write
        guard.insert(sym, qty);
    }
}

Message passing with channels

use std::sync::mpsc;

enum PosMsg {
    Add(String, f64),
    Get(String, mpsc::Sender<f64>),
}

fn position_manager() -> mpsc::Sender<PosMsg> {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let mut positions = std::collections::HashMap::new();
        for msg in rx {
            match msg {
                PosMsg::Add(sym, qty) => {
                    *positions.entry(sym).or_insert(0.0) += qty;
                }
                PosMsg::Get(sym, reply) => {
                    let qty = positions.get(&sym).copied().unwrap_or(0.0);
                    let _ = reply.send(qty);
                }
            }
        }
    });

    tx  // single owner of state - no races possible
}
Rust's ownership system makes data races a compile error, not a runtime bug. The Send and Sync traits are automatically derived — a type that is not safe to share across threads simply will not compile if you try. This eliminates the entire class of race conditions that ThreadSanitiser detects in C++.

Go thread safety

Goroutines and channels — "share by communicating"

package main

import (
    "fmt"
    "sync"
)

// BAD: shared mutable state without synchronisation
type UnsafePosition struct {
    Quantity float64  // data race: multiple goroutines read/write
}

// GOOD: mutex protection
type SafePosition struct {
    mu       sync.RWMutex
    quantity float64
}

func (p *SafePosition) Add(qty float64) {
    p.mu.Lock()
    defer p.mu.Unlock()
    p.quantity += qty
}

func (p *SafePosition) Get() float64 {
    p.mu.RLock()
    defer p.mu.RUnlock()
    return p.quantity
}

// BEST: channel-based message passing
type posMsg struct {
    sym string
    qty float64
}

func positionManager(updates <-chan posMsg, queries <-chan string, replies chan<- float64) {
    positions := make(map[string]float64)
    for {
        select {
        case msg := <-updates:
            positions[msg.sym] += msg.qty
        case sym := <-queries:
            replies <- positions[sym]
        }
    }
}

// Race detector: go run -race main.go
// Detects data races at runtime with ~5-10x overhead.
// Run ALL tests with: go test -race ./...
Go's motto is "don't communicate by sharing memory; share memory by communicating." Channels are the primary concurrency primitive. The -race flag enables the built-in race detector — run it on every test. Unlike Rust, Go does not prevent races at compile time; the race detector is a runtime tool.

TypeScript thread safety

Single-threaded by default — races still possible

// JavaScript is single-threaded (event loop), so classical data races
// between threads are impossible in the main thread.

// BUT: logical races still happen with async code
class UnsafeCache {
  private cache = new Map<string, number>();

  async getOrFetch(key: string): Promise<number> {
    if (this.cache.has(key)) return this.cache.get(key)!;
    // Two concurrent calls can both reach here before either caches
    const value = await fetchPrice(key);  // yields to event loop
    this.cache.set(key, value);           // duplicate fetch
    return value;
  }
}

// Fix: track in-flight promises
class SafeCache {
  private cache = new Map<string, number>();
  private inflight = new Map<string, Promise<number>>();

  async getOrFetch(key: string): Promise<number> {
    if (this.cache.has(key)) return this.cache.get(key)!;
    if (this.inflight.has(key)) return this.inflight.get(key)!;

    const promise = fetchPrice(key).then((value) => {
      this.cache.set(key, value);
      this.inflight.delete(key);
      return value;
    });
    this.inflight.set(key, promise);
    return promise;
  }
}

// Worker threads: true shared memory races ARE possible
import { Worker, isMainThread, parentPort } from "worker_threads";

// SharedArrayBuffer + Atomics for thread-safe shared memory
if (isMainThread) {
  const buffer = new SharedArrayBuffer(8);
  const view = new Float64Array(buffer);
  Atomics.store(view, 0, 100.0);  // atomic write

  const worker = new Worker(__filename, { workerData: buffer });
} else {
  // worker reads atomically
  const view = new Float64Array(workerData);
  const price = Atomics.load(view, 0);  // atomic read
}
JavaScript's single-threaded event loop eliminates classical data races, but async logical races (TOCTOU) are common and subtle. With Worker threads and SharedArrayBuffer, true data races become possible — use Atomics for synchronisation. There is no built-in race detector for JavaScript.

Key observations

  • Rust is the only language that prevents data races at compile time. The Send/Sync trait system makes it impossible to share mutable state across threads without synchronisation. This is a fundamental advantage over every other language here.
  • Go has the best runtime race detector and the most ergonomic channel-based concurrency. The -race flag should be on for every test run. Go's motto "share by communicating" is the right default.
  • C++ ThreadSanitiser (TSan) is powerful but has ~5x overhead and can miss races that do not occur during the test run. It detects, not prevents.
  • TypeScript avoids most races by being single-threaded, but async logical races (double-fetch, TOCTOU) are common. Worker threads with SharedArrayBuffer reintroduce true data races.
  • Python's GIL prevents true data races between threads for CPython, but this is an implementation detail, not a guarantee. Multiprocessing and async code can still have logical races.
  • C# has good concurrent collections (ConcurrentDictionary, Channel<T>) but no compile-time or runtime race detector built into the platform.

Decision guide

  • Race-free by construction: Rust (compile-time prevention via ownership)
  • Best runtime safety net: Go (-race detector) or C++ (ThreadSanitiser)
  • High-concurrency services: Go (goroutines + channels) or Rust (async + ownership)
  • Avoid concurrency complexity: TypeScript (single-threaded event loop)
  • Enterprise with concurrent collections: C# (ConcurrentDictionary, Channel<T>)
