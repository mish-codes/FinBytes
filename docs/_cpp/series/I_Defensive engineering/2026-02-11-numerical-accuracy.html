Silent wrong answers

Crashes announce themselves. Numerical errors do not. A pricer returning 10.4523 when the answer is 10.4521 passes casual inspection but fails desk reconciliation.

Catastrophic cancellation: subtracting two nearly-equal numbers destroys significant digits. Black-Scholes d1-d2 for deep ITM/OTM options, variance E[X^2] - E[X]^2, barrier option payoffs near the barrier - all can suffer this.

NaN and Inf propagation: NaN and Inf propagate silently through arithmetic. A single bad input can corrupt an entire risk report without any exception. Always validate inputs and outputs at system boundaries.

The fix: switching from float to double postpones the problem, it does not solve it. Reformulate the calculation to avoid cancellation, use compensated summation, and validate at boundaries.

#include <cmath>
#include <stdexcept>
#include <vector>
#include <string>

// NaN and Inf propagate silently
double x = std::numeric_limits<double>::quiet_NaN();
bool b  = (x > 0.0);   // false - no exception
bool eq = (x == x);    // false - NaN != NaN

// Validate at system boundaries
void validate_inputs(double S, double K, double T, double sigma) {
    auto check = [](double v, const std::string& name) {
        if (!std::isfinite(v)) throw std::invalid_argument(name + " is not finite");
        if (v <= 0) throw std::invalid_argument(name + " must be positive");
    };
    check(S, "S"); check(K, "K"); check(T, "T"); check(sigma, "sigma");
}

// Welford online algorithm - numerically stable variance
double variance_stable(const std::vector<double>& x) {
    double mean = 0, M2 = 0;
    for (std::size_t n = 1; n <= x.size(); ++n) {
        double delta = x[n-1] - mean;
        mean += delta / n;
        M2   += delta * (x[n-1] - mean);
    }
    return M2 / (x.size() - 1);
}

// Kahan compensated sum
double kahan_sum(const std::vector<double>& v) {
    double sum = 0, c = 0;
    for (double xi : v) {
        double y = xi - c, t = sum + y;
        c = (t - sum) - y; sum = t;
    }
    return sum;
}
Enable -fno-fast-math to prevent the compiler reordering floating-point ops in ways that break Kahan summation. Use -fsanitize=float-divide-by-zero to catch division by zero during development.

Rust numerical accuracy

NaN and Inf handling

// Rust f64 follows IEEE 754, same NaN/Inf behaviour as C++
let x: f64 = f64::NAN;
println!("{}", x + 1.0);     // NaN
println!("{}", x > 0.0);     // false - no panic!
println!("{}", x == x);      // false (NaN != NaN)
println!("{}", x.is_nan());  // true

// f64 does NOT implement Ord (because of NaN).
// This is a compile-time safety net:
// let mut v = vec![1.0, 2.0, f64::NAN];
// v.sort();  // COMPILE ERROR - f64 doesn't implement Ord

// Use total_cmp for sorting (available since Rust 1.62)
let mut v = vec![1.0, f64::NAN, 2.0];
v.sort_by(f64::total_cmp);  // NaN sorts to the end

// Validate at boundaries
fn validate_inputs(s: f64, k: f64, t: f64, sigma: f64) -> Result<(), String> {
    for (name, val) in [("S", s), ("K", k), ("T", t), ("sigma", sigma)] {
        if !val.is_finite() {
            return Err(format!("{name} is not finite: {val}"));
        }
        if val <= 0.0 {
            return Err(format!("{name} must be positive: {val}"));
        }
    }
    Ok(())
}

Kahan summation and stable variance

// Kahan compensated summation
fn kahan_sum(values: &[f64]) -> f64 {
    let (mut sum, mut c) = (0.0_f64, 0.0_f64);
    for &v in values {
        let y = v - c;
        let t = sum + y;
        c = (t - sum) - y;
        sum = t;
    }
    sum
}

// Welford online variance (numerically stable)
fn variance_stable(x: &[f64]) -> f64 {
    let mut mean = 0.0;
    let mut m2   = 0.0;
    for (n, &xi) in x.iter().enumerate() {
        let n = (n + 1) as f64;
        let delta = xi - mean;
        mean += delta / n;
        m2   += delta * (xi - mean);
    }
    m2 / (x.len() as f64 - 1.0)
}

// Newtype for validated prices
#[derive(Debug, Clone, Copy)]
struct FinitePrice(f64);

impl FinitePrice {
    fn new(v: f64) -> Result<Self, String> {
        if v.is_finite() && v >= 0.0 { Ok(Self(v)) }
        else { Err(format!("Invalid price: {v}")) }
    }
    fn value(self) -> f64 { self.0 }
}
Rust's type system offers a unique advantage: f64 does not implement Ord, so you cannot accidentally sort or compare floats without handling NaN. Use newtypes like FinitePrice to encode "validated and finite" in the type system, making invalid states unrepresentable.

Go numerical accuracy

NaN and Inf checking

package main

import (
    "fmt"
    "math"
)

func main() {
    x := math.NaN()
    fmt.Println(x + 1)           // NaN
    fmt.Println(x > 0)           // false - no panic
    fmt.Println(x == x)          // false (NaN != NaN)
    fmt.Println(math.IsNaN(x))   // true
    fmt.Println(math.IsInf(x, 0)) // false
}

// Validate at boundaries
func validateInputs(S, K, T, sigma float64) error {
    params := map[string]float64{"S": S, "K": K, "T": T, "sigma": sigma}
    for name, val := range params {
        if math.IsNaN(val) || math.IsInf(val, 0) {
            return fmt.Errorf("%s is not finite: %f", name, val)
        }
        if val <= 0 {
            return fmt.Errorf("%s must be positive: %f", name, val)
        }
    }
    return nil
}

// Kahan compensated summation
func kahanSum(values []float64) float64 {
    sum, c := 0.0, 0.0
    for _, v := range values {
        y := v - c
        t := sum + y
        c = (t - sum) - y
        sum = t
    }
    return sum
}

// Welford online variance
func varianceStable(x []float64) float64 {
    mean, m2 := 0.0, 0.0
    for i, xi := range x {
        n := float64(i + 1)
        delta := xi - mean
        mean += delta / n
        m2 += delta * (xi - mean)
    }
    return m2 / float64(len(x)-1)
}
Go follows IEEE 754 for all float operations. Unlike Rust, Go allows direct comparison and sorting of floats (NaN sorts unpredictably). There is no compile-time warning for NaN-unsafe operations — boundary validation and explicit math.IsNaN checks are your only defence.

TypeScript numerical accuracy

NaN and Inf handling

// JavaScript/TypeScript follows IEEE 754
const x = NaN;
console.log(x + 1);        // NaN
console.log(x > 0);        // false - no error!
console.log(x === x);      // false (NaN !== NaN)
console.log(Number.isNaN(x));      // true (prefer over global isNaN)
console.log(Number.isFinite(x));   // false

// DANGER: global isNaN vs Number.isNaN
console.log(isNaN("hello"));       // true (coerces string!)
console.log(Number.isNaN("hello")); // false (correct)

// Validate at boundaries
function validateInputs(
  S: number, K: number, T: number, sigma: number
): void {
  const params = { S, K, T, sigma };
  for (const [name, val] of Object.entries(params)) {
    if (!Number.isFinite(val)) {
      throw new Error(`${name} is not finite: ${val}`);
    }
    if (val <= 0) {
      throw new Error(`${name} must be positive: ${val}`);
    }
  }
}

// Kahan compensated summation
function kahanSum(values: number[]): number {
  let sum = 0, c = 0;
  for (const v of values) {
    const y = v - c;
    const t = sum + y;
    c = (t - sum) - y;
    sum = t;
  }
  return sum;
}

// Branded type for validated numbers
type FinitePositive = number & { readonly __brand: "FinitePositive" };

function toFinitePositive(v: number): FinitePositive {
  if (!Number.isFinite(v) || v <= 0) {
    throw new Error(`Not a finite positive number: ${v}`);
  }
  return v as FinitePositive;
}
TypeScript has only one number type (number = IEEE 754 double). There is no float vs double distinction. Use Number.isFinite (not isFinite) and Number.isNaN (not isNaN) to avoid type-coercion traps. Branded types can encode "validated" in the type system, similar to Rust newtypes.

Key observations

  • Rust is the only language where NaN comparison is a compile-time concern: f64 does not implement Ord, preventing accidental NaN-unsafe sorting. Newtypes can encode "finite and positive" directly in the type system.
  • C++ gives maximum precision control (float, double, long double, compiler flags) but -ffast-math can silently break Kahan summation and NaN checks. This is the biggest numerical footgun in any language here.
  • Python has arbitrary precision integers and decimal.Decimal for exact decimal arithmetic — useful for settlement calculations where floating-point is unacceptable.
  • C# offers decimal (128-bit exact decimal) as a first-class type, ideal for financial calculations where rounding matters.
  • Go has math/big for arbitrary precision but no special compile-time NaN protection. Validation is purely runtime.
  • TypeScript has a single number type (64-bit double), no float/int distinction, and a notorious isNaN vs Number.isNaN trap.

Decision guide

  • Settlement / accounting (exact decimals): C# decimal or Python Decimal
  • Monte Carlo / pricing (speed + precision): C++ or Rust (f64, Kahan summation)
  • Research / prototyping: Python (NumPy handles precision; focus on correctness first)
  • Type-safe numerical pipelines: Rust (newtypes prevent NaN propagation at compile time)
  • Full-stack dashboards displaying prices: TypeScript (but validate on the server side)
