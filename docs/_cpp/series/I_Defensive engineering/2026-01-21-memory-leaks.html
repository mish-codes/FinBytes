Three memory problems

Memory leaks: you allocate and never free. The process grows until killed. Memory fragmentation: repeated alloc/free leaves the heap full of unusable gaps. Memory bloat: you use far more memory than needed - large containers, unnecessary copies, objects that live too long.

A risk engine that leaks 1KB per trade processing a million trades per day uses 1GB of extra memory by end of day. A market data handler that fragments the heap will eventually fail an allocation mid-session. A pricer that copies large vol surfaces unnecessarily will be slower than it needs to be.

Detection tools: C++: Valgrind (Linux), AddressSanitizer (-fsanitize=address). Python: tracemalloc, memory_profiler. C#: dotMemory, Visual Studio Diagnostic Tools, BenchmarkDotNet [MemoryDiagnoser].

C++ memory tracking and prevention

// Build with AddressSanitizer
// g++ -fsanitize=address,leak -fno-omit-frame-pointer -g -O1 pricer.cpp
// Valgrind: valgrind --leak-check=full ./pricer

// Common leak: raw pointer escape
void bad_function() {
    double* prices = new double[1000];  // leak if exception thrown
    compute(prices);                     // if this throws, prices leaks
    delete[] prices;
}

// Fix: RAII
void good_function() {
    std::vector<double> prices(1000);  // freed on exit, always
    compute(prices.data());
}

// Pool allocator: prevent fragmentation, eliminate heap calls
#include <array>
template<typename T, std::size_t N>
class PoolAllocator {
    std::array<T, N>    pool_;
    std::array<bool, N> used_{};
public:
    T* allocate() {
        for (std::size_t i = 0; i < N; ++i)
            if (!used_[i]) { used_[i] = true; return &pool_[i]; }
        return nullptr;
    }
    void deallocate(T* p) {
        auto idx = p - pool_.data();
        if (idx >= 0 && idx < (int)N) used_[idx] = false;
    }
};
For a pricing loop creating and destroying many small objects, a pool allocator can outperform the system allocator by 5-10x. Pre-allocate a fixed block at startup; allocation becomes a pointer increment. No heap calls, no fragmentation.

Rust memory safety

Ownership eliminates leaks by design

// Rust's ownership system means memory is freed when the owner goes out of scope.
// No garbage collector, no manual free, no leaks (in safe code).

fn good_function() {
    let prices: Vec<f64> = vec![0.0; 1000]; // heap-allocated
    compute(&prices);
}   // prices dropped here - memory freed automatically

// The equivalent of C++ raw pointer escape is impossible in safe Rust:
// fn bad_function() {
//     let prices = Box::new([0.0; 1000]);
//     compute(&prices);
//     // forget to free? Compiler handles it. Exception? Still freed.
// }

// Move semantics are the default - no accidental copies
fn transfer_ownership() {
    let data = vec![1.0, 2.0, 3.0];
    let moved = data;           // data is MOVED, not copied
    // println!("{:?}", data);  // compile error: data was moved
    println!("{:?}", moved);    // only moved is valid
}

Arena / pool allocator with bumpalo

use bumpalo::Bump;

fn pricing_loop(trades: &[Trade]) {
    let arena = Bump::new();  // arena allocator

    for trade in trades {
        // Allocate from the arena - no individual frees
        let result = arena.alloc(PricingResult::default());
        price_into(trade, result);
    }
}   // entire arena freed at once - zero fragmentation

// Detecting leaks: Rc/Arc cycles are the one exception
use std::rc::Rc;
use std::cell::RefCell;

struct Node {
    parent: Option<Rc<RefCell<Node>>>,  // Rc cycle = leak
}
// Fix: use Weak references for back-pointers
use std::rc::Weak;
struct SafeNode {
    parent: Option<Weak<RefCell<SafeNode>>>,  // Weak breaks the cycle
}
Rust eliminates use-after-free, double-free, and most memory leaks at compile time through ownership and borrowing. The only way to leak in safe Rust is via Rc/Arc reference cycles or explicit std::mem::forget. Use Weak references to break cycles.

Go memory management

Garbage collector handles the basics

package main

import (
    "fmt"
    "runtime"
)

// Go's GC frees unreachable memory automatically.
// Leaks happen when references are held longer than needed.

// Common leak: goroutine that never exits
func leakyGoroutine() {
    ch := make(chan int)        // unbuffered channel
    go func() {
        val := <-ch            // blocks forever if nobody sends
        fmt.Println(val)
    }()
    // ch is never sent to - goroutine leaks (and anything it references)
}

// Fix: use context for cancellation
func safeGoroutine(ctx context.Context) {
    ch := make(chan int)
    go func() {
        select {
        case val := <-ch:
            fmt.Println(val)
        case <-ctx.Done():
            return  // goroutine exits cleanly
        }
    }()
}

// Monitor GC and memory stats
func printMemStats() {
    var m runtime.MemStats
    runtime.ReadMemStats(&m)
    fmt.Printf("Alloc: %d MB, Sys: %d MB, NumGC: %d\n",
        m.Alloc/1024/1024, m.Sys/1024/1024, m.NumGC)
}

// Reduce allocations: sync.Pool for hot-path objects
import "sync"

var resultPool = sync.Pool{
    New: func() interface{} { return &PricingResult{} },
}

func priceWithPool(trade Trade) float64 {
    result := resultPool.Get().(*PricingResult)
    defer resultPool.Put(result)  // returned to pool after use
    result.Reset()
    return computePrice(trade, result)
}
Go's most common memory leak is a goroutine that blocks forever, holding references the GC cannot collect. Always use context.Context for cancellation. Use runtime.ReadMemStats and pprof to detect leaks. sync.Pool reduces GC pressure for frequently allocated objects.

TypeScript memory tracking

Common leak patterns in Node.js / browser

// Leak 1: event listeners that are never removed
class LeakyPricer {
  constructor(private feed: EventEmitter) {
    // Every time we construct, we add a listener - but never remove it
    feed.on("tick", (price: number) => this.onTick(price));
  }
  private onTick(price: number): void { /* ... */ }
}

// Fix: remove listeners on cleanup
class SafePricer {
  private handler: (price: number) => void;

  constructor(private feed: EventEmitter) {
    this.handler = (price) => this.onTick(price);
    feed.on("tick", this.handler);
  }

  dispose(): void {
    this.feed.off("tick", this.handler);  // prevent leak
  }

  private onTick(price: number): void { /* ... */ }
}

// Leak 2: closures capturing large objects
function createPricer(hugeDataset: number[]): () => number {
  // The closure captures hugeDataset even if it only needs length
  return () => hugeDataset.length;  // hugeDataset stays in memory
}
// Fix: capture only what you need
function createPricerFixed(hugeDataset: number[]): () => number {
  const len = hugeDataset.length;
  return () => len;  // hugeDataset can be GC'd
}

// Leak 3: Maps/Sets that grow without bound
const cache = new Map<string, number>();
function getCachedPrice(key: string): number {
  if (!cache.has(key)) cache.set(key, computePrice(key));
  return cache.get(key)!;  // cache grows forever
}

// Fix: WeakRef or LRU cache
class LRUCache<K, V> {
  private map = new Map<K, V>();
  constructor(private maxSize: number) {}
  get(key: K): V | undefined { return this.map.get(key); }
  set(key: K, value: V): void {
    if (this.map.size >= this.maxSize) {
      const oldest = this.map.keys().next().value!;
      this.map.delete(oldest);
    }
    this.map.set(key, value);
  }
}

Key observations

  • Rust prevents use-after-free, double-free, and most leaks at compile time. The borrow checker is the strictest memory safety enforcer of any systems language. The trade-off: you fight the compiler until ownership patterns become second nature.
  • C++ gives maximum control over allocation (custom allocators, placement new, pool allocators) but requires discipline. RAII makes it manageable; raw new/delete does not.
  • Go has a low-latency GC (sub-millisecond pauses) but the most common leak pattern — blocked goroutines — is unique to Go and invisible to the GC.
  • Python and TypeScript have the least control over memory layout. Circular references (Python) and retained closures (TypeScript) are the primary leak vectors.
  • C# offers the best middle ground: managed GC with Span<T> and ArrayPool for hot paths, plus excellent profiling tools (dotMemory, VS Diagnostics).

Decision guide

  • Zero GC pauses required (HFT, real-time): C++ or Rust
  • High throughput with predictable latency: Go (low-pause GC) or Rust
  • Enterprise backend with good tooling: C# (managed + profiling)
  • Rapid prototyping, data pipelines: Python
  • Full-stack / browser-based dashboards: TypeScript
  • Memory-critical embedded or pricing loops: Rust (safety) or C++ (control)
