When code stops making progress

Infinite loops and deadlocks are insidious: the process is running but not producing results. No exception is thrown, no crash occurs. Without monitoring they go undetected until someone notices the system is unresponsive.

Common causes: loop condition that never becomes false, floating-point comparison that never terminates (NaN != NaN always), off-by-one in convergence check, deadlock (two threads each hold a lock the other needs), livelock (threads keep changing state in response to each other but make no progress).

Prevention: always impose an upper bound on iterations. Use timeouts on blocking operations. For numerical algorithms (Newton-Raphson, vol calibration), cap the iteration count. Never rely on convergence alone. Check std::isfinite(x) in convergence loops.

Python loop guards and watchdog

import signal, time, threading, os
from contextlib import contextmanager

# Iteration cap for numerical algorithms
def newton_raphson(f, df, x0: float, tol=1e-8, max_iter=100) -> float:
    x = x0
    for i in range(max_iter):
        fx = f(x)
        if abs(fx) < tol: return x
        dfx = df(x)
        if abs(dfx) < 1e-15: raise ValueError("Zero derivative")
        x -= fx / dfx
    raise RuntimeError(f"No convergence after {max_iter} iterations")

# Timeout context manager (Unix only)
@contextmanager
def timeout(seconds: float):
    def handler(signum, frame): raise TimeoutError(f"Timed out after {seconds}s")
    old = signal.signal(signal.SIGALRM, handler)
    signal.setitimer(signal.ITIMER_REAL, seconds)
    try: yield
    finally: signal.setitimer(signal.ITIMER_REAL, 0); signal.signal(signal.SIGALRM, old)

# Watchdog thread
class Watchdog:
    def __init__(self, timeout_s: float):
        self._timeout = timeout_s
        self._last    = time.monotonic()
        threading.Thread(target=self._watch, daemon=True).start()

    def heartbeat(self) -> None: self._last = time.monotonic()

    def _watch(self) -> None:
        while True:
            time.sleep(1.0)
            if time.monotonic() - self._last > self._timeout:
                print(f"WATCHDOG: no heartbeat - killing")
                os.kill(os.getpid(), signal.SIGABRT)

C++ watchdog and loop guards

#include <atomic>
#include <thread>
#include <chrono>
#include <cmath>
#include <stdexcept>
#include <string>

// Iteration cap with NaN guard
double newton_raphson(auto f, auto df, double x0, double tol=1e-8, int max_iter=100) {
    double x = x0;
    for (int i = 0; i < max_iter; ++i) {
        if (!std::isfinite(x)) throw std::runtime_error("NaN/Inf in iteration");
        double fx  = f(x);
        if (std::abs(fx) < tol) return x;
        double dfx = df(x);
        if (std::abs(dfx) < 1e-15) throw std::runtime_error("zero derivative");
        x -= fx / dfx;
    }
    throw std::runtime_error("no convergence after " + std::to_string(max_iter) + " iterations");
}

// Watchdog thread
class Watchdog {
    std::atomic<int64_t> last_beat_{0};
    std::atomic<bool>    running_{true};
    std::thread          watcher_;
    int64_t              timeout_ns_;
public:
    explicit Watchdog(std::chrono::milliseconds timeout) {
        timeout_ns_ = std::chrono::nanoseconds(timeout).count();
        heartbeat();
        watcher_ = std::thread([this] {
            while (running_.load()) {
                std::this_thread::sleep_for(std::chrono::milliseconds(500));
                auto now  = std::chrono::steady_clock::now().time_since_epoch().count();
                auto last = last_beat_.load(std::memory_order_relaxed);
                if (now - last > timeout_ns_) { std::cerr << "WATCHDOG: aborting
"; std::abort(); }
            }
        });
    }
    void heartbeat() {
        last_beat_.store(std::chrono::steady_clock::now().time_since_epoch().count(),
                         std::memory_order_relaxed);
    }
    ~Watchdog() { running_ = false; if (watcher_.joinable()) watcher_.join(); }
};

NaN causes infinite loops in numerical code. NaN != NaN is always true, so while (abs(x) > tol) loops forever if x becomes NaN. Always add std::isfinite(x) to convergence checks.

Rust loop guards and timeouts

Iteration caps and NaN guards

/// Newton-Raphson with iteration cap and NaN guard.
fn newton_raphson(
    f:  impl Fn(f64) -> f64,
    df: impl Fn(f64) -> f64,
    x0: f64,
    tol: f64,
    max_iter: usize,
) -> Result<f64, String> {
    let mut x = x0;
    for i in 0..max_iter {
        if !x.is_finite() {
            return Err(format!("NaN/Inf at iteration {i}"));
        }
        let fx = f(x);
        if fx.abs() < tol { return Ok(x); }
        let dfx = df(x);
        if dfx.abs() < 1e-15 {
            return Err("Zero derivative".into());
        }
        x -= fx / dfx;
    }
    Err(format!("No convergence after {max_iter} iterations"))
}

// Watchdog via crossbeam channel
use std::sync::atomic::{AtomicI64, Ordering};
use std::sync::Arc;
use std::thread;
use std::time::{Duration as StdDuration, Instant};

struct Watchdog {
    last_beat: Arc<AtomicI64>,
}

impl Watchdog {
    fn new(timeout: StdDuration) -> Self {
        let last_beat = Arc::new(AtomicI64::new(
            Instant::now().elapsed().as_nanos() as i64
        ));
        let lb = Arc::clone(&last_beat);
        let timeout_ns = timeout.as_nanos() as i64;

        thread::spawn(move || {
            let start = Instant::now();
            loop {
                thread::sleep(StdDuration::from_millis(500));
                let now = start.elapsed().as_nanos() as i64;
                let last = lb.load(Ordering::Relaxed);
                if now - last > timeout_ns {
                    eprintln!("WATCHDOG: no heartbeat - aborting");
                    std::process::abort();
                }
            }
        });
        Self { last_beat }
    }

    fn heartbeat(&self) {
        // Update with current elapsed time
        self.last_beat.store(0, Ordering::Relaxed); // simplified
    }
}
Rust's Result type forces callers to handle convergence failure explicitly — you cannot accidentally ignore a "no convergence" error. The tokio::time::timeout wrapper makes async timeouts a one-liner. For CPU-bound loops, use iteration caps just like C++.

Go timeout and loop guards

Context-based timeouts

package main

import (
    "context"
    "fmt"
    "math"
    "time"
)

// Newton-Raphson with iteration cap
func newtonRaphson(f, df func(float64) float64, x0, tol float64, maxIter int) (float64, error) {
    x := x0
    for i := 0; i < maxIter; i++ {
        if math.IsNaN(x) || math.IsInf(x, 0) {
            return 0, fmt.Errorf("NaN/Inf at iteration %d", i)
        }
        fx := f(x)
        if math.Abs(fx) < tol { return x, nil }
        dfx := df(x)
        if math.Abs(dfx) < 1e-15 {
            return 0, fmt.Errorf("zero derivative at iteration %d", i)
        }
        x -= fx / dfx
    }
    return 0, fmt.Errorf("no convergence after %d iterations", maxIter)
}

// Context timeout for async operations
func priceWithTimeout(ctx context.Context, trade Trade) (float64, error) {
    ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()

    resultCh := make(chan float64, 1)
    errCh    := make(chan error, 1)

    go func() {
        price, err := computePrice(trade)
        if err != nil { errCh <- err; return }
        resultCh <- price
    }()

    select {
    case price := <-resultCh:
        return price, nil
    case err := <-errCh:
        return 0, err
    case <-ctx.Done():
        return 0, fmt.Errorf("pricing timed out: %w", ctx.Err())
    }
}

// "fatal error: all goroutines are asleep - deadlock!"
// Use go run -race to detect data races that might cause hangs
Go's context.Context is the standard way to propagate timeouts and cancellation through call chains. The select statement with ctx.Done() makes it impossible to forget the timeout path. Go's runtime also detects global deadlocks automatically.

TypeScript timeout and loop guards

AbortController and iteration caps

// Newton-Raphson with iteration cap
function newtonRaphson(
  f:  (x: number) => number,
  df: (x: number) => number,
  x0: number,
  tol = 1e-8,
  maxIter = 100,
): number {
  let x = x0;
  for (let i = 0; i < maxIter; i++) {
    if (!Number.isFinite(x)) throw new Error(`NaN/Inf at iteration ${i}`);
    const fx = f(x);
    if (Math.abs(fx) < tol) return x;
    const dfx = df(x);
    if (Math.abs(dfx) < 1e-15) throw new Error("Zero derivative");
    x -= fx / dfx;
  }
  throw new Error(`No convergence after ${maxIter} iterations`);
}

// Async timeout with AbortController
async function priceWithTimeout(
  trade: Trade,
  timeoutMs = 5000,
): Promise<number> {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), timeoutMs);

  try {
    return await computePrice(trade, controller.signal);
  } catch (err) {
    if (controller.signal.aborted) {
      throw new Error(`Pricing timed out after ${timeoutMs}ms`);
    }
    throw err;
  } finally {
    clearTimeout(timer);
  }
}

// Promise.race as timeout wrapper
function withTimeout<T>(promise: Promise<T>, ms: number): Promise<T> {
  const timeout = new Promise<never>((_, reject) =>
    setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms)
  );
  return Promise.race([promise, timeout]);
}

const price = await withTimeout(computePrice(trade), 5000);
JavaScript is single-threaded, so true deadlocks between threads are impossible. However, the event loop can hang if a synchronous computation runs too long. Use AbortController (modern standard) for cancellation, and Promise.race for simple timeouts. Worker threads can deadlock on shared SharedArrayBuffer locks.

Key observations

  • Go has the best timeout story — context.Context propagates deadlines through every layer of the call stack. The runtime also detects global deadlocks automatically (all goroutines blocked).
  • Rust forces you to handle timeout results via Result — you cannot accidentally ignore a timeout. The borrow checker also prevents some deadlock patterns by disallowing shared mutable state without synchronisation.
  • C# CancellationToken is well-integrated into the async ecosystem. Every async API in .NET accepts one, making timeout propagation natural.
  • TypeScript is single-threaded, so classical deadlocks are rare, but the event loop can freeze on synchronous computation. AbortController is the modern cancellation primitive.
  • NaN-induced infinite loops affect all languages equally — IEEE 754 is universal. Every language needs explicit isFinite/IsNaN checks in convergence loops.
  • C++ has the weakest built-in timeout support; you must build watchdog threads manually. ThreadSanitiser helps detect deadlocks but is a development-only tool.

Decision guide

  • Microservices with cascading timeouts: Go (context propagation) or C# (CancellationToken)
  • Numerical algorithms: all languages need iteration caps + NaN guards — no language is exempt
  • Real-time systems: C++ or Rust (watchdog threads, no GC pauses that mask hangs)
  • Async I/O pipelines: TypeScript (AbortController) or Rust (tokio)
