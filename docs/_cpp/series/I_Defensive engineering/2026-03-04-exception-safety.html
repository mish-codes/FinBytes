Three levels of exception safety

  • No-throw: the function never throws. Destructors, swap, moves. Mark with noexcept.
  • Strong: if an exception escapes, program state is unchanged - like a database transaction. Commit or roll back, nothing in between.
  • Basic: if an exception escapes, the program is in a valid but unspecified state. No resources are leaked, no invariants violated.

A portfolio update that partially modifies positions before throwing leaves the book inconsistent. The risk engine sees half-updated positions and computes wrong Greeks. Strong exception safety prevents this: either the entire update succeeds, or nothing changes.

Copy-and-swap: work on a copy, then swap atomically. Since swap is no-throw, the commit is guaranteed to succeed once the risky work (copying) is done.

Python exception safety

# Strong safety: all-or-nothing via copy-then-replace
class Portfolio:
    def __init__(self): self._positions: dict[str, float] = {}

    # Basic safety - partial update possible on failure
    def update_basic(self, updates: dict[str, float]) -> None:
        for sym, qty in updates.items():
            validate(sym, qty)           # may raise
            self._positions[sym] = qty   # partial update if later raises

    # Strong safety - all-or-nothing
    def update_strong(self, updates: dict[str, float]) -> None:
        new_positions = dict(self._positions)  # copy
        for sym, qty in updates.items():
            validate(sym, qty)           # may raise - original unchanged
            new_positions[sym] = qty
        self._positions = new_positions  # atomic replace - does not raise

def validate(sym: str, qty: float) -> None:
    if not sym:  raise ValueError("empty symbol")
    if qty < 0:  raise ValueError(f"negative quantity: {qty}")

C# exception safety

public class Portfolio {
    private Dictionary<string, double> _positions = new();

    // Strong safety via copy-then-replace
    public void UpdateStrong(Dictionary<string, double> updates) {
        var copy = new Dictionary<string, double>(_positions);
        foreach (var (sym, qty) in updates) {
            Validate(sym, qty);           // may throw - original untouched
            copy[sym] = qty;
        }
        _positions = copy;                // atomic replace - does not throw
    }

    // No-throw swap
    public void SwapWith(Portfolio other) {
        (_positions, other._positions) = (other._positions, _positions);
    }

    static void Validate(string sym, double qty) {
        if (string.IsNullOrEmpty(sym)) throw new ArgumentException("empty symbol");
        if (qty < 0) throw new ArgumentException($"negative quantity: {qty}");
    }
}

C++ copy-and-swap for strong guarantee

#include <unordered_map>
#include <string>
#include <stdexcept>
#include <utility>

class Portfolio {
    std::unordered_map<std::string, double> positions_;

    static void validate(const std::string& sym, double qty) {
        if (sym.empty()) throw std::invalid_argument("empty symbol");
        if (qty < 0)     throw std::invalid_argument("negative quantity");
    }

public:
    // Basic guarantee
    void update_basic(const std::unordered_map<std::string, double>& updates) {
        for (const auto& [sym, qty] : updates) {
            validate(sym, qty);         // may throw - partial update possible
            positions_[sym] = qty;
        }
    }

    // Strong guarantee via copy-and-swap
    void update_strong(const std::unordered_map<std::string, double>& updates) {
        auto copy = positions_;         // copy - risky part
        for (const auto& [sym, qty] : updates) {
            validate(sym, qty);         // may throw - original untouched
            copy[sym] = qty;
        }
        using std::swap;
        swap(positions_, copy);         // noexcept commit
    }

    // No-throw swap
    friend void swap(Portfolio& a, Portfolio& b) noexcept {
        using std::swap;
        swap(a.positions_, b.positions_);
    }

    // Copy-and-swap assignment operator
    Portfolio& operator=(Portfolio other) noexcept {
        swap(*this, other);
        return *this;
    }
};
Mark destructors, swap, and move operations noexcept. These are the building blocks of exception safety. std::vector uses move during reallocation only if the move constructor is noexcept - without it, it falls back to copying, which is slower.

Rust error handling — no exceptions

Result<T, E> replaces exceptions entirely

use std::collections::HashMap;

#[derive(Debug)]
enum PortfolioError {
    EmptySymbol,
    NegativeQuantity(f64),
}

struct Portfolio {
    positions: HashMap<String, f64>,
}

impl Portfolio {
    fn new() -> Self { Self { positions: HashMap::new() } }

    // "Basic safety" is the default in Rust - no partial state possible
    // because Result forces all-or-nothing at each step.

    // Strong guarantee via clone-then-replace
    fn update_strong(
        &mut self,
        updates: &[(String, f64)],
    ) -> Result<(), PortfolioError> {
        let mut copy = self.positions.clone();  // copy - the risky part
        for (sym, qty) in updates {
            Self::validate(sym, *qty)?;          // ? returns early on error
            copy.insert(sym.clone(), *qty);      // original untouched
        }
        self.positions = copy;                   // commit - infallible
        Ok(())
    }

    fn validate(sym: &str, qty: f64) -> Result<(), PortfolioError> {
        if sym.is_empty()  { return Err(PortfolioError::EmptySymbol); }
        if qty < 0.0       { return Err(PortfolioError::NegativeQuantity(qty)); }
        Ok(())
    }
}

The ? operator and error propagation

// The ? operator propagates errors without try/catch boilerplate.
// It is the Rust equivalent of exception propagation, but explicit.

fn process_trade(trade: &Trade) -> Result<f64, Box<dyn std::error::Error>> {
    let validated = validate_trade(trade)?;       // propagates error
    let market_data = fetch_market_data()?;       // propagates error
    let price = compute_price(&validated, &market_data)?;
    Ok(price)  // only reached if all steps succeeded
}

// Panics are Rust's equivalent of "unrecoverable crash" - not for control flow.
// catch_unwind provides a boundary similar to C++ catch(...):
use std::panic;

fn safe_price_handler(trade: &Trade) -> Option<f64> {
    panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
        compute_price(trade)
    })).ok()
}
Rust has no exceptions. All recoverable errors use Result<T, E>, which the compiler forces you to handle. This eliminates the "exception safety" problem entirely — there is no way for an exception to leave an object in a partially-updated state. Panics exist for unrecoverable bugs (like C++ abort), not for normal error handling.

Go error handling — explicit error returns

Errors are values, not control flow

package main

import (
    "errors"
    "fmt"
)

type Portfolio struct {
    positions map[string]float64
}

func NewPortfolio() *Portfolio {
    return &Portfolio{positions: make(map[string]float64)}
}

// Strong guarantee via copy-then-replace
func (p *Portfolio) UpdateStrong(updates map[string]float64) error {
    // Copy current state
    copy := make(map[string]float64, len(p.positions))
    for k, v := range p.positions {
        copy[k] = v
    }

    // Apply updates to copy - original untouched on error
    for sym, qty := range updates {
        if err := validate(sym, qty); err != nil {
            return fmt.Errorf("validation failed for %s: %w", sym, err)
        }
        copy[sym] = qty
    }

    p.positions = copy  // atomic commit
    return nil
}

var (
    ErrEmptySymbol      = errors.New("empty symbol")
    ErrNegativeQuantity = errors.New("negative quantity")
)

func validate(sym string, qty float64) error {
    if sym == "" { return ErrEmptySymbol }
    if qty < 0   { return fmt.Errorf("%w: %f", ErrNegativeQuantity, qty) }
    return nil
}

// defer for cleanup - Go's version of RAII
func processFile(path string) error {
    f, err := os.Open(path)
    if err != nil { return err }
    defer f.Close()  // guaranteed cleanup, like a C++ destructor

    // process f...
    return nil
}
Go uses explicit error returns instead of exceptions. The if err != nil pattern is verbose but makes error handling visible at every call site. defer provides cleanup guarantees similar to RAII. Go's panic/recover exists but is reserved for truly unrecoverable situations — never use it for normal error handling.

TypeScript exception safety

Strong guarantee via copy-then-replace

class Portfolio {
  private positions = new Map<string, number>();

  // Strong safety: all-or-nothing
  updateStrong(updates: Map<string, number>): void {
    const copy = new Map(this.positions);  // shallow copy

    for (const [sym, qty] of updates) {
      validate(sym, qty);       // throws on failure - original untouched
      copy.set(sym, qty);
    }

    this.positions = copy;      // atomic commit - does not throw
  }

  // Swap (no-throw equivalent)
  swapWith(other: Portfolio): void {
    [this.positions, other.positions] = [other.positions, this.positions];
  }
}

function validate(sym: string, qty: number): void {
  if (!sym) throw new Error("Empty symbol");
  if (qty < 0) throw new Error(`Negative quantity: ${qty}`);
  if (!Number.isFinite(qty)) throw new Error(`Non-finite quantity: ${qty}`);
}

// TypeScript Result pattern (for when exceptions feel wrong)
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function safePrice(trade: Trade): Result<number> {
  try {
    const price = computePrice(trade);
    return { ok: true, value: price };
  } catch (err) {
    return { ok: false, error: err as Error };
  }
}

// Using the result
const result = safePrice(trade);
if (result.ok) {
  publish(result.value);
} else {
  logError(result.error);
}

finally for resource cleanup

// TypeScript has no RAII - use try/finally or Disposable (TC39 stage 3)
async function processWithCleanup(path: string): Promise<void> {
  const handle = await openFile(path);
  try {
    await processData(handle);
  } finally {
    await handle.close();  // always runs, even on exception
  }
}

// Symbol.dispose (new in TypeScript 5.2+)
{
  using handle = await openFile(path);  // auto-disposed at end of block
  await processData(handle);
}   // handle[Symbol.dispose]() called automatically
TypeScript uses exceptions like C# and Python. The copy-then-replace pattern works identically for strong guarantees. For code that wants explicit error handling without exceptions, use a discriminated union Result type — TypeScript's type narrowing makes it ergonomic. The new using declaration (TC39 Explicit Resource Management) brings RAII-like cleanup to JavaScript.

Key observations

  • Rust eliminates the exception safety problem entirely. Since there are no exceptions, there is no risk of an exception leaving an object in a partially-updated state. Result<T, E> forces explicit handling at every call site. The ? operator makes this ergonomic.
  • Go takes a similar approach with explicit error returns. Errors cannot be silently ignored (the compiler warns on unused variables). defer provides cleanup guarantees without RAII's automatic destructor model.
  • C++ RAII is the strongest cleanup mechanism — destructors run unconditionally. But exception safety guarantees (no-throw, strong, basic) must be manually reasoned about for every function. noexcept is critical for performance (move semantics).
  • Python, C#, and TypeScript all use exceptions. The copy-then-replace pattern for strong safety is identical across all three. The risk is that any function might throw, making it easy to leave state inconsistent.
  • Rust and C++ both have RAII (Rust calls it Drop). Go has defer. Python/C#/TypeScript rely on with/using/finally — forgetting the cleanup block leaks resources.

Decision guide

  • Strongest error handling guarantees: Rust (Result + ownership = no partial updates, no ignored errors)
  • Best resource cleanup: C++ or Rust (RAII — destructors are automatic and unconditional)
  • Most explicit error visibility: Go (every error checked at every call site)
  • Fastest to write: Python/TypeScript (exceptions are implicit — less boilerplate, more risk)
  • Enterprise standard: C# (exceptions + using + nullable reference types)
