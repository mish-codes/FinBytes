Resources are finite

A process can have at most ~65535 open file descriptors on Linux by default. A thread pool has a maximum size. A database connection pool has a configured limit. Exhaust any of these and new operations fail - often with cryptic errors far from the actual cause.

Common patterns: file handle leak (open files in a loop, forget to close - eventually open() returns EMFILE); thread pool saturation (queue tasks faster than they complete, latency spikes); connection pool starvation (long transactions hold connections, new requests wait indefinitely).

Prevention: RAII for everything. Set explicit limits. Monitor pool utilisation. Reject new work when approaching limits (backpressure) rather than queuing infinitely. Alert at 70% utilisation; page at 85%.

Python resource management

import resource
from contextlib import contextmanager
from queue import Queue, Empty
import time

# File handle exhaustion - always use context managers
def bad_reader(path: str):
    f = open(path)
    data = f.read()  # if this raises, f is never closed
    f.close()

def good_reader(path: str):
    with open(path) as f:   # guaranteed close even on exception
        return f.read()

# Check current limits
soft, hard = resource.getrlimit(resource.RLIMIT_NOFILE)
print(f"File descriptor limit: {soft}/{hard}")

# Thread pool with explicit limit
from concurrent.futures import ThreadPoolExecutor
with ThreadPoolExecutor(max_workers=8) as executor:
    futures = [executor.submit(price_trade, t) for t in trades]

# Connection pool
class ConnectionPool:
    def __init__(self, size: int = 10, timeout: float = 5.0):
        self._pool    = Queue(maxsize=size)
        self._timeout = timeout
        for _ in range(size): self._pool.put(self._create_connection())

    @contextmanager
    def get(self):
        try:
            conn = self._pool.get(timeout=self._timeout)
        except Empty:
            raise RuntimeError("Connection pool exhausted")
        try: yield conn
        finally: self._pool.put(conn)  # always returned, even on exception

    def _create_connection(self): return object()

C++ resource limits and RAII

#include <fstream>
#include <semaphore>
#include <queue>
#include <mutex>
#include <condition_variable>
#include <stdexcept>

// RAII: file always closed
std::string read_file(const std::string& path) {
    std::ifstream f(path);
    if (!f.is_open()) throw std::runtime_error("Cannot open: " + path);
    return {std::istreambuf_iterator<char>(f), std::istreambuf_iterator<char>()};
}   // f.~ifstream() called - file closed even on exception

// Semaphore as concurrency limiter (C++20)
std::counting_semaphore<64> sem(10);  // max 10 concurrent pricers

double price_with_limit(const Trade& trade) {
    sem.acquire();   // blocks if 10 already running
    struct Guard { std::counting_semaphore<64>& s; ~Guard() { s.release(); } } g{sem};
    return price(trade);
}   // sem released via RAII

// Connection pool
template<typename Conn>
class ConnectionPool {
    std::queue<std::unique_ptr<Conn>> pool_;
    std::mutex mtx_; std::condition_variable cv_;
public:
    explicit ConnectionPool(int size) {
        for (int i = 0; i < size; ++i) pool_.push(std::make_unique<Conn>());
    }
    std::unique_ptr<Conn> acquire(std::chrono::milliseconds timeout = std::chrono::seconds(5)) {
        std::unique_lock lock(mtx_);
        if (!cv_.wait_for(lock, timeout, [&]{ return !pool_.empty(); }))
            throw std::runtime_error("Connection pool exhausted");
        auto conn = std::move(pool_.front()); pool_.pop(); return conn;
    }
    void release(std::unique_ptr<Conn> conn) {
        std::lock_guard lock(mtx_); pool_.push(std::move(conn)); cv_.notify_one();
    }
};
Set ulimit -n (file descriptors) and ulimit -u (threads) explicitly before launching production processes. Know your limits before you hit them. Instrument pool utilisation - 80% is a capacity warning, not yet a failure.

Rust resource management

Drop trait = automatic cleanup

use std::fs::File;
use std::io::{self, Read};

// Files are closed automatically when dropped - no leak possible
fn read_file(path: &str) -> io::Result<String> {
    let mut f = File::open(path)?;  // if this fails, no resource acquired
    let mut s = String::new();
    f.read_to_string(&mut s)?;
    Ok(s)
}   // f dropped here - file closed even on early return via ?

// Semaphore for concurrency limiting
use tokio::sync::Semaphore;
use std::sync::Arc;

async fn price_with_limit(
    sem: Arc<Semaphore>,
    trade: Trade,
) -> Result<f64, String> {
    let _permit = sem.acquire().await.map_err(|e| e.to_string())?;
    // permit dropped at end of scope - semaphore released via RAII
    compute_price(&trade).await
}

// Connection pool with deadpool
use deadpool::managed::{Manager, Pool};

struct DbManager;

impl Manager for DbManager {
    type Type = DbConnection;
    type Error = io::Error;

    async fn create(&self) -> Result<DbConnection, io::Error> {
        DbConnection::connect("postgres://...").await
    }

    async fn recycle(&self, conn: &mut DbConnection, _: &deadpool::managed::Metrics)
        -> deadpool::managed::RecycleResult<io::Error>
    {
        conn.ping().await.map_err(Into::into)
    }
}

async fn use_pool(pool: &Pool<DbManager>) -> Result<(), String> {
    let conn = pool.get().await.map_err(|e| e.to_string())?;
    // conn returned to pool when dropped
    conn.query("SELECT 1").await.map_err(|e| e.to_string())
}
Rust's ownership system guarantees that every resource is released exactly once. File handles, mutex guards, semaphore permits, and connection pool leases are all released via the Drop trait when they go out of scope. Forgetting to release is a compile-time impossibility in safe Rust.

Go resource management

defer for guaranteed cleanup

package main

import (
    "context"
    "database/sql"
    "fmt"
    "os"
    "time"

    "golang.org/x/sync/semaphore"
)

// File handle: always defer Close
func readFile(path string) (string, error) {
    f, err := os.Open(path)
    if err != nil { return "", err }
    defer f.Close()  // guaranteed close even if ReadAll fails

    data, err := io.ReadAll(f)
    if err != nil { return "", err }
    return string(data), nil
}

// Semaphore for concurrency limiting
var sem = semaphore.NewWeighted(10)  // max 10 concurrent pricers

func priceWithLimit(ctx context.Context, trade Trade) (float64, error) {
    if err := sem.Acquire(ctx, 1); err != nil {
        return 0, fmt.Errorf("semaphore acquire: %w", err)
    }
    defer sem.Release(1)
    return computePrice(trade)
}

// Connection pool (database/sql has built-in pooling)
func setupDB() (*sql.DB, error) {
    db, err := sql.Open("postgres", "postgres://...")
    if err != nil { return nil, err }

    db.SetMaxOpenConns(25)              // hard limit
    db.SetMaxIdleConns(5)               // keep 5 warm
    db.SetConnMaxLifetime(5 * time.Minute) // recycle stale connections
    return db, nil
}

// Monitor pool stats
func monitorPool(db *sql.DB) {
    stats := db.Stats()
    fmt.Printf("Open: %d, InUse: %d, Idle: %d, WaitCount: %d\n",
        stats.OpenConnections, stats.InUse, stats.Idle, stats.WaitCount)
    if float64(stats.InUse)/float64(stats.MaxOpenConnections) > 0.8 {
        fmt.Println("WARNING: connection pool > 80% utilised")
    }
}
Go's defer is the primary cleanup mechanism. Unlike RAII, defer is explicit — you must remember to write it. But the convention of defer f.Close() immediately after os.Open is well-established. The database/sql package has built-in connection pooling with configurable limits.

TypeScript resource management

try/finally and the new using declaration

import { createReadStream } from "fs";
import { readFile } from "fs/promises";

// File handles: use promises (auto-closed) or try/finally
async function readSafe(path: string): Promise<string> {
  return readFile(path, "utf-8");  // handle opened and closed internally
}

// For manual handles: try/finally
async function processFile(path: string): Promise<void> {
  const handle = await open(path);
  try {
    const data = await handle.readFile("utf-8");
    // process data...
  } finally {
    await handle.close();  // guaranteed cleanup
  }
}

// Explicit Resource Management (TypeScript 5.2+)
// Symbol.dispose for sync, Symbol.asyncDispose for async
class ConnectionPool implements Disposable {
  private pool: Connection[] = [];

  constructor(private maxSize: number) {
    for (let i = 0; i < maxSize; i++) {
      this.pool.push(createConnection());
    }
  }

  acquire(): Connection {
    const conn = this.pool.pop();
    if (!conn) throw new Error("Pool exhausted");
    return conn;
  }

  release(conn: Connection): void {
    this.pool.push(conn);
  }

  [Symbol.dispose](): void {
    for (const conn of this.pool) conn.close();
    this.pool.length = 0;
  }
}

// Usage with 'using' keyword
{
  using pool = new ConnectionPool(10);
  const conn = pool.acquire();
  try {
    await processData(conn);
  } finally {
    pool.release(conn);
  }
}   // pool[Symbol.dispose]() called automatically

// Concurrency limiter
class Semaphore {
  private queue: (() => void)[] = [];
  private count: number;

  constructor(max: number) { this.count = max; }

  async acquire(): Promise<void> {
    if (this.count > 0) { this.count--; return; }
    return new Promise<void>((resolve) => this.queue.push(resolve));
  }

  release(): void {
    const next = this.queue.shift();
    if (next) next(); else this.count++;
  }
}
TypeScript historically lacked RAII. The new Explicit Resource Management proposal (using keyword, Symbol.dispose) brings RAII-like cleanup to JavaScript. Until it is widely adopted, try/finally is the standard pattern. In Node.js, unclosed handles can exhaust file descriptors — always pair open with close.

Key observations

  • Rust and C++ RAII guarantees are the strongest: resources are released the instant the owning scope ends, unconditionally. Rust adds ownership enforcement — you cannot accidentally alias a resource and forget to release one copy.
  • Go has excellent built-in connection pooling (database/sql) with configurable limits. defer is reliable but explicit — you must remember to write it. The convention of defer close() immediately after open is well-established.
  • C# using blocks provide reliable cleanup, and the IDisposable pattern is enforced by analyzers. Connection pooling in ADO.NET is mature and configurable.
  • Python context managers (with) are reliable when used, but nothing forces you to use them. A bare open() without with can leak.
  • TypeScript is catching up with the using declaration (Explicit Resource Management), but it is not yet widely supported. try/finally remains the standard cleanup pattern.
  • Backpressure is critical for resource exhaustion. Go channels with bounded capacity and Rust's bounded async channels both provide natural backpressure. Unbounded queues are a resource exhaustion bug waiting to happen.

Decision guide

  • Leak-proof resource management: Rust (ownership + Drop) or C++ (RAII)
  • Best connection pooling out of the box: Go (database/sql) or C# (ADO.NET)
  • Natural backpressure: Go (bounded channels) or Rust (bounded async channels)
  • Managed environments with GC: C# or Python (GC handles memory; focus on non-memory resources)
  • Browser / Node.js services: TypeScript (adopt using when available; try/finally until then)
