What this is

A single-page reference for defensive coding practices across Python, C#, and C++. Use it as a code review checklist for any system handling financial data.

Input validation (preconditions)

Check at the boundary - the entry point to your system. Do not let bad data travel deep before failing.

# Python
def price_option(S, K, T, sigma):
    assert S > 0,          f"S must be positive: {S}"
    assert K > 0,          f"K must be positive: {K}"
    assert T > 0,          f"T must be positive: {T}"
    assert sigma > 0,      f"sigma must be positive: {sigma}"
    import math
    assert all(math.isfinite(v) for v in (S, K, T, sigma)), "non-finite input"
// C++
double price_option(double S, double K, double T, double sigma) {
    assert(S > 0     && "S must be positive");
    assert(K > 0     && "K must be positive");
    assert(T > 0     && "T must be positive");
    assert(sigma > 0 && "sigma must be positive");
    assert(std::isfinite(S) && std::isfinite(K) && std::isfinite(T) && std::isfinite(sigma));
    // ...
}

Output validation (postconditions)

Validate results before returning. A price of -5 or NaN should never leave the pricer.

def price_option(S, K, T, sigma) -> float:
    result = _price_impl(S, K, T, sigma)
    import math
    assert math.isfinite(result), f"Non-finite result: {result}"
    assert result >= 0,           f"Negative option price: {result}"
    return result

Class invariants

A class invariant is a condition that must be true after every public method. Document them and check in debug builds.

class Portfolio {
    void check_invariants() const {
        assert(!name_.empty()  && "Portfolio must have a name");
        for (const auto& [sym, qty] : positions_)
            assert(qty != 0.0 && "Zero-quantity positions must be removed");
    }
public:
    void add(const std::string& sym, double qty) {
        positions_[sym] += qty;
        if (positions_[sym] == 0.0) positions_.erase(sym);
        check_invariants();  // compile out in release: #ifdef DEBUG
    }
};

Error propagation checklist

  • Every function that can fail: does the caller check the return?
  • Every exception: is there a handler at the appropriate level?
  • Every std::optional: is the empty case handled?
  • Every thread: is there a top-level exception handler?
  • Every callback/async: are errors surfaced, not swallowed?

Code review red flags

Returning raw pointers from functions - who owns this?
Catching all exceptions (except Exception, catch (...)) without logging or re-raising
Shared mutable state with no synchronisation
Numeric operations without NaN/Inf check at boundaries
Loops with no iteration bound
Resource acquisition with no corresponding release
RAII for all resources - destructor releases unconditionally
Fail fast at boundaries - validate inputs before processing
Immutable data passed between threads - no synchronisation needed
Every public API has documented exception/error behaviour

Rust defensive checklist

Input validation (preconditions)

fn price_option(s: f64, k: f64, t: f64, sigma: f64) -> Result<f64, String> {
    // Validate at the boundary - fail fast
    for (name, val) in [("S", s), ("K", k), ("T", t), ("sigma", sigma)] {
        if !val.is_finite() { return Err(format!("{name} is not finite: {val}")); }
        if val <= 0.0       { return Err(format!("{name} must be positive: {val}")); }
    }
    let result = price_impl(s, k, t, sigma);
    // Output validation (postcondition)
    if !result.is_finite() { return Err(format!("Non-finite result: {result}")); }
    if result < 0.0        { return Err(format!("Negative price: {result}")); }
    Ok(result)
}

Null / missing value handling

// Option<T> replaces null - compiler forces handling
let price: Option<f64> = feed.get(sym);
let safe = price.unwrap_or(last_known);  // fallback
let safe = price.ok_or("no price")?;     // propagate error

// Exhaustive enums prevent missing cases
enum TradeStatus { Pending, Filled, Rejected }
fn handle(status: TradeStatus) {
    match status {
        TradeStatus::Pending  => queue_for_execution(),
        TradeStatus::Filled   => update_position(),
        TradeStatus::Rejected => log_rejection(),
    }   // adding a new variant without a match arm = compile error
}

Class invariants via newtype pattern

// Encode invariants in the type system - invalid states are unrepresentable
struct PositiveF64(f64);

impl PositiveF64 {
    fn new(val: f64) -> Result<Self, String> {
        if val > 0.0 && val.is_finite() { Ok(Self(val)) }
        else { Err(format!("Not positive and finite: {val}")) }
    }
    fn value(&self) -> f64 { self.0 }
}

fn price_option(s: PositiveF64, k: PositiveF64, t: PositiveF64, sigma: PositiveF64) -> f64 {
    // No validation needed here - types guarantee the invariant
    price_impl(s.value(), k.value(), t.value(), sigma.value())
}

Error propagation checklist

  • Every function that can fail returns Result<T, E> — the compiler forces callers to handle it
  • Every Option<T> is matched — unwrap() in production code is a code smell
  • Every thread has catch_unwind at its boundary for panic isolation
  • Every async task has error handling — dropped futures do not silently swallow errors
Rust's type system encodes defensive programming at compile time. Newtypes make invalid states unrepresentable. Result makes error handling mandatory. Exhaustive matching prevents missed cases. The compiler is your defensive coding checklist.

Go defensive checklist

Input validation (preconditions)

func priceOption(S, K, T, sigma float64) (float64, error) {
    // Validate at the boundary
    params := map[string]float64{"S": S, "K": K, "T": T, "sigma": sigma}
    for name, val := range params {
        if math.IsNaN(val) || math.IsInf(val, 0) {
            return 0, fmt.Errorf("%s is not finite: %f", name, val)
        }
        if val <= 0 {
            return 0, fmt.Errorf("%s must be positive: %f", name, val)
        }
    }

    result := priceImpl(S, K, T, sigma)

    // Output validation (postcondition)
    if math.IsNaN(result) || math.IsInf(result, 0) {
        return 0, fmt.Errorf("non-finite result: %f", result)
    }
    if result < 0 {
        return 0, fmt.Errorf("negative price: %f", result)
    }
    return result, nil
}

Null / missing value handling

// Go has nil for pointers, slices, maps, channels, interfaces.
// Always check before use.
func safeGet(m map[string]float64, key string, fallback float64) float64 {
    val, ok := m[key]
    if !ok { return fallback }
    return val
}

// Pointer nil checks at function entry
func processOrder(order *Order) error {
    if order == nil { return errors.New("nil order") }
    if order.Symbol == "" { return errors.New("empty symbol") }
    if order.Quantity <= 0 { return errors.New("non-positive quantity") }
    // proceed with validated order...
    return nil
}

Error propagation checklist

  • Every function that returns error: is the error checked at the call site?
  • Every goroutine: does it have defer recover() at the top?
  • Every channel: is it closed by the sender when done?
  • Every context: is cancel() called via defer?
  • Run go vet and staticcheck in CI — they catch unchecked errors and common mistakes

Code review red flags (Go-specific)

// RED FLAG: ignoring errors
result, _ := riskyFunction()      // _ discards the error

// RED FLAG: goroutine without recovery
go func() { riskyWork() }()      // panic crashes the process

// RED FLAG: unbounded goroutine creation
for _, item := range items {
    go process(item)              // 1M items = 1M goroutines
}

// GREEN: bounded worker pool
sem := make(chan struct{}, 10)
for _, item := range items {
    sem <- struct{}{}
    go func(it Item) {
        defer func() { <-sem }()
        process(it)
    }(item)
}
Go's simplicity means fewer language-level safety nets. The go vet and staticcheck tools are essential — they catch unused error returns, unreachable code, and common concurrency mistakes. Run them in CI alongside go test -race.

TypeScript defensive checklist

Input validation (preconditions)

// Runtime validation with Zod (for data from external boundaries)
import { z } from "zod";

const TradeSchema = z.object({
  symbol:   z.string().min(1),
  quantity: z.number().positive().finite(),
  price:    z.number().positive().finite(),
});

type Trade = z.infer<typeof TradeSchema>;

function processOrder(raw: unknown): Trade {
  return TradeSchema.parse(raw);  // throws ZodError with details
}

// Type guards for narrowing
function isFinitePositive(n: number): n is number {
  return Number.isFinite(n) && n > 0;
}

function priceOption(S: number, K: number, T: number, sigma: number): number {
  if (!isFinitePositive(S)) throw new Error(`Bad S: ${S}`);
  if (!isFinitePositive(K)) throw new Error(`Bad K: ${K}`);
  if (!isFinitePositive(T)) throw new Error(`Bad T: ${T}`);
  if (!isFinitePositive(sigma)) throw new Error(`Bad sigma: ${sigma}`);

  const result = priceImpl(S, K, T, sigma);
  if (!Number.isFinite(result) || result < 0) {
    throw new Error(`Invalid result: ${result}`);
  }
  return result;
}

Null / missing value handling

// tsconfig.json: "strict": true (includes strictNullChecks)

// Optional chaining + nullish coalescing
const price = feed.get(sym) ?? lastKnown;
const name  = order?.customer?.name ?? "unknown";

// Exhaustive checking with discriminated unions
type OrderResult =
  | { type: "filled"; price: number }
  | { type: "rejected"; reason: string }
  | { type: "pending" };

function handle(result: OrderResult): void {
  switch (result.type) {
    case "filled":   updatePosition(result.price); break;
    case "rejected": logRejection(result.reason); break;
    case "pending":  queueForRetry(); break;
    default: {
      const _: never = result;  // compile error if new variant added
    }
  }
}

Error propagation checklist

  • Is "strict": true enabled in tsconfig.json?
  • Is every Promise await-ed or has a .catch()?
  • Is external data validated at the boundary (Zod, io-ts, or manual checks)?
  • Are unhandled rejections caught via process.on("unhandledRejection")?
  • Are discriminated unions used instead of string literals for state?
  • Is ESLint configured with @typescript-eslint/no-floating-promises and @typescript-eslint/no-unsafe-assignment?

Code review red flags (TypeScript-specific)

// RED FLAG: type assertion without validation
const trade = rawData as Trade;        // bypasses type checking

// RED FLAG: non-null assertion on external data
const price = response.data!.price;    // crashes if data is null

// RED FLAG: floating promises
someAsyncOperation();                  // error silently swallowed

// GREEN: validated external data
const trade = TradeSchema.parse(rawData);  // runtime validation

// GREEN: awaited with error handling
try { await someAsyncOperation(); }
catch (err) { handleError(err); }
TypeScript's compile-time checks are powerful but erased at runtime. The defensive boundary is where external data enters your system (API responses, user input, file reads). Use runtime validation libraries (Zod, io-ts) at these boundaries. Inside the boundary, trust the type system.

Key observations

  • Rust requires the least defensive coding effort because the compiler does it for you. Null safety, memory safety, race prevention, and exhaustive error handling are all enforced at compile time. The trade-off is a steeper learning curve.
  • C++ requires the most defensive discipline. The language gives you maximum power but zero guardrails. Sanitisers (ASan, TSan, UBSan) are essential but only catch what they see at runtime.
  • Go strikes a pragmatic balance: simple language, good tooling (go vet, staticcheck, race detector), and explicit error handling. The weakness is nil pointers and no generics for defensive patterns pre-1.18.
  • TypeScript with strict mode catches many bugs at compile time, but type erasure means runtime boundaries need separate validation. The ecosystem (Zod, ESLint rules) fills the gap.
  • Python relies most heavily on developer discipline. Type hints + mypy help, but nothing is enforced at runtime unless you add it. Best suited for prototyping and research where speed-of-development matters more than safety.
  • C# is the most balanced for enterprise use: nullable reference types, analyzers, good IDE support, and a mature ecosystem of defensive patterns.

Decision guide

  • Maximum safety by construction: Rust (compiler prevents the most bug categories)
  • Maximum performance with manual discipline: C++ (sanitisers + RAII + code review)
  • Pragmatic safety + simplicity: Go (simple language, good tools, explicit errors)
  • Enterprise standard: C# (balanced safety, tooling, and productivity)
  • Full-stack with shared types: TypeScript (strict mode + Zod at boundaries)
  • Research / prototyping: Python (fast iteration, add safety later)
