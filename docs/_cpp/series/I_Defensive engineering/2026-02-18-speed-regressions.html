The silent performance killer

A refactor that moves a std::vector to a std::map for clarity. A copy constructor called where a move was expected. A unit test that runs fine but adds 50ms of latency in production. Performance regressions sneak in through normal development and accumulate until something complains.

The solution: treat performance like correctness. Benchmark critical paths in CI. If a PR makes the pricer 20% slower, fail the build. This requires stable benchmark infrastructure, baseline tracking, and agreed thresholds.

What to benchmark: focus on hot paths only - the single-trade pricing function, the inner loop of the batch pricer, queue throughput. Benchmarking everything is noise; benchmark what matters for latency and throughput.

Python performance regression testing

import timeit, json, os
import pytest

BASELINE_FILE = ".perf_baseline.json"
THRESHOLD     = 1.20  # 20% slowdown fails the test

def bench(fn, n=1000) -> float:
    t = timeit.timeit(fn, number=n)
    return t / n * 1e6  # microseconds per call

def load_baseline() -> dict:
    if os.path.exists(BASELINE_FILE):
        with open(BASELINE_FILE) as f: return json.load(f)
    return {}

def save_baseline(data: dict) -> None:
    with open(BASELINE_FILE, "w") as f: json.dump(data, f, indent=2)

def test_pricer_performance():
    from my_pricer import price_option
    elapsed_us = bench(lambda: price_option(100, 100, 1, 0.05, 0.2))

    baseline = load_baseline()
    key = "price_option"
    if key not in baseline:
        baseline[key] = elapsed_us
        save_baseline(baseline)
        pytest.skip("Baseline not set - stored new baseline")

    ratio = elapsed_us / baseline[key]
    assert ratio < THRESHOLD, (
        f"Regression: {elapsed_us:.1f}us vs baseline {baseline[key]:.1f}us "
        f"({ratio:.1%} exceeds {THRESHOLD:.0%})"
    )

C++ Google Benchmark in CI

// bench_pricer.cpp
#include <benchmark/benchmark.h>

static void BM_SinglePrice(benchmark::State& state) {
    BlackScholesPricer pricer;
    OptionTrade trade{100, 100, 1, 0.05, 0.2};
    for (auto _ : state)
        benchmark::DoNotOptimize(pricer.price(trade));
    state.SetItemsProcessed(state.iterations());
}
BENCHMARK(BM_SinglePrice);
BENCHMARK_MAIN();
# Run and export JSON
./bench_pricer --benchmark_format=json > bench_results.json

# Compare against stored baseline (Python helper)
import json, sys
curr = json.load(open(sys.argv[1]))["benchmarks"]
base = json.load(open(sys.argv[2]))["benchmarks"]
threshold = 1.15
failures = []
for c, b in zip(curr, base):
    ratio = c["real_time"] / b["real_time"]
    if ratio > threshold:
        failures.append(f"{c['name']}: {ratio:.2%} slower than baseline")
if failures:
    print("REGRESSION FAILURES:"); [print(f) for f in failures]; sys.exit(1)
Store benchmark baselines in version control. Run benchmarks on a dedicated CI runner (not a shared agent) for reproducible results. Use statistical tests (Mann-Whitney U) rather than a simple ratio for robust detection of genuine regressions vs noise.

Rust performance regression testing

Criterion.rs for statistical benchmarking

// Cargo.toml:
// [dev-dependencies]
// criterion = { version = "0.5", features = ["html_reports"] }
//
// [[bench]]
// name = "pricer_bench"
// harness = false

use criterion::{criterion_group, criterion_main, Criterion};

fn bench_single_price(c: &mut Criterion) {
    let pricer = BlackScholesPricer::new();
    let trade = OptionTrade { s: 100.0, k: 100.0, t: 1.0, r: 0.05, sigma: 0.2 };

    c.bench_function("single_price", |b| {
        b.iter(|| pricer.price(&trade))
    });
}

fn bench_batch_price(c: &mut Criterion) {
    let pricer = BlackScholesPricer::new();
    let trades: Vec<OptionTrade> = (0..1000)
        .map(|i| OptionTrade { s: 100.0 + i as f64, k: 100.0, t: 1.0, r: 0.05, sigma: 0.2 })
        .collect();

    c.bench_function("batch_1000", |b| {
        b.iter(|| {
            trades.iter().map(|t| pricer.price(t)).sum::<f64>()
        })
    });
}

criterion_group!(benches, bench_single_price, bench_batch_price);
criterion_main!(benches);

CI integration with cargo-criterion

# Run benchmarks and compare against baseline
cargo bench -- --save-baseline main
# After changes:
cargo bench -- --baseline main
# Criterion reports: "Performance has regressed by +15.2%"

# In CI (GitHub Actions):
# - Store baseline as artifact
# - Compare PR benchmarks against main baseline
# - Fail if regression exceeds threshold
Criterion.rs uses statistical analysis (bootstrap confidence intervals) to distinguish real regressions from noise — no manual threshold tuning needed. It also generates HTML reports with histograms. Rust's zero-cost abstractions mean benchmark results closely match production performance.

Go benchmark regression testing

Built-in testing.B benchmarks

package pricer

import "testing"

func BenchmarkSinglePrice(b *testing.B) {
    pricer := NewBlackScholesPricer()
    trade := OptionTrade{S: 100, K: 100, T: 1, R: 0.05, Sigma: 0.2}
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        pricer.Price(trade)
    }
}

func BenchmarkBatch1000(b *testing.B) {
    pricer := NewBlackScholesPricer()
    trades := make([]OptionTrade, 1000)
    for i := range trades {
        trades[i] = OptionTrade{S: 100 + float64(i), K: 100, T: 1, R: 0.05, Sigma: 0.2}
    }
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        for _, t := range trades {
            pricer.Price(t)
        }
    }
    b.ReportAllocs()  // also track allocations
}

benchstat for regression detection

# Run benchmarks and save results
go test -bench=. -count=10 -benchmem ./... > old.txt
# After changes:
go test -bench=. -count=10 -benchmem ./... > new.txt
# Compare with benchstat
benchstat old.txt new.txt
# Output:
# name          old time/op  new time/op  delta
# SinglePrice   1.23µs       1.45µs       +17.89%  (p=0.001 n=10+10)
# Batch1000     1.15ms       1.18ms        +2.61%  (p=0.043 n=10+10)
Go's benchstat tool uses statistical significance testing (p-values) to determine whether performance changed. Always use -count=10 or higher for stable results. b.ReportAllocs() catches allocation regressions that cause GC pressure even if raw speed looks fine.

TypeScript performance regression testing

Benchmark.js or tinybench

import { Bench } from "tinybench";
import { readFileSync, writeFileSync, existsSync } from "fs";

const BASELINE_FILE = ".perf_baseline.json";
const THRESHOLD = 1.20;  // 20% regression threshold

async function runBenchmarks(): Promise<Record<string, number>> {
  const bench = new Bench({ time: 1000 });

  bench.add("single_price", () => {
    priceOption(100, 100, 1, 0.05, 0.2);
  });

  bench.add("batch_1000", () => {
    for (let i = 0; i < 1000; i++) {
      priceOption(100 + i, 100, 1, 0.05, 0.2);
    }
  });

  await bench.run();

  const results: Record<string, number> = {};
  for (const task of bench.tasks) {
    results[task.name] = task.result!.mean * 1e6; // microseconds
    console.log(`${task.name}: ${results[task.name].toFixed(1)}µs`);
  }
  return results;
}

async function checkRegression(): Promise<void> {
  const current = await runBenchmarks();

  if (!existsSync(BASELINE_FILE)) {
    writeFileSync(BASELINE_FILE, JSON.stringify(current, null, 2));
    console.log("Baseline stored.");
    return;
  }

  const baseline = JSON.parse(readFileSync(BASELINE_FILE, "utf-8"));
  const failures: string[] = [];

  for (const [name, elapsed] of Object.entries(current)) {
    if (!(name in baseline)) continue;
    const ratio = elapsed / baseline[name];
    if (ratio > THRESHOLD) {
      failures.push(
        `${name}: ${elapsed.toFixed(1)}µs vs baseline ${baseline[name].toFixed(1)}µs ` +
        `(${(ratio * 100).toFixed(0)}% exceeds ${(THRESHOLD * 100).toFixed(0)}%)`
      );
    }
  }

  if (failures.length > 0) {
    console.error("REGRESSION FAILURES:");
    failures.forEach((f) => console.error(f));
    process.exit(1);
  }
}
JavaScript benchmarking is noisy due to JIT warm-up and GC pauses. Run benchmarks with --expose-gc and call global.gc() between runs to reduce variance. For Node.js, use --predictable flag to disable background optimisation threads. Always warm up the JIT before measuring.

Key observations

  • Rust (Criterion.rs) and Go (benchstat) both use proper statistical tests rather than simple ratio comparisons. This is the gold standard for detecting real regressions vs measurement noise.
  • C# (BenchmarkDotNet) is the most feature-rich: memory diagnoser, GC stats, disassembly, and multiple runtime comparisons in one run. Excellent for JIT-compiled languages where warm-up matters.
  • C++ and Rust produce the most stable benchmarks because there is no JIT or GC jitter. What you measure is what you get in production.
  • Python and TypeScript have the least stable benchmarks due to interpreter overhead, JIT warm-up, and GC pauses. Use higher iteration counts and statistical filtering.
  • Go's b.ReportAllocs() catches a common regression pattern: a refactor that is the same speed but allocates more, causing GC pressure that shows up as latency spikes in production.

Decision guide

  • Lowest-noise benchmarks: C++ (Google Benchmark) or Rust (Criterion.rs) — no runtime variance
  • Best built-in benchmark tooling: Go (go test -bench + benchstat, zero setup)
  • Most comprehensive analysis: C# (BenchmarkDotNet with memory diagnoser)
  • CI-first approach: Rust (Criterion baselines) or Go (benchstat in scripts)
  • All languages: store baselines in version control, run on dedicated hardware, use statistical tests
