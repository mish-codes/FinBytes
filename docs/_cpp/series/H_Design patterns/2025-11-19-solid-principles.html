Why SOLID matters in quant code

SOLID is not abstract theory. It is a set of heuristics that make code easier to test, extend, and reason about. In a pricing system, violations cluster in predictable places: a class that prices and persists and logs (SRP), a pricer that needs modifying every time a new model is added (OCP), or a risk engine that takes a concrete database rather than an interface (DIP).

  • S - Single Responsibility: a class should have one reason to change
  • O - Open/Closed: open for extension, closed for modification
  • L - Liskov Substitution: subtypes must be substitutable for their base types
  • I - Interface Segregation: clients should not depend on interfaces they do not use
  • D - Dependency Inversion: depend on abstractions, not concretions

All examples use a simplified options pricing system. The same Pricer evolves through each principle to show before and after.

Python SOLID examples

S - Single Responsibility

# BAD: one class, three reasons to change
class BadPricer:
    def price(self, trade): ...
    def save_to_db(self, trade): ...
    def log_result(self, result): ...

# GOOD: separated concerns
class Pricer:         def price(self, trade) -> float: ...
class TradeRepository:def save(self, trade) -> None: ...
class PricingLogger:  def log(self, result) -> None: ...

O - Open/Closed

from abc import ABC, abstractmethod

class PricingModel(ABC):
    @abstractmethod
    def price(self, trade) -> float: ...

class BlackScholes(PricingModel):
    def price(self, trade) -> float: ...

class MonteCarlo(PricingModel):
    def price(self, trade) -> float: ...

# Add Binomial without touching existing code
class Binomial(PricingModel):
    def price(self, trade) -> float: ...

D - Dependency Inversion

from typing import Protocol

class PositionStore(Protocol):
    def get_positions(self, portfolio_id: str) -> list: ...

class RiskEngine:
    def __init__(self, store: PositionStore):
        self.store = store  # injected: testable, swappable

C# SOLID examples

S - Single Responsibility

public class Pricer          { public double Price(Trade t) { ... } }
public class TradeRepository { public void Save(Trade t)   { ... } }
public class PricingLogger   { public void Log(double r)   { ... } }

O - Open/Closed

public interface IPricingModel { double Price(Trade trade); }

public class BlackScholes : IPricingModel { public double Price(Trade t) { ... } }
public class MonteCarlo   : IPricingModel { public double Price(Trade t) { ... } }

public class Pricer {
    private readonly IPricingModel _model;
    public Pricer(IPricingModel model) => _model = model;
    public double Price(Trade t) => _model.Price(t);
}

I - Interface Segregation

// BAD: fat interface forces implementors to stub unused methods
public interface IBadFeed {
    double GetSpot(string sym);
    double GetVol(string sym);
    OrderBook GetBook(string sym);  // not all consumers need this
}

// GOOD: split into focused interfaces
public interface ISpotFeed { double GetSpot(string sym); }
public interface IVolFeed  { double GetVol(string sym);  }

public class Pricer {
    public Pricer(ISpotFeed spots, IVolFeed vols) { ... }
}

L - Liskov Substitution

// BAD: subtype throws where base does not
public class VolSurface {
    public virtual double GetVol(double strike, double expiry) => 0.2;
}
public class EmptyVolSurface : VolSurface {
    public override double GetVol(double strike, double expiry)
        => throw new NotSupportedException();  // VIOLATION
}

// GOOD: return nullable instead
public double? TryGetVol(double strike, double expiry) => null;

C++ SOLID examples

S - Single Responsibility

class Pricer          { public: double price(const Trade& t); };
class TradeRepository { public: void   save(const Trade& t);  };
class PricingLogger   { public: void   log(double result);    };

O - Open/Closed: runtime polymorphism

struct IPricingModel {
    virtual ~IPricingModel() = default;
    virtual double price(const Trade& t) const = 0;
};

struct BlackScholes : IPricingModel {
    double price(const Trade& t) const override { return 0; }
};

class Pricer {
    std::unique_ptr<IPricingModel> model_;
public:
    explicit Pricer(std::unique_ptr<IPricingModel> m) : model_(std::move(m)) {}
    double price(const Trade& t) const { return model_->price(t); }
};

O - Open/Closed: compile-time (zero-overhead)

template<typename Model>
class StaticPricer {
    Model model_;
public:
    double price(const Trade& t) const { return model_.price(t); }
};

StaticPricer<BlackScholes> bs_pricer;
StaticPricer<MonteCarlo>   mc_pricer;

D - Dependency Inversion

struct IPositionStore {
    virtual ~IPositionStore() = default;
    virtual std::vector<Position> get_positions(const std::string& id) const = 0;
};

class RiskEngine {
    const IPositionStore& store_;
public:
    explicit RiskEngine(const IPositionStore& store) : store_(store) {}
};
C++ gives you two open/closed strategies: runtime polymorphism (virtual functions, testable, flexible) and compile-time polymorphism (templates, zero overhead). Use runtime when the model is chosen at runtime. Use templates when it is fixed at compile time and performance matters.

Rust SOLID examples

S — Single Responsibility

struct Pricer;
struct TradeRepository;
struct PricingLogger;

impl Pricer          { fn price(&self, trade: &Trade) -> f64 { todo!() } }
impl TradeRepository { fn save(&self, trade: &Trade)         { todo!() } }
impl PricingLogger   { fn log(&self, result: f64)            { todo!() } }

O — Open/Closed

trait PricingModel {
    fn price(&self, trade: &Trade) -> f64;
}

struct BlackScholes;
struct MonteCarlo;

impl PricingModel for BlackScholes {
    fn price(&self, trade: &Trade) -> f64 { todo!() }
}
impl PricingModel for MonteCarlo {
    fn price(&self, trade: &Trade) -> f64 { todo!() }
}

// Add Binomial without touching existing code
struct Binomial;
impl PricingModel for Binomial {
    fn price(&self, trade: &Trade) -> f64 { todo!() }
}

D — Dependency Inversion

trait PositionStore {
    fn get_positions(&self, portfolio_id: &str) -> Vec<Position>;
}

struct RiskEngine<S: PositionStore> {
    store: S,  // generic over any store implementation
}

impl<S: PositionStore> RiskEngine<S> {
    fn new(store: S) -> Self { Self { store } }
}

I — Interface Segregation

// Separate traits instead of one fat trait
trait SpotFeed { fn get_spot(&self, sym: &str) -> f64; }
trait VolFeed  { fn get_vol(&self, sym: &str)  -> f64; }

struct Pricer<S: SpotFeed, V: VolFeed> {
    spots: S,
    vols: V,
}
Rust enforces DIP at compile time through trait bounds. There is no runtime cost — the compiler monomorphises each generic into concrete code. You can also use dyn Trait for runtime polymorphism when you need it, similar to C++ virtual dispatch.

Go SOLID examples

S — Single Responsibility

type Pricer struct{}
func (p *Pricer) Price(t Trade) float64 { return 0 }

type TradeRepository struct{}
func (r *TradeRepository) Save(t Trade) {}

type PricingLogger struct{}
func (l *PricingLogger) Log(result float64) {}

O — Open/Closed

type PricingModel interface {
    Price(trade Trade) float64
}

type BlackScholes struct{}
func (bs BlackScholes) Price(t Trade) float64 { return 0 }

type MonteCarlo struct{}
func (mc MonteCarlo) Price(t Trade) float64 { return 0 }

// Add Binomial — no existing code touched
type Binomial struct{}
func (b Binomial) Price(t Trade) float64 { return 0 }

D — Dependency Inversion

type PositionStore interface {
    GetPositions(portfolioID string) []Position
}

type RiskEngine struct {
    store PositionStore  // accepts any implementation
}

func NewRiskEngine(store PositionStore) *RiskEngine {
    return &RiskEngine{store: store}
}

I — Interface Segregation

// Go interfaces are implicitly satisfied — no "implements" keyword
type SpotFeed interface { GetSpot(sym string) float64 }
type VolFeed  interface { GetVol(sym string)  float64 }

type Pricer struct {
    spots SpotFeed
    vols  VolFeed
}
Go interfaces are satisfied implicitly — any type that has the right methods automatically implements the interface. This makes Interface Segregation natural: small interfaces are cheap to define and easy to satisfy. The Go proverb is "the bigger the interface, the weaker the abstraction."

TypeScript SOLID examples

S — Single Responsibility

class Pricer          { price(trade: Trade): number { return 0; } }
class TradeRepository { save(trade: Trade): void    { /* ... */ } }
class PricingLogger   { log(result: number): void   { /* ... */ } }

O — Open/Closed

interface PricingModel {
  price(trade: Trade): number;
}

class BlackScholes implements PricingModel {
  price(trade: Trade): number { /* ... */ return 0; }
}

class MonteCarlo implements PricingModel {
  price(trade: Trade): number { /* ... */ return 0; }
}

class Pricer {
  constructor(private model: PricingModel) {}
  price(trade: Trade): number { return this.model.price(trade); }
}

D — Dependency Inversion

interface PositionStore {
  getPositions(portfolioId: string): Position[];
}

class RiskEngine {
  constructor(private store: PositionStore) {}
}

I — Interface Segregation

interface SpotFeed { getSpot(sym: string): number; }
interface VolFeed  { getVol(sym: string):  number; }

class Pricer {
  constructor(private spots: SpotFeed, private vols: VolFeed) {}
}
TypeScript interfaces are structural (like Go), not nominal (like C#/Java). Any object with the right shape satisfies the interface — no implements keyword required. This makes ISP natural but means you need to be careful: a typo in a method name silently creates a non-conforming type instead of a compile error.

Key observations

  • Rust enforces SOLID at compile time more strictly than any other language here. Trait bounds are checked before your code runs. The cost: more verbose type signatures and occasional fights with the borrow checker when interfaces involve references.
  • Go makes ISP the most natural — small, implicit interfaces are idiomatic. But Go lacks generics for most DIP patterns (pre-1.18), and even with generics the syntax is clunky compared to Rust or C++ templates.
  • TypeScript has the best DX for testing (structural typing = easy mocks) but the weakest runtime guarantees — interfaces are erased at runtime, so a wrong type at a system boundary won't be caught until it blows up.
  • Python is the fastest to prototype with but relies on discipline rather than the compiler. Protocol (PEP 544) gives you structural typing like Go/TS, but only if you run a type checker.
  • C++ uniquely offers compile-time polymorphism (templates) alongside runtime polymorphism (virtual). No other language in this list gives you zero-cost abstraction at the SOLID level.
  • C# is the most balanced — strong typing, good IDE support, readable syntax. The enterprise standard for a reason.

When to pick what

  • Prototyping / data science: Python
  • Enterprise backend / microservices: C# or Go
  • Systems / embedded / zero-cost: Rust or C++
  • High-concurrency services: Go or Rust
  • Full-stack with shared types: TypeScript
  • Hot pricing loop (millions/sec): C++ templates or Rust generics
