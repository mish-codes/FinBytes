Separating domain from persistence

Without a repository, domain objects fetch themselves from the database. A Trade class calls SELECT * FROM trades. Testing domain logic requires a real database. Changing from Postgres to a message queue requires changing the domain class.

A repository presents a collection-like interface. Domain code calls repo.find_by_id() and gets back a Trade. The repository handles all persistence details. A data mapper translates between domain objects and database rows - the domain object knows nothing about the database.

The real win: with a repository interface, tests inject an in-memory fake. No database setup, no teardown, no slow I/O. The entire domain test suite runs in milliseconds.

Python repository pattern

from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional

@dataclass
class Trade:
    id:       str
    symbol:   str
    notional: float
    status:   str = "PENDING"

class ITradeRepository(ABC):
    @abstractmethod
    def find_by_id(self, trade_id: str) -> Optional[Trade]: ...
    @abstractmethod
    def find_by_symbol(self, symbol: str) -> list[Trade]: ...
    @abstractmethod
    def save(self, trade: Trade) -> None: ...

# In-memory fake for tests
class InMemoryTradeRepo(ITradeRepository):
    def __init__(self): self._store: dict[str, Trade] = {}
    def find_by_id(self, tid: str) -> Optional[Trade]: return self._store.get(tid)
    def find_by_symbol(self, sym: str) -> list[Trade]:
        return [t for t in self._store.values() if t.symbol == sym]
    def save(self, trade: Trade) -> None: self._store[trade.id] = trade

class TradeService:
    def __init__(self, repo: ITradeRepository): self._repo = repo
    def execute_trade(self, trade: Trade) -> None:
        trade.status = "EXECUTED"
        self._repo.save(trade)

# Test with fake - no database
repo    = InMemoryTradeRepo()
service = TradeService(repo)
trade   = Trade("T001", "AAPL", 100_000)
service.execute_trade(trade)
assert repo.find_by_id("T001").status == "EXECUTED"

C# repository pattern

public record Trade(string Id, string Symbol, double Notional, string Status = "PENDING");

public interface ITradeRepository {
    Trade?       FindById(string id);
    List<Trade>  FindBySymbol(string symbol);
    void         Save(Trade trade);
}

public class InMemoryTradeRepository : ITradeRepository {
    private readonly Dictionary<string, Trade> _store = new();
    public Trade?      FindById(string id)      => _store.GetValueOrDefault(id);
    public List<Trade> FindBySymbol(string sym) => _store.Values.Where(t => t.Symbol == sym).ToList();
    public void        Save(Trade t)            => _store[t.Id] = t;
}

// EF Core implementation (same interface)
public class EFTradeRepository : ITradeRepository {
    private readonly TradingDbContext _ctx;
    public EFTradeRepository(TradingDbContext ctx) => _ctx = ctx;
    public Trade? FindById(string id) => _ctx.Trades.Find(id);
    public List<Trade> FindBySymbol(string sym) => _ctx.Trades.Where(t => t.Symbol == sym).ToList();
    public void Save(Trade t) { _ctx.Trades.Update(t); _ctx.SaveChanges(); }
}

C++ repository pattern

#include <string>
#include <vector>
#include <unordered_map>
#include <optional>
#include <memory>
#include <stdexcept>

struct Trade { std::string id, symbol; double notional; std::string status = "PENDING"; };

struct ITradeRepository {
    virtual ~ITradeRepository() = default;
    virtual std::optional<Trade> find_by_id(const std::string& id) const = 0;
    virtual std::vector<Trade>   find_by_symbol(const std::string& sym) const = 0;
    virtual void                  save(const Trade& t) = 0;
};

class InMemoryTradeRepository : public ITradeRepository {
    std::unordered_map<std::string, Trade> store_;
public:
    std::optional<Trade> find_by_id(const std::string& id) const override {
        auto it = store_.find(id);
        if (it == store_.end()) return std::nullopt;
        return it->second;
    }
    std::vector<Trade> find_by_symbol(const std::string& sym) const override {
        std::vector<Trade> result;
        for (const auto& [id, t] : store_)
            if (t.symbol == sym) result.push_back(t);
        return result;
    }
    void save(const Trade& t) override { store_[t.id] = t; }
};

class TradeService {
    ITradeRepository& repo_;
public:
    explicit TradeService(ITradeRepository& repo) : repo_(repo) {}
    void execute(const std::string& trade_id) {
        auto opt = repo_.find_by_id(trade_id);
        if (!opt) throw std::runtime_error("Trade not found: " + trade_id);
        Trade t = *opt;
        t.status = "EXECUTED";
        repo_.save(t);
    }
};
std::optional<Trade> is cleaner than a raw pointer (which the caller might forget to null-check) or an exception (which is not appropriate for "not found"). Always return optional for queries that may find nothing.

Rust repository pattern

use std::collections::HashMap;

#[derive(Clone)]
struct Trade {
    id:       String,
    symbol:   String,
    notional: f64,
    status:   String,
}

trait TradeRepository {
    fn find_by_id(&self, id: &str) -> Option<&Trade>;
    fn find_by_symbol(&self, symbol: &str) -> Vec<&Trade>;
    fn save(&mut self, trade: Trade);
}

struct InMemoryTradeRepo {
    store: HashMap<String, Trade>,
}

impl InMemoryTradeRepo {
    fn new() -> Self { Self { store: HashMap::new() } }
}

impl TradeRepository for InMemoryTradeRepo {
    fn find_by_id(&self, id: &str) -> Option<&Trade> {
        self.store.get(id)
    }

    fn find_by_symbol(&self, symbol: &str) -> Vec<&Trade> {
        self.store.values().filter(|t| t.symbol == symbol).collect()
    }

    fn save(&mut self, trade: Trade) {
        self.store.insert(trade.id.clone(), trade);
    }
}

struct TradeService<R: TradeRepository> {
    repo: R,
}

impl<R: TradeRepository> TradeService<R> {
    fn new(repo: R) -> Self { Self { repo } }

    fn execute_trade(&mut self, trade_id: &str) -> Result<(), String> {
        let trade = self.repo.find_by_id(trade_id)
            .ok_or_else(|| format!("Trade not found: {trade_id}"))?
            .clone();
        let mut updated = trade;
        updated.status = "EXECUTED".into();
        self.repo.save(updated);
        Ok(())
    }
}

// Test with fake — no database
let mut repo = InMemoryTradeRepo::new();
repo.save(Trade {
    id: "T001".into(), symbol: "AAPL".into(),
    notional: 100_000.0, status: "PENDING".into(),
});
let mut service = TradeService::new(repo);
service.execute_trade("T001").unwrap();
Rust's trait-based repository returns Option<&Trade> — the caller must handle the "not found" case or the code does not compile. The generic TradeService<R: TradeRepository> is monomorphised at compile time, so there is no virtual dispatch cost. For runtime flexibility (e.g., swapping repo implementations via config), use Box<dyn TradeRepository> instead.

Go repository pattern

package main

import "fmt"

type Trade struct {
    ID       string
    Symbol   string
    Notional float64
    Status   string
}

type TradeRepository interface {
    FindByID(id string) (*Trade, error)
    FindBySymbol(symbol string) []Trade
    Save(trade Trade) error
}

type InMemoryTradeRepo struct {
    store map[string]Trade
}

func NewInMemoryTradeRepo() *InMemoryTradeRepo {
    return &InMemoryTradeRepo{store: make(map[string]Trade)}
}

func (r *InMemoryTradeRepo) FindByID(id string) (*Trade, error) {
    t, ok := r.store[id]
    if !ok { return nil, fmt.Errorf("trade not found: %s", id) }
    return &t, nil
}

func (r *InMemoryTradeRepo) FindBySymbol(symbol string) []Trade {
    var result []Trade
    for _, t := range r.store {
        if t.Symbol == symbol { result = append(result, t) }
    }
    return result
}

func (r *InMemoryTradeRepo) Save(trade Trade) error {
    r.store[trade.ID] = trade
    return nil
}

type TradeService struct {
    repo TradeRepository
}

func (s *TradeService) ExecuteTrade(tradeID string) error {
    t, err := s.repo.FindByID(tradeID)
    if err != nil { return err }
    t.Status = "EXECUTED"
    return s.repo.Save(*t)
}

// Test with fake
repo := NewInMemoryTradeRepo()
repo.Save(Trade{ID: "T001", Symbol: "AAPL", Notional: 100_000, Status: "PENDING"})
svc := &TradeService{repo: repo}
svc.ExecuteTrade("T001")
Go's implicit interfaces make the repository pattern particularly clean. The TradeService depends on the TradeRepository interface — any struct with the right methods satisfies it. In tests, pass an in-memory fake. In production, pass a Postgres implementation. No registration, no DI container needed.

TypeScript repository pattern

interface Trade {
  id: string;
  symbol: string;
  notional: number;
  status: string;
}

interface TradeRepository {
  findById(id: string): Trade | undefined;
  findBySymbol(symbol: string): Trade[];
  save(trade: Trade): void;
}

class InMemoryTradeRepo implements TradeRepository {
  private store = new Map<string, Trade>();

  findById(id: string): Trade | undefined {
    return this.store.get(id);
  }

  findBySymbol(symbol: string): Trade[] {
    return [...this.store.values()].filter(t => t.symbol === symbol);
  }

  save(trade: Trade): void {
    this.store.set(trade.id, { ...trade });
  }
}

class TradeService {
  constructor(private repo: TradeRepository) {}

  executeTrade(tradeId: string): void {
    const trade = this.repo.findById(tradeId);
    if (!trade) throw new Error(`Trade not found: ${tradeId}`);
    this.repo.save({ ...trade, status: "EXECUTED" });
  }
}

// Test with fake — no database
const repo = new InMemoryTradeRepo();
repo.save({ id: "T001", symbol: "AAPL", notional: 100_000, status: "PENDING" });
const svc = new TradeService(repo);
svc.executeTrade("T001");
console.log(repo.findById("T001")?.status); // "EXECUTED"
TypeScript's structural typing means any object with findById, findBySymbol, and save methods satisfies TradeRepository — no implements keyword required (though it helps catch errors early). The spread operator ({ ...trade }) creates shallow copies, avoiding shared-reference mutations between the repo and caller.

Key observations

  • C# with EF Core is the gold standard for the repository pattern in enterprise systems. LINQ queries, change tracking, migrations, and strong typing all work together. The repository abstraction layer is thin because EF Core already provides collection-like semantics.
  • Go's implicit interfaces make testing trivial. Define a small interface with only the methods your service needs. The in-memory fake satisfies it automatically. No DI framework, no mocking library — just a struct with the right methods.
  • Rust's Option<T> forces callers to handle "not found" — you cannot accidentally dereference a null result. Combined with Result<T, E> for database errors, error handling is exhaustive at compile time.
  • TypeScript's structural typing allows quick test doubles. Pass any object literal that matches the interface shape — no class needed. This makes tests extremely concise but can hide mismatches if a method is accidentally omitted.
  • C++ has the weakest ORM ecosystem. Most C++ repository implementations use raw SQL or thin wrappers. The upside: full control over query performance. The downside: more boilerplate and no change tracking.

Decision guide for the Repository pattern

  • Enterprise CRUD with rich queries: C# + EF Core or Python + SQLAlchemy
  • Microservice with simple persistence: Go (minimal boilerplate, easy testing)
  • Type-safe SQL at compile time: Rust + SQLx (queries checked against the schema at build time)
  • Full-stack shared models: TypeScript + Prisma (schema-first, generated types)
  • High-performance, low-level control: C++ with manual SQL or memory-mapped storage
