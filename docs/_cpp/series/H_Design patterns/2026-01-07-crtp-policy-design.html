Polymorphism without virtual

CRTP (Curiously Recurring Template Pattern) achieves compile-time polymorphism - the flexibility of inheritance without the runtime cost of virtual dispatch. Used extensively in performance-critical C++ libraries such as Eigen, Boost, and STL internals.

A base class template is parameterised by its derived class: template<typename Derived> class Base. The derived class inherits from Base<Derived>. The base can call derived methods without virtual functions, resolved at compile time.

Policy-based design composes behaviour from interchangeable policies - similar to Strategy but at compile time. A pricer parameterised by rate curve policy, vol surface policy, and day count policy, each independently swappable, with zero runtime overhead.

Use CRTP when virtual dispatch overhead is measured to be a problem, when you need static interface enforcement, or when building a framework where users provide their own implementation of hot methods. Otherwise, virtual functions are clearer and adequate.

Python - duck typing handles this

# Python does not need CRTP - duck typing handles compile-time polymorphism
# If it has the right methods, it works

class FlatVolSurface:
    def get_vol(self, strike: float, expiry: float) -> float: return 0.2

class SkewVolSurface:
    def get_vol(self, strike: float, expiry: float) -> float:
        return 0.2 + 0.01 * abs(strike - 100) / 100

class USDAct360:
    def day_fraction(self, days: float) -> float: return days / 360

# Policy-style composition via injection
class Pricer:
    def __init__(self, vol_surface, day_count):
        self._vol = vol_surface
        self._dc  = day_count
    def price(self, S, K, days):
        T   = self._dc.day_fraction(days)
        vol = self._vol.get_vol(K, T)
        return max(S - K, 0) * vol

p1 = Pricer(FlatVolSurface(), USDAct360())
p2 = Pricer(SkewVolSurface(), USDAct360())

C# - generics with constraints

interface IVolSurface { double GetVol(double strike, double expiry); }
interface IDayCount   { double DayFraction(double days); }

class FlatVol : IVolSurface { public double GetVol(double K, double T) => 0.2; }
class SkewVol : IVolSurface { public double GetVol(double K, double T) => 0.2 + 0.01*Math.Abs(K-100)/100; }
class Act360  : IDayCount   { public double DayFraction(double d) => d/360.0; }
class Act365  : IDayCount   { public double DayFraction(double d) => d/365.0; }

class Pricer<TVol, TDc> where TVol : IVolSurface, new()
                          where TDc  : IDayCount,   new()
{
    private readonly TVol _vol = new();
    private readonly TDc  _dc  = new();
    public double Price(double S, double K, double days) {
        double T   = _dc.DayFraction(days);
        double vol = _vol.GetVol(K, T);
        return Math.Max(S - K, 0) * vol;
    }
}

var p1 = new Pricer<FlatVol, Act360>();
var p2 = new Pricer<SkewVol, Act365>();

C++ CRTP and policy-based design

// CRTP: base calls derived methods without virtual
template<typename Derived>
class PricerBase {
public:
    double price(double S, double K, double T) const {
        return static_cast<const Derived*>(this)->price_impl(S, K, T);
    }
    double pv01(double S, double K, double T, double notional) const {
        return price(S, K, T) * notional * 0.0001;
    }
};

struct BlackScholesPricer : PricerBase<BlackScholesPricer> {
    double price_impl(double S, double K, double T) const {
        return std::max(S - K, 0.0) * 1.05;
    }
};

struct IntrinsicPricer : PricerBase<IntrinsicPricer> {
    double price_impl(double S, double K, double T) const {
        return std::max(S - K, 0.0);
    }
};

BlackScholesPricer bs;
double px = bs.price(100, 100, 1);    // no virtual call
double pv = bs.pv01(100, 100, 1, 1'000'000);

// Policy-based design
struct FlatVol { double get_vol(double K, double T) const { return 0.2; } };
struct SkewVol { double get_vol(double K, double T) const { return 0.2 + 0.01*std::abs(K-100)/100; } };
struct Act360  { double day_fraction(double d) const { return d/360.0; } };
struct Act365  { double day_fraction(double d) const { return d/365.0; } };

template<typename VolPolicy, typename DayCountPolicy>
class PolicyPricer : private VolPolicy, private DayCountPolicy {
public:
    double price(double S, double K, double days) const {
        double T   = DayCountPolicy::day_fraction(days);
        double vol = VolPolicy::get_vol(K, T);
        return std::max(S - K, 0.0) * vol;
    }
};

PolicyPricer<FlatVol, Act360> p1;  // zero overhead - policies inlined
PolicyPricer<SkewVol, Act365> p2;
CRTP is an advanced pattern. Measure whether virtual dispatch is your bottleneck before using it. For most pricing code, virtual functions are adequate. CRTP shines when you call the same virtual method millions of times in a tight loop, or when building a framework where users plug in their own hot implementations.

Rust — traits and generics replace CRTP

// Rust does not need CRTP — traits with default methods achieve the same thing
trait PricerBase {
    fn price_impl(&self, s: f64, k: f64, t: f64) -> f64;

    // Default method calls price_impl — like CRTP base calling derived
    fn price(&self, s: f64, k: f64, t: f64) -> f64 {
        self.price_impl(s, k, t)
    }
    fn pv01(&self, s: f64, k: f64, t: f64, notional: f64) -> f64 {
        self.price(s, k, t) * notional * 0.0001
    }
}

struct BlackScholesPricer;
impl PricerBase for BlackScholesPricer {
    fn price_impl(&self, s: f64, k: f64, _t: f64) -> f64 {
        (s - k).max(0.0) * 1.05
    }
}

struct IntrinsicPricer;
impl PricerBase for IntrinsicPricer {
    fn price_impl(&self, s: f64, k: f64, _t: f64) -> f64 {
        (s - k).max(0.0)
    }
}

let bs = BlackScholesPricer;
let px = bs.price(100.0, 100.0, 1.0);      // no virtual call
let pv = bs.pv01(100.0, 100.0, 1.0, 1e6);  // calls price() -> price_impl()

// Policy-based design with generics
trait VolSurface { fn get_vol(&self, k: f64, t: f64) -> f64; }
trait DayCount   { fn day_fraction(&self, days: f64) -> f64; }

struct FlatVol;
impl VolSurface for FlatVol {
    fn get_vol(&self, _k: f64, _t: f64) -> f64 { 0.2 }
}

struct SkewVol;
impl VolSurface for SkewVol {
    fn get_vol(&self, k: f64, _t: f64) -> f64 { 0.2 + 0.01 * (k - 100.0).abs() / 100.0 }
}

struct Act360;
impl DayCount for Act360 { fn day_fraction(&self, d: f64) -> f64 { d / 360.0 } }

struct Act365;
impl DayCount for Act365 { fn day_fraction(&self, d: f64) -> f64 { d / 365.0 } }

struct PolicyPricer<V: VolSurface, D: DayCount> {
    vol: V,
    dc:  D,
}

impl<V: VolSurface, D: DayCount> PolicyPricer<V, D> {
    fn price(&self, s: f64, k: f64, days: f64) -> f64 {
        let t   = self.dc.day_fraction(days);
        let vol = self.vol.get_vol(k, t);
        (s - k).max(0.0) * vol
    }
}

let p1 = PolicyPricer { vol: FlatVol, dc: Act360 };
let p2 = PolicyPricer { vol: SkewVol, dc: Act365 };
Rust's generics achieve exactly what C++ CRTP and policy-based design achieve — compile-time polymorphism with zero overhead — but with a cleaner syntax. The compiler monomorphises each combination of V and D into concrete code. Unlike C++ CRTP, there is no static_cast trick — Rust traits express the relationship directly.

Go — embedding and interfaces

package main

import "math"

// Go has no generics-based CRTP equivalent.
// The closest idiom: define an interface + helper functions,
// or use struct embedding for shared behaviour.

type Pricer interface {
    Price(S, K, T float64) float64
}

// Shared behaviour via standalone function (accepts any Pricer)
func PV01(p Pricer, S, K, T, notional float64) float64 {
    return p.Price(S, K, T) * notional * 0.0001
}

type BlackScholesPricer struct{}
func (bs BlackScholesPricer) Price(S, K, T float64) float64 {
    return math.Max(S-K, 0) * 1.05
}

type IntrinsicPricer struct{}
func (ip IntrinsicPricer) Price(S, K, T float64) float64 {
    return math.Max(S-K, 0)
}

pv := PV01(BlackScholesPricer{}, 100, 100, 1, 1_000_000)

// Policy-based design via interfaces
type VolSurface interface { GetVol(K, T float64) float64 }
type DayCount   interface { DayFraction(days float64) float64 }

type FlatVol struct{}
func (f FlatVol) GetVol(K, T float64) float64 { return 0.2 }

type SkewVol struct{}
func (s SkewVol) GetVol(K, T float64) float64 { return 0.2 + 0.01*math.Abs(K-100)/100 }

type Act360 struct{}
func (a Act360) DayFraction(d float64) float64 { return d / 360 }

type Act365 struct{}
func (a Act365) DayFraction(d float64) float64 { return d / 365 }

type PolicyPricer struct {
    Vol VolSurface
    DC  DayCount
}

func (p PolicyPricer) Price(S, K, days float64) float64 {
    T   := p.DC.DayFraction(days)
    vol := p.Vol.GetVol(K, T)
    return math.Max(S-K, 0) * vol
}

p1 := PolicyPricer{Vol: FlatVol{}, DC: Act360{}}
p2 := PolicyPricer{Vol: SkewVol{}, DC: Act365{}}
Go cannot eliminate virtual dispatch at compile time — all interface calls go through an itab (interface table), similar to C++ vtable. For most applications this is fast enough. If you need zero-overhead policies in Go, the only option is code generation or accepting the interface dispatch cost. Go's philosophy: clarity over micro-optimisation.

TypeScript — generics and mixins

// TypeScript generics approximate policy-based design
// No compile-time monomorphisation (it all erases to JS), but the types help at dev time

interface VolSurface { getVol(K: number, T: number): number; }
interface DayCount   { dayFraction(days: number): number; }

class FlatVol implements VolSurface {
  getVol(K: number, T: number): number { return 0.2; }
}

class SkewVol implements VolSurface {
  getVol(K: number, T: number): number { return 0.2 + 0.01 * Math.abs(K - 100) / 100; }
}

class Act360 implements DayCount {
  dayFraction(d: number): number { return d / 360; }
}

class Act365 implements DayCount {
  dayFraction(d: number): number { return d / 365; }
}

class PolicyPricer<V extends VolSurface, D extends DayCount> {
  constructor(private vol: V, private dc: D) {}

  price(S: number, K: number, days: number): number {
    const T   = this.dc.dayFraction(days);
    const vol = this.vol.getVol(K, T);
    return Math.max(S - K, 0) * vol;
  }
}

const p1 = new PolicyPricer(new FlatVol(), new Act360());
const p2 = new PolicyPricer(new SkewVol(), new Act365());

// Mixin pattern — TypeScript's closest equivalent to CRTP
type Constructor<T = {}> = new (...args: any[]) => T;

function WithPV01<TBase extends Constructor<{ price(S: number, K: number, T: number): number }>>(
  Base: TBase
) {
  return class extends Base {
    pv01(S: number, K: number, T: number, notional: number): number {
      return this.price(S, K, T) * notional * 0.0001;
    }
  };
}

class BSPricer {
  price(S: number, K: number, T: number): number { return Math.max(S - K, 0) * 1.05; }
}

const BSWithPV01 = WithPV01(BSPricer);
const bs = new BSWithPV01();
const pv = bs.pv01(100, 100, 1, 1_000_000);
TypeScript's mixin pattern is the closest analogue to CRTP. A function takes a base class and returns an extended class with additional methods. Unlike C++ CRTP, there is no performance benefit — TypeScript generics are erased at compile time, so all dispatch is dynamic. The value is purely in type checking: the compiler verifies that the base has a price() method before allowing the mixin.

Key observations

  • Only C++ and Rust offer true zero-overhead policy composition. The compiler generates specialised code for each policy combination — no vtable, no indirection. For a pricer evaluating millions of options, this eliminates ~5-15ns per call of virtual dispatch overhead.
  • Rust's approach is cleaner than C++ CRTP. Traits with default methods achieve the same result without the static_cast<Derived*>(this) trick. Error messages from the Rust compiler are also far more readable than C++ template errors.
  • Python does not need CRTP — duck typing means any object with the right methods works. The trade-off: no compile-time enforcement and no performance benefit.
  • Go uses embedding for code reuse but cannot eliminate interface dispatch. Go's philosophy prioritises simplicity and readability over micro-optimisation. If you need zero-overhead dispatch in Go, profile first — the itab lookup is typically under 2ns.
  • TypeScript mixins provide type safety but no runtime benefit. Generics erase to plain JavaScript. The value is catching policy mismatches at compile time, not performance.
  • C#'s JIT can sometimes devirtualise generic method calls when the concrete type is known, giving partial zero-overhead behaviour without explicit CRTP. But it is not guaranteed.

Decision guide for CRTP / Policy design

  • Hot inner loop (millions of calls/sec): C++ CRTP or Rust generics — measure the vtable cost first
  • Framework where users provide implementations: C++ CRTP or Rust traits with default methods
  • Composable behaviour via policies: any language's interface injection works; C++/Rust for zero-cost
  • Readability over performance: Python duck typing or Go interfaces
  • Type-safe code reuse: TypeScript mixins or C# generic constraints
