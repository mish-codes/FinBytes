Adding behaviour without modifying classes

The Decorator pattern wraps an object with another that adds behaviour while preserving the same interface. A caching decorator wraps a pricer and returns cached results for repeated inputs. A logging decorator wraps it again. The underlying pricer is unchanged.

The Pipeline pattern passes data through a sequence of transformations, each independent and composable. A raw tick flows through normalisation, validation, pricing, and output stages. Each stage is a function or class with a known interface.

Both patterns express composition over inheritance. A CachingPricer that inherits from BlackScholes is brittle - you cannot use the caching with Monte Carlo. A CachingPricer that wraps any IPricingModel works with everything.

Python decorator and pipeline

import time

class IPricer:
    def price(self, S: float, K: float, T: float) -> float: ...

class BSPricer(IPricer):
    def price(self, S, K, T) -> float: return max(S - K, 0) * 1.05

class CachingPricer(IPricer):
    def __init__(self, inner: IPricer):
        self._inner = inner; self._cache: dict = {}
    def price(self, S, K, T) -> float:
        key = (round(S,4), round(K,4), round(T,4))
        if key not in self._cache:
            self._cache[key] = self._inner.price(S, K, T)
        return self._cache[key]

class TimingPricer(IPricer):
    def __init__(self, inner: IPricer): self._inner = inner
    def price(self, S, K, T) -> float:
        t0 = time.perf_counter()
        result = self._inner.price(S, K, T)
        print(f"price() took {(time.perf_counter()-t0)*1e6:.1f}us")
        return result

# Stack decorators - order matters
pricer = TimingPricer(CachingPricer(BSPricer()))
pricer.price(100, 100, 1)

# Functional pipeline
from typing import Callable
def pipeline(*fns: Callable) -> Callable:
    def run(x):
        for fn in fns: x = fn(x)
        return x
    return run

normalise = lambda t: {**t, "price": round(t["price"], 4)}
enrich    = lambda t: {**t, "mid": t["price"] * 1.001}
process   = pipeline(normalise, enrich)
result    = process({"price": 100.123456, "symbol": "AAPL"})

C# decorator and pipeline

interface IPricer { double Price(double S, double K, double T); }
class BSPricer : IPricer { public double Price(double S, double K, double T) => Math.Max(S-K,0)*1.05; }

class CachingPricer : IPricer {
    private readonly IPricer _inner;
    private readonly Dictionary<(double,double,double), double> _cache = new();
    public CachingPricer(IPricer inner) => _inner = inner;
    public double Price(double S, double K, double T) {
        var key = (Math.Round(S,4), Math.Round(K,4), Math.Round(T,4));
        if (!_cache.TryGetValue(key, out var v)) _cache[key] = v = _inner.Price(S, K, T);
        return v;
    }
}

class LoggingPricer : IPricer {
    private readonly IPricer _inner;
    public LoggingPricer(IPricer inner) => _inner = inner;
    public double Price(double S, double K, double T) {
        var result = _inner.Price(S, K, T);
        Console.WriteLine($"Priced S={S} K={K} T={T} -> {result:F4}");
        return result;
    }
}

IPricer pricer = new LoggingPricer(new CachingPricer(new BSPricer()));

C++ decorator and pipeline

#include <memory>
#include <unordered_map>
#include <functional>
#include <chrono>
#include <iostream>
#include <initializer_list>
#include <cmath>

struct IPricer {
    virtual ~IPricer() = default;
    virtual double price(double S, double K, double T) const = 0;
};

struct BSPricer : IPricer {
    double price(double S, double K, double T) const override { return std::max(S-K,0.0)*1.05; }
};

class CachingPricer : public IPricer {
    std::unique_ptr<IPricer> inner_;
    mutable std::unordered_map<std::size_t, double> cache_;
    static std::size_t key(double S, double K, double T) {
        std::size_t h = std::hash<int>{}((int)(S*100));
        h ^= std::hash<int>{}((int)(K*100)) << 16;
        h ^= std::hash<int>{}((int)(T*100)) << 32;
        return h;
    }
public:
    explicit CachingPricer(std::unique_ptr<IPricer> inner) : inner_(std::move(inner)) {}
    double price(double S, double K, double T) const override {
        auto k = key(S, K, T);
        auto it = cache_.find(k);
        if (it != cache_.end()) return it->second;
        return cache_[k] = inner_->price(S, K, T);
    }
};

// Functional pipeline
template<typename T>
T pipeline(T value, std::initializer_list<std::function<T(T)>> stages) {
    for (auto& fn : stages) value = fn(value);
    return value;
}

double result = pipeline<double>(100.123456, {
    [](double x) { return std::round(x * 10000) / 10000; },
    [](double x) { return x * 1.001; },
    [](double x) { return x + 0.5;   },
});
Decorators in C++ must manage ownership carefully - use unique_ptr and move semantics. The outermost decorator owns the next one, which owns the next. The whole stack is freed when the outermost goes out of scope.

Rust decorator and pipeline

use std::collections::HashMap;
use std::time::Instant;

trait Pricer {
    fn price(&self, s: f64, k: f64, t: f64) -> f64;
}

struct BSPricer;
impl Pricer for BSPricer {
    fn price(&self, s: f64, k: f64, _t: f64) -> f64 { (s - k).max(0.0) * 1.05 }
}

// Caching decorator wraps any Pricer
struct CachingPricer<P: Pricer> {
    inner: P,
    cache: std::cell::RefCell<HashMap<(i64, i64, i64), f64>>,
}

impl<P: Pricer> CachingPricer<P> {
    fn new(inner: P) -> Self {
        Self { inner, cache: std::cell::RefCell::new(HashMap::new()) }
    }
}

impl<P: Pricer> Pricer for CachingPricer<P> {
    fn price(&self, s: f64, k: f64, t: f64) -> f64 {
        let key = ((s * 1e4) as i64, (k * 1e4) as i64, (t * 1e4) as i64);
        if let Some(&v) = self.cache.borrow().get(&key) { return v; }
        let v = self.inner.price(s, k, t);
        self.cache.borrow_mut().insert(key, v);
        v
    }
}

// Timing decorator
struct TimingPricer<P: Pricer> { inner: P }

impl<P: Pricer> Pricer for TimingPricer<P> {
    fn price(&self, s: f64, k: f64, t: f64) -> f64 {
        let start = Instant::now();
        let result = self.inner.price(s, k, t);
        println!("price() took {:?}", start.elapsed());
        result
    }
}

// Stack decorators
let pricer = TimingPricer { inner: CachingPricer::new(BSPricer) };
let px = pricer.price(100.0, 100.0, 1.0);

// Functional pipeline
fn pipeline<T>(mut value: T, stages: &[&dyn Fn(T) -> T]) -> T {
    for stage in stages { value = stage(value); }
    value
}

let result = pipeline(100.123456_f64, &[
    &|x| (x * 10000.0).round() / 10000.0,
    &|x| x * 1.001,
]);
Rust decorators use generics rather than trait objects, so the entire decorator stack is resolved at compile time — zero virtual dispatch overhead. The RefCell in CachingPricer is needed because price(&self, ...) takes a shared reference but the cache must be mutated. This is interior mutability — Rust's way of saying "I promise this is safe even though it looks mutable through an immutable reference."

Go decorator and pipeline

package main

import (
    "fmt"
    "math"
    "sync"
    "time"
)

type Pricer interface {
    Price(S, K, T float64) float64
}

type BSPricer struct{}
func (bs BSPricer) Price(S, K, T float64) float64 { return math.Max(S-K, 0) * 1.05 }

// Caching decorator
type CachingPricer struct {
    inner Pricer
    mu    sync.RWMutex
    cache map[[3]int64]float64
}

func NewCachingPricer(inner Pricer) *CachingPricer {
    return &CachingPricer{inner: inner, cache: make(map[[3]int64]float64)}
}

func (c *CachingPricer) Price(S, K, T float64) float64 {
    key := [3]int64{int64(S * 1e4), int64(K * 1e4), int64(T * 1e4)}
    c.mu.RLock()
    if v, ok := c.cache[key]; ok { c.mu.RUnlock(); return v }
    c.mu.RUnlock()
    v := c.inner.Price(S, K, T)
    c.mu.Lock()
    c.cache[key] = v
    c.mu.Unlock()
    return v
}

// Logging decorator
type LoggingPricer struct{ inner Pricer }

func (l LoggingPricer) Price(S, K, T float64) float64 {
    start := time.Now()
    result := l.inner.Price(S, K, T)
    fmt.Printf("Price(%.0f,%.0f,%.1f) = %.4f [%v]\n", S, K, T, result, time.Since(start))
    return result
}

// Stack decorators
pricer := LoggingPricer{inner: NewCachingPricer(BSPricer{})}
pricer.Price(100, 100, 1)

// Functional pipeline
type Stage func(float64) float64

func Pipeline(value float64, stages ...Stage) float64 {
    for _, fn := range stages {
        value = fn(value)
    }
    return value
}

result := Pipeline(100.123456,
    func(x float64) float64 { return math.Round(x*1e4) / 1e4 },
    func(x float64) float64 { return x * 1.001 },
)
Go decorators are straightforward — each wrapper struct holds the inner Pricer interface. The caching decorator uses sync.RWMutex for thread-safe concurrent access. Go's interface satisfaction is implicit, so any struct with a Price method works as a decorator without declaring it.

TypeScript decorator and pipeline

interface Pricer {
  price(S: number, K: number, T: number): number;
}

class BSPricer implements Pricer {
  price(S: number, K: number, T: number): number { return Math.max(S - K, 0) * 1.05; }
}

class CachingPricer implements Pricer {
  private cache = new Map<string, number>();
  constructor(private inner: Pricer) {}

  price(S: number, K: number, T: number): number {
    const key = `${S.toFixed(4)}|${K.toFixed(4)}|${T.toFixed(4)}`;
    if (this.cache.has(key)) return this.cache.get(key)!;
    const v = this.inner.price(S, K, T);
    this.cache.set(key, v);
    return v;
  }
}

class LoggingPricer implements Pricer {
  constructor(private inner: Pricer) {}

  price(S: number, K: number, T: number): number {
    const start = performance.now();
    const result = this.inner.price(S, K, T);
    console.log(`price(${S},${K},${T}) = ${result.toFixed(4)} [${(performance.now()-start).toFixed(3)}ms]`);
    return result;
  }
}

// Stack decorators
const pricer: Pricer = new LoggingPricer(new CachingPricer(new BSPricer()));
pricer.price(100, 100, 1);

// Functional pipeline
type Stage<T> = (value: T) => T;

function pipeline<T>(value: T, ...stages: Stage<T>[]): T {
  return stages.reduce((v, fn) => fn(v), value);
}

const result = pipeline(
  100.123456,
  x => Math.round(x * 1e4) / 1e4,
  x => x * 1.001,
);
TypeScript's Array.reduce makes pipelines concise. The decorator classes use constructor injection — private inner: Pricer in the constructor both declares and assigns the field. TypeScript also supports native decorators (@decorator syntax) for method-level decoration, but the wrapper-class approach shown here is more explicit and works at the object level.

Key observations

  • Rust decorators are zero-cost when using generics. The compiler monomorphises the entire decorator stack into a single function — no virtual dispatch at any layer. C++ achieves the same with templates, but Rust's approach also guarantees memory safety.
  • Python and TypeScript have native decorator syntax (@decorator) that simplifies method-level decoration. However, object-level wrapping (shown here) is more powerful and works across all languages.
  • Go's implicit interfaces make decorators natural. Any struct with a Price method satisfies the interface — no need to declare "implements Pricer". This makes adding a new decorator layer trivial.
  • Thread-safe caching is hardest in C++ and easiest in Rust. C++ requires manual mutex management with no compiler help. Rust's RefCell (single-thread) or RwLock (multi-thread) are enforced by the type system — you cannot accidentally share a non-thread-safe cache across threads.
  • Pipelines are most natural in functional-friendly languages — Python's list comprehension style and TypeScript's reduce are concise. C++ requires std::function overhead or template metaprogramming.

Decision guide for Decorator / Pipeline

  • Cross-cutting concerns (logging, timing, caching): Decorator in any language — pick by team expertise
  • Performance-critical decorator stack: Rust generics or C++ templates (zero dispatch overhead)
  • Data transformation chain: Pipeline — Python or TypeScript for conciseness, Go for concurrency
  • Thread-safe memoisation: Rust RwLock or Go sync.RWMutex
  • Rapid prototyping: Python @functools.lru_cache for instant caching without a decorator class
