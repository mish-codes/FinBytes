Three ways to create objects

Creational patterns address who creates objects and how. The wrong construction strategy leads to tight coupling, untestable code, and hidden dependencies.

  • Factory: create objects without specifying exact class. A PricerFactory reads config and returns the right pricer.
  • Builder: construct complex objects step by step. A TradeBuilder enforces required fields and makes construction readable with many optional fields.
  • Singleton: ensure only one instance exists. Legitimate uses are rare - shared resources like connection pools. In C++ the singleton has specific thread-safety concerns.

Python creational patterns

from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional
import threading

# Factory
class PricingModel(ABC):
    @abstractmethod
    def price(self, S: float, K: float, T: float) -> float: ...

class BlackScholes(PricingModel):
    def price(self, S, K, T) -> float: return max(S - K, 0)

class MonteCarlo(PricingModel):
    def __init__(self, n_paths: int = 10_000): self.n_paths = n_paths
    def price(self, S, K, T) -> float: return max(S - K, 0) * 0.99

def pricer_factory(model_type: str, **kwargs) -> PricingModel:
    models = {"black_scholes": BlackScholes, "monte_carlo": MonteCarlo}
    if model_type not in models:
        raise ValueError(f"Unknown model: {model_type}")
    return models[model_type](**kwargs)

# Builder
@dataclass
class Trade:
    symbol:   str
    notional: float
    maturity: float
    coupon:   float = 0.0
    currency: str   = "GBP"

class TradeBuilder:
    def __init__(self): self._t = {}
    def symbol(self, s):   self._t["symbol"]   = s;   return self
    def notional(self, n): self._t["notional"] = n;   return self
    def maturity(self, m): self._t["maturity"] = m;   return self
    def coupon(self, c):   self._t["coupon"]   = c;   return self
    def build(self) -> Trade:
        for req in ("symbol", "notional", "maturity"):
            if req not in self._t: raise ValueError(f"Missing: {req}")
        return Trade(**self._t)

trade = TradeBuilder().symbol("GILT2030").notional(1_000_000).maturity(5.0).build()

# Singleton (thread-safe)
class ConnectionPool:
    _instance: Optional["ConnectionPool"] = None
    _lock = threading.Lock()
    def __new__(cls):
        with cls._lock:
            if cls._instance is None:
                cls._instance = super().__new__(cls)
        return cls._instance

C# creational patterns

// Factory
interface IPricingModel { double Price(double S, double K, double T); }
record BlackScholes : IPricingModel { public double Price(double S, double K, double T) => Math.Max(S-K, 0); }
record MonteCarlo(int NPaths) : IPricingModel { public double Price(double S, double K, double T) => Math.Max(S-K,0)*0.99; }

static IPricingModel PricerFactory(string type, int nPaths = 10_000) => type switch {
    "black_scholes" => new BlackScholes(),
    "monte_carlo"   => new MonteCarlo(nPaths),
    _               => throw new ArgumentException($"Unknown: {type}")
};

// Builder
class TradeBuilder {
    private string? _symbol; private double? _notional; private double? _maturity;
    public TradeBuilder Symbol(string s)   { _symbol   = s; return this; }
    public TradeBuilder Notional(double n) { _notional = n; return this; }
    public TradeBuilder Maturity(double m) { _maturity = m; return this; }
    public Trade Build() {
        if (_symbol is null || _notional is null || _maturity is null)
            throw new InvalidOperationException("Required fields missing");
        return new Trade(_symbol, _notional.Value, _maturity.Value);
    }
}

var trade = new TradeBuilder().Symbol("GILT2030").Notional(1_000_000).Maturity(5.0).Build();

// Singleton via Lazy<T>
public sealed class ConnectionPool {
    private static readonly Lazy<ConnectionPool> _instance = new(() => new ConnectionPool());
    public static ConnectionPool Instance => _instance.Value;
    private ConnectionPool() { }
}

C++ creational patterns

#include <memory>
#include <string>
#include <stdexcept>

// Factory
struct IPricingModel {
    virtual ~IPricingModel() = default;
    virtual double price(double S, double K, double T) const = 0;
};
struct BlackScholes : IPricingModel {
    double price(double S, double K, double T) const override { return std::max(S-K, 0.0); }
};
struct MonteCarlo : IPricingModel {
    int n_paths;
    explicit MonteCarlo(int n = 10'000) : n_paths(n) {}
    double price(double S, double K, double T) const override { return std::max(S-K,0.0)*0.99; }
};

std::unique_ptr<IPricingModel> pricer_factory(const std::string& type, int n = 10'000) {
    if (type == "black_scholes") return std::make_unique<BlackScholes>();
    if (type == "monte_carlo")   return std::make_unique<MonteCarlo>(n);
    throw std::invalid_argument("Unknown model: " + type);
}

// Builder
struct Trade { std::string symbol; double notional; double maturity; double coupon = 0.0; };

class TradeBuilder {
    Trade t_; bool hs_=false, hn_=false, hm_=false;
public:
    TradeBuilder& symbol(std::string s)  { t_.symbol   = std::move(s); hs_=true; return *this; }
    TradeBuilder& notional(double n)     { t_.notional = n;            hn_=true; return *this; }
    TradeBuilder& maturity(double m)     { t_.maturity = m;            hm_=true; return *this; }
    TradeBuilder& coupon(double c)       { t_.coupon   = c;                      return *this; }
    Trade build() {
        if (!hs_ || !hn_ || !hm_) throw std::logic_error("Required fields missing");
        return t_;
    }
};

auto trade = TradeBuilder().symbol("GILT2030").notional(1'000'000).maturity(5.0).build();

// Singleton - Meyers singleton (C++11 thread-safe)
class ConnectionPool {
public:
    static ConnectionPool& instance() {
        static ConnectionPool inst;  // guaranteed thread-safe since C++11
        return inst;
    }
    ConnectionPool(const ConnectionPool&)            = delete;
    ConnectionPool& operator=(const ConnectionPool&) = delete;
private:
    ConnectionPool() {}
};
The static local variable singleton is the Meyers Singleton, guaranteed thread-safe since C++11. The double-checked locking pattern seen in older code is unnecessary. Use this instead.

Rust creational patterns

Factory

trait PricingModel {
    fn price(&self, s: f64, k: f64, t: f64) -> f64;
}

struct BlackScholes;
impl PricingModel for BlackScholes {
    fn price(&self, s: f64, k: f64, _t: f64) -> f64 { (s - k).max(0.0) }
}

struct MonteCarlo { n_paths: usize }
impl PricingModel for MonteCarlo {
    fn price(&self, s: f64, k: f64, _t: f64) -> f64 { (s - k).max(0.0) * 0.99 }
}

fn pricer_factory(model_type: &str, n_paths: usize) -> Box<dyn PricingModel> {
    match model_type {
        "black_scholes" => Box::new(BlackScholes),
        "monte_carlo"   => Box::new(MonteCarlo { n_paths }),
        other           => panic!("Unknown model: {other}"),
    }
}

Builder

struct Trade {
    symbol:   String,
    notional: f64,
    maturity: f64,
    coupon:   f64,
    currency: String,
}

#[derive(Default)]
struct TradeBuilder {
    symbol:   Option<String>,
    notional: Option<f64>,
    maturity: Option<f64>,
    coupon:   Option<f64>,
    currency: Option<String>,
}

impl TradeBuilder {
    fn symbol(mut self, s: &str)    -> Self { self.symbol   = Some(s.into()); self }
    fn notional(mut self, n: f64)   -> Self { self.notional = Some(n);        self }
    fn maturity(mut self, m: f64)   -> Self { self.maturity = Some(m);        self }
    fn coupon(mut self, c: f64)     -> Self { self.coupon   = Some(c);        self }
    fn build(self) -> Trade {
        Trade {
            symbol:   self.symbol.expect("symbol required"),
            notional: self.notional.expect("notional required"),
            maturity: self.maturity.expect("maturity required"),
            coupon:   self.coupon.unwrap_or(0.0),
            currency: self.currency.unwrap_or_else(|| "GBP".into()),
        }
    }
}

let trade = TradeBuilder::default()
    .symbol("GILT2030")
    .notional(1_000_000.0)
    .maturity(5.0)
    .build();

Singleton

use std::sync::OnceLock;

struct ConnectionPool { /* ... */ }

impl ConnectionPool {
    fn instance() -> &'static ConnectionPool {
        static POOL: OnceLock<ConnectionPool> = OnceLock::new();
        POOL.get_or_init(|| ConnectionPool { /* ... */ })
    }
}
Rust's OnceLock (stable since 1.70) provides a safe, lazy singleton without any unsafe. The builder pattern is extremely common in Rust — the Option<T> fields plus .expect() in build() enforce required vs optional fields at runtime. For compile-time enforcement, use the typestate builder pattern with generic marker types.

Go creational patterns

Factory

package main

import "fmt"

type PricingModel interface {
    Price(S, K, T float64) float64
}

type BlackScholes struct{}
func (bs BlackScholes) Price(S, K, T float64) float64 { return max(S-K, 0) }

type MonteCarlo struct{ NPaths int }
func (mc MonteCarlo) Price(S, K, T float64) float64 { return max(S-K, 0) * 0.99 }

func PricerFactory(modelType string, nPaths int) (PricingModel, error) {
    switch modelType {
    case "black_scholes":
        return BlackScholes{}, nil
    case "monte_carlo":
        return MonteCarlo{NPaths: nPaths}, nil
    default:
        return nil, fmt.Errorf("unknown model: %s", modelType)
    }
}

Builder (functional options)

type Trade struct {
    Symbol   string
    Notional float64
    Maturity float64
    Coupon   float64
    Currency string
}

type TradeOption func(*Trade)

func WithCoupon(c float64)   TradeOption { return func(t *Trade) { t.Coupon = c } }
func WithCurrency(c string)  TradeOption { return func(t *Trade) { t.Currency = c } }

func NewTrade(symbol string, notional, maturity float64, opts ...TradeOption) Trade {
    t := Trade{Symbol: symbol, Notional: notional, Maturity: maturity, Currency: "GBP"}
    for _, opt := range opts {
        opt(&t)
    }
    return t
}

trade := NewTrade("GILT2030", 1_000_000, 5.0, WithCoupon(0.03))

Singleton

import "sync"

type ConnectionPool struct{ /* ... */ }

var (
    poolInstance *ConnectionPool
    poolOnce     sync.Once
)

func GetPool() *ConnectionPool {
    poolOnce.Do(func() {
        poolInstance = &ConnectionPool{ /* ... */ }
    })
    return poolInstance
}
Go favours the "functional options" pattern over a classical builder. Required fields are constructor parameters; optional fields use variadic option functions. sync.Once guarantees the singleton initialiser runs exactly once, even under concurrent access — the Go equivalent of Meyers' Singleton.

TypeScript creational patterns

Factory

interface PricingModel {
  price(S: number, K: number, T: number): number;
}

class BlackScholes implements PricingModel {
  price(S: number, K: number, T: number): number { return Math.max(S - K, 0); }
}

class MonteCarlo implements PricingModel {
  constructor(private nPaths: number = 10_000) {}
  price(S: number, K: number, T: number): number { return Math.max(S - K, 0) * 0.99; }
}

function pricerFactory(type: string, nPaths = 10_000): PricingModel {
  switch (type) {
    case "black_scholes": return new BlackScholes();
    case "monte_carlo":   return new MonteCarlo(nPaths);
    default: throw new Error(`Unknown model: ${type}`);
  }
}

Builder

interface Trade {
  symbol: string; notional: number; maturity: number;
  coupon: number; currency: string;
}

class TradeBuilder {
  private _symbol?:   string;
  private _notional?: number;
  private _maturity?: number;
  private _coupon     = 0;
  private _currency   = "GBP";

  symbol(s: string)    { this._symbol   = s; return this; }
  notional(n: number)  { this._notional = n; return this; }
  maturity(m: number)  { this._maturity = m; return this; }
  coupon(c: number)    { this._coupon   = c; return this; }
  currency(c: string)  { this._currency = c; return this; }

  build(): Trade {
    if (!this._symbol || !this._notional || !this._maturity)
      throw new Error("Required fields missing");
    return {
      symbol: this._symbol, notional: this._notional,
      maturity: this._maturity, coupon: this._coupon, currency: this._currency,
    };
  }
}

const trade = new TradeBuilder().symbol("GILT2030").notional(1_000_000).maturity(5).build();

Singleton

class ConnectionPool {
  private static _instance: ConnectionPool;
  private constructor() { /* ... */ }
  static get instance(): ConnectionPool {
    if (!ConnectionPool._instance) ConnectionPool._instance = new ConnectionPool();
    return ConnectionPool._instance;
  }
}
TypeScript's private constructor prevents external instantiation, enforcing the singleton. For the builder, TypeScript's structural typing means you can also use a plain object literal — the builder adds value when you need validation or defaults that go beyond what a simple interface provides.

Key observations

  • Rust's builder is the most rigorous. The typestate builder pattern (using generic markers for set/unset fields) can make it a compile error to call .build() without setting required fields. No other language offers this without code generation.
  • Go replaces the builder with functional options. Required fields are constructor parameters (can't forget them), optional fields use variadic functions. Less ceremony than a builder class, more idiomatic in Go.
  • Python's factory is the simplest — a dict mapping names to classes. But with no type enforcement, a typo in the model name is a runtime error. Type checkers help but are optional.
  • C++ factory returns unique_ptr — ownership is explicit. The caller knows it owns the created object. In Python/Go/TS, ownership is implicit (GC handles it), which is simpler but hides memory semantics.
  • Singleton is controversial in all languages. Prefer dependency injection. When you must use a singleton (connection pools, config), Rust's OnceLock, C++'s Meyers pattern, C#'s Lazy<T>, and Go's sync.Once are all thread-safe by default.

Decision guide for creational patterns

  • Object type chosen at runtime (config, user input): Factory in any language
  • Many optional fields, complex validation: Builder (Rust for strictest, Go functional options for lightest)
  • Shared resource (pool, config): Singleton — but inject it rather than accessing the global
  • Performance-critical object creation: C++ or Rust — stack allocation, no GC overhead
  • Rapid prototyping: Python dict-factory or TypeScript factory function
