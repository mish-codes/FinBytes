Decoupling producers from consumers

The Observer pattern lets one object (publisher) notify many others (subscribers) when its state changes, without knowing who those others are. It is the backbone of market data distribution: a feed publishes price updates; pricers, risk engines, and dashboards all subscribe independently.

Push vs pull: Push sends data to observers immediately. Pull notifies observers something changed; they fetch what they need. Push is simpler and lower-latency. Pull avoids sending data to observers that do not need it.

A market data event arrives. Multiple subscribers handle it: the option pricer re-prices, the delta hedger recomputes the hedge, the risk monitor updates PnL. None of them need to know about each other.

Python observer pattern

from typing import Callable, Protocol
from dataclasses import dataclass

@dataclass
class PriceUpdate:
    symbol: str
    price:  float
    volume: int

class PriceObserver(Protocol):
    def on_price(self, update: PriceUpdate) -> None: ...

class MarketDataFeed:
    def __init__(self): self._observers: list[PriceObserver] = []
    def subscribe(self, obs: PriceObserver) -> None: self._observers.append(obs)
    def unsubscribe(self, obs: PriceObserver) -> None: self._observers.remove(obs)
    def publish(self, update: PriceUpdate) -> None:
        for obs in self._observers: obs.on_price(update)

class OptionPricer:
    def on_price(self, u: PriceUpdate) -> None:
        print(f"Pricer: repricing {u.symbol} @ {u.price}")

class RiskMonitor:
    def on_price(self, u: PriceUpdate) -> None:
        print(f"Risk: updating PnL for {u.symbol}")

feed = MarketDataFeed()
feed.subscribe(OptionPricer())
feed.subscribe(RiskMonitor())
feed.publish(PriceUpdate("AAPL", 151.25, 10_000))

# Callable observer variant
class SimpleEventBus:
    def __init__(self): self._handlers: list[Callable] = []
    def on(self, fn: Callable) -> None: self._handlers.append(fn)
    def emit(self, *args) -> None:
        for h in self._handlers: h(*args)

C# events and delegates

public record PriceUpdate(string Symbol, double Price, int Volume);

public class MarketDataFeed {
    public event EventHandler<PriceUpdate>? PriceReceived;
    public void Publish(PriceUpdate update) => PriceReceived?.Invoke(this, update);
}

public class OptionPricer {
    public void OnPrice(object? sender, PriceUpdate u) =>
        Console.WriteLine($"Pricer: repricing {u.Symbol} @ {u.Price}");
}

var feed   = new MarketDataFeed();
var pricer = new OptionPricer();
feed.PriceReceived += pricer.OnPrice;
feed.PriceReceived += (_, u) => Console.WriteLine($"LOG: {u.Symbol}");
feed.Publish(new PriceUpdate("AAPL", 151.25, 10_000));
feed.PriceReceived -= pricer.OnPrice;  // unsubscribe

C++ observer - interface-based and callback-based

#include <vector>
#include <functional>
#include <string>
#include <algorithm>

struct PriceUpdate { std::string symbol; double price; int volume; };

// Interface-based
struct IPriceObserver {
    virtual ~IPriceObserver() = default;
    virtual void on_price(const PriceUpdate&) = 0;
};

class MarketDataFeed {
    std::vector<IPriceObserver*> observers_;
public:
    void subscribe(IPriceObserver* obs) { observers_.push_back(obs); }
    void unsubscribe(IPriceObserver* obs) {
        observers_.erase(std::remove(observers_.begin(), observers_.end(), obs), observers_.end());
    }
    void publish(const PriceUpdate& u) {
        for (auto* obs : observers_) obs->on_price(u);
    }
};

// Callback-based (accepts lambdas, member functions, free functions)
class EventFeed {
    using Handler = std::function<void(const PriceUpdate&)>;
    std::vector<Handler> handlers_;
public:
    void on(Handler h)           { handlers_.push_back(std::move(h)); }
    void emit(const PriceUpdate& u) { for (const auto& h : handlers_) h(u); }
};

EventFeed feed;
feed.on([](const PriceUpdate& u) { /* reprice */ });
feed.on([](const PriceUpdate& u) { /* risk update */ });
feed.emit({"AAPL", 151.25, 10'000});
Raw observer pointers work if observer lifetimes are managed carefully (unsubscribe before destruction). For shared ownership, use weak_ptr observers - the publisher skips any that have expired, avoiding dangling pointer crashes.

Rust observer pattern

use std::sync::{Arc, Mutex};

struct PriceUpdate {
    symbol: String,
    price:  f64,
    volume: i64,
}

trait PriceObserver: Send {
    fn on_price(&self, update: &PriceUpdate);
}

struct MarketDataFeed {
    observers: Vec<Arc<dyn PriceObserver>>,
}

impl MarketDataFeed {
    fn new() -> Self { Self { observers: Vec::new() } }

    fn subscribe(&mut self, obs: Arc<dyn PriceObserver>) {
        self.observers.push(obs);
    }

    fn publish(&self, update: &PriceUpdate) {
        for obs in &self.observers {
            obs.on_price(update);
        }
    }
}

struct OptionPricer;
impl PriceObserver for OptionPricer {
    fn on_price(&self, u: &PriceUpdate) {
        println!("Pricer: repricing {} @ {:.2}", u.symbol, u.price);
    }
}

struct RiskMonitor;
impl PriceObserver for RiskMonitor {
    fn on_price(&self, u: &PriceUpdate) {
        println!("Risk: updating PnL for {}", u.symbol);
    }
}

// Callback-based variant using closures
struct EventFeed {
    handlers: Vec<Box<dyn Fn(&PriceUpdate) + Send>>,
}

impl EventFeed {
    fn new() -> Self { Self { handlers: Vec::new() } }
    fn on(&mut self, f: impl Fn(&PriceUpdate) + Send + 'static) {
        self.handlers.push(Box::new(f));
    }
    fn emit(&self, update: &PriceUpdate) {
        for h in &self.handlers { h(update); }
    }
}
Rust requires observers to be Send if the feed might be shared across threads. Arc<dyn PriceObserver> gives shared ownership with reference counting — no dangling pointers. For mutable observers, wrap in Arc<Mutex<T>>. The closure-based variant avoids trait objects entirely, which is often simpler for small projects.

Go observer pattern

package main

import "fmt"

type PriceUpdate struct {
    Symbol string
    Price  float64
    Volume int
}

type PriceObserver interface {
    OnPrice(update PriceUpdate)
}

type MarketDataFeed struct {
    observers []PriceObserver
}

func (f *MarketDataFeed) Subscribe(obs PriceObserver) {
    f.observers = append(f.observers, obs)
}

func (f *MarketDataFeed) Publish(update PriceUpdate) {
    for _, obs := range f.observers {
        obs.OnPrice(update)
    }
}

type OptionPricer struct{}
func (p OptionPricer) OnPrice(u PriceUpdate) {
    fmt.Printf("Pricer: repricing %s @ %.2f\n", u.Symbol, u.Price)
}

type RiskMonitor struct{}
func (r RiskMonitor) OnPrice(u PriceUpdate) {
    fmt.Printf("Risk: updating PnL for %s\n", u.Symbol)
}

// Channel-based observer (idiomatic Go for concurrent consumers)
type ChannelFeed struct {
    subscribers []chan PriceUpdate
}

func (f *ChannelFeed) Subscribe() chan PriceUpdate {
    ch := make(chan PriceUpdate, 64)
    f.subscribers = append(f.subscribers, ch)
    return ch
}

func (f *ChannelFeed) Publish(update PriceUpdate) {
    for _, ch := range f.subscribers {
        select {
        case ch <- update:
        default: // drop if subscriber is slow
        }
    }
}

// Usage:
// ch := feed.Subscribe()
// go func() { for u := range ch { process(u) } }()
Go's channels provide a natural observer mechanism. Each subscriber gets its own channel, and the publisher fans out updates. The select/default idiom drops messages to slow subscribers rather than blocking the publisher — a common pattern in real-time market data systems where stale data is worse than missing data.

TypeScript observer pattern

interface PriceUpdate {
  symbol: string;
  price:  number;
  volume: number;
}

interface PriceObserver {
  onPrice(update: PriceUpdate): void;
}

class MarketDataFeed {
  private observers: PriceObserver[] = [];

  subscribe(obs: PriceObserver): void { this.observers.push(obs); }

  unsubscribe(obs: PriceObserver): void {
    this.observers = this.observers.filter(o => o !== obs);
  }

  publish(update: PriceUpdate): void {
    for (const obs of this.observers) obs.onPrice(update);
  }
}

class OptionPricer implements PriceObserver {
  onPrice(u: PriceUpdate): void {
    console.log(`Pricer: repricing ${u.symbol} @ ${u.price}`);
  }
}

// Generic typed event emitter
class EventEmitter<T> {
  private handlers: Array<(data: T) => void> = [];

  on(handler: (data: T) => void): () => void {
    this.handlers.push(handler);
    return () => { this.handlers = this.handlers.filter(h => h !== handler); };
  }

  emit(data: T): void {
    for (const h of this.handlers) h(data);
  }
}

// Usage
const priceEvents = new EventEmitter<PriceUpdate>();
const unsub = priceEvents.on(u => console.log(`${u.symbol}: ${u.price}`));
priceEvents.emit({ symbol: "AAPL", price: 151.25, volume: 10_000 });
unsub(); // clean unsubscribe
The generic EventEmitter<T> is a reusable building block. The on() method returns an unsubscribe function — a pattern borrowed from React/RxJS that avoids the "forgot to unsubscribe" memory leak. TypeScript's type parameter ensures subscribers receive correctly typed data at compile time.

Key observations

  • Go channels are the most natural fit for observer in concurrent systems. Each subscriber runs in its own goroutine, receiving events through a buffered channel. No shared mutable state, no locks. The publisher does not block if a subscriber is slow — it drops or buffers.
  • C# has first-class language support via the event keyword and delegates. Subscribe with +=, unsubscribe with -=. This is the most ergonomic observer syntax of all six languages.
  • C++ observer is dangerous if observer lifetimes are not managed. A destroyed observer that is still subscribed causes a dangling pointer crash. Use weak_ptr or ensure unsubscribe-before-destroy discipline.
  • Rust enforces lifetime safety at compile time. You cannot hold a reference to a destroyed observer — the compiler rejects it. The cost: more explicit ownership (Arc, Box) and trait bounds (Send).
  • TypeScript's unsubscribe-function pattern (returning a cleanup closure from on()) is the cleanest way to avoid forgotten unsubscriptions. Borrowed from the React/RxJS ecosystem.

Decision guide for the Observer pattern

  • Real-time market data fan-out: Go channels or Rust with tokio::broadcast
  • UI event binding: C# events or TypeScript EventEmitter
  • Simple in-process pub/sub: Python Protocol-based or callback list
  • High-frequency with many subscribers: C++ with careful lifetime management
  • Cross-thread notification: Go channels (built-in) or Rust Arc<Mutex>
