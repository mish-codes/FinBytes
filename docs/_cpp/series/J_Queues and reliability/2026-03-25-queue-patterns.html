Four queue types, four use cases

  • Bounded FIFO: market data feed to pricer. Bounded provides backpressure when the pricer cannot keep up.
  • Priority queue: order router. High-urgency cancel/replace jumps the queue ahead of new orders.
  • Delay queue: retry scheduler. Failed connections retry after an exponentially increasing delay.
  • Work-stealing deque: parallel batch pricer. Threads steal work from each other when their own queue is empty.

Bounded vs unbounded: unbounded queues hide backpressure problems - the producer keeps adding work, memory grows without limit, the system eventually OOMs. Bounded queues force the producer to slow down (block), shed load (drop), or signal the caller (return error). Always prefer bounded in production.

C++ bounded thread-safe queue

#include <queue>
#include <mutex>
#include <condition_variable>
#include <optional>
#include <chrono>

template<typename T>
class BoundedQueue {
    std::queue<T>           q_;
    std::mutex              mtx_;
    std::condition_variable not_full_, not_empty_;
    std::size_t             capacity_;
public:
    explicit BoundedQueue(std::size_t cap) : capacity_(cap) {}

    bool push(T item, std::chrono::milliseconds timeout = std::chrono::seconds(1)) {
        std::unique_lock lock(mtx_);
        if (!not_full_.wait_for(lock, timeout, [&]{ return q_.size() < capacity_; }))
            return false;   // timeout - caller can drop or count
        q_.push(std::move(item));
        not_empty_.notify_one();
        return true;
    }

    std::optional<T> pop(std::chrono::milliseconds timeout = std::chrono::seconds(5)) {
        std::unique_lock lock(mtx_);
        if (!not_empty_.wait_for(lock, timeout, [&]{ return !q_.empty(); }))
            return std::nullopt;
        T item = std::move(q_.front()); q_.pop();
        not_full_.notify_one();
        return item;
    }

    std::size_t size() const { std::lock_guard l(mtx_); return q_.size(); }
};

// Priority queue
#include <vector>
struct Order { int priority; std::string id; };
auto cmp = [](const Order& a, const Order& b) { return a.priority > b.priority; };
std::priority_queue<Order, std::vector<Order>, decltype(cmp)> order_pq(cmp);
For ultra-low-latency use an SPSC (single-producer single-consumer) lock-free queue instead of the mutex-based one above. The SPSC queue is appropriate for the 10-50ns market data hot path; the mutex version is fine for background threads.

Rust queue patterns

use std::sync::Arc;
use tokio::sync::mpsc;
use tokio::time::{sleep, Duration, Instant};
use std::collections::BinaryHeap;
use std::cmp::Reverse;

// Bounded FIFO — tokio::sync::mpsc is bounded by default
async fn bounded_fifo() {
    let (tx, mut rx) = mpsc::channel::<Tick>(1024); // bounded to 1024

    // Producer — send() awaits when full (backpressure built in)
    let producer = tokio::spawn(async move {
        for tick in ticks {
            if tx.send(tick).await.is_err() {
                break; // receiver dropped
            }
        }
    });

    // Consumer
    while let Some(tick) = rx.recv().await {
        process_tick(tick);
    }
}

// Priority queue — wrap BinaryHeap with tokio::sync::Mutex
use tokio::sync::Mutex;

#[derive(Eq, PartialEq)]
struct OrderMsg {
    priority: u32,
    order: Order,
}
impl Ord for OrderMsg {
    fn cmp(&self, other: &Self) -> std::cmp::Ordering {
        other.priority.cmp(&self.priority) // min-heap: lower = higher priority
    }
}
impl PartialOrd for OrderMsg {
    fn partial_cmp(&self, other: &Self) -> Option<std::cmp::Ordering> {
        Some(self.cmp(other))
    }
}

let pq: Arc<Mutex<BinaryHeap<OrderMsg>>> = Arc::new(Mutex::new(BinaryHeap::new()));
pq.lock().await.push(OrderMsg { priority: 1, order: cancel_order }); // high
pq.lock().await.push(OrderMsg { priority: 5, order: new_order });

// Delay queue — sleep until the earliest item is due
struct DelayedTask {
    run_at: Instant,
    task: Box<dyn FnOnce() + Send>,
}

async fn delay_queue_loop(mut rx: mpsc::Receiver<DelayedTask>) {
    let mut heap: BinaryHeap<Reverse<(Instant, usize)>> = BinaryHeap::new();
    let mut tasks: Vec<Option<Box<dyn FnOnce() + Send>>> = Vec::new();

    loop {
        let timeout = heap.peek().map(|Reverse((t, _))| *t);
        tokio::select! {
            Some(dt) = rx.recv() => {
                let idx = tasks.len();
                heap.push(Reverse((dt.run_at, idx)));
                tasks.push(Some(dt.task));
            }
            _ = async {
                match timeout {
                    Some(t) => sleep(t.saturating_duration_since(Instant::now())).await,
                    None => std::future::pending::<()>().await,
                }
            } => {
                while let Some(Reverse((t, idx))) = heap.peek() {
                    if *t > Instant::now() { break; }
                    let (_, idx) = heap.pop().unwrap().0;
                    if let Some(task) = tasks[idx].take() { task(); }
                }
            }
        }
    }
}

// Work-stealing — use rayon's thread pool (built-in work stealing)
use rayon::prelude::*;

fn batch_price(trades: &[Trade]) -> Vec<f64> {
    trades.par_iter().map(|t| price_trade(t)).collect()
}
Rust's mpsc::channel with a capacity gives you bounded FIFO with built-in backpressure — send().await suspends the producer when the buffer is full, no manual condition variables needed. For work-stealing, rayon is the go-to crate and mirrors C++'s TBB.

Go queue patterns

package main

import (
    "container/heap"
    "sync"
    "time"
)

// Bounded FIFO — a Go buffered channel IS a bounded queue
func boundedFIFO() {
    feed := make(chan Tick, 1024) // bounded to 1024

    // Producer — blocks when channel is full (backpressure built in)
    go func() {
        for tick := range ticks {
            feed <- tick // blocks if 1024 items buffered
        }
        close(feed)
    }()

    // Consumer — range drains until channel is closed
    for tick := range feed {
        processTick(tick)
    }
}

// Non-blocking push with drop counting
var dropped uint64

func producerWithDrop(feed chan<- Tick, ticks []Tick) {
    for _, tick := range ticks {
        select {
        case feed <- tick:
            // sent
        default:
            dropped++ // channel full — drop and count
        }
    }
}

// Priority queue — use container/heap
type OrderMsg struct {
    Priority int
    Order    Order
    index    int
}

type OrderHeap []*OrderMsg

func (h OrderHeap) Len() int            { return len(h) }
func (h OrderHeap) Less(i, j int) bool   { return h[i].Priority < h[j].Priority }
func (h OrderHeap) Swap(i, j int)        { h[i], h[j] = h[j], h[i]; h[i].index = i; h[j].index = j }
func (h *OrderHeap) Push(x interface{})  { *h = append(*h, x.(*OrderMsg)) }
func (h *OrderHeap) Pop() interface{}    { old := *h; n := len(old); item := old[n-1]; *h = old[:n-1]; return item }

func priorityQueueExample() {
    pq := &OrderHeap{}
    heap.Init(pq)
    heap.Push(pq, &OrderMsg{Priority: 1, Order: cancelOrder}) // high priority
    heap.Push(pq, &OrderMsg{Priority: 5, Order: newOrder})
    next := heap.Pop(pq).(*OrderMsg) // gets priority-1 first
    _ = next
}

// Delay queue — time.AfterFunc or timer + heap
func delayQueue() {
    type delayed struct {
        runAt time.Time
        task  func()
    }
    var mu sync.Mutex
    var tasks []delayed

    schedule := func(task func(), delay time.Duration) {
        mu.Lock()
        tasks = append(tasks, delayed{time.Now().Add(delay), task})
        mu.Unlock()
    }

    // Drain due tasks
    go func() {
        ticker := time.NewTicker(100 * time.Millisecond)
        defer ticker.Stop()
        for range ticker.C {
            now := time.Now()
            mu.Lock()
            var remaining []delayed
            for _, t := range tasks {
                if t.runAt.Before(now) {
                    go t.task()
                } else {
                    remaining = append(remaining, t)
                }
            }
            tasks = remaining
            mu.Unlock()
        }
    }()

    schedule(func() { reconnect() }, 5*time.Second)
}

// Work-stealing — Go's runtime scheduler does this automatically
// goroutines are multiplexed across OS threads with work stealing
func batchPrice(trades []Trade) []float64 {
    results := make([]float64, len(trades))
    var wg sync.WaitGroup
    for i, t := range trades {
        wg.Add(1)
        go func(idx int, trade Trade) {
            defer wg.Done()
            results[idx] = priceTrade(trade)
        }(i, t)
    }
    wg.Wait()
    return results
}
Go's buffered channels are the language's native bounded queue — no wrapper class needed. The select statement with a default case gives you non-blocking try-send in one line. Go's runtime scheduler already uses work-stealing across OS threads, so goroutines get it for free.

TypeScript queue patterns

// Bounded FIFO with backpressure
class BoundedQueue<T> {
  private queue: T[] = [];
  private waiters: Array<(value: T) => void> = [];
  private pushWaiters: Array<() => void> = [];

  constructor(private capacity: number) {}

  async push(item: T): Promise<void> {
    while (this.queue.length >= this.capacity) {
      await new Promise<void>(resolve => this.pushWaiters.push(resolve));
    }
    this.queue.push(item);
    if (this.waiters.length > 0) this.waiters.shift()!(this.queue.shift()!);
  }

  tryPush(item: T): boolean {
    if (this.queue.length >= this.capacity) return false;
    this.queue.push(item);
    if (this.waiters.length > 0) this.waiters.shift()!(this.queue.shift()!);
    return true;
  }

  async pop(): Promise<T> {
    if (this.queue.length > 0) {
      const item = this.queue.shift()!;
      if (this.pushWaiters.length > 0) this.pushWaiters.shift()!();
      return item;
    }
    return new Promise<T>(resolve => this.waiters.push(resolve));
  }

  get size(): number { return this.queue.length; }
}

// Usage
const feedQueue = new BoundedQueue<Tick>(1024);
let dropped = 0;

async function producer(ticks: AsyncIterable<Tick>): Promise<void> {
  for await (const tick of ticks) {
    if (!feedQueue.tryPush(tick)) dropped++;
  }
}

// Priority queue using sorted insert
class PriorityQueue<T> {
  private heap: Array<{ priority: number; item: T }> = [];

  enqueue(item: T, priority: number): void {
    this.heap.push({ priority, item });
    this.heap.sort((a, b) => a.priority - b.priority);
  }

  dequeue(): T | undefined {
    return this.heap.shift()?.item;
  }
}

const orderPQ = new PriorityQueue<Order>();
orderPQ.enqueue(cancelOrder, 1);  // high priority
orderPQ.enqueue(newOrder, 5);

// Delay queue using setTimeout
class DelayQueue {
  schedule(task: () => void, delayMs: number): void {
    setTimeout(task, delayMs);
  }

  scheduleRecurring(task: () => void, intervalMs: number): NodeJS.Timeout {
    return setInterval(task, intervalMs);
  }
}

// Work distribution — worker_threads for CPU-bound pricing
import { Worker, isMainThread, parentPort, workerData } from "worker_threads";

async function batchPrice(trades: Trade[]): Promise<number[]> {
  const cpuCount = require("os").cpus().length;
  const chunkSize = Math.ceil(trades.length / cpuCount);
  const promises = Array.from({ length: cpuCount }, (_, i) => {
    const chunk = trades.slice(i * chunkSize, (i + 1) * chunkSize);
    return new Promise<number[]>((resolve, reject) => {
      const worker = new Worker(__filename, { workerData: chunk });
      worker.on("message", resolve);
      worker.on("error", reject);
    });
  });
  return (await Promise.all(promises)).flat();
}
Node.js is single-threaded, so bounded queues are simpler (no mutex needed) but also less powerful — you cannot block a producer in the same way. The tryPush pattern with a drop counter is the idiomatic approach. For CPU-bound work-stealing, use worker_threads to distribute across cores.

Key observations

  • Go's buffered channel is the gold standard for queue ergonomics. A bounded FIFO with backpressure is one line: make(chan T, n). Non-blocking send is select/default. No wrapper class, no condition variables, no mutex. This is where Go shines most.
  • Rust's mpsc::channel is nearly as clean and adds compile-time ownership guarantees — you cannot accidentally share a receiver across threads without Arc. The downside: priority queues require more boilerplate than other languages.
  • C++ gives you maximum control and maximum responsibility. You write the mutex, condition variable, and capacity logic yourself. The payoff: lock-free SPSC queues for the hot path that no other language can match at 10-50ns per operation.
  • TypeScript's single-threaded model simplifies queues (no data races) but limits parallelism. For CPU-bound batch pricing, you must reach for worker_threads, which adds IPC serialisation overhead.
  • Go's runtime work-stealing is invisible. Every goroutine benefits from it automatically. In C++ and Rust you choose between manual thread pools (TBB, rayon) or rolling your own.

Decision guide for queue patterns

  • Market data feed to pricer (bounded FIFO): Go channel or Rust mpsc for clean backpressure; C++ SPSC for ultra-low-latency
  • Order routing with priority: C++ or C# have the best stdlib priority queues; Go requires container/heap boilerplate
  • Retry scheduling (delay queue): all languages need manual implementation; Go's time.AfterFunc is the most concise
  • Parallel batch pricing (work-stealing): Go goroutines or Rust rayon — both handle thousands of tasks effortlessly
  • Prototype or scripting: Python's queue.Queue is simple and correct; TypeScript if the team is JS-native
