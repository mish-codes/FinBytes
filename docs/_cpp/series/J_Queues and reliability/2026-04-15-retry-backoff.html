Why naive retry is dangerous

A service goes down. One thousand clients each retry immediately. All one thousand hit the service at the same instant it recovers. The burst overwhelms it and takes it down again. This is the thundering herd problem.

Exponential backoff: after each failure, wait twice as long before retrying: 1s, 2s, 4s, 8s, 16s... With many clients all starting at different times, load is spread across time rather than concentrated into a spike.

Jitter: even with exponential backoff, clients that start at the same time will retry together. Adding random jitter - a random fraction of the backoff delay - desynchronises them.

Dead letter queue: some failures are permanent. After N retries, stop and route the message to a DLQ for manual inspection. Without a DLQ, permanently-failing messages loop forever.

Python retry with backoff

import time, random, logging, json
from typing import Callable, TypeVar
from pathlib import Path

T = TypeVar("T")

def retry_with_backoff(
    fn: Callable[[], T], max_attempts: int = 5,
    base_delay_s: float = 1.0, max_delay_s: float = 60.0, jitter: bool = True,
) -> T:
    for attempt in range(1, max_attempts + 1):
        try:
            return fn()
        except Exception as e:
            if attempt == max_attempts: raise
            delay = min(base_delay_s * (2 ** (attempt - 1)), max_delay_s)
            if jitter: delay *= random.uniform(0.5, 1.5)
            logging.warning(f"Attempt {attempt} failed: {e}. Retrying in {delay:.1f}s")
            time.sleep(delay)

def connect_to_feed():
    return MarketDataFeed.connect("tcp://feed:5555")

feed = retry_with_backoff(connect_to_feed, max_attempts=5, base_delay_s=1.0)

# Dead letter queue
class DeadLetterQueue:
    def __init__(self, path: str = "dlq.jsonl"): self._path = Path(path)
    def push(self, msg: dict, reason: str) -> None:
        entry = {"msg": msg, "reason": reason, "ts": time.time()}
        with open(self._path, "a") as f: f.write(json.dumps(entry) + "
")
        logging.error(f"DLQ: {reason}")

C++ retry with backoff

#include <chrono>
#include <thread>
#include <random>
#include <functional>
#include <stdexcept>
#include <cmath>
#include <iostream>

struct RetryConfig {
    int    max_attempts = 5;
    double base_delay_s = 1.0;
    double max_delay_s  = 60.0;
    bool   jitter       = true;
};

template<typename F>
auto retry_with_backoff(F fn, const RetryConfig& cfg = {}) -> decltype(fn()) {
    std::mt19937 rng(std::random_device{}());
    std::uniform_real_distribution<double> jitter_dist(0.5, 1.5);

    for (int attempt = 1; attempt <= cfg.max_attempts; ++attempt) {
        try {
            return fn();
        } catch (const std::exception& e) {
            if (attempt == cfg.max_attempts) throw;
            double delay = std::min(cfg.base_delay_s * std::pow(2.0, attempt-1), cfg.max_delay_s);
            if (cfg.jitter) delay *= jitter_dist(rng);
            std::cerr << "Attempt " << attempt << " failed: " << e.what()
                      << " - retrying in " << delay << "s
";
            std::this_thread::sleep_for(std::chrono::duration<double>(delay));
        }
    }
    throw std::runtime_error("unreachable");
}

auto feed = retry_with_backoff(
    []{ return MarketDataFeed::connect("tcp://feed:5555"); },
    {.max_attempts = 5, .base_delay_s = 1.0}
);
Equal jitter (delay/2 + random(0, delay/2)) prevents very short retries while still desynchronising clients. Full jitter (random(0, delay)) provides maximum spread but can result in very short waits. For financial reconnects, equal jitter is usually the right choice.

Rust retry with backoff

use std::time::Duration;
use tokio::time::sleep;
use rand::Rng;

struct RetryConfig {
    max_attempts: u32,
    base_delay:   Duration,
    max_delay:    Duration,
    jitter:       bool,
}

impl Default for RetryConfig {
    fn default() -> Self {
        Self {
            max_attempts: 5,
            base_delay: Duration::from_secs(1),
            max_delay: Duration::from_secs(60),
            jitter: true,
        }
    }
}

async fn retry_with_backoff<F, Fut, T, E>(
    mut f: F,
    cfg: &RetryConfig,
) -> Result<T, E>
where
    F: FnMut() -> Fut,
    Fut: std::future::Future<Output = Result<T, E>>,
    E: std::fmt::Display,
{
    let mut rng = rand::thread_rng();
    for attempt in 1..=cfg.max_attempts {
        match f().await {
            Ok(val) => return Ok(val),
            Err(e) => {
                if attempt == cfg.max_attempts {
                    eprintln!("All {} attempts failed: {}", cfg.max_attempts, e);
                    return Err(e);
                }
                let exp = cfg.base_delay * 2u32.pow(attempt - 1);
                let delay = exp.min(cfg.max_delay);
                let delay = if cfg.jitter {
                    delay.mul_f64(rng.gen_range(0.5..1.5))
                } else {
                    delay
                };
                eprintln!("Attempt {} failed: {}. Retrying in {:?}", attempt, e, delay);
                sleep(delay).await;
            }
        }
    }
    unreachable!()
}

// Usage
async fn connect_to_feed() -> Result<MarketDataFeed, Box<dyn std::error::Error>> {
    let feed = retry_with_backoff(
        || async { MarketDataFeed::connect("tcp://feed:5555").await },
        &RetryConfig::default(),
    ).await?;
    Ok(feed)
}

// Dead letter queue — append to file
use tokio::fs::OpenOptions;
use tokio::io::AsyncWriteExt;

struct DeadLetterQueue {
    path: String,
}

impl DeadLetterQueue {
    async fn push(&self, msg: &str, reason: &str) -> std::io::Result<()> {
        let entry = format!(
            "{{\"msg\":\"{}\",\"reason\":\"{}\",\"ts\":{}}}\n",
            msg, reason, std::time::SystemTime::now()
                .duration_since(std::time::UNIX_EPOCH).unwrap().as_secs_f64()
        );
        let mut file = OpenOptions::new().create(true).append(true)
            .open(&self.path).await?;
        file.write_all(entry.as_bytes()).await
    }
}
Rust's async retry function uses generics over the future type, so the same function works with any async operation — TCP connect, HTTP request, database query. The compiler ensures the retry closure is Send-safe if you spawn it across threads.

Go retry with backoff

package main

import (
    "context"
    "fmt"
    "log"
    "math"
    "math/rand"
    "os"
    "time"
)

type RetryConfig struct {
    MaxAttempts int
    BaseDelay   time.Duration
    MaxDelay    time.Duration
    Jitter      bool
}

var DefaultRetryConfig = RetryConfig{
    MaxAttempts: 5,
    BaseDelay:   time.Second,
    MaxDelay:    60 * time.Second,
    Jitter:      true,
}

func RetryWithBackoff(ctx context.Context, cfg RetryConfig, fn func() error) error {
    for attempt := 1; attempt <= cfg.MaxAttempts; attempt++ {
        err := fn()
        if err == nil {
            return nil
        }
        if attempt == cfg.MaxAttempts {
            return fmt.Errorf("all %d attempts failed: %w", cfg.MaxAttempts, err)
        }

        delay := float64(cfg.BaseDelay) * math.Pow(2, float64(attempt-1))
        if delay > float64(cfg.MaxDelay) {
            delay = float64(cfg.MaxDelay)
        }
        if cfg.Jitter {
            delay *= 0.5 + rand.Float64() // [0.5, 1.5)
        }

        log.Printf("Attempt %d failed: %v. Retrying in %v", attempt, err, time.Duration(delay))

        select {
        case <-ctx.Done():
            return ctx.Err()
        case <-time.After(time.Duration(delay)):
        }
    }
    return nil // unreachable
}

// Usage — context cancellation stops retries cleanly
func connectToFeed(ctx context.Context) (*MarketDataFeed, error) {
    var feed *MarketDataFeed
    err := RetryWithBackoff(ctx, DefaultRetryConfig, func() error {
        var e error
        feed, e = MarketDataFeed.Connect("tcp://feed:5555")
        return e
    })
    return feed, err
}

// Dead letter queue — append to file
type DLQ struct {
    f *os.File
}

func NewDLQ(path string) (*DLQ, error) {
    f, err := os.OpenFile(path, os.O_APPEND|os.O_CREATE|os.O_WRONLY, 0644)
    if err != nil {
        return nil, err
    }
    return &DLQ{f: f}, nil
}

func (d *DLQ) Push(msg, reason string) {
    entry := fmt.Sprintf("{\"msg\":\"%s\",\"reason\":\"%s\",\"ts\":%f}\n",
        msg, reason, float64(time.Now().UnixNano())/1e9)
    d.f.WriteString(entry)
    log.Printf("DLQ: %s", reason)
}
Go's context.Context integration is the standout feature here. Pass a context with a deadline, and retry stops cleanly when the deadline expires or the caller cancels. The select on ctx.Done() and time.After is a pattern you will see in every production Go service.

TypeScript retry with backoff

interface RetryConfig {
  maxAttempts: number;
  baseDelayMs: number;
  maxDelayMs:  number;
  jitter:      boolean;
}

const defaultRetryConfig: RetryConfig = {
  maxAttempts: 5, baseDelayMs: 1000, maxDelayMs: 60_000, jitter: true,
};

async function retryWithBackoff<T>(
  fn: () => Promise<T>,
  config: Partial<RetryConfig> = {},
  signal?: AbortSignal,
): Promise<T> {
  const cfg = { ...defaultRetryConfig, ...config };

  for (let attempt = 1; attempt <= cfg.maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (attempt === cfg.maxAttempts) throw err;
      if (signal?.aborted) throw new Error("Retry aborted");

      let delay = Math.min(cfg.baseDelayMs * 2 ** (attempt - 1), cfg.maxDelayMs);
      if (cfg.jitter) delay *= 0.5 + Math.random();

      console.warn(`Attempt ${attempt} failed: ${err}. Retrying in ${delay.toFixed(0)}ms`);
      await new Promise<void>((resolve, reject) => {
        const timer = setTimeout(resolve, delay);
        signal?.addEventListener("abort", () => {
          clearTimeout(timer);
          reject(new Error("Retry aborted"));
        }, { once: true });
      });
    }
  }
  throw new Error("unreachable");
}

const feed = await retryWithBackoff(

The AbortSignal parameter lets the caller cancel retries mid-flight — essential for request handlers that have their own timeout. Without it, a retry loop can outlive the request that started it, wasting resources on a result nobody is waiting for.

Key observations

  • Go's context.Context is the best cancellation mechanism for retry loops. Pass a context with a deadline, and the retry stops cleanly when time runs out or the caller cancels. The select on ctx.Done() and time.After is a two-line pattern that every Go developer knows.
  • C# Polly composes retry + circuit breaker + fallback as chained policies. No other language has a single library that wraps all three patterns into a composable pipeline with this level of maturity.
  • Rust's async retry is generic over the future type, so one function covers TCP connect, HTTP request, and database query. The compiler checks that the closure is thread-safe at compile time — no runtime surprises.
  • TypeScript's AbortSignal is critical. Without it, a retry loop can outlive the HTTP request handler that started it. Always pass a signal so retries stop when the caller gives up.
  • C++ has no standard retry library. Every team writes their own. The jitter distribution choice (full vs equal vs decorrelated) matters for thundering herd prevention — equal jitter is usually the safest default for financial reconnects.

Decision guide for retry and backoff

  • Microservice with timeouts and cancellation: Go with context.Context — cleanest cancellation propagation
  • Complex resilience pipeline (retry + breaker + fallback): C# Polly — most composable
  • Type-safe async retry: Rust — compiler-checked thread safety on the retry closure
  • Always add jitter. Without it, clients that fail at the same time will retry at the same time, recreating the thundering herd
  • Always set a max retry count and a DLQ. Without both, permanently-failing messages loop forever and poison your system
