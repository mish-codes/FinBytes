The at-least-once problem

In any distributed system, message delivery is either at-most-once (may lose messages) or at-least-once (may deliver duplicates). At-least-once is usually safer - better to process twice than not at all - but only if your processing is idempotent.

Idempotency: an operation is idempotent if applying it multiple times has the same effect as applying it once. Setting a position to 100 is idempotent. Adding 100 to a position is not.

Finance examples: a fill message delivered twice must not double the position. A trade confirmation processed after a system restart must be recognised as already processed. An EOD job that fails halfway and restarts must not produce duplicate records.

Solution - idempotency keys: assign a unique ID to each operation. Before processing, check if the ID has been seen. If yes, return the cached result. Store processed IDs in a persistent store. Use database unique constraints as enforcement.

Python idempotency

import hashlib, json

class IdempotentTradeProcessor:
    def __init__(self): self._processed: dict[str, dict] = {}

    def process_fill(self, fill: dict) -> dict:
        key = fill.get("fill_id") or self._hash_fill(fill)
        if key in self._processed:
            return {**self._processed[key], "idempotent": True}
        result = self._apply_fill(fill)
        self._processed[key] = result
        return result

    def _apply_fill(self, fill: dict) -> dict:
        symbol   = fill["symbol"]
        quantity = fill["quantity"]   # absolute, not delta
        self._positions[symbol] = quantity  # SET not ADD
        return {"symbol": symbol, "quantity": quantity, "status": "processed"}

    def _hash_fill(self, fill: dict) -> str:
        canonical = json.dumps(fill, sort_keys=True).encode()
        return hashlib.sha256(canonical).hexdigest()[:16]

public class IdempotentTradeProcessor {

    private readonly Dictionary<string, double> _positions = new();

C++ idempotent processing

#include <string>
#include <unordered_set>
#include <unordered_map>

struct Fill { std::string fill_id, symbol; double quantity; };
struct ProcessResult { std::string symbol; double quantity; bool was_duplicate; };

class IdempotentFillProcessor {
    std::unordered_set<std::string>        processed_ids_;  // use persistent store in production
    std::unordered_map<std::string, double> positions_;
public:
    ProcessResult process(const Fill& fill) {
        if (processed_ids_.count(fill.fill_id))
            return {fill.symbol, positions_[fill.symbol], true};
        positions_[fill.symbol] = fill.quantity;   // SET not ADD
        processed_ids_.insert(fill.fill_id);
        return {fill.symbol, fill.quantity, false};
    }
};
Design your data model for idempotency from the start. Store absolute state (position = 100) rather than deltas (position += 10). Require idempotency keys on all mutating operations. Use database unique constraints as enforcement - they are the only reliable deduplication mechanism when multiple processes can run concurrently.

Rust idempotent processing

use std::collections::{HashMap, HashSet};
use std::sync::Arc;
use tokio::sync::Mutex;

#[derive(Clone, Debug)]
struct Fill {
    fill_id:  String,
    symbol:   String,
    quantity: f64,
}

#[derive(Clone, Debug)]
struct ProcessResult {
    symbol:        String,
    quantity:      f64,
    was_duplicate: bool,
}

struct IdempotentFillProcessor {
    processed_ids: Arc<Mutex<HashSet<String>>>,
    positions:     Arc<Mutex<HashMap<String, f64>>>,
}

impl IdempotentFillProcessor {
    fn new() -> Self {
        Self {
            processed_ids: Arc::new(Mutex::new(HashSet::new())),
            positions: Arc::new(Mutex::new(HashMap::new())),
        }
    }

    async fn process(&self, fill: &Fill) -> ProcessResult {
        let mut ids = self.processed_ids.lock().await;
        let mut pos = self.positions.lock().await;

        if ids.contains(&fill.fill_id) {
            return ProcessResult {
                symbol: fill.symbol.clone(),
                quantity: *pos.get(&fill.symbol).unwrap_or(&0.0),
                was_duplicate: true,
            };
        }

        // SET not ADD — idempotent by design
        pos.insert(fill.symbol.clone(), fill.quantity);
        ids.insert(fill.fill_id.clone());

        ProcessResult {
            symbol: fill.symbol.clone(),
            quantity: fill.quantity,
            was_duplicate: false,
        }
    }
}

// Content-based deduplication using hash
use std::hash::{Hash, Hasher};
use std::collections::hash_map::DefaultHasher;

fn hash_fill(fill: &Fill) -> String {
    let mut hasher = DefaultHasher::new();
    fill.fill_id.hash(&mut hasher);
    fill.symbol.hash(&mut hasher);
    fill.quantity.to_bits().hash(&mut hasher);
    format!("{:x}", hasher.finish())
}

// Database-backed idempotency (sqlx)
// In production, use: INSERT INTO processed_fills (fill_id)
// VALUES ($1) ON CONFLICT DO NOTHING
// Returns 0 rows if duplicate — the database is the single
// source of truth for deduplication
Rust's ownership system prevents a subtle idempotency bug: if you accidentally clone a Fill and process both copies, the HashSet catches the duplicate. The Arc<Mutex<_>> pattern ensures thread-safe access to the processed-IDs set. In production, replace the in-memory set with a database unique constraint.

Go idempotent processing

package main

import (
    "crypto/sha256"
    "fmt"
    "sync"
)

type Fill struct {
    FillID   string
    Symbol   string
    Quantity float64
}

type ProcessResult struct {
    Symbol       string
    Quantity     float64
    WasDuplicate bool
}

type IdempotentFillProcessor struct {
    mu           sync.Mutex
    processedIDs map[string]bool
    positions    map[string]float64
}

func NewIdempotentFillProcessor() *IdempotentFillProcessor {
    return &IdempotentFillProcessor{
        processedIDs: make(map[string]bool),
        positions:    make(map[string]float64),
    }
}

func (p *IdempotentFillProcessor) Process(fill Fill) ProcessResult {
    p.mu.Lock()
    defer p.mu.Unlock()

    if p.processedIDs[fill.FillID] {
        return ProcessResult{
            Symbol:       fill.Symbol,
            Quantity:     p.positions[fill.Symbol],
            WasDuplicate: true,
        }
    }

    // SET not ADD — idempotent by design
    p.positions[fill.Symbol] = fill.Quantity
    p.processedIDs[fill.FillID] = true

    return ProcessResult{
        Symbol:   fill.Symbol,
        Quantity: fill.Quantity,
    }
}

// Content-based deduplication
func hashFill(fill Fill) string {
    data := fmt.Sprintf("%s|%s|%f", fill.FillID, fill.Symbol, fill.Quantity)
    h := sha256.Sum256([]byte(data))
    return fmt.Sprintf("%x", h[:8])
}

// Idempotent HTTP handler — check before processing
func idempotentHandler(w http.ResponseWriter, r *http.Request) {
    key := r.Header.Get("Idempotency-Key")
    if key == "" {
        http.Error(w, "Idempotency-Key header required", http.StatusBadRequest)
        return
    }

    processor.mu.Lock()
    if result, ok := processor.results[key]; ok {
        processor.mu.Unlock()
        json.NewEncoder(w).Encode(result) // return cached result
        return
    }
    processor.mu.Unlock()

    result := processRequest(r)

    processor.mu.Lock()
    processor.results[key] = result
    processor.mu.Unlock()

    json.NewEncoder(w).Encode(result)
}
Go's HTTP handler pattern for idempotency is clean: check the Idempotency-Key header, look up the cached result, and return it if found. The mutex protects against concurrent requests with the same key. In production, replace the in-memory map with a Redis or PostgreSQL lookup with a TTL to prevent unbounded memory growth.

TypeScript idempotent processing

import { createHash } from "crypto";

interface Fill {
  fillId:   string;
  symbol:   string;
  quantity: number;
}

interface ProcessResult {
  symbol:       string;
  quantity:     number;
  wasDuplicate: boolean;
}

class IdempotentFillProcessor {
  private processedIds = new Set<string>();
  private positions = new Map<string, number>();

  process(fill: Fill): ProcessResult {
    if (this.processedIds.has(fill.fillId)) {
      return {
        symbol: fill.symbol,
        quantity: this.positions.get(fill.symbol) ?? 0,
        wasDuplicate: true,
      };
    }

    // SET not ADD — idempotent by design
    this.positions.set(fill.symbol, fill.quantity);
    this.processedIds.add(fill.fillId);

    return { symbol: fill.symbol, quantity: fill.quantity, wasDuplicate: false };
  }
}

// Content-based deduplication
function hashFill(fill: Fill): string {
  return createHash("sha256")
    .update(`${fill.fillId}|${fill.symbol}|${fill.quantity}`)
    .digest("hex")
    .slice(0, 16);
}

// Express middleware for idempotent API endpoints
import express from "express";

const idempotencyCache = new Map<string, { status: number; body: unknown }>();

function idempotent(
  handler: (req: express.Request) => Promise<{ status: number; body: unknown }>
) {
  return async (req: express.Request, res: express.Response) => {
    const key = req.headers["idempotency-key"] as string;
    if (!key) {
      return res.status(400).json({ error: "Idempotency-Key header required" });
    }

    const cached = idempotencyCache.get(key);
    if (cached) {
      return res.status(cached.status).json(cached.body);
    }

    const result = await handler(req);
    idempotencyCache.set(key, result);
    res.status(result.status).json(result.body);
  };
}

// Usage
const app = express();
app.post("/fills", idempotent(async (req) => {
  const result = processor.process(req.body as Fill);
  return { status: result.wasDuplicate ? 200 : 201, body: result };
}));
The Express middleware approach wraps any handler with idempotency checks. Because Node.js is single-threaded, you do not need a mutex to protect the Set and Map. However, you still need a persistent store (Redis, database) in production — in-memory state is lost on process restart.

Key observations

  • TypeScript's single-threaded model makes in-memory idempotency trivial. No mutex, no race conditions, no concurrent duplicate processing. A Set and a Map are all you need. The tradeoff: state is lost on restart, so a persistent store is mandatory for production.
  • Go's HTTP handler pattern for idempotency is exceptionally clean. Check the Idempotency-Key header, lookup in map, return cached result or process. The mutex protects against concurrent requests with the same key arriving on different goroutines.
  • Rust's ownership model prevents accidental double-processing. If a Fill is moved into the processor, it cannot be processed again by accident — the original variable is no longer valid. This is a compile-time guarantee no other language offers.
  • The database is the real deduplication layer. In all languages, in-memory sets are a first line of defence. The INSERT ... ON CONFLICT DO NOTHING pattern in PostgreSQL (or equivalent) is the only reliable deduplication when multiple processes run concurrently.
  • Design for idempotency from day one. Store absolute state (position = 100) not deltas (position += 10). Retrofitting idempotency into a delta-based system is extremely difficult and error-prone.

Decision guide for idempotency

  • Single-process, in-memory first: TypeScript (no locks needed) or Python (simple dict/set)
  • Multi-process production: database unique constraint — language does not matter, the DB is authoritative
  • HTTP API idempotency: Go or C# — clean middleware patterns, battle-tested in production
  • Compile-time safety against double-processing: Rust — ownership prevents re-use of consumed values
  • Always require an idempotency key on mutating operations. If the message does not carry one, generate a content hash — but explicit keys are always more reliable
